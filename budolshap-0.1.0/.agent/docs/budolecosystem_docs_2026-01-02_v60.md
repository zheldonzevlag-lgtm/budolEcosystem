## đźŹ—ď¸Ź Architectural Decisions

### Metadata-First Webhook Matching
**Decision**: Prioritize `metadata.orderId` over `payment_intent_id` or `source_id` in webhook handlers.
**Reason**: While payment IDs are unique, they are generated by the provider. Injecting our internal `orderId` into the provider's metadata creates a definitive, immutable link that survives retries and partial failures.

### Decoupled QR Lifecycle
**Decision**: Moved QR payment logic from a component-level modal to a page-level route.
**Reason**: Modals are fragile in Next.js when parent components (like the Cart) undergo significant state changes (like clearing items). A dedicated page provides a stable, long-lived environment for polling and user interaction.
