🚀 Implementation Plan - v60

QR Flow Stabilization & Webhook Enhancements

🎯 Purpose

v60 focuses on stabilizing the QR payment lifecycle, ensuring definitive order matching through enhanced webhook metadata, and optimizing payment status validation.

🛠️ Technical Implementation

1. Webhook Metadata Integration

Enhanced PayMongo webhook handler to use metadata.orderId as the primary key for order matching.

  • paymongo-adapter.js: Injects orderId into metadata for all intent creations.
  • paymongo/route.js: Prioritizes metadata matching in database for incoming events.

2. Robust QR State Recovery

Implemented multi-channel state recovery for the dedicated QR payment page.

  • OrderSummary.jsx: Now passes amount and provider in addition to payment_intent_id and orderId.
  • QRPaymentPage.jsx: Recovers state from URL parameters if localStorage is cleared, including a mock fallback for BudolPay QR codes.

3. Unified DB-First Status Check

Optimized the /api/paymongo/status endpoint to be provider-agnostic and more efficient.

  • Checks the local database first for a "paid" status before calling external APIs.
  • Supports both PayMongo and BudolPay (via DB lookup) using the same polling endpoint.
  • Reduced external API dependency for already processed transactions.

🏗️ Architectural Decisions

Metadata-First Webhook Matching

Using internal IDs in provider metadata creates an immutable link that survives retries and partial failures, ensuring 100% accurate order fulfillment.

Redundant State Recovery

By combining localStorage with URL parameters, we ensure the user never loses their payment QR code, even if they refresh the page or clear their cache mid-transaction.

🚀 Future Enhancements

  • Implementation of a unified transaction ledger for multi-gateway reconciliation.
  • Real-time webhook monitoring dashboard for failed event retries.
  • Enhanced error reporting for cross-origin payment failures.
