🐛Issues Discovered & Fixes Implemented

BUG #1Missing "Book Lalamove" Button

Discovered: When viewing a newly placed, paid order in the Seller Dashboard, the "Book Lalamove Delivery" button was not visible.

Root Cause: The rendering logic in app/store/orders/page.jsx only displayed the button if selectedOrder.shipping already existed. For orders paid via GCash (where shipping is selected post-payment), this condition was false.

Impact: Sellers could not book Lalamove delivery for orders that didn't have pre-existing shipping information.

FIX #1Added Conditional Rendering for Book Button

File Modified: app/store/orders/page.jsx

Solution: Added a new conditional rendering block to display the "Book Lalamove Delivery" button for orders that are selectedOrder.isPaid and !selectedOrder.shipping.

// Added between Payment section and Shipping Details {selectedOrder.isPaid && !selectedOrder.shipping && ( <div className="mt-6"> <button onClick={handleBookLalamove} className="w-full bg-orange-500 text-white..." > 📦 Book Lalamove Delivery </button> </div> )}

Status: ✅ Deployed to Production

BUG #3Lalamove Booking API Validation Error

Discovered: When attempting to book Lalamove for order cmiq1h2j80006l204batceqe1, the API returned: "Booking failed: Order does not have Lalamove shipping selected"

Root Cause: The /api/shipping/lalamove/book endpoint expected pre-existing Lalamove shipping information in the order, but orders paid via GCash didn't have this data.

Impact: Sellers could not book Lalamove for any orders that didn't have shipping pre-selected during checkout.

BUG #4Buyer Login Causes 404 Error

Discovered: When a non-seller user (buyer) logged in, the application crashed with a client-side exception due to a 404 error from /api/stores/user/${user.id}.

Root Cause: The Navbar.jsx component attempted to fetch store information for all logged-in users. Since buyers don't have stores, the API returned 404, causing the fetch to fail and crash the UI.

Impact: Buyers could not use the application after logging in.

FIX #4Graceful Handling of Non-Seller Users

Files Modified:

  • app/api/stores/user/[userId]/route.js
  • components/Navbar.jsx

Solution:

  1. Modified the API endpoint to return 200 OK with { found: false } instead of 404 Not Found when no store exists
  2. Updated Navbar.jsx to check for data.found before attempting to use store data
// API Route Change if (!store) { return NextResponse.json( { found: false }, { status: 200 } // Changed from 404 ); } // Navbar.jsx Change if (data.found && data.store) { setStore(data.store); }

Status: ✅ Deployed to Production

ISSUE #5Lalamove Webhooks Not Received

Root Cause: Lalamove's sandbox environment shows "Destination host unreachable" for the webhook URL https://budolshap.vercel.app/api/webhooks/lalamove.

Impact: Automatic status synchronization between Lalamove and Budolshap was not working in sandbox.

WORKAROUND #5Manual Webhook Simulation Scripts

  • scripts/trigger-completed-webhook.js

Solution: Created Node.js scripts that manually send properly signed webhook requests to the production endpoint, simulating Lalamove's webhook events.

// Usage: node scripts/trigger-pickup-webhook.js // Simulates PICKED_UP event → Updates order to SHIPPED node scripts/trigger-completed-webhook.js // Simulates COMPLETED event → Updates order to DELIVERED

Analysis

  • Lalamove's sandbox servers cannot reach the Vercel-hosted endpoint

  2. DNS/SSL Issues: Sandbox might not resolve certain domains properly

💡Key Learnings & Best Practices

1. Graceful Error Handling

Instead of returning 404 errors that crash the client, return 200 with metadata indicating the resource wasn't found. This allows the client to handle the absence gracefully.

2. Flexible Booking Logic

Don't assume all orders will have pre-existing shipping information. Implement fallback logic to generate quotes on-demand when sellers initiate booking.

3. Webhook Reliability

Always have a backup plan for webhooks. Sandbox environments may have connectivity issues, so manual triggers or admin panels can serve as reliable alternatives.

4. Comprehensive Testing

Test the complete user journey from both buyer and seller perspectives. Browser-based testing reveals UI/UX issues that unit tests might miss.

5. Status Transition Logic

Not all webhook events should change the main order status. Events like ON_GOING (driver assigned) should update metadata without changing the primary status.
