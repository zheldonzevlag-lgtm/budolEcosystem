API Gateway Implementation Recommendation

Project: Budolshap (Next.js Application)

Deployment: Vercel Hobby Account

Date: [Current Date]

Executive Summary

Based on the review of your Next.js codebase, which includes extensive API routes for authentication, orders, payments, shipping, and admin functions, this document recommends implementing an API Gateway to centralize request handling, improve security, and enhance scalability. The plan considers Vercel Hobby account limitations (100GB bandwidth, 1000 serverless function invocations/month) and focuses on lightweight, efficient solutions using Next.js middleware and edge functions.

Current Architecture Overview

Your application uses Next.js API routes under app/api/ for various functionalities:

  • Authentication: Login, registration, token verification
  • E-commerce: Orders, payments (PayMongo, GCash), cart management
  • Shipping: Lalamove integration, manual shipping
  • Admin: User management, analytics, webhooks
  • System: Caching, error tracking, real-time updates

Current middleware (middleware.js) handles basic authentication and route protection but lacks advanced gateway features like rate limiting, centralized logging, or request routing.

Current Architecture Diagram

Client → Vercel Edge → Next.js Middleware → API Routes (app/api/*) → Database/External Services

Proposed API Gateway Architecture

The recommended API Gateway will be implemented using Next.js middleware and a central gateway route, leveraging Vercel's edge network for optimal performance within Hobby account constraints.

Key Components

  • Edge Middleware: Handles authentication, rate limiting, CORS, and request preprocessing
  • Gateway Route: Central API endpoint for complex routing and aggregation
  • Internal APIs: Existing routes remain, with enhanced security
  • Monitoring: Centralized logging and error tracking

Proposed Architecture Diagram

Client → Vercel Edge Middleware (Auth, Rate Limiting, CORS) ↓ Gateway Route (/api/gateway) → Route to Internal APIs ↓ Internal API Routes (app/api/*) → Database/External Services ↓ Response ← Centralized Logging & Monitoring

Scalability Features

  • Edge Computing: Utilize Vercel's global edge network for low-latency responses
  • Caching: Implement response caching to reduce function invocations
  • Load Balancing: Distribute requests across multiple regions
  • Horizontal Scaling: Stateless design allows easy scaling

Implementation Plan

Phase 1: Enhanced Middleware (Week 1-2)

  1. Extend middleware.js to include rate limiting using a lightweight library like @upstash/ratelimit
  2. Add CORS headers for cross-origin requests
  3. Implement request logging with structured data
  4. Add request/response size validation

Phase 2: Gateway Route Implementation (Week 3-4)

  1. Create app/api/gateway/route.js as the central entry point
  2. Implement request routing logic based on path and method
  3. Add request aggregation for batch operations
  4. Integrate with existing authentication middleware

Phase 3: Security Enhancements (Week 5-6)

  1. Implement API key authentication for external integrations
  2. Add request signing/verification for webhooks
  3. Set up centralized error handling and responses
  4. Implement IP whitelisting for admin routes

Phase 4: Monitoring and Optimization (Week 7-8)

  1. Integrate with Sentry for error tracking
  2. Add performance monitoring and metrics
  3. Implement caching strategies (Redis/Upstash)
  4. Optimize for Vercel Hobby limits (reduce function invocations)

Phase 5: Testing and Deployment (Week 9-10)

  1. Comprehensive testing of all API routes through gateway
  2. Load testing to ensure scalability within Hobby limits
  3. Gradual rollout with feature flags
  4. Monitor performance and adjust as needed

Technical Considerations

Vercel Hobby Account Constraints

  • Bandwidth: 100GB/month - Optimize responses and implement compression
  • Function Invocations: 1000/month - Use caching and batching to reduce calls
  • No Custom Domains: Use Vercel-provided domains
  • Edge Functions: Leverage for global performance

Code Examples

Enhanced Middleware (middleware.js)

import { NextResponse } from 'next/server' import { Ratelimit } from '@upstash/ratelimit' import { Redis } from '@upstash/redis' const ratelimit = new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(100, '1 m'), // 100 requests per minute }) export async function middleware(request) { const ip = request.ip ?? '127.0.0.1' const { success } = await ratelimit.limit(ip) if (!success) { return NextResponse.json({ error: 'Rate limit exceeded' }, { status: 429 }) } // Existing auth logic... return NextResponse.next() }

Gateway Route (app/api/gateway/route.js)

import { NextRequest, NextResponse } from 'next/server' export async function POST(request: NextRequest) { const { path, method, body } = await request.json() // Route to appropriate internal API const internalUrl = new URL(path, process.env.NEXT_PUBLIC_API_URL) const response = await fetch(internalUrl, { method, body: JSON.stringify(body), headers: { 'Content-Type': 'application/json', // Forward auth headers } }) return NextResponse.json(await response.json(), { status: response.status }) }
