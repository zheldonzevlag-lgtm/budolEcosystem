1. Architecture Overview

The BudolEcosystem is built on a modern stack leveraging Next.js 14 (App Router), Redux Toolkit for state management, and Prisma ORM for database interactions.

  • Frontend: React, Tailwind CSS, Redux Toolkit
  • Backend: Next.js API Routes, Prisma (SQLite/PostgreSQL)
  • Authentication: Custom JWT-based auth with HTTP-only cookies

2. Cart System Implementation

The shopping cart uses a Hybrid Persistence Strategy to ensure seamless user experience across guest and logged-in states.

Core Components

  • cartSlice.js: Redux logic for cart operations.
  • PublicLayout.jsx: Orchestrates cart syncing and merging.
  • /api/cart/route.js: Backend endpoint for atomic cart updates.

Synchronization Logic

When a user logs in, the local (guest) cart is merged with the server cart. The larger quantity for any given item is preserved.

Open PublicLayout.jsx // Merge Logic in PublicLayout.jsx const localCart = cartItemsRef.current || {}; const mergedCart = { ...serverCart }; Object.keys(localCart).forEach(itemKey => { if (mergedCart[itemKey]) { // Use the larger quantity to prevent data loss mergedCart[itemKey] = Math.max(Number(mergedCart[itemKey] || 0), Number(localCart[itemKey] || 0)); } else { mergedCart[itemKey] = localCart[itemKey]; } });

3. Authentication & Logout Flow

A critical race condition was identified where logging out would trigger a sync of an empty cart, wiping the user's data. This has been resolved by checking userId transitions.

The Fix

We use a prevUserIdRef to detect the exact moment a user logs out (transition from ID to null) and clear the local state without triggering a server sync.

Open PublicLayout.jsx // Handle cart clearing on logout useEffect(() => { // If we had a user ID before, but now we don't, it means we logged out if (prevUserIdRef.current && !userId) { console.log('[Layout] User logged out, clearing cart state'); dispatch(clearCart()); localStorage.removeItem('budolshap_cart'); } prevUserIdRef.current = userId; }, [userId, dispatch]);

4. Database Schema (Prisma)

The cart system uses a composite key of productId and variationId to handle product variants correctly.

Open schema.prisma model CartItem { id String @id @default(cuid()) cartId String productId String variationId String? // Nullable for products without variations quantity Int cart Cart @relation(fields: [cartId], references: [id], onDelete: Cascade) product Product @relation(fields: [productId], references: [id]) @@unique([cartId, productId, variationId]) // Composite unique constraint }
