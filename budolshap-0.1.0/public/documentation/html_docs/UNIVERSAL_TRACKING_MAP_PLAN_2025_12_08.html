Proposed Refactor

Universal Tracking Map Implementation Plan

Date: December 8, 2025 | Author: Antigravity

Objective

Decouple the frontend tracking logic from specific providers (Lalamove, Standard, future integrations). Create a unified interface that automatically renders the correct map component based on the shipping provider configuration.

1. The Problem

Currently, app/(public)/orders/[orderId]/page.jsx has hardcoded logic:

{/* Hardcoded Checks */}
{order.shipping?.provider === 'lalamove' ? (
    <LalamoveTracking /> 
) : (
    <StandardDeliveryMap />
)}

If we add Grab or J&T, we have to modify the main order page every time. This scales poorly.

2. The Solution: UniversalTrackingMap

We will introduce a Factory Component pattern on the frontend.

New Component Structure

  • File: components/UniversalTrackingMap.jsx
  • Responsibility: Receive order object, detect provider, render correct sub-component.
  • Scalability: Adding a new provider only requires updating this one file, not the main pages.

3. Implementation Steps

Step 1: Create Universal Component

Create components/UniversalTrackingMap.jsx. It will import LalamoveTracking and StandardDeliveryMap.

Step 2: Implement Factory Logic

Write switch/case logic to match order.shipping.provider strings to components.

Step 3: Refactor Order Page

Replace the conditional rendering in page.jsx with a single <UniversalTrackingMap order={order} />.

Step 4: Verification

Verify that existing Lalamove orders still show Lalamove map, and standard orders show standard map.

4. Timeframe Estimate

Total Estimated Time: 15 - 20 Minutes

  • Code Creation: 5 mins
  • Refactoring: 5 mins
  • Testing & Verification: 5-10 mins
