PayMongo Redirect Issue: Summary & Resolution

Date: December 15, 2025
Environment: Next.js (Localhost & Production)

1. The Issue

Symptom: When testing payments (Maya, GrabPay, GCash) locally, the user was redirected to the production URL (budolshap.vercel.app) instead of localhost:3000 after authorization.

Root Cause:

  • The application logic prioritized the NEXT_PUBLIC_BASE_URL environment variable.
  • During local development, if this variable was loaded (e.g., from a shared .env file or default fallback), the app would construct the successUrl using the production domain.
  • This caused the PayMongo API to receive a production return URL, breaking the local testing loop.

2. The Fix

Resolution: Implemented a "Localhost Priority" logic in the backend route handlers.

We modified the following files:

  • app/api/payment/checkout/route.js
  • app/api/paymongo/create-intent/route.js

Implemented Logic

We now inspect the request headers before checking environment variables:

// Dynamic Return URL Strategy
const host = request.headers.get('host');
const protocol = request.headers.get('x-forwarded-proto') || 'https';

// 1. Force localhost if detected in host header
if (host && (host.includes('localhost') || host.includes('127.0.0.1'))) {
    baseUrl = `http://${host}`; // e.g., http://localhost:3000
} 
// 2. Use Environment Variable (if not local)
else if (process.env.NEXT_PUBLIC_BASE_URL) {
    baseUrl = process.env.NEXT_PUBLIC_BASE_URL;
} 
// 3. Fallback to Vercel URL or Host Header
else {
    baseUrl = `https://${process.env.VERCEL_URL || host || 'budolshap-v3.vercel.app'}`;
}

3. How it Works Now

  • Local Development: The system sees localhost:3000 in the header and immediately forces the return URL to match. Environment variables are ignored for this specific purpose.
  • Production/Preview: The system sees the actual domain (e.g., budolshap.vercel.app) and uses it, or falls back to the configured environment variable.

4. Future Recommendations

Status: Not Started

Currently, the logic above is duplicated in two different route files. It should be extracted into a shared utility function, e.g., lib/utils/url-helper.js.

export function getBaseUrl(request) { ... }

🔹 Recommendation 2: Strict Environment Variable Separation

Status: Ongoing

Ensure that local development does not accidentally load production-specific environment files (like .env.connection.prod) unless explicitly intended. Validate .env loading order in package.json scripts.

🔹 Recommendation 3: Automated Testing for Redirects

Status: Recommended

Implement a unit test for the URL generation logic to ensure that passing a localhost header always returns a http://localhost URL, preventing future regressions.
