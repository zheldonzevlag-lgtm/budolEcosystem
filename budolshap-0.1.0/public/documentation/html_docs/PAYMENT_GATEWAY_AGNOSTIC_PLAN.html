🛡️ Payment Gateway Agnostic Architecture

To answer your questions: The previous plan was NOT gateway agnostic. It was tightly coupled to PayMongo.
This new plan introduces an Adapter Pattern to ensure robustness and allow easy addition of future gateways (Xendit, Stripe, PayPal) without rewriting frontend logic.

🏗️ Architecture Overview

We will introduce a PaymentService layer that abstracts specific provider logic.

graph TD Client[Frontend PaymentComponent] -->|POST /api/payment/checkout| UnifiedAPI UnifiedAPI[Unified API Route] -->|initiatePayment| PaymentService PaymentService -->|Select Provider| AdapterFactory subgraph Adapters AdapterFactory --> PayMongo[PayMongo Adapter] AdapterFactory --> Xendit["Xendit Adapter (Future)"] AdapterFactory --> Stripe["Stripe Adapter (Future)"] end PayMongo -->|API Call| PayMongoAPI[PayMongo External API]

🛠 Implementation Steps

1. Create Payment Adapter Interface

File: lib/payment/adapters/base-adapter.js NEW

  • Define the contract: createPaymentIntent(amount, currency, method, billing)
  • Define the return structure: { checkoutUrl, paymentIntentId, status } (Normalized)

2. Implement PayMongo Adapter

File: lib/payment/adapters/paymongo-adapter.js NEW

  • Move logic from lib/paymongo.js into this class.
  • Handle mapping of generic methods (gcash, card) to PayMongo specific payloads.

3. Create Unified Payment Service

File: lib/payment/service.js NEW

  • initiatePayment(provider, amount, method, billing)
  • Loads the correct adapter based on config or request.

4. Create Unified API Endpoint

File: app/api/payment/checkout/route.js NEW

  • Replaces specific create-intent routes.
  • Accepts: { provider: 'paymongo', method: 'qrph', amount: 1000, ... }
  • Returns: { checkoutUrl: '...' }

5. Update Frontend Component

File: components/payment/UniversalPaymentButton.jsx NEW

  • Accepts a provider prop (defaulting to 'paymongo' for now).
  • Renders available options dynamically.
  • Calls the unified /api/payment/checkout.

🚀 Future Extensibility

To add a new gateway (e.g., Xendit) in the future, you only need to:

  1. Create lib/payment/adapters/xendit-adapter.js implementing the base interface.
  2. Add 'xendit' to the service.js factory.
  3. No changes required in the frontend component (just pass provider="xendit").
  4. No changes required in the API route logic.
