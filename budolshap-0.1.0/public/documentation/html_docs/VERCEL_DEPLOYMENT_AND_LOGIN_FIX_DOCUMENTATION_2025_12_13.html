Vercel Deployment & Login Fix Documentation

Date: December 13, 2025

Status: RESOLVED

1. Executive Summary

This document details the successful resolution of critical deployment and authentication issues affecting the budolshap-v3 production environment on Vercel. The primary challenges included a build failure due to missing environment variables, a persistent login failure due to database connection mismatches, and a redirection loop caused by domain configuration.

2. Issue Analysis

2.1 Issue 1: Vercel Build Failure

Symptom: The specific error Error: P1001: Can't reach database server at... during usage of prisma generate.

Root Cause: The DIRECT_URL environment variable was missing from the Vercel Production environment. Prisma requires a direct connection (bypassing connection poolers) for schema migrations and client generation.

2.2 Issue 2: Login Failure ("Invalid Credentials")

Symptom: Users could not log in to the production site despite believing the admin account existed.

Root Cause: The application was pointing to a Legacy Database (3e537...), while the User was viewing a different Production Database (c0999...) in their Vercel dashboard. This created a "phantom data" scenario where the app was validly checking an empty database.

2.3 Issue 3: Domain Redirection Loop

Symptom: Successfully logging in at budolshap.vercel.app/admin/login redirected the user back to the home page without persisting the session.

Root Cause: The application middleware relies on the NEXT_PUBLIC_APP_URL environment variable to determine valid redirection targets. Its absence caused the security logic to reject the cross-path redirect.

3. Technical Implementation & Fixes

3.1 Database Reconciliation Strategy

Discovery: Using prisma studio and custom scripts (list-users.js), we identified that the Vercel environment was configured with a seemingly valid but "hidden" database URL different from the one displayed in the Vercel Storage console.
Alignment: We extracted the visible database credentials (starting with c0999...) and forcefully updated the Vercel Production Environment Variables to match this specific instance.
Seed: We ran scripts/create-admin.js targeting the aligned database to ensure the initial Admin user (email: admin@budolshap.com) was created exactly where the app (and the user) could see it.

3.2 Environment Variable Configuration

The following variables were critical to the solution:

Variable Purpose Configuration Strategy
DATABASE_URL Primary Application Connection Set to the c0999... instance (Direct connection without pooling for now to prevent header mismatch errors).
DIRECT_URL Prisma Migration/Generation Set identical to DATABASE_URL to ensure build tools can connect.
NEXT_PUBLIC_APP_URL Frontend/Middleware Logic Set explicitly to https://budolshap.vercel.app to authorize redirects on the main domain.

3.3 Admin User Creation Script

We enhanced the scripts/create-admin.js script to support environment context switching. It now intelligently loads from .env.local (Production Config) or .env (Dev Config) based on execution context.

// Key logic for robust admin creation
const admin = await prisma.user.create({
    data: {
        email: 'admin@budolshap.com',
        accountType: 'ADMIN',
        isAdmin: true,
        emailVerified: true // Critical for bypassing verification checks
        // ...
    }
})

4. Verification Results

4.1 Deployment Success

The Vercel deployment pipeline is now Green (Success). The build process successfully generating the Prisma Client using the provided DIRECT_URL.

  • Credential Test: admin@budolshap.com / admin123 checks PASSED.

5. Recommendations & Future Enhancements

5.1 Enable Connection Pooling

Currently, we are connecting directly to the database. For high-traffic production use, you should enable the Prisma Accelerate or PgBouncer connection string (the one starting with prisma+postgres://...). This was temporarily bypassed to rule out pooling issues but should be reinstated once stable.

5.2 Automated Smoke Tests

Implement a GitHub Action that runs a simple "Health Check" script after deployment. This script should attempt a login API call to verify that critical infrastructure (Auth + DB) is online.

5.3 Environment Variable Locking

Use Vercel's "Protect System Environment Variables" or simply document the c0999... database ID as the Single Source of Truth to prevent future developers from accidentally connecting to test databases.
