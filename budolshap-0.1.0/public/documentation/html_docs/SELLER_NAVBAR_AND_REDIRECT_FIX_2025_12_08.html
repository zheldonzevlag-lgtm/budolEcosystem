🔴Issue #1: Seller Menu & Badge Not Displaying

Problem Description

When sellers logged into the platform, they experienced the following issues:

  • ❌ Seller menu options were not visible (Store Dashboard, Add Product, Manage Products, Store Orders)
  • ❌ Seller badge was not displayed next to their name
  • ❌ Only generic buyer options were shown ("My Orders" and "Become A Seller")
  • ❌ Store logo was not appearing in the navbar

User Impact: Sellers could not access their store management features, severely limiting their ability to manage products and orders.

🔍 Root Cause Analysis

Issue 1.1: Incorrect API Response Handling

❌ BEFORE (Incorrect)

// Line 40-50 in Navbar.jsx fetch(`/api/stores/user/${user.id}`) .then(res => res.json()) .then(data => { if (data && data.found) { // ❌ 'found' doesn't exist setIsSeller(true) setStoreStatus(data.status) ... } })

Problem: The code checked for data.found property that doesn't exist in the API response.

✅ AFTER (Fixed)

// Line 40-59 in Navbar.jsx fetch(`/api/stores/user/${user.id}`) .then(res => { if (!res.ok) { // ✅ Check HTTP status setIsSeller(false) setStoreStatus(null) return null } return res.json() }) .then(data => { if (data && data.id) { // ✅ Check for store ID setIsSeller(true) setStoreStatus(data.status) ... } })

Solution: Properly check HTTP response status and verify store existence using data.id.

Issue 1.2: Wrong Database Field Used

⚠️ Field Confusion

The Store model has two similar fields that serve different purposes:

Field Name Type Purpose Values Used For
status String Approval status "pending", "approved", "rejected" ✅ UI Logic (badges, menus)
isActive Boolean Active/Inactive state true, false ❌ Not for UI display

❌ WRONG FIELD

setStoreStatus(data.isActive) // Returns: true or false // UI expects: "pending" or "approved"

✅ CORRECT FIELD

setStoreStatus(data.status) // Returns: "pending" or "approved" // UI can now properly check status

💡Recommendations

🔒 Security Recommendations

  1. Add Server-Side Authorization: While the UI correctly hides seller menus, ensure all seller API endpoints verify the user's seller status server-side.
  2. Validate Store Status: API endpoints should check both status === 'approved' AND isActive === true before allowing seller actions.
  3. Rate Limiting: Consider adding rate limiting to the /api/stores/user/[userId] endpoint to prevent abuse.

🎨 UX Improvements

  1. Loading State: Add a skeleton loader while fetching store data to improve perceived performance.
  2. Error Handling: Display user-friendly error messages if store data fails to load.
  3. Badge Tooltip: Add tooltips to badges explaining what "Pending" means (e.g., "Your store is awaiting admin approval").
  4. Countdown Animation: Consider adding a visual countdown timer (circular progress) for the 7-second redirect.
  5. Cancel Redirect: Add a "Stay on this page" button to cancel the automatic redirect.

⚡ Performance Optimizations

  1. Cache Store Data: Consider caching store data in localStorage or React Context to reduce API calls.
  2. Debounce Store Checks: If the user object changes frequently, debounce the store status check.
  3. Lazy Load Menus: Only fetch detailed store data when the user opens the dropdown menu.
  4. Optimize Images: Ensure store logos are properly optimized and use Next.js Image component.

📚Key Learnings

  1. Always Verify API Response Structure:

    Don't assume properties exist without checking the actual endpoint. Use browser DevTools or API documentation to verify response format.

  2. Understand Database Schema:

    Know the difference between similar fields (status vs isActive). Read the schema comments and understand each field's purpose.

  3. Handle HTTP Errors Properly:

    Check res.ok before parsing JSON. A 404 response will still return JSON, but you need to handle it differently than a 200.

  4. Provide Clear User Feedback:

    Users should always know their status. The "Pending" badge helps sellers understand why they can't access certain features.

  5. Test All User Roles:

    When fixing authentication/authorization issues, test with all user types: buyer, pending seller, approved seller, and admin.

  6. Document Your Changes:

    Comprehensive documentation helps future developers (and your future self) understand why changes were made.
