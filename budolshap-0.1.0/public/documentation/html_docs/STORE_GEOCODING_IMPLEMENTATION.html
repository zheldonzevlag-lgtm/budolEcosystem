🌍 Store Geocoding Implementation

Using Actual Store Locations for Lalamove Deliveries

Implemented Production Ready Requires Migration

📋 Overview

This implementation adds geocoding support for store locations, replacing hardcoded default coordinates with actual store addresses. This ensures accurate pickup locations for Lalamove deliveries, resulting in correct delivery quotes and successful bookings.

🎯 Objective: Use real store coordinates instead of default Manila location (14.5505, 121.0260) for all Lalamove quote and booking operations.

⚠️ Problem Statement

Previously, the system used hardcoded coordinates for all store pickups:

// Old implementation - hardcoded coordinates let pickup = { address: "Glorietta 4, Ayala Center, Makati, Metro Manila", coordinates: { lat: 14.5505, lng: 121.0260 }, contactName: "BudolShap Store", contactPhone: "+639170000000" };

Issues with this approach:

  • ❌ Inaccurate delivery quotes (wrong distance calculations)
  • ❌ Incorrect pickup locations shown to drivers
  • ❌ Poor user experience for customers
  • ❌ Potential delivery failures

🏗️ Solution Architecture

1. Database Schema Changes

Added geocoding cache fields to the Store model:

model Store { // ... existing fields ... // Geocoding cache for store location latitude Float? longitude Float? }
Field Type Description
latitude Float (nullable) Cached latitude coordinate from geocoding
longitude Float (nullable) Cached longitude coordinate from geocoding

2. Geocoding API Endpoint

Endpoint: PATCH /api/stores/[storeId]/geocode

Purpose: Geocode store address and cache coordinates in the database

Features:

  • ✅ Checks for cached coordinates first (avoids redundant API calls)
  • ✅ Uses Nominatim OpenStreetMap API for geocoding
  • ✅ Automatically caches results in database
  • ✅ Returns standardized response format

3. Updated Components

Component/File Changes
LalamoveWidget.jsx Fetches store coordinates, geocodes if needed
app/api/shipping/lalamove/book/route.js Uses actual store coordinates when refreshing quotes
prisma/schema.prisma Added latitude/longitude fields to Store model

🔄 Implementation Flow

Quote Request Flow

1 User selects Lalamove delivery

LalamoveWidget component is activated

2 Fetch store details

GET /api/stores/[storeId] - Retrieve store information

3 Check for cached coordinates

If store.latitude && store.longitude exist → Use cached coordinates

4 Geocode if needed

If no cached coordinates → PATCH /api/stores/[storeId]/geocode

5 Request quote with actual coordinates

POST /api/shipping/lalamove/quote with real pickup location

6 Display accurate quote

Show delivery fee based on actual distance

💻 Code Examples

LalamoveWidget.jsx - Store Location Fetching

// Check if store has cached coordinates if (storeData.latitude && storeData.longitude) { console.log('[Store Location] Using cached coordinates'); pickup = { address: storeData.address, coordinates: { lat: storeData.latitude, lng: storeData.longitude }, contactName: storeData.name, contactPhone: storeData.contact || "+639170000000" }; } else if (storeData.address) { // Geocode store address and cache it const geocodeRes = await fetch(`/api/stores/${storeId}/geocode`, { method: 'PATCH' }); if (geocodeRes.ok) { const geocodeData = await geocodeRes.json(); pickup = { address: storeData.address, coordinates: geocodeData.coordinates, contactName: storeData.name, contactPhone: storeData.contact }; } }

Booking Route - Quote Refresh with Store Coordinates

// Try to use cached store coordinates if (order.store?.latitude && order.store?.longitude) { pickupCoordinates = { lat: order.store.latitude.toString(), lng: order.store.longitude.toString() }; } else if (order.store?.address) { // Geocode store address const geocodeRes = await fetch( `/api/stores/${order.storeId}/geocode`, { method: 'PATCH' } ); if (geocodeRes.ok) { const geocodeData = await geocodeRes.json(); pickupCoordinates = { lat: geocodeData.coordinates.lat.toString(), lng: geocodeData.coordinates.lng.toString() }; } }

🔧 Technical Details

Geocoding Service

Provider: Nominatim (OpenStreetMap)

Endpoint: https://nominatim.openstreetmap.org/search

Usage Policy: Nominatim requires a User-Agent header and recommends rate limiting (max 1 request per second). Our implementation respects these guidelines.

Caching Strategy

  • Cache Location: Database (Store table)
  • Cache Duration: Permanent (until address changes)
  • Cache Invalidation: Manual (when store address is updated)

Error Handling

Error Type Handling Strategy
Geocoding API failure Use default coordinates, log warning
Invalid address Return error, suggest address verification
Network timeout Retry with exponential backoff
Database error Continue without caching, log error
