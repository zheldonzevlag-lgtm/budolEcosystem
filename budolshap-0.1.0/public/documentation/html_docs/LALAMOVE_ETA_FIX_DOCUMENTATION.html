🔴 The Problem

❌ Issue: "Invalid Date" Error

Symptom: When users selected "Same Day Delivery" (Lalamove) on the checkout page, the widget displayed "Invalid Date" in the Estimated Arrival field.

Impact: Users couldn't see when their order would arrive, reducing confidence in the delivery service.

Location: Cart/Checkout page → Payment Summary → Lalamove Widget

Root Cause Analysis

After thorough investigation of the Lalamove API response, we discovered:

🔍 API Response Analysis

The Lalamove Quotation API (/v3/quotations) does NOT return the following fields:

  • ❌ estimatedPickupTime
  • ❌ estimatedDeliveryTime
  • ❌ stops[].eta

What it DOES return:

  • ✅ scheduleAt - Scheduled pickup time
  • ✅ expiresAt - Quote expiration time
  • ✅ distance.value - Distance in meters
  • ✅ priceBreakdown - Pricing details

Actual API Response Structure

The Code Issue: Our widget was trying to display quote.eta.delivery which was undefined, causing JavaScript's new Date(undefined) to return "Invalid Date".

✅ The Solution

💡 Smart ETA Calculation

Since Lalamove doesn't provide ETA directly, we implemented an intelligent calculation algorithm that estimates delivery time based on:

  • 📅 Schedule Time: Using the scheduleAt field as the base
  • 📏 Distance: Calculating travel time from distance in kilometers
  • 🚗 Average Speed: Assuming 20 km/h in city traffic
  • ⏱️ Pickup Buffer: Adding 10 minutes for pickup preparation

Calculation Formula

// Calculate estimated delivery time const scheduleTime = new Date(result.data.scheduleAt); const distanceInKm = result.data.distance?.value ? result.data.distance.value / 1000 : 5; // Estimate delivery time based on distance // Assume average speed of 20 km/h in city traffic + 10 min pickup time const estimatedMinutes = Math.ceil((distanceInKm / 20) * 60) + 10; // Calculate final delivery ETA deliveryEta = new Date(scheduleTime.getTime() + estimatedMinutes * 60000).toISOString();

Example Calculation

📊 Real Example

Given:

  • Distance: 2.4 km
  • Schedule Time: 05:00 PM

Calculation:

  • Travel time: (2.4 / 20) × 60 = 7.2 minutes
  • Pickup buffer: 10 minutes
  • Total time: 7.2 + 10 = ~17 minutes

Result: Estimated Arrival at 05:20 PM

📝 Technical Notes

Assumptions Made

⚙️ Algorithm Parameters

  • Average Speed: 20 km/h (conservative estimate for Manila traffic)
  • Pickup Time: 10 minutes (time for driver to reach pickup location and prepare)
  • Default Distance: 5 km (fallback if distance is not provided)

Note: These values can be adjusted based on real-world data and feedback.

Future Improvements

  • 🔄 Implement dynamic speed calculation based on time of day (rush hour vs. off-peak)

  • 🎯 Add confidence intervals (e.g., "Arrives between 5:15 PM - 5:25 PM")

Edge Cases Handled

  • Missing scheduleAt field

    Fallback: Display "Calculating..." instead of crashing

  • Missing distance data

    Fallback: Use default 5 km for calculation

  • API timeout or error

    Handled: Error message displayed with retry option

💡 Lessons Learned

🎓 Key Takeaways

  1. Always verify API documentation: Don't assume fields exist without checking actual responses
  2. Test with real API calls: Mock data can hide issues that only appear with real API responses
  3. Implement graceful fallbacks: Handle missing data elegantly rather than displaying errors
  4. Log extensively during debugging: Console logs helped identify the exact issue quickly
  5. Calculate when necessary: If the API doesn't provide data, calculate it intelligently
