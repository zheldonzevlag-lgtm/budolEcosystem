Database Migration Execution Report

Topic: Migrating Vercel PostgreSQL to Local MySQL (Side-by-Side)

Date Executed: December 13, 2025

Timeframe: 12:15 PM - 12:30 PM

Executive Summary: The migration was executed using a "Parallel" strategy. Data was successfully exported from PostgreSQL and 90% of it was restored to a local MySQL database. A specific issue with Order data structure prevented 4 order records from being fully restored. The application remains fully functional on the original PostgreSQL database.

1. Processes & Execution Log

Phase 1: Preparation & Backup (12:15 PM)

Goal: Extract data from PostgreSQL in a format suitable for granular restoration.

  • Issue Encountered: The original backup-database.js script produced a single monolithic JSON file. This was risky for large migrations and hard to debug.
  • Fix Applied: Refactored the script to create a timestamped directory and save each table as an individual JSON file (e.g., users.json, products.json).
  • Result: Successfully created backup at: c:\wamp64\www\budolshap-v3\backups\2025-12-13_12-19\
Phase 2: Schema Conversion (12:19 PM)

Goal: Create a MySQL-compatible schema without breaking the Postgres app.

  • Created prisma/schema.mysql.prisma.
  • Technical Detail: Added @db.Text to fields like `description`, `content`, `review`, `reason` to handle long text that exceeds MySQL's default varchar limits.
  • Command: npx prisma db push --schema=prisma/schema.mysql.prisma
Phase 3: Data Restoration (12:20 PM - 12:22 PM)

Goal: Load JSON data into the new MySQL database.

  • Issue Encountered: The restoration script crashed when trying to insert Orders.
  • Root Cause: Complex nested data structure in `Order` (specifically `shipping` JSON or `orderItems` relations) caused a Prisma Client validation error.
  • Fix Applied: Added `try-catch` blocks around the Order insertion loop in `restore-database.js` to log errors individually and allow other records to proceed.
  • Command: node scripts/restore-database.js 2025-12-13_12-19
