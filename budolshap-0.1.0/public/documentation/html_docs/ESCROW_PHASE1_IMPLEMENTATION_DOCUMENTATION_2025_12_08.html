3️⃣ Issues Encountered

❌

Issue #1: Prisma Schema Conflict

Error: Backup script failed because Prisma client was using the NEW schema (with pendingBalance) but the database still had the OLD schema (without pendingBalance).

PrismaClientKnownRequestError: Unknown column 'Wallet.pendingBalance'

Impact: Could not run backup using Prisma ORM methods
❌

Issue #2: Missing DIRECT_URL Environment Variable

Error: Prisma migrate deploy failed because DIRECT_URL was not configured in Vercel environment variables.

Error code: P1012
error: Environment variable not found: DIRECT_URL

Impact: Could not run Prisma migrations
❌

Issue #3: Previous Failed Migration

Error: A previous migration (20251203_add_order_lifecycle_timestamps) had failed and was blocking new migrations.

Migration `20251203_add_order_lifecycle_timestamps` started at 2025-12-06 04:07:41 UTC failed

Impact: Could not deploy new migrations until resolved

4️⃣ Solutions & Fixes

✅

Fix #1: Raw SQL Backup Script

Created pre-migration-backup.js that uses raw SQL queries instead of Prisma ORM to avoid schema conflicts.

Solution Code:

// Use raw SQL to avoid schema conflicts
const wallets = await prisma.$queryRaw`SELECT * FROM "Wallet"`
const transactions = await prisma.$queryRaw`SELECT * FROM "Transaction"`
// ... backup other tables
Result: Successfully backed up all data (66KB) before migration
✅

Fix #2: Added DIRECT_URL to Vercel

Manually added DIRECT_URL environment variable in Vercel dashboard with the Prisma Accelerate connection string.

Configuration:

Key: DIRECT_URL

Value: prisma+postgres://[connection-string]

Environment: All (Production, Preview, Development)

Result: Prisma migrations can now access the database directly
✅

Fix #3: Resolved Failed Migration + Direct SQL Execution

Marked the failed migration as rolled back, then used direct SQL execution to apply the new migration.

Solution Steps:

# Step 1: Mark failed migration as rolled back
npx prisma migrate resolve --rolled-back 20251203_add_order_lifecycle_timestamps

# Step 2: Create SQL migration file
# migrations/add_pending_balance.sql
ALTER TABLE "Wallet" ADD COLUMN IF NOT EXISTS "pendingBalance" DOUBLE PRECISION NOT NULL DEFAULT 0;

# Step 3: Execute SQL directly
npx prisma db execute --file migrations/add_pending_balance.sql

✅ Script executed successfully
Result: Migration applied successfully, bypassing Prisma migration system

8️⃣ Lessons Learned

🔧 Technical Lessons

📌
Prisma Schema vs Database State

When Prisma schema is updated but migration hasn't run yet, the Prisma client will expect the new schema. Use raw SQL queries for operations that need to work with the current database state.

📌
Environment Variable Management

Vercel's environment variables need to be explicitly configured. The DIRECT_URL is required for Prisma migrations but not automatically included when using Prisma Accelerate.

📌
Failed Migration Handling

Failed migrations block future migrations. They must be resolved (either rolled back or applied) before new migrations can proceed. Consider using prisma migrate resolve.

📋 Process Lessons

📌
Backup Before Schema Changes

Always create backups BEFORE modifying the schema in code. This ensures the backup script works with the current database state.

📌
Direct SQL as Fallback

When Prisma migrations fail, direct SQL execution via prisma db execute is a reliable fallback. Document these manual interventions thoroughly.

📌
Verification is Critical

Always run comprehensive tests after migration. Don't assume success based on "no errors" alone.

💬 Communication & Documentation

📌
Document Everything

Every issue, workaround, and manual intervention should be documented. This documentation is invaluable for troubleshooting similar issues in the future.

📌
Create Runbooks

Migration procedures should be documented as step-by-step runbooks that anyone can follow.
