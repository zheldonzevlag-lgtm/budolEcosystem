🚀 GitHub Push Guide

Complete Guide to Pushing Your Budolshap Application to GitHub

📋 Prerequisites

Before you begin, make sure you have the following:

  • Git installed: Download from git-scm.com
  • GitHub account: Sign up at github.com
  • Git configured: Your name and email should be set up
ℹ️ Check Git Installation

Verify Git is installed by running:

Command
git --version

Configure Git (First Time Setup)

If you haven't configured Git yet, set your name and email:

Configuration
git config --global user.name "Your Name" git config --global user.email "your.email@example.com"

1 Create a New GitHub Repository

Option A: Via GitHub Website (Recommended for Beginners)

  1. Go to GitHub.com and log in
  2. Click the "+" icon in the top right corner
  3. Select "New repository"
  4. Fill in the repository details:
    • Repository name: budolshap
    • Description: "Budolshap - Best budol deal ever"
    • Visibility: Choose Public or Private
    • ⚠️ Important: Do NOT initialize with README, .gitignore, or license (since you already have a local repository)
  5. Click "Create repository"
⚠️ Important Note

Since you already have a local Git repository with commits, make sure to create an empty repository on GitHub (don't initialize it with any files).

Option B: Via GitHub CLI (Advanced)

GitHub CLI
gh repo create budolshap --public --source=. --remote=origin

2 Connect Your Local Repository to GitHub

Check Current Remote

First, check if you already have a remote configured:

PowerShell
git remote -v

If No Remote Exists

Add the GitHub repository as your remote origin:

Add Remote
git remote add origin https://github.com/YOUR_USERNAME/budolshap.git

If Remote Already Exists (Update It)

If you need to change the remote URL:

Update Remote
git remote set-url origin https://github.com/YOUR_USERNAME/budolshap.git
💡 Pro Tip

Replace YOUR_USERNAME with your actual GitHub username. You can find this in the repository URL that GitHub shows you after creating the repository.

3 Prepare Your Code for Push

Check Repository Status

Check Status
git status

Create/Update .gitignore

Make sure you have a .gitignore file to exclude sensitive and unnecessary files:

.gitignore
# Dependencies node_modules/ .pnp .pnp.js # Testing coverage/ # Next.js .next/ out/ build/ dist/ # Production build/ # Misc .DS_Store *.pem # Debug npm-debug.log* yarn-debug.log* yarn-error.log* # Local env files .env .env*.local .env.local .env.development.local .env.test.local .env.production.local # Vercel .vercel # Prisma prisma/migrations/migration_lock.toml # IDE .vscode/ .idea/ *.swp *.swo *~ # OS Thumbs.db .DS_Store

Stage All Changes (If Needed)

If you have uncommitted changes:

Stage Changes
git add .

Commit Changes (If Needed)

Commit
git commit -m "Initial commit: Budolshap application"
✅ Current Status

Your repository shows: "working tree clean" - This means all your changes are already committed and ready to push!

4 Push Your Code to GitHub

First Time Push

Push your code and set the upstream branch:

Initial Push
git push -u origin main
ℹ️ What does -u mean?

The -u flag (or --set-upstream) sets the remote branch as the default upstream for your local branch. After this, you can simply use git push without specifying the remote and branch.

Authentication Methods

Method 1: HTTPS with Personal Access Token (Recommended)

  1. Go to GitHub Settings → Developer settings → Personal access tokens → Tokens (classic)
  2. Click "Generate new token (classic)"
  3. Give it a name (e.g., "Budolshap Development")
  4. Select scopes: repo (full control of private repositories)
  5. Click "Generate token"
  6. Copy the token (you won't see it again!)
  7. When pushing, use the token as your password
🔐 Security Note

Never commit your Personal Access Token to your repository! Store it securely in a password manager.

Method 2: SSH (Advanced)

Generate an SSH key and add it to GitHub:

Generate SSH Key
ssh-keygen -t ed25519 -C "your.email@example.com"

Then add the public key to GitHub Settings → SSH and GPG keys

Update your remote to use SSH:

SSH Remote
git remote set-url origin git@github.com:YOUR_USERNAME/budolshap.git

5 Verify Your Push

Check Push Status

Verify
git status

You should see: "Your branch is up to date with 'origin/main'"

View on GitHub

  1. Go to your repository on GitHub: https://github.com/YOUR_USERNAME/budolshap
  2. Verify all your files are there
  3. Check the commit history
  4. Review the README (if you have one)
🎉 Success!

If you can see your code on GitHub, congratulations! You've successfully pushed your Budolshap application to GitHub.

⚠️ Common Issues & Solutions

Issue 1: "Repository not found"

❌ Error
fatal: repository 'https://github.com/username/repo.git/' not found

Solutions:

  • Make sure the repository exists on GitHub
  • Check that the repository URL is correct
  • Verify you have access to the repository
  • Ensure your GitHub username is correct in the URL

Issue 2: Authentication Failed

❌ Error
remote: Support for password authentication was removed

Solution: Use a Personal Access Token instead of your password (see Step 4)

Issue 3: Rejected Push (Non-Fast-Forward)

❌ Error
! [rejected] main -> main (non-fast-forward)

Solution: Pull the remote changes first:

Pull First
git pull origin main --rebase git push origin main

Issue 4: Large Files

❌ Error
remote: error: File is too large

Solution: Use Git LFS for large files or add them to .gitignore:

Git LFS
git lfs install git lfs track "*.psd" git add .gitattributes git commit -m "Add Git LFS tracking" git push origin main

🔄 Making Future Updates

Regular Workflow

After your initial push, use this workflow for updates:

Update Workflow
# 1. Check what changed git status # 2. Stage your changes git add . # 3. Commit with a descriptive message git commit -m "Add new feature: user authentication" # 4. Push to GitHub git push

Best Practices for Commit Messages

  • Be descriptive: "Add user authentication" instead of "Update files"
  • Use present tense: "Add feature" not "Added feature"
  • Keep it concise: First line should be 50 characters or less
  • Use conventional commits:
    • feat: New feature
    • fix: Bug fix
    • docs: Documentation changes
    • style: Code style changes
    • refactor: Code refactoring
    • test: Adding tests
    • chore: Maintenance tasks
Example Commits
git commit -m "feat: add GCash payment integration" git commit -m "fix: resolve product image stretching issue" git commit -m "docs: update deployment guide" git commit -m "refactor: optimize database queries"
