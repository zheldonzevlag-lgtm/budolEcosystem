💡 Key Learnings

1. Prisma Schema Consistency is Critical

The most significant issue was the inconsistency between Prisma relation names in the schema (capitalized) and their usage in code (lowercase). This caused:

  • API 500 errors when querying with incorrect relation names
  • Frontend components unable to access data
  • Confusion during debugging

Lesson: Always verify relation names match exactly between schema and code.

2. Prisma Schema Documentation is Essential

Every field must be verified against the schema before creating data:

  • Check required vs optional fields
  • Verify field names (e.g., inStock not stock)
  • Understand default values and auto-generation
  • Know which fields have defaults (like cuid())

3. Composite Keys Require Special Handling

OrderItem uses a composite key @@id([orderId, productId]), which means:

  • Don't provide explicit id in nested creates
  • Prisma handles ID generation automatically
  • Attempting to set ID manually causes conflicts

4. Iterative Testing is Necessary

The script required 20+ iterations to get right because:

  • Prisma validation errors appear one at a time
  • Each fix reveals the next issue
  • Schema complexity requires careful attention

Lesson: Budget time for iterative debugging when working with complex schemas.

5. Frontend-Backend Consistency

When API structure changes, all consuming components must be updated:

  • API routes
  • Frontend components
  • Type definitions (if using TypeScript)

🎯 Recommendations for Future Development

1. Establish Naming Conventions

  • Document whether relations should be capitalized or lowercase
  • Enforce consistency across the entire codebase
  • Use linting rules to catch naming inconsistencies

2. Create Schema Documentation

  • Maintain a visual diagram of the database schema
  • Document all required fields and their purposes
  • List all relations and their cardinality

3. Implement Type Safety

  • Use TypeScript with Prisma-generated types
  • This would have caught relation name errors at compile time
  • Reduces runtime errors significantly

4. Add Integration Tests

  • Test API endpoints with real database queries
  • Verify relation loading works correctly
  • Catch schema mismatches early

5. Create Seed Scripts

  • Maintain official seed scripts for development
  • Include comprehensive test data
  • Version control seed data
