🔧 Technical Implementation

1. StoreAddressModal Component

File: components/StoreAddressModal.jsx

Key Features:

  • Structured Input Fields:
    • Phone Number (with country code dropdown)
    • Region (with auto-suggestions)
    • Province (optional)
    • City/Municipality (with auto-suggestions)
    • Barangay (with auto-suggestions)
    • Detailed Address (Street, Building, House No.)
    • Postal Code
    • Label/Notes (optional)
  • Validation: Client-side validation for required fields and E.164 phone format
  • Address Formatting: Converts structured data to formatted string
  • Address Parsing: Extracts structured data from formatted string for editing

Address Parsing Logic:

// Parse formatted address string back into structured components // Format: "Detailed Address, Barangay, City, Province, Region, Zip, Country" const parseFormattedAddress = (formattedAddress) => { if (!formattedAddress) return null const parts = formattedAddress.split(',').map(p => p.trim()) if (parts.length < 4) return null // Extract components (working backwards from the end) const country = parts[parts.length - 1] || 'Philippines' const zip = parts[parts.length - 2] || '' const district = parts[parts.length - 3] || '' // Region const province = parts.length > 6 ? parts[parts.length - 4] : '' // City is either 4th from end (if province exists) or 3rd from end const cityIndex = parts.length > 6 ? parts.length - 5 : parts.length - 4 const city = parts[cityIndex] || '' // Barangay is before city const barangay = parts[cityIndex - 1] || '' // Everything before barangay is detailed address const detailedAddress = parts.slice(0, cityIndex - 1).join(', ') return { district, province, city, barangay, detailedAddress, zip, country } }

🐛 Issues Encountered and Fixes

Issue 1: File Corruption During Multi-Replace

Problem:

Using multi_replace_file_content with inaccurate target content caused severe file corruption with duplicate code blocks and broken JSX structure.

Symptoms:

  • Duplicate imports and state definitions
  • Nested component definitions
  • Missing closing tags
  • Hundreds of TypeScript lint errors

Solution:

Restored files from git and used write_to_file with Overwrite: true to completely rewrite the files with correct content.

git checkout HEAD -- "app/store/settings/page.jsx" "app/(public)/create-store/page.jsx"

Issue 2: Empty Fields When Editing Address

Problem:

When clicking "Edit Address" in settings, only the phone number was pre-filled. All other fields (Region, City, Barangay, etc.) were empty.

Root Cause:

The database stores the address as a single formatted string (e.g., "123 Web Lane, Spider Tower, Intramuros, Manila, NCR - National Capital Region, 1002, Philippines"). The modal needs structured data but was only receiving the formatted string.

Solution:

Implemented parseFormattedAddress function that intelligently parses the formatted string back into structured components by working backwards from the end of the string.

Parsing Strategy:

  1. Split the string by commas
  2. Extract components from right to left (Country → Zip → Region → Province → City → Barangay → Detailed Address)
  3. Handle optional province field
  4. Join remaining parts as detailed address

Issue 3: Contact Information Section Clutter

Problem:

The modal displayed a "CONTACT INFORMATION" section with Full Name and Phone Number fields, which was unnecessary for store address input.

Solution:

Removed the "CONTACT INFORMATION" section header and Full Name field. Moved Phone Number field under the "ADDRESS" section as it's required for the store contact.

Issue 4: Redundant Email and Contact Inputs

Problem:

The create-store and settings forms had separate email and contact number input fields, creating redundancy with the address modal.

Solution:

  • Removed email input - automatically use logged-in user's email
  • Removed contact number input - capture from address modal's phone field
  • Updated validation to ensure address (and thus contact) is required

🔄 Data Flow

Creating a New Store

  1. User clicks "+ Add Address" button
  2. StoreAddressModal opens with empty fields
  3. User fills in:
    • Phone Number
    • Region, City, Barangay
    • Detailed Address
    • Postal Code
  4. User clicks "Save Address"
  5. Modal formats address into string: "Detailed, Barangay, City, Province, Region, Zip, Country"
  6. Modal calls onSave(formattedAddress, addressObj)
  7. Parent component updates:
    • storeInfo.address = formattedAddress
    • storeInfo.contact = addressObj.phone
  8. On form submit:
    • email = user.email (from logged-in user)
    • contact = storeInfo.contact
    • address = storeInfo.address

Editing Existing Store Address

  1. User clicks "Edit Address" button
  2. StoreAddressModal opens
  3. emptyAddress(initialData) is called:
    • Calls parseFormattedAddress(initialData.address)
    • Extracts: Region, City, Barangay, Detailed Address, Zip
    • Gets phone from initialData.contact
  4. All fields are pre-filled with parsed data
  5. User modifies fields as needed
  6. User clicks "Save Address"
  7. Same save flow as creating new address

💾 Database Schema

Store Model

The Store model in prisma/schema.prisma has the following address-related fields:

Field Type Description
email String Store email (from user account)
contact String Store phone number in E.164 format
address String Formatted address string
Note: The address is stored as a single formatted string rather than separate fields. This maintains compatibility with the existing database schema while still providing a structured input experience.
