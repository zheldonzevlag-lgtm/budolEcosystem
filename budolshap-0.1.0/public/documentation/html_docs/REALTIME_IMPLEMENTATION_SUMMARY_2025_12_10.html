Realtime Architecture & Implementation Guide

Date: December 10, 2025
Context: Transitioning BudolShap Order Management to a Scalable Realtime System.

1. Issues & Challenges Addressed

❌ The Friction Points

Prior to this update, the application relied on "Fetch Once" logic. This caused several issues:

  • Stale Data: Sellers had to manually refresh the page to see if a new order arrived.
  • Lalamove Disconnect: When a driver status changed (e.g., "Picked Up"), the UI remained static.

2. Implemented Solutions (The Tri-brid Fix)

✅ The "Tri-brid" Realtime Engine

We implemented a system that supports Three Modes, switchable at runtime.

Mode Mechanism Cost Use Case
POLLING SWR Hook fetches data every 5 seconds. FREE Default. Good for MVP.
PUSHER Server pushes events via WebSockets. PAID / LIMITED Production Standard (Fast + Easy).
SOCKET.IO Dedicated Node.js Microservice. SERVER COST Enterprise Scale (Unlimited Conns).

4. Validated Socket.io Microservice

🚀 The /websocket-server Folder

We created a standalone Node.js application inside the project structure. It contains:

  • server.js: The Express + Socket.io logic.
  • Dockerfile: Ready-to-deploy configuration container.
  • Endpoint: POST /trigger (called by Next.js backend to broadcast messages).

5. Detailed Deployment Guide (Socket.io)

Since Vercel cannot host persistent WebSocket servers, choose one of the providers below for the /websocket-server folder.

Option A: Render (Easiest)

  1. Log in to Render Dashboard.
  2. Click New + > Web Service > Connect your Repo.
  3. IMPORTANT Config:
    • Root Directory: websocket-server
    • Runtime: Docker
  4. Click Create. Copy the onrender.com URL.

Option B: Railway (Fastest)

  1. Go to Railway.app and "Start a New Project".
  2. Select Deploy from GitHub repo.
  3. Once the repo is added, click on the project card to open Settings.
  4. Go to Settings > Build > Root Directory and set it to: /websocket-server.
  5. Railway will auto-detect the Dockerfile/Node app and redeploy.
  6. Go to Settings > Networking and generate a Domain to get your URL.

Option C: Fly.io (CLI Based)

  1. Install Fly CLI: iwr https://fly.io/install.ps1 -useb | iex (Windows).
  2. Open terminal in your project: cd websocket-server.
  3. Run fly launch. it will detect the Dockerfile.
  4. Follow the prompts (Yes to copy config, No to Tweak settings).
  5. Uncomment the EXPOSE 4000 line in Dockerfile if not detected.
  6. Run fly deploy. Your app runs on https://your-app-name.fly.dev.

Option D: DigitalOcean App Platform

  1. Go to DO App Platform.
  2. Click Create App and select GitHub.
  3. Select your repository.
  4. Resources: Click "Edit" next to the detected web service.
  5. Source Directory: Ensure it is set to websocket-server.
  6. Autodetect: It should see "Docker" or "Node.js". Docker is safer.
  7. Click Next until "Create Resources". Copy the live URL.

Final Step (For All Providers)

Go to Admin > Realtime Settings. Switch provider to Socket.io and paste your new valid URL.

6. Verification & Troubleshooting

🔍 Verifying Deployment

After deploying, check the following:

  1. Admin Panel: Go to Realtime Settings. You should see all 3 options (Polling, Pusher, Socket.io).
  2. Socket Connection: If using Socket.io, open the browser console. You should see 🔌 Connecting to Socket.io... and then ✅ Socket connected.

🐞 Common Issues Log

  • Build Error (Exit Code 1): Check lib/realtime.js. Ensure all braces } are closed properly, especially after the Pusher block.
  • Option Not Showing: If "Socket.io" is missing in the dropdown, ensure you have redeployed the Frontend (Vercel) after the code merge.
  • Connection Failed: "Connection Refused" usually means the Render/Railway service is asleep or the URL is incorrect (ensure no trailing slash).

7. Future Recommendations & Enhancements

⭐ Optimistic UI Updates (Recommended)

Goal: Zero-latency feel.

Currently, when a realtime event fires, the frontend calls mutate() to re-fetch the fresh data from the API. To make the app feel "instant", the UI should update its local state immediately based on the event payload (e.g., flip status to "Shipped") even before the server confirms.

📦 Granular Event Payloads

Goal: Reduce bandwidth and API calls.

Current: Event says "Update Available" -> Client fetches ALL orders.
Future: The event should contain the actual data: { orderId: 101, status: 'DELIVERED' }. The client can then update just that single row in the React state without a network call.

🔄 Redis Adapter for Socket.io

Goal: Horizontal Scaling.

If you deploy multiple instances of your Socket.io server (e.g., 2 Render nodes to handle 50k users), they won't talk to each other by default. You will need to add a Redis Adapter to sync events across all nodes.
