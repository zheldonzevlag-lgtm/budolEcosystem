1. UniversalTrackingMap Component

NEW components/UniversalTrackingMap.jsx

'use client';

import LalamoveTracking from './LalamoveTracking';
import StandardDeliveryMap from './StandardDeliveryMap';

export default function UniversalTrackingMap({ order }) {
    if (!order || !order.shipping) {
        return <StandardDeliveryMap order={order} />;
    }

    const provider = order.shipping.provider?.toLowerCase();

    switch (provider) {
        case 'lalamove':
            return <LalamoveTracking order={order} />;
        
        // Future providers
        // case 'grab':
        //     return <GrabTracking order={order} />;
        
        default:
            return <StandardDeliveryMap order={order} />;
    }
}

🐛 Issues Identified & Fixes Applied

Issue 1: Frontend Not Provider-Agnostic

Problem: The order details page had hardcoded conditional rendering for Lalamove vs Standard maps. Adding new providers would require modifying the main page every time.
Solution: Created UniversalTrackingMap component that uses a switch statement to select the correct map component based on order.shipping.provider.

Issue 2: No Visibility for Cancelled Deliveries

Problem: Sellers had no way to track orders that were cancelled by third-party delivery providers (e.g., Lalamove driver cancellations).
Solution: Added a dedicated "Cancelled" tab with red color coding to the Store Orders dashboard, making these problem orders immediately visible.

Issue 3: Inconsistent Status Badge Colors

Problem: Status badges were showing blue for any order with shipping info, even if the order was delivered or cancelled. This created visual confusion.
Solution: Refactored badge logic to prioritize status-based colors: Green for Delivered, Red for Cancelled, Purple for Shipped, Blue for others.

Issue 4: "SHIPPED" vs "IN TRANSIT" Terminology

Problem: The status badge showed "SHIPPED" while the tab and delivery column said "In Transit", creating inconsistency.
Solution: Updated the badge text to display "IN TRANSIT" for shipped orders, maintaining the purple color.

export function getShippingProvider(providerName) {
    switch (providerName.toLowerCase()) {
        case 'lalamove':
            return new Lalamove();
        // Future providers:
        // case 'grab':
        //     return new Grab();
        // case 'jnt':

✅ Backend Verdict: The backend is already well-structured for multi-provider support. Adding new providers only requires creating a new service class (e.g., services/Grab.js) and adding a case to the factory.

Webhook Handler

app/api/webhooks/lalamove/route.js correctly handles provider-specific events:

  • Maps Lalamove statuses (PICKED_UP, COMPLETED, CANCELLED) to internal order statuses
  • Updates order.shipping.status with provider-specific status
  • Preserves shareLink and trackingUrl for map display
  • Sends email notifications on status changes

2. Provider-Specific Status Mapping

Priority: Medium

Different providers may use different status terminology. Consider creating a status mapping layer: 

// services/statusMapper.js
export function mapProviderStatus(provider, providerStatus) {
    const mappings = {
        lalamove: {
            'PICKED_UP': 'SHIPPED',
            'COMPLETED': 'DELIVERED',
            'CANCELLED': 'CANCELLED'
        },
        grab: {
            'DRIVER_PICKED_UP': 'SHIPPED',
            'ORDER_DELIVERED': 'DELIVERED'
        }
    };
    return mappings[provider]?.[providerStatus] || providerStatus;
}
