PayMongo Integration Session Documentation

Session Date: December 15, 2024

Overview

This session focused on finalizing the integration of Maya, GrabPay, and QRPh payment methods via PayMongo. The primary objectives were to ensure reliable QR code display, fix production deployment issues, synchronize the database schema, and verify the end-to-end payment flow including the success page details.

Process & Issues Log

1. QR Code Modal Display Feature

Issue: The QR Code modal was disappearing immediately or not showing at all due to component unmounting in OrderSummary.jsx (caused by premature cart clearing).
Fix: Inline Modal & Smart Cart Logic
Restored the inline modal within components/OrderSummary.jsx:
  • Portal Implementation: Used ReactDOM.createPortal to render the modal at the document body level, ensuring it overlays correctly.
  • Cart Persistence: modified handlePlaceOrder to only clear the cart immediately for non-QR payments. For QR payments, the cart (and thus the component) remains active until the payment is confirmed.
  • Z-Index: Increased modal z-index to 9999 to prevent navbar overlap.

2. Vercel Build Failure Bug Fix

Issue: Deployment failed with error: useSearchParams() should be wrapped in a suspense boundary.
Fix: Suspense Boundary
Refactored pages using search params to wrap content in Suspense. (Note: The dedicated QR page was removed in favor of the inline modal, but this pattern was applied elsewhere).

3. Production Database Schema Sync DevOps

Issue: Runtime error "invalid input value for enum" for PaymentMethod.
Fix: Remote Schema Push
Executed npx prisma db push to align the production database enum values with the codebase.

4. Payment Success Page Details Bug Fix

Issue: Order ID, Amount, and Status were missing on the success page for Maya/GrabPay.
Fix: Robust Order Lookup
Updated app/api/orders/update-status/route.js to look up orders by `paymentId` instead of relying solely on description parsing.

5. Automatic Payment Verification New Feature

Implementation: Polling
Added a polling mechanism to components/payment/QRCodeModal.jsx:
  • Checks /api/paymongo/status every 3 seconds while the modal is open.
  • Automatically redirects the user to the success page upon detecting succeeded status.
  • Benefit: Reduces friction; users don't need to manually click "I've Paid".

6. UI Polish & Branding UX

Enhancements:
  • Branding: Added colorful "budolShap." logo to the modal header (Green 'budol', Slate 'Shap', Green dot) and updated labels (e.g., "budolShap Owner").
  • Responsiveness: Added max-h-[90vh] and scroll support for smaller screens.
  • Cleanliness: Hidden visual scrollbar using CSS utilities while maintaining functionality.

Technical Details

Payment Flow Architecture

  • GCash: Redirect Flow (PayMongo Hosted Page)
  • Maya / GrabPay / QRPh: Inline QR Code Modal -> Auto-poll Verification -> Success Page

Key Files

  • components/OrderSummary.jsx: Main checkout logic & modal trigger.
  • components/payment/QRCodeModal.jsx: QR UI & Polling Logic.
  • app/api/orders/update-status/route.js: Verification webhook/callback handler.

Recommendations for Future Enhancement

  1. Webhook Implementation: Currently, we rely on the client redirecting to /payment/return to update the order status. Implementing full server-side webhooks (listening to PayMongo's `source.chargeable` or `payment_intent.succeeded` events) would ensure orders are marked as paid even if the user closes their browser immediately after payment.
  2. Abandoned Checkout Recovery: Capture partial order details before payment initiation to allow retargeting users who drop off at the payment modal or gateway page.
  3. Payment Method Icons: Centralize payment method icons in a UI component library to ensure consistency across the checkout, modal, and footer.
