💳 Payment Error Handling Guide

E-Commerce Best Practices for Budolshap

🔴 The Problem

What Happened?

After authorizing a GCash payment through PayMongo's test environment, users were redirected to a 404 error page instead of a proper payment status page.

Root Causes:

  • Missing Environment Variable: NEXT_PUBLIC_SITE_URL not set in production
  • No Fallback Handling: Application didn't gracefully handle redirect failures
  • Incomplete Error States: Limited error handling for edge cases
  • No Retry Mechanism: Users couldn't easily retry failed payments

✅ The Solution

What We Implemented:

  • ✅ Enhanced /payment/return page with comprehensive error handling
  • ✅ Created /payment/pending page for in-progress payments
  • ✅ Added 5 distinct payment states with proper UX
  • ✅ Implemented automatic retry mechanisms
  • ✅ Added timeout protection (60 seconds max)
  • ✅ Preserved order context for recovery

🏪 E-Commerce Best Practices

1. Never Show Raw Errors

❌ Before (Bad) ✅ After (Good)
404 Not Found Unable to verify payment. Please check your orders.
500 Internal Server Error We're having technical difficulties. Your payment is safe.
Error: undefined Payment verification incomplete. Please contact support.

2. Always Provide Next Steps

Every error page now answers three critical questions:

  • What happened? Clear status message
  • Why did it happen? Possible reasons
  • What can I do? Action buttons

📊 5 Payment States Implemented

⏳

CHECKING

When: User just returned from PayMongo

  • Animated spinner
  • Polls API every 2s
  • Max 60 seconds
  • Progress indicator
✅

SUCCEEDED

When: Payment confirmed

  • Green checkmark
  • Order details shown
  • DB updated
  • Email sent
❌

FAILED

When: Payment declined

  • Red X icon
  • Failure reasons
  • Retry button
  • Order preserved
⏰

TIMEOUT

When: Verification > 60s

  • Clock icon
  • Check later message
  • Retry option
  • Support guidance
⚠️

ERROR

When: Technical issues

  • Alert icon
  • Error explanation
  • Payment safety note
  • Support contact

🔧 Technical Implementation

Automatic Retry Logic

// Retry failed API calls up to 3 times const pollStatus = async (id, oid, count = 0) => { try { const resp = await fetch(`/api/paymongo/status?intent_id=${id}`); // ... process response } catch (e) { if (count < 3) { // Retry after 3 seconds setTimeout(()=> pollStatus(id, oid, count + 1), 3000); } else { // Give up after 3 retries setStatus('error'); } } };

Timeout Protection

const MAX_POLLS = 30; // 30 × 2 seconds = 60 seconds if (count >= MAX_POLLS) { setStatus('timeout'); setErrorMessage('Verification taking longer than expected'); }

🎓 What We Learned from Big E-Commerce

Amazon's Approach:

  • Never shows technical errors to customers
  • Always provides order tracking
  • Multiple retry options
  • 24/7 support links on error pages

Shopify's Approach:

  • Clear payment status pages
  • Immediate email confirmations
  • Order recovery flows
  • Detailed but user-friendly error messages

Stripe's Approach:

  • Comprehensive error codes
  • Automatic retry logic with exponential backoff
  • Webhook verification for security
  • Idempotent API requests
