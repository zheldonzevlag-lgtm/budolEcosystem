Implementation Plan: Lalamove Tracking Sync

Date: December 14, 2025

Status: Pending Implementation

1. Goal Description

Sync the buyer order progress with the seller order tracking context. Specifically, ensure that when the Lalamove status updates to PICKED_UP, the system-wide order status transitions to IN_TRANSIT immediately (matching the seller's view validation).

Additionally, the buyer's tracking history should display an "In Transit" status with a clear remark indicating the driver has picked up the package.

2. User Review Required

IMPORTANT: The Lalamove PICKED_UP event will now trigger the IN_TRANSIT order status instead of SHIPPED. This effectively makes the "In Transit" stage active immediately upon pickup, skipping the intermediate "Shipped" visual state if it was distinct, which aligns better with same-day delivery flows.

3. Proposed Changes

Core Logic Refactoring (Robustness)

  • File: lib/shipping/config.js [NEW]
    • Change: Create a centralized configuration file.
    • Detail: Define SHIPPING_PROVIDERS and PROVIDER_STAGES to allow scalable additions of new couriers (e.g., Grab, Borzo) without modifying UI components.

App Logic Updates

  • File: app/api/webhooks/lalamove/route.js
    • Change: Update the logic for PICKED_UP events.
    • Detail: Set newStatus = 'IN_TRANSIT' instead of 'SHIPPED' when a driver picks up the package.

UI Component Updates

  • File: components/OrderProgressTracker.jsx
    • Change: Refactor to use dynamic configuration.
    • Detail: Remove hardcoded provider checks. Instead, import stages from lib/shipping/config.js matching the order's provider. Map PICKED_UP to the In Transit stage via config.
  • File: components/TrackingTimeline.jsx
    • Change: Generalize timeline generation.
    • Detail: Use the new configuration to determine if the current status warrants the "In Transit" remark: "Driver has picked up your package. Package is on the way."

4. Future Enhancements

The current implementation establishes a solid foundation for tracking. Future iterations can focus on:

  • Real-time Map Integration: Embed a live map view (e.g., Google Maps or Lalamove widget) for "In Transit" orders.
  • Driver Chat: Direct communication link with the assigned driver.
  • SMS/Email Notifications: Trigger notifications on specific status changes (e.g., "Out for Delivery").
  • Multi-webhook Support: Standardize webhook handlers for other providers to auto-map their events to our stages.

5. Developer Guide: Adding a New Courier

The system is designed to be courier-agnostic. To add a new shipping provider (e.g., "Grab"), follow these steps:

Step 1: Update Configuration

Edit lib/shipping/config.js:

// 1. Add to SHIPPING_PROVIDERS
export const SHIPPING_PROVIDERS = {
    // ...
    GRAB: 'grab'
};

// 2. Define Tracking Stages (Progress Bar)
export const PROVIDER_STAGES = {
    [SHIPPING_PROVIDERS.GRAB]: [
        { label: 'Booking Confirmed', status: 'ORDER_PLACED', icon: '📝' },
        { label: 'Driver Found', status: 'PROCESSING', specificStatus: 'DRIVER_ALLOCATED', icon: '👤' },
        // ... define other stages
    ]
};

// 3. Define Timeline Events (History)
export const PROVIDER_TIMELINE_EVENTS = {
    [SHIPPING_PROVIDERS.GRAB]: {
        PROCESSING: { title: 'Driver Assigned', description: 'We found a driver for you.' },
        IN_TRANSIT: { 
            title: 'On the Way', 
            description: 'Your Grab delivery is arriving.',
            triggerStatuses: ['IN_DELIVERY']
        }
    }
};

Step 2: Ensure Backend Compatibility

Ensure your webhook or API logic sets the correct provider string in order.shipping.provider (must match value in Step 1) and updates order.shipping.status or order.shipping.lastEvent with the provider-specific status codes used in your config.
