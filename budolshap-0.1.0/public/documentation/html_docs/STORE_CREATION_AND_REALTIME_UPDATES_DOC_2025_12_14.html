Store Address Fix, Realtime Status & UI Updates

Date: December 14, 2025

Summary: This document details the technical implementation required to fix the "Legacy Address" issue for new stores, implement realtime responsiveness for store approval, and update product page branding.

1. Store Address & Geocoding Fix FIX

Issue: New stores were saving addresses as simple strings, causing them to be flagged as "Legacy" and lacking Geocoordinates (Lat/Long) required for couriers like Lalamove.

Technical Implementation

  • Frontend Geocoding (components/StoreAddressModal.jsx):
    Integrated OpenStreetMap Nominatim API to fetch coordinates before saving. 
    const response = await fetch(`https://nominatim.openstreetmap.org/search?q=${query}&format=json&limit=1`)
  • Data Payload (app/(public)/create-store/page.jsx):
    Updated the form submission to include the full structuredAddress object containing latitude, longitude, city, district, etc.
  • Backend Storage (app/api/stores/route.js):
    Modified the POST endpoint to create a StoreAddress record linked to the Store.
    Impact: New stores now have a properly structured address and are compatible with courier APIs.

2. Realtime Store Status Updates FEATURE

Issue: After a user submitted a store application, the page would remain "Pending" even after Admin approval until a manual refresh. Additionally, the Admin API was taking a shortcut by only updating verificationStatus but not the legacy status field.

Technical Implementation

  • Database Synchronization:
    Updated app/api/admin/stores/[id]/verify/route.js to sync the status field (string) whenever verificationStatus (enum) changes. This ensures the frontend (which checks status) sees the update immediately.
  • Realtime Trigger (Server):
    Added triggerRealtimeEvent to the verification endpoint. 
    await triggerRealtimeEvent(`user-${store.userId}`, 'store-status-updated', { status: 'APPROVED' })
  • Realtime Listener (Client):
    Updated app/(public)/create-store/page.jsx to listen for the store-status-updated event.
    Architecture: Implemented a Provider-Agnostic pattern that dynamically uses Pusher, Socket.io, or falls back to Polling based on /api/system/settings.

3. Product Branding UI Updates UI

Objective: Update the shipping text to reflect the "budol" sub-brands with specific color coding.

Changes (components/ProductDetails.jsx)

  • Text: "Free shipping via budolExpress & payment via budolPay"
  • Styling:
    • Whole Brand: font-semibold
    • "budol": text-green-700 (Green)
    • "Express"/"Pay": Default text color
    • Formatting: Ensured no spaces between "budol" and suffix (e.g., budolExpress).

4. Recommendations for Future Enhancement

Component Recommendation Benefits
Address Input Implement a visual Map Picker (Leaflet or Google Maps) instead of text-search geocoding. Higher accuracy for delivery pins; better UX for users who don't know their exact text address format.
Realtime Hooks Extract the listener logic from create-store/page.jsx into a reusable hook useStoreStatus(userId). Clean code; reusability for other dashboard pages that might need to know store status changes.
Courier Validation Add a "Check Service Availability" button in the address modal using the Lalamove API. Prevents users from saving addresses that are technically valid but out of courier service range.
Brand Components Create <BudolBrand type="Express" /> reusable components. Ensures consistent coloring and font-weight across the entire site without repeating manual HTML/CSS.
