Database Synchronization Protocol

Incident Report & Process Guide | Generated: Dec 13, 2025

Status: RESOLVED - Bidirectional synchronization is now enabled.

Direction 1: Production -> Local (Downstream)

Use this to pull the latest live data into your development environment.

node scripts/sync-db.js
  • Source: .env.check_prod (Verified "Rich" DB)
  • Dest: .env.connection.local (Localhost)
  • Safety: Wipes Local DB before copy.

Direction 2: Local -> Production (Upstream)

⚠️ DANGER ZONE: This operation is destructive.

Use this to overwrite the live production database with your local data. Useful for resetting production to a known state.

node scripts/push-to-prod.js
  • Source: .env.connection.local (Localhost)
  • Dest: .env.check_prod (Verified "Rich" DB)
  • Safety: Requires typing CONFIRM_PROD_OVERWRITE to proceed.

Work Context Switching (Detailed Guide)

You can instantly switch your application's active database connection without changing code. This determines which database the app talks to when you run npm run dev.

Run this command to switch and start the server:
npm run db:local && npm run dev (Development)
OR
npm run db:prod && npm run dev (Production Review)
Option A: Switch to Local (Development)

Use this for day-to-day coding, testing new features, and breaking things safely.

npm run db:local

This copies configuration from .env.connection.local to .env.local.

Default Connection: postgresql://postgres:password@localhost:5432/budolshap_v3

Option B: Switch to Production (Live Data)

Use this to verify bugs against real data or check the state of the live system from your local machine.

npm run db:prod

This copies configuration from .env.connection.prod to .env.local.

Command Breakdown (What happens behind the scenes?)
  • npm run db:local

    1. Runs node scripts/env-manager.js local.

    2. Reads .env.connection.local (Source).

    3. Reads your current .env.local (Target).

    4. Merges them: Updates DATABASE_URL and DIRECT_URL to point to Localhost, but keeps all your other API keys (Lalamove, keys, etc.) intact.


  • npm run db:prod

    1. Runs node scripts/env-manager.js prod.

    2. Reads .env.connection.prod (Source).

    3. Reads your current .env.local (Target).

    4. Merges them: Updates DATABASE_URL and DIRECT_URL to point to the Verified Production DB, keeping other keys intact.

File Breakdown (What is each file for?)
  • .env.connection.local

    Holds ONLY your local database credentials (postgres://localhost...). This is the "blueprint" for your local connection.

  • .env.connection.prod

    Holds ONLY your verified production database credentials. This is the "blueprint" for your production connection.

  • scripts/env-manager.js

    The "Switcher" utility. It performs the smart merge logic. It ensures that when you switch databases, you don't accidentally wipe out your Lalamove or Cloudinary keys.

Troubleshooting Deployments

If your Vercel deployment fails with database errors, ensure you have added DIRECT_URL to your Vercel Environment Variables.

  • Key: DIRECT_URL
  • Value: Same as your POSTGRES_URL (or POSTGRES_URL_NON_POOLING).

Critical Safety Notes

Database Identity Verification

Both scripts rely on .env.check_prod rather than the standard Vercel environment variables.

Why? We discovered a "Phantom Database" issue where Vercel's default DATABASE_URL pointed to an empty instance, while the live app used a different one.

Rule: Always ensure .env.check_prod contains the verified, "Rich" database credentials before running these scripts. Never push to production if you are unsure.
