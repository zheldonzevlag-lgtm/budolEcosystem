1. Executive Overview

Project Goal

Migrate BudolShap from a modular monolith to a microservices architecture while maintaining full functionality, improving scalability, and enabling independent deployment of services.

1.1 Current State

Aspect Current
Architecture Next.js monolith with 80+ API routes
Deployment Single Vercel project
Database Single PostgreSQL database (shared)
Realtime POLLING (default), Pusher-ready, Socket.io server available
Data Fetching SWR (perfect for microservices!)

1.2 Target State

Service Responsibility Deployment
API Gateway Frontend + Composition APIs Vercel (main app)
Auth Service Authentication & User Management Vercel (separate project)
Catalog Service Stores, Products, Coupons Vercel (separate project)
Order Service Orders, Cart, Ratings, Returns Vercel (separate project)
Payment Service Payment Processing, Webhooks Vercel (separate project)
Shipping Service Lalamove, Manual Shipping, Tracking Vercel (separate project)
Admin Service Analytics, Admin Operations Vercel (main app or separate)
Realtime Service Pusher (hosted) or Socket.io Pusher Cloud (recommended)

1.3 Key Principles

  • Incremental Migration: Extract services one at a time, test thoroughly
  • Zero Downtime: Maintain production functionality throughout
  • Backward Compatible: Old endpoints work until fully migrated
  • SWR Compatible: Frontend code requires minimal changes
  • Vercel Native: All services deployable on Vercel

2. Prerequisites & Setup (Week 0)

📋 Pre-Migration Checklist

Duration: 3-5 days

2.1 Environment Setup

✅ Set up Pusher Account
  • Create account at pusher.com
  • Create app (choose cluster: ap1 for Philippines)
  • Get credentials: App ID, Key, Secret, Cluster
  • Time: 30 minutes
✅ Configure Pusher in Admin Panel
  • Navigate to Admin → Settings → Realtime Updates
  • Select "Pusher" provider
  • Enter all 4 credentials
  • Test connection
  • Time: 15 minutes
✅ Set up Vercel Projects (Optional - Can do later)
  • Create separate Vercel projects for each service (or wait until extraction)
  • Configure environment variables
  • Set up domain mappings
  • Time: 1-2 hours (can be done per-phase)
✅ Create Centralized API Fetcher
  • Create lib/api/fetcher.js
  • Create lib/api/serviceUrls.js for service URL management
  • Add environment variables for service URLs
  • Time: 2-3 hours
✅ Set up Monitoring & Logging
  • Configure Vercel Analytics (if not already)
  • Set up error tracking (Sentry or similar)
  • Configure logging aggregation
  • Time: 2-3 hours

✅ Deliverables

  • Pusher account configured and tested
  • Centralized API fetcher created
  • Service URL configuration system in place
  • Monitoring tools configured
  • Development environment ready for Phase 1

3. Phase 1: Modularization (Weeks 1-2)

🎯 Phase 1 Goal

Transform the monolith into a modular monolith with clear service boundaries, while keeping everything in one codebase and deployment.

Duration: 2 weeks (10 working days)

Risk Level: Low

Week 1: Service Layer Creation

  • Days 1-2: Create service layer structure
  • Days 3-4: Extract business logic from API routes
  • Day 5: Standardize responses and error handling

Week 2: Refactoring & Testing

  • Days 6-7: Refactor API routes to use services
  • Days 8-9: Enforce domain boundaries
  • Day 10: Testing, documentation, code review

3.1 Week 1 Tasks

Day 1-2: Create Service Layer Structure
  • Create lib/services/ directory
  • Create service modules:
    • lib/services/authService.js
    • lib/services/ordersService.js
    • lib/services/paymentService.js
    • lib/services/shippingService.js
    • lib/services/catalogService.js
    • lib/services/adminService.js
  • Define service interfaces (JSDoc or TypeScript types)
  • Estimated Time: 8-10 hours
Day 3-4: Extract Business Logic
  • Move Prisma queries from API routes to services
  • Extract validation logic
  • Extract business rules (e.g., order creation logic)
  • Keep API routes as thin controllers
  • Estimated Time: 16-20 hours
Day 5: Standardize Responses
  • Create common response format
  • Standardize error handling
  • Add request/response logging
  • Create error utility functions
  • Estimated Time: 4-6 hours

3.2 Week 2 Tasks

Day 6-7: Refactor API Routes
  • Update all API routes to call service methods
  • Remove direct Prisma calls from routes
  • Ensure routes are thin controllers
  • Test each route after refactoring
  • Estimated Time: 16-20 hours
Day 8-9: Enforce Domain Boundaries
  • Ensure services only access their own Prisma models
  • Create service-to-service interfaces
  • Add validation to prevent cross-domain access
  • Document service dependencies
  • Estimated Time: 12-16 hours
Day 10: Testing & Documentation
  • Run full test suite
  • Manual testing of all features
  • Update documentation
  • Code review
  • Deploy to staging
  • Estimated Time: 8-10 hours

✅ Phase 1 Deliverables

  • ✅ Service layer modules created and tested
  • ✅ All API routes refactored to use services
  • ✅ Domain boundaries enforced
  • ✅ Standardized response format
  • ✅ All tests passing
  • ✅ Code deployed to staging
  • ✅ Documentation updated
🎯 Milestone 1: Modular Monolith Complete
⚠️ Risks & Mitigation:
  • Risk: Breaking existing functionality
  • Mitigation: Test each route after refactoring, maintain backward compatibility
  • Risk: Service boundaries not clear
  • Mitigation: Code review, document boundaries clearly

4. Phase 2: Internal Service Boundaries (Weeks 3-4)

🎯 Phase 2 Goal

Introduce HTTP-based communication between services, even though they're still in the same codebase. This prepares for actual service extraction.

Duration: 2 weeks (10 working days)

Risk Level: Medium

Week 3: Internal API Structure

  • Days 11-12: Create internal API structure
  • Days 13-14: Define service interfaces
  • Day 15: Implement service-to-service HTTP calls

Week 4: Integration & Testing

  • Days 16-17: Update services to use HTTP calls
  • Days 18-19: Add health checks and monitoring
  • Day 20: Testing and documentation

4.1 Week 3 Tasks

Day 11-12: Create Internal API Structure
  • Create app/api/internal/ directory
  • Move domain routes to app/api/internal/{service}/*
  • Keep public routes in app/api/public/* or root
  • Set up routing structure
  • Estimated Time: 8-10 hours
Day 13-14: Define Service Interfaces
  • Document API contracts for each service
  • Create OpenAPI/Swagger specs (optional)
  • Define request/response formats
  • Define error codes and status codes
  • Estimated Time: 12-16 hours
Day 15: Implement Service-to-Service Calls
  • Create HTTP client utilities
  • Implement service-to-service calls via HTTP
  • Use environment variables for service URLs
  • In dev: use http://localhost:3000/api/internal/...
  • Estimated Time: 8-10 hours

4.2 Week 4 Tasks

Day 16-17: Update Service Calls
  • Replace direct service method calls with HTTP calls
  • Update all cross-service interactions
  • Add retry logic and error handling
  • Test each integration
  • Estimated Time: 16-20 hours
Day 18-19: Health Checks & Monitoring
  • Create GET /api/internal/{service}/health endpoints
  • Add health check monitoring
  • Implement circuit breakers (optional)
  • Add request tracing
  • Estimated Time: 8-12 hours
Day 20: Testing & Documentation
  • End-to-end testing
  • Performance testing
  • Update documentation
  • Deploy to staging
  • Estimated Time: 8-10 hours

✅ Phase 2 Deliverables

  • ✅ Internal service structure defined
  • ✅ Service interfaces documented
  • ✅ Service-to-service HTTP calls implemented
  • ✅ Health checks in place
  • ✅ All tests passing
  • ✅ Monitoring configured
🎯 Milestone 2: Internal Service Boundaries Complete

5. Phase 3: Payment Service Extraction (Weeks 5-6)

🎯 Phase 3 Goal

Extract Payment Service as the first independent microservice. This is the highest-value extraction due to security and isolation requirements.

Duration: 2 weeks (10 working days)

Risk Level: High (Critical service)

Week 5: Service Creation

  • Days 21-22: Create payment service repository
  • Days 23-24: Copy and adapt payment code
  • Day 25: Deploy to Vercel

Week 6: Integration & Testing

  • Days 26-27: Update main app to call payment service
  • Days 28-29: Test payment flows end-to-end
  • Day 30: Production deployment

5.1 Week 5 Tasks

Day 21-22: Create Payment Service Repository
  • Create new repository: budolshap-payment-service
  • Initialize Next.js project
  • Set up Prisma (shared DB or separate schema)
  • Configure environment variables
  • Set up Vercel project
  • Estimated Time: 8-10 hours
Day 23-24: Copy and Adapt Payment Code
  • Copy payment-related files:
    • app/api/payment/*
    • app/api/paymongo/*
    • app/api/webhooks/paymongo/*
    • lib/paymongo.js
    • lib/payment/service.js
    • lib/payment/adapters/*
    • lib/escrow.js (or coordinate with Order Service)
  • Adapt code for standalone service
  • Update imports and dependencies
  • Test locally
  • Estimated Time: 12-16 hours
Day 25: Deploy to Vercel
  • Deploy to: payment.budolshap.com
  • Configure environment variables
  • Set up PayMongo webhook URL
  • Test deployment
  • Estimated Time: 4-6 hours

5.2 Week 6 Tasks

Day 26-27: Update Main App
  • Replace local payment code with HTTP calls
  • Update app/api/payment/* to proxy to payment service
  • Update lib/payment/service.js to call external API
  • Update environment variables
  • Test integration
  • Estimated Time: 12-16 hours
Day 28-29: End-to-End Testing
  • Test payment flow (GCash, Maya, GrabPay, QRPH)
  • Test webhook handling
  • Test error scenarios
  • Test with real PayMongo (sandbox)
  • Load testing
  • Estimated Time: 12-16 hours
Day 30: Production Deployment
  • Deploy payment service to production
  • Update main app production environment
  • Monitor for issues
  • Rollback plan ready
  • Estimated Time: 4-6 hours

✅ Phase 3 Deliverables

  • ✅ Payment service deployed independently
  • ✅ Main app calls payment service via HTTP
  • ✅ All payment tests passing
  • ✅ Webhooks functioning correctly
  • ✅ Production deployment successful
🎯 Milestone 3: First Microservice Extracted (Payment)
⚠️ Critical Considerations:
  • Payment is critical - test thoroughly before production
  • Have rollback plan ready
  • Monitor webhook delivery closely
  • Keep old code as backup for 1-2 weeks

11. Risk Management

11.1 Risk Matrix

Risk Probability Impact Mitigation
Breaking existing functionality Medium High Thorough testing, gradual migration, rollback plan
Service communication failures Medium High Retry logic, circuit breakers, graceful degradation
Increased latency Low Medium Optimize calls, use caching, minimize cross-service calls
Database connection limits Low Medium Connection pooling, monitor usage
Deployment complexity Medium Medium Automate deployments, use CI/CD
Team knowledge gap Low Medium Training, documentation, code reviews

11.2 Contingency Plans

Rollback Strategy

  • Keep old code in separate branch for 2-4 weeks after migration
  • Maintain feature flags to switch between old/new implementations
  • Have database migration rollback scripts ready
  • Document rollback procedures for each service

12. Success Criteria

✅ Migration Complete When:

  • All services extracted and deployed independently
  • All functionality working as before
  • Performance equal or better than monolith
  • Monitoring and observability in place
  • Documentation complete
  • Team trained on new architecture
  • Zero critical bugs in production
