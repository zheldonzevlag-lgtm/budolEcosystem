3) Process & Steps

  1. Added service layer modules (placeholders for domains; real implementation for SystemSettings).
  2. Refactored system settings API routes to call service functions instead of direct DB calls.
  3. Built Phase 1 test suite:
    • DB connectivity check (PostgreSQL/MySQL compatible).
    • SystemSettings CRUD (create if missing, update, verify persistence, restore defaults).
    • Service file presence check.
    • Domain boundary sanity for SystemSettings service.
    • Optional API route smoke tests (skipped if dev server is not running).

4) Issues Encountered & Fixes

Module format mismatch (ESM vs CJS)
Issue: Node threw “Unexpected token export” when running tests.
Fix: Moved test runner to ESM (.mjs) and adjusted imports; avoided service imports in tests by using direct Prisma for SystemSettings checks.
Database() function not available on PostgreSQL
Issue: SELECT DATABASE() fails on Postgres.
Fix: Added fallback to current_database() and a generic connectivity check.
False-positive boundary check
Issue: Initial boundary test flagged SystemSettings service wrongly.
Fix: Updated check to verify it imports settings module and not other models.
API route tests dependency
Note: API smoke tests require npm run dev. They’re skipped gracefully if the dev server isn’t running.

8) How to Re-run Phase 1 Tests

  1. Ensure local DB is running and .env has DATABASE_URL.
  2. Run migrations if needed: npm run prisma:migrate.
  3. Optional: start dev server for API smoke tests: npm run dev.
  4. Run tests: npm run test:phase1.
