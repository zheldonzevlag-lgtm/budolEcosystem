4. Architecture Changes

4.1 Before Phase 3

Public API Route → Direct Payment Service Call → PayMongo Adapter
                ↓
           Database (Order Update)

4.2 After Phase 3

Public API Route → Payment Service Layer → PayMongo Adapter
                ↓
           Database (Order Update)

Internal API Route → Payment Service Layer → PayMongo Adapter
                ↓
           Database (Order Update)

Service A → HTTP → Internal Payment API → Payment Service → PayMongo
         (via serviceClient)
💡 Key Insight:
Payment operations are now abstracted through a service layer, making it easy to:
  • Call payment operations via HTTP (internal API)
  • Test payment logic independently
  • Extract payment service to separate deployment in future
  • Maintain consistent payment handling across the application

6. Implementation Details

6.1 Payment Service Implementation

Payment Initiation Flow:

1. Validate input (amount, method)
2. Normalize payment method (MAYA → paymaya, etc.)
3. Call paymentServiceInstance.initiatePayment()
4. Link payment intent to order (if orderId provided)
5. Return checkout URL and payment intent ID

6.2 Order Linking

Critical for Webhook Processing:

// Link payment intent to order
await linkPaymentToOrder(orderId, paymentIntentId);

// Updates order with:
// - paymentId: paymentIntentId
// - paymentStatus: 'awaiting_payment'

// This allows webhooks to find orders by paymentIntentId

6.3 Health Check Implementation

Payment Service Health:

export async function GET() {
    const checks = {
        paymentService: 'available',
        paymongo: process.env.PAYMONGO_SECRET_KEY ? 'configured' : 'missing_key'
    };
    
    return {
        status: checks.paymongo === 'configured' ? 'healthy' : 'degraded',
        service: 'payment',
        checks
    };
}

10. Payment Flow

10.1 Complete Payment Flow

  1. User initiates checkout → Frontend calls POST /api/payment/checkout
  2. Payment Service → Creates payment intent via PayMongo
  3. Order Linking → Links payment intent ID to order
  4. Redirect → User redirected to PayMongo checkout URL
  5. Payment Processing → User completes payment on PayMongo
  6. Webhook → PayMongo sends webhook to /api/webhooks/paymongo
  7. Order Update → Webhook handler updates order status
  8. Escrow Credit → Seller's pending balance credited
  9. Real-time Update → Order status updated via Pusher/Socket.io

10.2 Webhook Processing

// Webhook handler (still in main app)
POST /api/webhooks/paymongo

1. Verify webhook signature (via paymentService.verifyWebhookSignature)
2. Parse event type (payment_intent.succeeded, payment.paid, etc.)
3. Find order by paymentIntentId
4. Update order status
5. Credit seller's escrow balance
6. Send real-time update
7. Send email notification
