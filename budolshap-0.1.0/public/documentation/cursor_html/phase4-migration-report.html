4. Architecture Changes

4.1 Before Phase 4

Public API Route → Direct Prisma Calls → Database
                ↓
           Email & Realtime Events

4.2 After Phase 4

Public API Route → Orders Service Layer → Prisma → Database
                ↓
           Email & Realtime Events

Internal API Route → Orders Service Layer → Prisma → Database
                ↓
           Email & Realtime Events

Payment Service → HTTP → Internal Orders API → Orders Service → Database
         (via serviceClient)
💡 Key Insight:
Order operations are now abstracted through a service layer, making it easy to:
  • Call order operations via HTTP (internal API)
  • Test order logic independently
  • Extract orders service to separate deployment in future
  • Maintain consistent order handling across the application
  • Integrate with payment service via HTTP

6. Implementation Details

6.1 Order Creation Flow

Complete Order Creation Process:

1. Validate input (userId, addressId, orderItems, paymentMethod)
2. Verify address belongs to user
3. Fetch product details
4. Validate products exist and are in stock
5. Group items by store
6. For each store:
   - Calculate subtotal
   - Calculate shipping cost
   - Apply coupon discount
   - Create order in transaction
7. Clear cart (for synchronous payments only)
8. Send emails (for non-async payments)
9. Trigger real-time events
10. Return created orders

9. Code Examples

9.1 Using Orders Service

import { getOrders, createOrder, updateOrderStatus } from '@/lib/services/ordersService';

// Get orders
const result = await getOrders({
    userId: 'user123',
    storeId: 'store456',
    status: 'PAID',
    page: 1,
    limit: 20
});

// Create order
const orders = await createOrder({
    userId: 'user123',
    addressId: 'addr789',
    orderItems: [
        { productId: 'prod1', quantity: 2 },
        { productId: 'prod2', quantity: 1 }
    ],
    paymentMethod: 'GCASH',
    shipping: { provider: 'lalamove', cost: 50 }
});

// Update order status
const updated = await updateOrderStatus('order123', {
    status: 'SHIPPED',
    isPaid: true
});

9.2 Calling Internal Orders API

// Get orders via internal API

    })

// Find order by payment intent

10. Order Flow

10.1 Complete Order Flow

  1. User initiates checkout → Frontend calls POST /api/orders
  2. Orders Service → Validates products, calculates totals, creates orders
  3. Payment Service → Creates payment intent
  4. Order Linking → Links payment intent ID to order
  5. Payment Processing → User completes payment
  6. Webhook → Payment service receives webhook
  7. Order Update → Orders service updates order status
  8. Escrow Credit → Seller's pending balance credited
  9. Real-time Update → Order status updated via Pusher/Socket.io

10.2 Service-to-Service Communication

Payment Service → HTTP → Orders Service
  (webhook handler)      (findOrderByPaymentIntent)
                        ↓
                   (updateOrderStatus)
                        ↓
                   Database Update
