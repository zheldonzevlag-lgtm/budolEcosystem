1. Why SWR Works Great with Microservices

1.1 SWR Doesn't Care About Backend Architecture

SWR is a client-side data fetching library. It makes HTTP requests to any URL and handles caching/revalidation. Whether your backend is:

  • ✅ A monolith (/api/orders)
  • ✅ Microservices (https://orders.budolshap.com/api/orders)
  • ✅ Serverless functions
  • ✅ External APIs

SWR works the same way! You just change the URL.

1.2 Key Features That Benefit Microservices

🔄 Request Deduplication

If multiple components call the same endpoint simultaneously, SWR makes only one request and shares the result. This is crucial when calling multiple microservices. 

// Component A and Component B both call this
useSWR('/api/orders?userId=123', fetcher)
// SWR makes only ONE request, both components get the data

⚡ Parallel Fetching

SWR can fetch from multiple services simultaneously without blocking. Perfect for microservices where you need data from multiple sources. 

// Fetch from multiple services at once
const { data: orders } = useSWR('/api/orders', fetcher)
const { data: products } = useSWR('/api/products', fetcher)
const { data: user } = useSWR('/api/users/me', fetcher)
// All three requests happen in parallel!

💾 Client-Side Caching

SWR caches responses automatically, reducing redundant calls to your microservices. This improves performance and reduces server load.

🔄 Automatic Revalidation

SWR can automatically refetch data when:

  • Window regains focus
  • Network reconnects
  • At specified intervals (polling)
  • Manually via mutate()

This works seamlessly with your realtime updates (Pusher/Socket.io).

3. How to Use SWR with Microservices

3.1 Pattern 1: Direct Service Calls (Recommended)

Call microservices directly from the frontend. This works great with SWR:

Example: Fetching from Multiple Services

// lib/api/client.js - Create a fetcher that handles auth
const fetcher = async (url) => {
  const response = await fetch(url, {
    headers: {
      'Authorization': `Bearer ${getToken()}`,
      'Content-Type': 'application/json'
    }
  });
  
  if (!response.ok) {
    throw new Error('Failed to fetch');
  }
  
  return response.json();
};

// In your component
function OrderDetails({ orderId }) {
  // Fetch from Order Service
  const { data: order } = useSWR(
    `https://orders.budolshap.com/api/orders/${orderId}`,
    fetcher
  );
  
  // Fetch from Payment Service (if order has paymentId)
  const { data: payment } = useSWR(
    order?.paymentId 
      ? `https://payment.budolshap.com/api/payments/${order.paymentId}`
      : null,
    fetcher
  );
  
  // Fetch from Shipping Service (if order has shippingId)
  const { data: shipping } = useSWR(
    order?.shipping?.bookingId
      ? `https://shipping.budolshap.com/api/shipping/track/${order.shipping.bookingId}`
      : null,
    fetcher
  );
  
  // All three requests happen in parallel!
  // SWR handles caching, deduplication, and error states
  
  return (
    <div>
      <h1>Order {order?.id}</h1>
      <p>Status: {order?.status}</p>
      <p>Payment: {payment?.status}</p>
      <p>Shipping: {shipping?.status}</p>
    </div>
  );
}

3.2 Pattern 2: API Gateway / BFF (Also Works Great)

Use your Next.js app as an API Gateway that aggregates data from multiple services:

Example: BFF Aggregation Endpoint

// app/api/orders/[orderId]/details/route.js
// This endpoint calls multiple microservices and aggregates data
export async function GET(request, { params }) {
  const { orderId } = await params;
  
  // Call multiple services in parallel
  const [order, payment, shipping] = await Promise.all([
    fetch(`https://orders.budolshap.com/api/orders/${orderId}`),
    fetch(`https://payment.budolshap.com/api/payments/by-order/${orderId}`),
    fetch(`https://shipping.budolshap.com/api/shipping/by-order/${orderId}`)
  ]);
  
  return NextResponse.json({
    order: await order.json(),
    payment: await payment.json(),
    shipping: await shipping.json()
  });
}

// Frontend - Single SWR call to BFF
function OrderDetails({ orderId }) {
  const { data } = useSWR(`/api/orders/${orderId}/details`, fetcher);
  
  // All data comes from one endpoint
  return (
    <div>
      <h1>Order {data?.order?.id}</h1>
      <p>Payment: {data?.payment?.status}</p>
      <p>Shipping: {data?.shipping?.status}</p>
    </div>
  );
}
💡 Recommendation:
Pattern 1 (Direct Calls) is better for:
  • Independent data that doesn't need aggregation
  • Better caching (each service cached separately)
  • Faster updates (only refetch what changed)
Pattern 2 (BFF) is better for:
  • Complex aggregations
  • Reducing client-side complexity
  • When you need to transform/combine data

4. Best Practices for SWR + Microservices

4.1 Create a Centralized Fetcher

// lib/api/fetcher.js
import { getToken } from '@/lib/auth-client';

// Base URL configuration
const SERVICE_URLS = {
  orders: process.env.NEXT_PUBLIC_ORDERS_SERVICE_URL || '/api/orders',
  payment: process.env.NEXT_PUBLIC_PAYMENT_SERVICE_URL || '/api/payment',
  shipping: process.env.NEXT_PUBLIC_SHIPPING_SERVICE_URL || '/api/shipping',
  catalog: process.env.NEXT_PUBLIC_CATALOG_SERVICE_URL || '/api/stores',
  auth: process.env.NEXT_PUBLIC_AUTH_SERVICE_URL || '/api/auth',
};

// Smart fetcher that handles auth and errors
export const apiFetcher = async (url) => {
  const token = getToken();
  
  const response = await fetch(url, {
    headers: {
      'Authorization': token ? `Bearer ${token}` : '',
      'Content-Type': 'application/json',
    },
    credentials: 'include', // For cookies if using session-based auth
  });
  
  if (!response.ok) {
    const error = new Error('An error occurred while fetching the data.');
    error.status = response.status;
    error.info = await response.json().catch(() => ({}));
    throw error;
  }
  
  return response.json();
};

// Helper to build service URLs
export const buildServiceUrl = (service, path) => {
  const baseUrl = SERVICE_URLS[service];
  if (!baseUrl) throw new Error(`Unknown service: ${service}`);
  
  // Handle both relative and absolute URLs
  if (baseUrl.startsWith('http')) {
    return `${baseUrl}${path}`;
  }
  return `${baseUrl}${path}`;
};

4.3 Handle Loading and Error States

function OrderList() {
  const { data, error, isLoading } = useSWR(
    buildServiceUrl('orders', '/orders'),
    apiFetcher,
    {
      onErrorRetry: (error, key, config, revalidate, { retryCount }) => {
        // Don't retry on 4xx errors
        if (error.status === 404 || error.status === 403) return;
        
        // Retry up to 3 times
        if (retryCount >= 3) return;
        
        // Retry after 5 seconds
        setTimeout(() => revalidate({ retryCount }), 5000);
      },
    }
  );
  
  if (isLoading) return <Loading />;
  if (error) return <Error message={error.message} />;
  
  return <OrderListComponent orders={data} />;
}

4.4 Combine with Realtime Updates

Your current pattern of using Pusher/Socket.io with SWR's mutate() works perfectly: 

// When realtime event arrives, trigger SWR revalidation
useEffect(() => {
  if (!pusher) return;
  
  const channel = pusher.subscribe(`store-${storeId}`);
  channel.bind('order-updated', (data) => {
    // Trigger SWR to refetch
    mutate(buildServiceUrl('orders', `/orders?storeId=${storeId}`));
  });
  
  return () => {
    channel.unbind_all();
    pusher.disconnect();
  };
}, [pusher, storeId, mutate]);

5. Migration Strategy: Monolith → Microservices

5.1 Phase 1: No Changes Needed!

Your current SWR code works as-is. Start extracting services, and SWR will automatically work with the new endpoints.

5.2 Phase 2: Update URLs Gradually

As you extract services, update environment variables. SWR code doesn't need to change:

Phase URL Pattern SWR Code
Current (Monolith) /api/orders useSWR('/api/orders', fetcher)
After Extraction https://orders.budolshap.com/api/orders useSWR(ordersServiceUrl, fetcher)
(Same code, just different URL variable)

5.3 Phase 3: Use Helper Functions

// Before (hardcoded)
const { data } = useSWR('/api/orders', fetcher);

// After (service-aware)
const { data } = useSWR(
  buildServiceUrl('orders', '/orders'),
  apiFetcher
);

7. Common Patterns & Examples

7.1 Fetching Dependent Data

function OrderDetails({ orderId }) {
  // First, fetch order
  const { data: order } = useSWR(
    buildServiceUrl('orders', `/orders/${orderId}`),
    apiFetcher
  );
  
  // Then fetch payment (only if order exists and has paymentId)
  const { data: payment } = useSWR(
    order?.paymentId 
      ? buildServiceUrl('payment', `/payments/${order.paymentId}`)
      : null,  // null = don't fetch
    apiFetcher
  );
  
  // SWR handles the dependency chain automatically!
  
  return (
    <div>
      {order && <OrderInfo order={order} />}
      {payment && <PaymentInfo payment={payment} />}
    </div>
  );
}

7.2 Optimistic Updates

function OrderStatus({ orderId }) {
  const { data, mutate } = useSWR(
    buildServiceUrl('orders', `/orders/${orderId}`),
    apiFetcher
  );
  
  const updateStatus = async (newStatus) => {
    // Optimistically update UI
    mutate({ ...data, status: newStatus }, false);
    
    // Then update server
    await fetch(buildServiceUrl('orders', `/orders/${orderId}/status`), {
      method: 'PUT',
      body: JSON.stringify({ status: newStatus }),
    });
    
    // Revalidate to get server state
    mutate();
  };
  
  return (
    <button onClick={() => updateStatus('COMPLETED')}>
      Mark as Completed
    </button>
  );
}

7.3 Pagination

function OrderList() {

    </div>
  );
}
