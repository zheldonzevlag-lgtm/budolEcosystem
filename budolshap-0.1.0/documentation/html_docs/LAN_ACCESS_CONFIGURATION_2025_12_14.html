1. Overview

This document details the configuration changes made to enable LAN (Local Area Network) access for the Budolshap v3 application. This allows the application to be accessed from other devices on the same network, such as mobile phones, tablets, and other computers.

📌 Purpose: Enable testing and development of the Budolshap application across multiple devices on the local network without deploying to a remote server.

Benefits

  • Mobile Testing: Test the application on actual mobile devices
  • Cross-Device Testing: Verify responsive design across different screen sizes
  • Team Collaboration: Allow team members to access the development server
  • Client Demos: Showcase features to clients on the same network
  • Real Device Testing: Test features like camera, geolocation, and touch interactions

3. Configuration Changes

Modified File: package.json

Before:

"scripts": { "dev": "next dev --turbopack", "build": "prisma generate && next build", "start": "next start", // ... other scripts }

After:

"scripts": { "dev": "next dev --turbopack -H 0.0.0.0", "build": "prisma generate && next build", "start": "next start -H 0.0.0.0", // ... other scripts }

What Changed?

Added Flag: -H 0.0.0.0

Purpose: Binds the Next.js server to all network interfaces instead of just localhost (127.0.0.1)

Scripts Modified:

  • dev - Development server
  • start - Production server

Technical Explanation

Binding Address Access Level Use Case
127.0.0.1 (localhost) Local machine only Default Next.js behavior - secure but limited
0.0.0.0 (all interfaces) Local machine + LAN devices Development testing across devices
192.168.1.6 (specific IP) Specific interface only Advanced configuration (not used here)

4. Setup Guide

Quick Start

  1. Start the Development Server
    npm run dev

    The server will start and bind to all network interfaces (0.0.0.0:3000)

  2. Verify Server is Running

    You should see output similar to:

    ▲ Next.js 16.0.7 - Local: http://localhost:3000 - Network: http://192.168.1.6:3000 ✓ Ready in 2.5s
  3. Connect Device to Same Network

    Ensure your mobile device, tablet, or other computer is connected to the same Wi-Fi network (192.168.1.x)

  4. Access from Other Device

    Open a browser on the device and navigate to:

    http://192.168.1.6:3000
  5. Test the Application

    You should now see the Budolshap application running on your device!

✅ Success Indicator: If you can see the application on your mobile device or other computer, the LAN access is working correctly!

5. Firewall Configuration

⚠️ Important: Windows Firewall may block incoming connections on port 3000. You need to allow this port for LAN access to work.

Method 1: Allow Through Windows Defender Firewall (Recommended)

  1. Open Windows Defender Firewall

    Press Win + R, type wf.msc, and press Enter

  2. Create Inbound Rule
    • Click "Inbound Rules" in the left panel
    • Click "New Rule..." in the right panel
  3. Configure Rule Type
    • Select "Port"
    • Click "Next"
  4. Specify Port
    • Select "TCP"
    • Select "Specific local ports"
    • Enter: 3000
    • Click "Next"
  5. Allow Connection
    • Select "Allow the connection"
    • Click "Next"
  6. Select Profiles
    • Check "Private" (for home/work networks)
    • Optionally check "Domain" if on a domain network
    • ⚠️ Avoid checking "Public" for security
    • Click "Next"
  7. Name the Rule
    • Name: Next.js Development Server (Port 3000)
    • Description: Allow LAN access to Budolshap development server
    • Click "Finish"

Method 2: Quick Test (Temporary)

⚠️ Not Recommended for Production: This method temporarily disables the firewall. Use only for quick testing.
  1. Open Windows Defender Firewall settings
  2. Click "Turn Windows Defender Firewall on or off"
  3. Temporarily turn off for Private networks
  4. Test LAN access
  5. Remember to turn it back on!

Method 3: Using PowerShell (Advanced)

# Run PowerShell as Administrator New-NetFirewallRule -DisplayName "Next.js Dev Server" ` -Direction Inbound ` -LocalPort 3000 ` -Protocol TCP ` -Action Allow ` -Profile Private

7. Troubleshooting

Common Issues and Solutions

❌ Issue 1: Cannot Connect from Mobile Device

Symptoms: Browser shows "Can't reach this page" or times out

Solutions:
  • Check Network: Ensure both devices are on the same Wi-Fi network (192.168.1.x)
  • Verify IP: Run ipconfig to confirm the host IP hasn't changed
  • Check Firewall: Ensure port 3000 is allowed through Windows Firewall
  • Restart Server: Stop and restart the dev server with npm run dev

❌ Issue 2: Connection Refused

Symptoms: "Connection refused" error

Solutions:
  • Verify Server: Check that the dev server is actually running
  • Check Binding: Ensure the server output shows "Network: http://192.168.1.6:3000"
  • Port Conflict: Make sure no other application is using port 3000

❌ Issue 3: IP Address Changed

Symptoms: Previously working URL no longer works

Solutions:
  • Check Current IP: Run ipconfig to get the current IP
  • DHCP: Your router may have assigned a new IP address
  • Static IP: Consider setting a static IP in your router settings

❌ Issue 4: Slow Performance on Mobile

Symptoms: App loads slowly or is laggy on mobile devices

Solutions:
  • Wi-Fi Signal: Ensure strong Wi-Fi signal on mobile device
  • Network Congestion: Check if other devices are using bandwidth
  • Development Mode: Remember that dev mode is slower than production
  • Hot Reload: Disable hot reload if not needed for testing

Diagnostic Commands

# Check your current IP address ipconfig # Test if port 3000 is listening netstat -an | findstr :3000 # Ping your own IP (should work) ping 192.168.1.6 # Check firewall rules netsh advfirewall firewall show rule name=all | findstr 3000

Testing from Command Line

# From another device on the network, test connectivity: # On Windows: Test-NetConnection -ComputerName 192.168.1.6 -Port 3000 # On Mac/Linux: nc -zv 192.168.1.6 3000 # Or use curl: curl http://192.168.1.6:3000

8. Security Considerations

⚠️ Important Security Notice: Binding to 0.0.0.0 makes your development server accessible to anyone on your local network. Follow these security best practices.

Security Best Practices

1. Network Security

  • Trusted Networks Only: Only use this configuration on trusted private networks (home/office)
  • Avoid Public Wi-Fi: Never run the dev server on public Wi-Fi networks
  • Password Protection: Ensure your Wi-Fi network has a strong password
  • Guest Network: Don't use this on guest Wi-Fi networks

2. Firewall Configuration

  • Private Profile Only: Only allow port 3000 on the "Private" firewall profile
  • Specific Port: Only open port 3000, not a range of ports
  • Remove When Done: Delete the firewall rule when not actively developing

3. Development vs Production

Environment Binding Security Level Use Case
Development (Local Only) 127.0.0.1 🔒 High Solo development
Development (LAN) 0.0.0.0 ⚠️ Medium Testing on devices, team collaboration
Production Reverse proxy (nginx/Apache) 🔒 High Public deployment

4. Sensitive Data

  • Environment Variables: Ensure .env files are not exposed
  • API Keys: Use development/test API keys, not production keys
  • Database: Use a local development database, not production data
  • Authentication: Test with dummy accounts, not real user credentials

5. When to Disable LAN Access

Revert to localhost-only when:
  • You're done testing on other devices
  • Working on a public/untrusted network
  • Handling sensitive production data
  • Not actively needing LAN access

To revert: Remove the -H 0.0.0.0 flag from your npm scripts

Security Checklist

✅ Before Enabling LAN Access:

  • ☐ Verify you're on a trusted private network
  • ☐ Confirm Wi-Fi network has strong password
  • ☐ Using development database (not production)
  • ☐ Using test API keys (not production)
  • ☐ Firewall rule is set to "Private" profile only
  • ☐ Team members are aware of the access URL

Alternative Methods

Using ngrok (For Remote Access)

If you need to share your development server with someone not on your local network, consider using ngrok:

# Install ngrok npm install -g ngrok # Start your dev server npm run dev # In another terminal, create a tunnel ngrok http 3000
⚠️ Security Warning: ngrok exposes your local server to the internet. Only use for temporary sharing and never with production data.

Using localtunnel

# Install localtunnel npm install -g localtunnel # Start your dev server npm run dev # Create a tunnel lt --port 3000
