Phase 2: Status Mapping & Logic

Detailed mapping between Lalamove event states and Budolshap order statuses.

Core Status Mapping Table

The following table defines how Lalamove statuses are translated into Budolshap order statuses to ensure UI consistency.

Lalamove Event / Status Budolshap Order Status Action / Description
ASSIGNING_DRIVER PROCESSING Initial state after booking creation.
DRIVER_ASSIGNED PROCESSING Driver info captured; shipping status set to ON_GOING.
PICKED_UP IN_TRANSIT Package is with the driver; shippedAt timestamp set.
COMPLETED DELIVERED Successfully delivered; deliveredAt set; Auto-complete timer starts (7 days).
CANCELED / EXPIRED PROCESSING Booking cleared; Order reverts to PROCESSING for re-booking.
REJECTED PROCESSING Driver rejected; booking archived; ready for new attempt.

State Machine Logic

The implementation uses a robust state machine in both the webhook and sync routes to prevent invalid transitions (e.g., an order cannot move from DELIVERED back to IN_TRANSIT).

  • Validation: Current order status is checked before every update.
  • Redundancy: Handles both eventType and data.order.status for maximum compatibility with Lalamove Webhook versions.
  • Payment Integration: For COD and BudolPay orders, the order is automatically marked as paid when the Lalamove status reaches COMPLETED.

Failure Recovery

When a Lalamove booking fails (Canceled, Expired, Rejected), the system:

  1. Archives the failed booking info into shipping.previousAttempts.
  2. Clears the active bookingId and driverInfo.
  3. Resets shipping status to PENDING.
  4. Notifies both the Seller and Buyer via email about the failed delivery attempt.
