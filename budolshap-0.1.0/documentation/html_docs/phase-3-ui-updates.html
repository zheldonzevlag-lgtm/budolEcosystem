Executive Summary

This phase addressed UI inconsistencies by updating the payment return page to explicitly display subtotal, shipping fee, and total amount. Modifications ensure the shipping fee is shown correctly using order.shippingCost || 0, maintaining consistency across success and failed payment sections.

Technical Details

Updated app/payment/return/page.jsx to include new div elements for displaying breakdown. Used conditional rendering and currency formatting.

Key code snippet:

{order && (
  <div className="bg-slate-50 rounded-lg p-4 mb-6 text-left">
    <div className="flex justify-between mb-2">
      <span className="text-slate-600">Subtotal:</span>
      <span className="font-medium text-slate-800">
        ₱{(order.total - (order.shippingCost || 0)).toLocaleString('en-PH', { minimumFractionDigits: 2 })}
      </span>
    </div>
    <div className="flex justify-between mb-2">
      <span className="text-slate-600">Shipping Fee:</span>
      <span className="font-medium text-slate-800">
        ₱{(order.shippingCost || 0).toLocaleString('en-PH', { minimumFractionDigits: 2 })}
      </span>
    </div>
  </div>
)}

Risks and Mitigation

  • Risk: UI breakage on different order states. Mitigation: Use fallback to 0.
  • Risk: Formatting issues. Mitigation: Use localeString with en-PH.

Testing Protocol

Manual UI testing on payment pages, verify display for various shipping costs. Success criteria: Correct fee shown without discrepancies.
