Security Implementation - Phase 3

COMPLETED

Focus: Database-Backed Rate Limiting for Auth Endpoints

Implementation Summary

We have implemented a robust, database-backed rate limiting system to protect against brute-force attacks and abuse of the authentication endpoints.

Architecture:

  • Storage: A new `RateLimit` table in Prisma stores hit counts per key.
  • Configuration: Limits are now fully configurable via the Admin > Settings page.
  • Key Strategy: Limits are tracked by a combination of IP Address and Email to prevent both targeted account attacks and generic spam.
  • Persistence: Unlike in-memory limiters, this solution works across serverless function invocations (Vercel).

Applied Configurations

🔒 Login Endpoint
limit: 10 attempts (Default, Adjustable)
window: 15 minutes
Prevents password guessing attacks.
📝 Registration Endpoint
limit: 5 attempts (Default, Adjustable)
window: 60 minutes
Prevents spam account creation bots.

Behavior

If a user exceeds the limit, the API returns a 429 Too Many Requests error with a Retry-After header indicating when they can try again.

Verification & Testing

Manual Verification Steps:

  1. Attempt to log in with an incorrect password 11 times in rapid succession.
  2. Expected Result: The 11th attempt should fail immediately with the message "Too many login attempts. Please try again later."
  3. Wait for 15 minutes (or manually clear the `RateLimit` table).
  4. Expected Result: Login should work again.
