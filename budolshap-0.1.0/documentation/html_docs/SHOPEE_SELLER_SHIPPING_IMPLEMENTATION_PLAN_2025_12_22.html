Executive Summary

  • Implement a seller-first “To Ship → To Process → Arrange Shipment” workflow that matches Shopee’s steps: review order, confirm buyer contact/address, choose shipment model (pickup/drop-off), produce a waybill/label + packing list, hand over package, and track logistics status.
  • Reuse existing budolShap shipping boundaries: service-layer orchestration in lib/services/shippingService.js, provider integration via services/shippingFactory.js, and seller UI entrypoints in app/store/orders/page.jsx.
  • Add missing capabilities: seller shipment “handover deadlines”, label/packing-list generation, guardrails against duplicate waybill/tracking numbers, and drop-off vs pickup guidance surfaced in UI.
  • Deliver incrementally behind a feature flag so the current booking flow continues working during rollout.

Shopee Shipping Flow (Source Requirements)

The BigSeller guide describes a seller flow in Shopee Seller Centre:

  • Navigate to My Shipment → To Ship → To Process, open the order, confirm item details + buyer phone + shipping address, and optionally contact the buyer via chat.
  • Click Arrange Shipment, then pick a shipment model: Pickup or Drop-off.
  • Prepare the parcel, stick the air waybill/label, then hand over via drop-off point or schedule pickup. The courier delivers to the buyer.
  • Track shipping status via the shipping page / logistics info; ship on time to avoid late shipment cancellation/penalties.
  • Operational constraints called out in the article snippet: avoid using the same air waybill across multiple parcels; pickup eligibility may require a minimum daily order threshold; sellers may print combined waybill + packing list and customize label information.

Source: How to Ship Shopee Orders – BigSeller (2024).

Current Codebase Baseline

Seller UX (Orders)

  • Seller order management lives in app/store/orders/page.jsx, including a “Needs Booking”/“Booked”/“In Transit” filter and booking actions (handleBookLalamove at app/store/orders/page.jsx:110 and handleManualBook at app/store/orders/page.jsx:265).
  • Existing flow already resembles Shopee’s “To Process → Arrange Shipment”, but it is framed as “booking” rather than “arrange shipment”.

Shipping Service Boundary

  • Quotes + bookings are orchestrated by lib/services/shippingService.js (getShippingQuote at lib/services/shippingService.js:22, bookShipping at lib/services/shippingService.js:131).
  • Lalamove booking and tracking update order state; manual shipping stores tracking number and sets order status to PROCESSING (lib/services/shippingService.js:158).
  • Shipping provider selection exists via services/shippingFactory.js but currently only returns Lalamove (services/shippingFactory.js:13).

Shipping Status Updates

  • Webhook-driven status transitions exist for Lalamove in app/api/webhooks/lalamove/route.js (state machine around app/api/webhooks/lalamove/route.js:183).
  • Seller can manually sync status in app/api/orders/[orderId]/sync/route.js (app/api/orders/[orderId]/sync/route.js:83).

Data Model

  • Order shipping is a flexible JSON column: Order.shipping Json? in prisma/schema.prisma:58. This allows adding “waybill”, “handover”, “pickup/dropoff” without DB migrations, but requires disciplined JSON contract management.

Phased Implementation Plan

Phase 0 — Define Seller Shipping SLA & Acceptance Criteria (0.5–1 day)
  • Define “ship-by” calculation policy (e.g., createdAt + N business days) and what happens on breach (auto-cancel, auto-alert, or disable booking).
  • Define shipment models and how they map to providers: e.g. PICKUP for Lalamove; DROPOFF for manual carriers; allow both per store if configured.
  • Define seller surface area: seller order list + order details modal in app/store/orders/page.jsx.
  • Exit gate: 1-page spec for SLA + shipping states + UI entry points.
Phase 1 — Formalize Shipping JSON Contract & Feature Flag (1–2 days)
  • Create a single shared “shipping schema” contract (JS object shape) for Order.shipping and standardize fields used across Lalamove + manual.
  • Add a feature flag (env-driven) to toggle the Shopee-aligned “Arrange Shipment” UI path.
  • Implement duplicate-waybill guardrails for manual shipping by rejecting a tracking number already attached to another active order.
  • Exit gate: manual booking still works; flag can enable/disable new UX; uniqueness check verified in dev.

Primary touchpoints: lib/services/shippingService.js:158, app/api/shipping/manual/book/route.js:10, prisma/schema.prisma:58.

Phase 2 — “Arrange Shipment” UX Alignment (2–4 days)
  • Rename/realign UI language: “Needs Booking” becomes “To Process” and “Book” becomes “Arrange Shipment”, keeping the underlying API calls.
  • Before shipment arrangement, show a “Verify buyer details” panel (address + phone + item list) sourced from order data and allow quick “Chat buyer” jump.
  • Add “Shipment model” selection (pickup vs drop-off) and show contextual instructions (pickup scheduling vs drop-off point guidance).
  • Exit gate: seller can reliably find orders that need action and complete arrangement with fewer support prompts.

Primary touchpoints: app/store/orders/page.jsx:18, app/store/orders/page.jsx:110, app/store/orders/page.jsx:265.

Phase 3 — Label/Waybill + Packing List Generation (3–6 days)
  • Add a server-side PDF generator for:
    • Packing list (items, quantities, order id, buyer name, store name).
    • Shipping label ("air waybill" equivalent) with recipient + sender blocks and optional QR/code.
    • Combined PDF bundle (packing list + label) to support “one print” workflows.
  • Persist generated artifacts in object storage (Cloudinary is present in dependencies; use existing upload patterns if available).
  • Add UI buttons: “Print Waybill”, “Print Packing List”, “Print Combined”.
  • Exit gate: seller can print documents; URLs are stable and access-controlled.
Phase 4 — Operational Controls: Deadlines, Reminders, Enforcement (2–4 days)
  • Compute and store shipByAt and handoverAt in Order.shipping when an order is created/paid.
  • Add reminders: realtime notification + email 24h before deadline and at deadline.
  • Add an enforcement policy job (cron route) to mark orders as “overdue shipment” (or auto-cancel per your business rule).
  • Exit gate: overdue cases are visible to seller/admin; measurable reduction in late handovers.
Phase 5 — Provider Expansion (Optional) (5–10 days)
  • Extend services/shippingFactory.js to support additional carriers (J&T, NinjaVan, 2GO), using a unified interface (getQuote, createOrder, trackOrder, cancelOrder).
  • For providers without API integration, keep them as “manual carriers” with normalized tracking links and timeline updates.
  • Exit gate: seller can choose multiple carriers without changing the order UX.

API endpoints to add

  • POST /api/shipping/documents: generate waybill/packing list PDFs for an order
  • POST /api/shipping/arrange: persist shipment model selection + mark as arranged
  • POST /api/orders/overdue-shipping/sweep (cron): enforce deadlines and emit alerts
Security and access control
  • Only the owning store/admin can generate or download waybill documents.
  • Document URLs should be signed or proxied via an authenticated download endpoint.
  • Never embed buyer phone/address in public, unauthenticated URLs.

Testing Strategy & Checkpoints

Unit/Service-level

  • Manual shipping uniqueness: attempting to reuse a tracking/waybill number fails with a clear error.
  • Shipping JSON contract: serialization/deserialization sanity checks for new fields.

Integration

  • Arrange shipment → book → webhook/sync updates status and timeline (Lalamove path).
  • Arrange shipment → manual carrier → seller can attach tracking URL and see status state in UI.
  • Document generation: PDFs are generated, stored, and retrievable only by authorized users.

Operational

  • Deadline reminders trigger at correct times for test orders.
  • Overdue sweep does not cancel orders incorrectly; actions are auditable.

Rollout, Monitoring, Rollback

Rollout strategy

  • Ship behind a feature flag, enable for internal admin + a single pilot store.
  • Gradually expand to 10%, 50%, 100% of stores once error rates and support tickets are stable.

Monitoring

  • Track: “orders in To Process”, “time-to-arrange”, “late shipment rate”, “booking failures”, and “duplicate waybill rejections”.
  • Ensure webhook processing reliability for status transitions (app/api/webhooks/lalamove/route.js).

Rollback

  • Disable feature flag to revert seller UI to current booking flow without data loss.
  • Leave generated documents in storage; keep links but hide UI actions when flag is off.
