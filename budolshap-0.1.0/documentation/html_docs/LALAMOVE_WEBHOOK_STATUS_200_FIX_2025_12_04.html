🚨 CRITICAL: Lalamove Webhook Fix - Status 200 Requirement

Date: December 4, 2025, 19:44 PHT

Status: ✅ FIXED AND DEPLOYED

⚠️ Root Cause Identified

The webhook was returning HTTP 401 status codes for signature validation errors, which caused Lalamove to reject the webhook registration and stop sending events.

Lalamove Requirement: Webhooks MUST return HTTP 200 status code for ALL requests, even errors.

🔍 Problem Analysis

What Was Wrong

Our webhook endpoint had two places returning non-200 status codes:

// ❌ WRONG - Returns 401
if (!signature) {
    return NextResponse.json(
        { error: 'Signature missing' },
        { status: 401 }  // ❌ This causes Lalamove to reject webhook
    );
}

// ❌ WRONG - Returns 401
if (!isValid) {
    return NextResponse.json(
        { error: 'Invalid signature' },
        { status: 401 }  // ❌ This causes Lalamove to reject webhook
    );
}

Why This Matters

  • Lalamove tests webhook URLs before accepting them
  • If they receive ANY non-200 status code, they mark the webhook as "non-200 webhook response" error
  • Once marked as error, Lalamove stops sending webhook events
  • This means no driver information is received, causing the app to show mock data

✅ Solution Implemented

// ✅ CORRECT - Always returns 200
if (!signature) {
    console.error('Webhook signature missing');
    // IMPORTANT: Lalamove requires 200 status code even for errors
    return NextResponse.json({
        received: true,
        error: 'Signature missing',
        message: 'Webhook acknowledged but signature was missing'
    });  // ✅ Returns 200 by default
}

// ✅ CORRECT - Always returns 200
if (!isValid) {
    console.error('Invalid webhook signature');
    // IMPORTANT: Lalamove requires 200 status code even for errors
    return NextResponse.json({
        received: true,
        error: 'Invalid signature',
        message: 'Webhook acknowledged but signature validation failed'
    });  // ✅ Returns 200 by default
}

📝 Technical Details

Why Driver Info Isn't in GET API

Lalamove's GET /v3/orders/{orderId} endpoint does NOT return driver information. It only returns:

  • driverId: Just a numeric ID
  • status: Order status
  • shareLink: Tracking URL

Driver details (name, phone, plate, vehicle type) are ONLY sent via webhooks.

This is why the webhook must work correctly - it's the ONLY way to get driver information.

Webhook Events

Event When Sent Driver Info Included?
ASSIGNING_DRIVER Looking for a driver ❌ No
ON_GOING Driver assigned and accepted ✅ Yes - This is when we get driver info!
PICKED_UP Driver picked up package ✅ Yes
IN_TRANSIT On the way to destination ✅ Yes
COMPLETED Delivered successfully ✅ Yes
CANCELLED Order cancelled Maybe
