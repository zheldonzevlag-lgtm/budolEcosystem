Phase 1: Analysis

Identification of schema-code discrepancies

Audit Results

After auditing the schema.prisma against the returnsService.js and Lalamove webhook handlers, several critical integrity issues were identified.

1. Missing Fields in Return Model

The resolveDispute function in returnsService.js attempts to update fields that do not exist in the Return model.

// From returnsService.js
await tx.return.update({
    where: { id: returnId },
    data: {
        status: newReturnStatus,
        adminId, // MISSING IN SCHEMA
        adminNotes, // MISSING IN SCHEMA
        isEscrowLocked: false,
        updatedAt: new Date()
    }
});

2. Missing Enum Value in ReturnStatus

The Lalamove webhook handler (v3) uses TO_PICKUP as a status for returns, but this value is missing from the ReturnStatus enum.

// From app/api/shipping/lalamove/webhook/route.js
} else if (!progressedStatuses.includes(activeReturn.status)) {
    returnStatusUpdate = 'TO_PICKUP'; // MISSING IN SCHEMA ENUM
}

3. Inconsistent Field Naming

The schema defines arbitrationNotes, but the code uses adminNotes. While arbitrationNotes exists, it remains unused in the current implementation of dispute resolution.

Recommendations

  • Add adminId (String?) and adminNotes (String?) to the Return model.
  • Add TO_PICKUP to the ReturnStatus enum.
  • Alternatively, sync the code to use arbitrationNotes instead of adminNotes, or keep both if needed for different purposes.
  • Verify if adminId should be a relation to the User model.
