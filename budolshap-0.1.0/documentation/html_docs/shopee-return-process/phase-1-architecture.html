Phase 1: Architecture & State Machine

This phase defines the core logic of the Shopee-style return process, focusing on state transitions and control ownership at each stage.

1. Control Ownership Summary

In a Shopee-style return, the platform (BudolShap) acts as the central orchestrator, while the courier handles physical logistics.

Stage Who Controls It Primary Action
Return Request Buyer + Platform Initial filing and fund locking (Escrow)
Approval / Rejection Seller Accepting return or proposing alternatives
Courier Selection Platform Lalamove (BudolShap default) assigned automatically
Pickup / Drop-off Courier Physical scan triggers status update
In-Transit Updates Courier Real-time webhook updates from Lalamove
Refund Release Platform Funds released from Escrow to Buyer

2. Return State Machine

The return process follows a strict state progression to ensure transparency and accountability.

PENDING (Buyer files) → APPROVED (Seller accepts) → BOOKING_REQUESTED (Buyer asks for pickup) → BOOKED (Seller/Platform confirms courier) → SHIPPED (Courier picks up) → RECEIVED (Seller gets item) → REFUNDED (Funds released)

3. Prisma Schema Implementation

The ReturnStatus enum in schema.prisma has been updated to reflect these states:

enum ReturnStatus {
  PENDING
  APPROVED
  BOOKING_REQUESTED
  BOOKED
  SHIPPED
  RECEIVED
  REFUNDED
  CANCELLED
  DISPUTED
}

4. Key Differences from Forward Delivery

  • Initiation: Buyer initiates the process, not the seller.
  • Escrow: Funds are locked immediately upon request and only released after verification.
  • SLA Rules: Platform enforces deadlines for seller response and buyer return shipment.
  • Courier Control: Platform (Shopee/BudolShap) decides the shipping provider to ensure tracking consistency.
