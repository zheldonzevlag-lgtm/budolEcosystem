Executive Summary

The Audit Trail feature is now fully operational. It tracks user logins and logouts with high-precision geolocation data. The system is designed to be robust across different hosting environments by using a multi-tiered fallback strategy for location detection.

Universal Geolocation Strategy

To ensure location data (City, Country, Latitude, Longitude) is always available, the system uses the following logic:

Tier 1: Hosting Headers (Fastest)

The system first checks for headers provided by the hosting platform (Vercel) to get instant location data without external API calls.

  • x-vercel-ip-city
  • x-vercel-ip-country
  • x-vercel-ip-latitude
  • x-vercel-ip-longitude

Tier 2: Universal Fallback (IP API)

If Tier 1 fails (e.g., hosting on AWS, DigitalOcean, or if headers are missing), the system automatically queries ip-api.com using the user's public IP address to retrieve the missing data.

if ((!city || !country || !latitude || !longitude) && isPublicIp) {
    // Fetch from ip-api.com

Privacy Note

The "Hostname" column was removed as browsers block access to the internal computer name (e.g., "JON-PC") for privacy reasons. The current implementation relies solely on public IP geolocation, which is the industry standard for security auditing.

Database Schema Final State

model AuditLog {
    id        String   @id @default(cuid())
    userId    String
    action    String   // "LOGIN" | "LOGOUT"
    ipAddress String?

    longitude String?  // Exact coord
