1. Database Schema Extensions

To support detailed comparative analysis, the Order model must be enhanced to track both initial quotes and final fulfillment costs.

model Order {
  // Existing fields...
  shippingCost         Float   @default(0) // Final cost charged to buyer
  quotedShippingCost   Float?  // Original estimate from Lalamove API
  actualLalamoveCharge Float?  // Final charge from Lalamove (after trip)
  
  // Absorption Tracking
  absorptionType       ShippingAbsorption @default(BUYER_PAYS)
  platformSubsidy      Float   @default(0)
  sellerSubsidy        Float   @default(0)
  
  // Metadata for Analysis
  shippingMetadata     Json?   // Stores route_id, surge_factor, vehicle_type
}

enum ShippingAbsorption {
  BUYER_PAYS
  SELLER_ABSORBS
  PLATFORM_ABSORBS
  HYBRID
}

2. Technical Components

Component Specification Tool/Library
Distance Calculation Road-routing distance via Haversine or Map API. Google Maps / Radar API
Pricing Engine Logic to compute Base + (KM * Rate) + Surge. Node.js Backend
Lalamove Webhook Capture ORDER_COMPLETED events for final billing. Next.js API Routes
Analytics Dashboard Compare quotedShippingCost vs actualLalamoveCharge. Prisma + Chart.js

3. Dependencies

  • Lalamove API: For real-time quotations and booking.
  • Map Provider: (Google/OSM) for accurate geocoding and pin placement.
  • Prisma: For database migrations and data access.
  • Redis: For caching shipping quotes to prevent redundant API calls during checkout.

Risk Assessment: Price Volatility

Risk: Lalamove quotes expire or change between cart view and checkout completion.

Mitigation: Implement a 5-minute cache for quotes. If the user stays on checkout longer, refresh the quote and notify the user of price changes.

Phase 2 Success Criteria

  • Database schema updated to track quoted vs. actual costs.
  • Clear definition of absorption enums and subsidy fields.
  • Mapping of Lalamove webhook payload to internal order fields.
