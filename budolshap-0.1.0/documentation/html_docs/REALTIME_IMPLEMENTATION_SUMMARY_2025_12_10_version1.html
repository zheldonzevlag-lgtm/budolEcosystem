Realtime Architecture & Implementation Guide

Date: Dece mber 10, 2025
Context: Transitioning BudolShap Order Management to a Realtime/Hybrid System.

1. Issues & Challenges Addressed

❌ The Friction Points

Prior to this update, the application relied on "Fetch Once" logic. This caused several issues:

  • Stale Data: Sellers had to manually refresh the page to see if a new order arrived.
  • Lalamove Disconnect: When a driver status changed (e.g., "Picked Up"), the UI remained static, causing confusion about delivery progress.
  • Infrastructure Cost: Implementing full WebSockets (Pusher) immediately would introduce monthly costs before the user base scales.

2. Implemented Solutions (The Hybrid Fix)

✅ The "Hybrid" Realtime Engine

We implemented a robust system that supports Two Modes, switchable at runtime without code changes.

Mode Mechanism Cost Latency
POLLING (Default) SWR Hook fetches data every 5 seconds. FREE ~3-5 Seconds
PUSHER (Pro) Server pushes events via WebSockets. PAID / LIMITED Instant (< 1s)

Critical Code Snippet (Hybrid Hook)

// hooks/useRealtimeOrders.js
export function useRealtimeOrders({ storeId }) {
    const { data: config } = useSWR('/api/system/realtime');
    
    // Mode Switch
    const isPolling = config?.provider === 'POLLING';
    
    // 1. Polling Logic
    const { data, mutate } = useSWR(url, fetcher, { 
        refreshInterval: isPolling ? 5000 : 0 
    });

    // 2. Pusher Logic
    useEffect(() => {
        if (config?.provider === 'PUSHER') {
            const channel = pusher.subscribe(`store-${storeId}`);
            channel.bind('order-updated', () => mutate());
        }
    }, [config]);

    return { orders: data?.orders, mutate };
}

5. Recommendations for "True Instant" Scale

Recommendation 1: Optimistic UI Updates

Goal: Zero-latency feel.

Instead of waiting for the server to reply "OK" and then re-fetching the list, the UI should immediately update the order status to "Processing" locally. If the server request fails, rollback the change. This makes the app feel instantly responsive regardless of network speed.

Recommendation 2: Dedicated WebSocket Microservice

Goal: Scale beyond Pusher limits.

If the app grows to >10,000 concurrent users, Pusher bills will rise. A dedicated Node.js server running Socket.io (hosted on EC2/DigitalOcean) can replace Pusher. This gives you unlimited connections for a fixed server cost.

Recommendation 3: Granular Event Payloads

Goal: Bandwidth efficiency.

Currently, the event tells the client to "Refetch All Orders". For maximum efficiency, the event payload should contain the actual data: { orderId: 101, status: 'DELIVERED', timestamp: ... }. The client can then update just that single row in the React state without a network call.
