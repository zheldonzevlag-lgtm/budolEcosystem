Session Management & Login Refactor Documentation

Date: December 13, 2025

Author: Antigravity (AI Assistant)

1. Executive Summary

This report documents the comprehensive refactoring of the authentication system, specifically focusing on session management, user interface consistency, and role-based security policies. The goal was to align the application's behavior with industry standards (similar to Shopee/Lazada) by implementing persistent sessions for buyers while maintaining strict security timeouts for high-privilege accounts (Admins and Sellers). Additionally, the login interface was unified across the application using a shared component.

2. Processes & Implementation

2.1 UI Refactoring: Shared AuthForm

The `LoginModal`, `StoreLoginPage`, and `AdminLoginPage` previously used disparate code for their forms. We refactored `AuthForm` to be a highly reusable component that handles both login and registration (where applicable), supports custom submit labels, and exposes an `onSuccess` callback for parent-specific logic.

2.2 Session Expiration Handling

We enhanced the `AuthContext` and `AuthUIContext` to handle session expiration gracefully. Instead of simple redirects or silent failures, the system now detects invalid tokens/sessions and triggers the Login Modal with a specific visual warning.

2.3 Idle Timeout Integration

For Admins and Sellers, we integrated the `useIdleTimeout` hook into their respective layouts (`AdminLayout`, `StoreLayout`). Upon 15 minutes of inactivity, the user is logged out and the "Not Authorized" view is triggered, overlaid immediately by the Login Modal.

2.4 Role-Based Session Durations

We updated the backend login API to issue JWTs with variable expiration times based on user roles:

  • Buyers: 30 Days (Persistent session for better UX)
  • Admins/Sellers: 24 Hours (Strict security + 15min idle timeout)

3. Issues & Fixes

Issue 1: Inconsistent Login UI

Problem: Store and Admin login pages had duplicated, divergent code from the main Login Modal.

Fixed Replaced all manual forms with the enhanced ` ` component. This ensures that any future changes to authentication logic only need to be made in one place.

Issue 2: Silent Logout on Expiry

Problem: When a session expired, users would simply find themselves unable to perform actions or redirected without explanation.

Fixed Implemented `loginMessage` in `AuthUIContext`. The Login Modal now displays a red warning banner: "Your session has expired. Please login again."

Issue 3: Generic Timeout for All Users

Problem: Buyers were subject to the same strict session limits as admins, causing frustration.

Fixed Modified `api/auth/login/route.js` to set `maxAge` dynamically. Buyer cookies now last 30 days, while Admin/Seller cookies expire in 24 hours.

Issue 4: Context Nesting Issue

Problem: `AuthProvider` could not use `AuthUIContext`'s `showLogin` method because it was a parent provider.

Fixed Swapped provider hierarchy in `app/layout.jsx`. `AuthUIProvider` now wraps `AuthProvider`.

4. Technical Details & Code

4.1 Dynamic JWT Generation

The login route now determines token lifespan based on user account type.

// app/api/auth/login/route.js

const isBuyer = !user.accountType || user.accountType === 'BUYER';
const tokenDuration = isBuyer ? '30d' : '24h'; 
const cookieMaxAge = isBuyer ? 60 * 60 * 24 * 30 : 60 * 60 * 24;

const token = generateToken({ ...payload }, tokenDuration);

response.cookies.set('token', token, {
    ...COOKIE_OPTIONS,
    maxAge: cookieMaxAge
});

4.2 Layout-Based Idle Timeout

High-security layouts enforce stricter inactivity rules.

// components/admin/AdminLayout.jsx

const { showLogin } = useAuthUI();

const handleLogout = useCallback(async () => {
    // 1. Perform server logout
    await fetch('/api/auth/logout', { method: 'POST' });
    
    // 2. Clear local admin state to trigger "Not Authorized" view
    setIsAdmin(false); 
    
    // 3. Show Re-login Modal
    setTimeout(() => {
        showLogin(null, 'You have been logged out due to inactivity.');
    }, 100);
}, [showLogin]);

5. Future Recommendations

  • Refresh Token Mechanism: Currently, we use long-lived access tokens for buyers. Implementing a Refresh Token rotation strategy (short-lived access token + long-lived httpOnly refresh cookie) would further improve security by allowing token revocation without waiting for expiration.
  • Force Logout Event: Implement a server-side "security stamp" check. If an admin changes a user's password or disables an account, the valid JWT should be invalidated immediately on the next request, not just when it expires.
  • "Remember Me" Checkbox: Give users explicit control over whether they want a persistent 30-day session or a transient session that ends when the browser closes.
