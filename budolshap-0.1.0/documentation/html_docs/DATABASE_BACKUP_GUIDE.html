📋 Overview

This guide covers two complementary backup methods for your Budolshap PostgreSQL database:

🔧 SQL Backup (pg_dump)

Creates complete database dumps using PostgreSQL's native tools. Perfect for full database restoration and migration.

Complete Production-Ready

📦 JSON Backup (Prisma)

Exports data as JSON files using Prisma. Ideal for data inspection, selective restoration, and development.

Data Only Human-Readable

⚙️ Prerequisites

🔧 Required Tools

Before running backups, ensure you have:

  • Node.js - For running Prisma-based JSON backups
  • PostgreSQL Client Tools - For SQL dumps (pg_dump, pg_restore)
  • PowerShell - For running backup scripts on Windows
  • Database Access - Valid credentials in your .env file

📥 Installing PostgreSQL Client Tools

If you don't have pg_dump installed:

  1. Download PostgreSQL

    Visit PostgreSQL Downloads and download the Windows installer.

  2. Install Client Tools

    During installation, ensure "Command Line Tools" is selected.

  3. Verify Installation
    pg_dump --version # Should output: pg_dump (PostgreSQL) 15.x

🚀 Quick Start

✨ One-Command Backup

Run the PowerShell script to create both SQL and JSON backups:

.\scripts\backup-postgres.ps1

This will create a timestamped backup directory containing:

  • 📄 database-backup.sql - Plain SQL dump
  • 📦 database-backup.dump - Compressed custom format
  • 📁 json-data/ - JSON files for each table
  • 📋 README.md - Backup summary and instructions

🎯 Backup Specific Format

# SQL backup only .\scripts\backup-postgres.ps1 -BackupType sql # JSON backup only .\scripts\backup-postgres.ps1 -BackupType json # Both formats (default) .\scripts\backup-postgres.ps1 -BackupType both

💾 SQL Backup (pg_dump)

The SQL backup method uses PostgreSQL's pg_dump utility to create complete database dumps.

📝 Manual SQL Backup

If you prefer to run pg_dump manually:

# Set your database password $env:PGPASSWORD = "your_password" # Create plain SQL dump pg_dump -h your_host -p 5432 -U your_user -d your_database ` -f backup.sql --no-owner --no-acl --clean --if-exists # Create compressed custom format dump pg_dump -h your_host -p 5432 -U your_user -d your_database ` -f backup.dump -F c --no-owner --no-acl # Clear password from environment Remove-Item Env:\PGPASSWORD

🔍 Understanding pg_dump Options

  • --no-owner - Don't output commands to set ownership
  • --no-acl - Don't output commands to set access privileges
  • --clean - Add DROP commands before CREATE
  • --if-exists - Use IF EXISTS when dropping objects
  • -F c - Custom format (compressed, for pg_restore)

🌐 Vercel Postgres Backup

For Vercel Postgres databases, the script automatically parses your connection string from .env:

# Your .env file should contain: POSTGRES_URL="postgres://user:password@host:5432/database" # or DATABASE_URL="postgres://user:password@host:5432/database"

📦 JSON Backup (Prisma)

The JSON backup method uses Prisma to export all data as human-readable JSON files.

🔧 Using npm Scripts

# Run JSON backup directly npm run db:backup # Or use the PowerShell script .\scripts\backup-postgres.ps1 -BackupType json

📊 What Gets Backed Up

The JSON backup includes all tables in your database:

👥 User Data

  • users.json
  • addresses.json
  • carts.json

🏪 Store Data

  • stores.json
  • products.json
  • wallets.json

🛒 Transaction Data

  • orders.json
  • ratings.json
  • returns.json

💬 Communication

  • chats.json
  • coupons.json
  • payout-requests.json

📋 Backup Metadata

Each backup includes a metadata.json file with:

{ "timestamp": "2025-11-24T08:00:00.000Z", "tables": { "users": 150, "stores": 45, "products": 320, "orders": 890, ... } }

♻️ Restore Instructions

⚠️ Important Warning

Always test restores in a non-production environment first! Restoring a backup will overwrite existing data.

🔄 Restoring from SQL Dump

Option 1: Plain SQL File

# Set password $env:PGPASSWORD = "your_password" # Restore from SQL file psql -h your_host -p 5432 -U your_user -d your_database -f database-backup.sql # Clear password Remove-Item Env:\PGPASSWORD

Option 2: Custom Format Dump

# Set password $env:PGPASSWORD = "your_password" # Restore from custom dump pg_restore -h your_host -p 5432 -U your_user -d your_database ` -F c --clean --if-exists database-backup.dump # Clear password Remove-Item Env:\PGPASSWORD

📥 Restoring from JSON Backup

# Use the restore script npm run db:restore # Or manually run the script node scripts/restore-database.js

🔍 JSON Restore Notes

  • Requires the database schema to already exist (run migrations first)
  • Best for selective data restoration
  • May need to handle foreign key constraints carefully
  • Useful for development and testing environments

🆕 Restoring to a New Database

  1. Create New Database
    CREATE DATABASE budolshap_restored;
  2. Restore from Backup
    pg_restore -h host -p 5432 -U user -d budolshap_restored ` -F c database-backup.dump
  3. Update Connection String

    Update your .env file to point to the new database.

  4. Verify Data

    Test the application to ensure all data was restored correctly.

⏰ Automation & Scheduling

🪟 Windows Task Scheduler

Set up automated daily backups using Windows Task Scheduler:

  1. Open Task Scheduler

    Press Win + R, type taskschd.msc, and press Enter.

  2. Create Basic Task

    Click "Create Basic Task" and name it "Budolshap Database Backup".

  3. Set Trigger

    Choose "Daily" and set your preferred time (e.g., 2:00 AM).

  4. Set Action

    Choose "Start a program" and configure:

    Program: powershell.exe Arguments: -ExecutionPolicy Bypass -File "C:\wamp64\www\budolshap\scripts\backup-postgres.ps1" Start in: C:\wamp64\www\budolshap
  5. Test the Task

    Right-click the task and select "Run" to test it.

📜 PowerShell Scheduled Job

# Create a scheduled job that runs daily at 2 AM $trigger = New-JobTrigger -Daily -At "2:00 AM" $scriptPath = "C:\wamp64\www\budolshap\scripts\backup-postgres.ps1" Register-ScheduledJob -Name "BudolshapBackup" ` -Trigger $trigger ` -FilePath $scriptPath ` -MaxResultCount 10 # View scheduled jobs Get-ScheduledJob # Remove scheduled job Unregister-ScheduledJob -Name "BudolshapBackup"

✅ Best Practices

🔒 Security

  • Never commit backups to version control
  • Encrypt sensitive backup data
  • Store backups in secure locations
  • Use environment variables for credentials
  • Regularly rotate backup encryption keys

📅 Frequency

  • Production: Daily automated backups
  • Critical data: Multiple daily backups
  • Before deployments: Always backup
  • Before migrations: Mandatory backup
  • Development: Weekly or as needed

💾 Storage

  • Keep multiple backup versions
  • Store backups off-site (cloud storage)
  • Implement backup retention policy
  • Monitor backup storage space
  • Use compression for large databases

🧪 Testing

  • Regularly test restore procedures
  • Verify backup integrity
  • Document restore processes
  • Time your restore operations
  • Practice disaster recovery scenarios

🔧 Troubleshooting

❌ pg_dump not found

Problem: PowerShell can't find the pg_dump command.

Solution:

  • Install PostgreSQL client tools
  • Add PostgreSQL bin directory to PATH
  • Typical path: C:\Program Files\PostgreSQL\15\bin
# Add to PATH temporarily $env:Path += ";C:\Program Files\PostgreSQL\15\bin" # Or add permanently via System Properties > Environment Variables

⚠️ Connection refused

Problem: Can't connect to the database.

Solution:

  • Verify database is running
  • Check connection string in .env
  • Ensure firewall allows connections
  • For Vercel Postgres, check IP whitelist

🔐 Authentication failed

Problem: Invalid credentials.

Solution:

  • Verify username and password in .env
  • Check for special characters in password
  • Ensure database user has necessary permissions
  • Try connecting manually with psql to test credentials

💥 Out of memory

Problem: Backup fails due to memory issues.

Solution:

  • Use custom format (-F c) instead of plain SQL
  • Backup tables individually for very large databases
  • Increase Node.js memory limit for JSON backups:
node --max-old-space-size=4096 scripts/backup-database.js

📁 Permission denied

Problem: Can't write to backup directory.

Solution:

  • Run PowerShell as Administrator
  • Check folder permissions
  • Ensure backup directory is not read-only
