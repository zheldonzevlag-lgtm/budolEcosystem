Lalamove Status Sync & Order UI Fix

Date: December 13, 2025

Version: 1.0 (Complete Guide)

Executive Summary: This document details the resolution of a critical mismatch between Lalamove delivery statuses and the platform's order display. It covers the strategy to fix the "Driver Assigned" confusion, the correction of Payment Status logic, and the addition of a new Order Status column.

1. Processes & Methodology

The resolution followed a systematic approach to preserve existing functionality while fixing the user experience:

  1. Diagnosis: Analyzed the webhook handler and database state to understand why cancelled orders were reverting to "Processing".
  2. Strategic Decision: Decided NOT to change the backend logic (splitting "Processing" into a new status) to avoid breaking the "Rebooking" flow. Instead, we implemented a UI-First Interpretation Layer.
  3. Implementation: Updated key React components to check for provider-specific failure flags details (shipping.failureReason).
  4. Verification: Simulated Lalamove webhooks to confirm the UI reacts correctly without requiring live orders.

2. Issues Identification

A. The "Processing" Trap

When a Lalamove driver cancels an order, the system's webhook handler (app/api/webhooks/lalamove/route.js) sets the order status back to PROCESSING. This was intended to allow sellers to try booking again. However, the UI blindly displayed "Processing" as "Driver Assigned", misleading both buyers and sellers.

B. Payment Status Mislabeling

The "Payment Status" column in the order list was simply mirroring the main status (e.g., displaying "Processing" instead of "Paid" or "Unpaid").

3. The Fixes (Implementation Details)

Fix 1: Seller Dashboard Sync

Goal: Ensure failed deliveries appear in the "Cancelled" tab.

  • Modified the getDeliveryStatus helper function to classify an order as "cancelled" if it has a failureReason, even if its database status is PROCESSING.

Fix 2: Buyer Progress Tracker

Goal: Stop showing "Driver Assigned" for failed orders.

  • Updated OrderProgressTracker.jsx to detect shipping.failureReason.
  • Added visual overrides to display a red "Delivery Cancelled" state.

Fix 3: Order List Columns

Goal: clear separation of Order and Payment states.

  • Payment Status Column: Rewrote logic to strictly check isPaid and paymentMethod (showing "Paid", "Unpaid", or "Pending COD").
  • Order Status Column: Added a NEW column that syncs with courier status. It displays "Delivery Cancelled" in red if the courier fails, otherwise showing the standard status.

4. Technical Code Details

Key code logic implemented across components:

The "Failure Reason" Check

This is the core logic used to override the confusing "Processing" status:

// Used in OrderItem.jsx and getDeliveryStatus
const isDeliveryFailed = order.status === 'PROCESSING' && order.shipping?.failureReason;

if (isDeliveryFailed) {
    return 'Delivery Cancelled'; // Displayed in Red
}

Payment Status Logic

// strictly checking payment fields, ignoring shipment status
{order.isPaid 
    ? 'Paid' 
    : order.paymentMethod === 'COD' 
        ? 'Pending (COD)' 
        : 'Unpaid'
}

5. Future Recommendations

Enhancement: To improve system robustness, consider the following for the next phase:
  • Dedicated Status Enum: Introduce a DELIVERY_FAILED status in the Prisma OrderStatus enum. This would remove the need for "interpretation layers" in the UI.
  • Webhook Refactor: Update the Lalamove webhook to set this new status instead of reverting to PROCESSING.
  • Rebook Flow Update: Update the "Rebook" button to explicitly transition an order from DELIVERY_FAILED back to PROCESSING only when the user clicks it.
