Phase 2: Technical Specification

This phase defines the technical architecture, data structures, and component hierarchy required to implement the Shopee-like address system.

1. Database Schema Unification

We need to unify Address and StoreAddress models to ensure consistency. The proposed unified structure is:

// Unified Address structure (mental model)
{
  region: string;     // e.g., "Metro Manila"
  province: string;   // e.g., "Metro Manila"
  city: string;       // e.g., "Quezon City"
  barangay: string;   // e.g., "Commonwealth"
  detailedAddress: string; // House #, Street, etc.
  latitude: float;
  longitude: float;
}

Note: We will map state in Address and district/province in StoreAddress to this standardized hierarchy.

2. Component Architecture (Geoapify Enhanced)

We will leverage Geoapify's specialized APIs to provide a superior "Shopee-like" experience.

2.1 AddressAutocomplete.jsx Geoapify API

  • Responsibility: Provides a search bar that suggests real addresses in the Philippines.
  • Logic: When a user selects a suggestion, it automatically fills the City, Barangay, and sets the Map coordinates.

2.2 MapPicker.jsx React Leaflet + Geoapify Tiles

  • Responsibility: Displays an interactive map with Geoapify-styled tiles.
  • Reverse Geocoding: Uses Geoapify Reverse Geocoding API to update the address fields if the pin is moved manually.

3. API & Data Strategy

We will use Geoapify as the primary data source, with a local PSGC JSON as a fallback for manual selection if the user cannot find their address via search.

  3. Install @geoapify/react-geocoder-autocomplete, react-leaflet, and leaflet.
