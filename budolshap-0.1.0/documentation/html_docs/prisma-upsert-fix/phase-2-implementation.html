Phase 2: Refactoring & Schema Sync

Implementation of robust defaults and cross-environment schema consistency.

Key Technical Changes

1. Centralized Defaults

A DEFAULT_SETTINGS constant was introduced in lib/settings.js to ensure all mandatory fields are accounted for in both get and update operations.

export const DEFAULT_SETTINGS = {
    id: "default",
    realtimeProvider: "POLLING",
    sessionTimeout: 15,
    // ... all other required fields
};

2. Robust Upsert Pattern

The updateSystemSettings function now merges incoming data with the full set of defaults in the create block:

create: { 
    ...DEFAULT_SETTINGS,
    ...data,
    id: "default"
}

3. Multi-Schema Synchronization

The SystemSettings model was synchronized across three different Prisma schema files to ensure that local development (MySQL) doesn't fall behind production (PostgreSQL):

  • prisma/schema.prisma (Main / Production)
  • prisma/schema.mysql.prisma (Local MySQL)
  • prisma/schema.local.prisma (Local PostgreSQL)

Impact

These changes eliminate "Missing field" validation errors during upserts and ensure that partial updates (like map settings) can safely initialize the entire settings record if it doesn't already exist.
