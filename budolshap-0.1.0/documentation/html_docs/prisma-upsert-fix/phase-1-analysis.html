Phase 1: Analysis & Diagnosis

Understanding the Prisma validation failure during settings updates.

The Error

The application encountered an Invalid prisma.systemSettings.upsert() invocation. The error trace indicated that during the create part of an upsert, Prisma expected several required fields (like realtimeProvider, sessionTimeout, etc.) that were not provided in the payload.

Root Cause

In lib/settings.js, the updateSystemSettings function used a spread of partial data for both update and create blocks:

// problematic code
create: { id: "default", ...data }

If data only contained map-related settings, the create block lacked other mandatory fields of the SystemSettings model. Even if these fields have @default values in the schema, Prisma Client requires them in the create object if they are not explicitly marked as optional (?) in the schema.

Technical Findings

  • Model: SystemSettings
  • Constraint: Mandatory fields without defaults in the create payload caused validation errors.
  • Environment Inconsistency: schema.mysql.prisma and schema.local.prisma were missing newer fields like map providers, leading to "Unknown argument" errors in local environments.
