Detailed Recommendations

🕒 1. Automated Unpaid Order Cancellation Phase 1: 1-2 Days

Implement a Cron Job that runs every 6-12 hours to cancel orders that meet the following criteria:

  • Status is ORDER_PLACED (Unpaid).
  • Payment method is NOT COD.
  • Created more than 48 hours ago (configurable cutoff).
Benefit: Prevents "ghost orders" from locking stock and clarifies the seller's active queue.

📸 2. Buyer Payment Proof Intake Phase 2: 3-5 Days

Add a "I have paid" button for buyers waiting for QRPH status updates. This allows them to:

  • Upload a screenshot of the GCash/Maya transaction.
  • Input the Reference ID manually.
  • Change order sub-status to "Verification Requested".

📢 3. Seller Escalation Dashboard Phase 2: 3-5 Days

Enhance the Seller Order Management page to highlight orders where the buyer has uploaded proof but status is still unpaid.

  • View uploaded proof images directly.
  • Flag order for Admin review if payment remains unverified.

⚖️ 4. COD Privilege Restriction Phase 3: 1 Week

Mirror Shopee's policy of penalizing buyers who refuse COD parcels.

  • Track "Return to Seller" (RTS) rates per buyer.
  • Automatically disable COD option if a buyer exceeds a failure threshold.

Shopee Return & Refund (R&R) Strategic Integration

To provide a premium marketplace experience, budolShap will adopt a Shopee-style Buyer Protection period and a multi-stage dispute resolution flow.

🔄 R&R Lifecycle (Escrow Integrated)

  • Protection Window: Funds are held for 3 days post-delivery. If the buyer doesn't click "Order Received" or file R&R, funds are auto-released.
  • Dual Path Requests: Buyers choose between Refund Only (low value/defective) or Return & Refund (high value/wrong item).
  • Evidence Mandate: Both buyers and sellers must upload unboxing/packing videos and photos during the request and response windows.
  • Arbitration Engine: An Admin dashboard will be created to review chat logs and evidence should a seller dispute a return.

Admin Configuration Control

To provide flexibility, a new "Unpaid Order Cancellation" submenu will be added to the Admin Settings. This allows administrators to manage the lifecycle of abandoned orders directly from the dashboard.

⚙️ Configurable Parameters

  • Auto-Cancellation Toggle: Enable or disable the background cleanup process entirely.
  • Cutoff Threshold (Hours/Days): Set the exact duration an order can remain unpaid before it is voided.
  • Contextual Hints: Inputs will include clear instructions (e.g., "Market standard is 48 hours for QRPH payments") to guide administrators.

🛡️ Escrow Protection Window (Admin Control)

  • Path: Settings > Escrow Settings
  • Configurable Window: Administrators can set the number of days funds are held post-delivery (e.g., set to 3 days).
  • Purpose Hint: "The Protection Window defines how many days the system holds funds in escrow after delivery before auto-releasing to the seller. Standard market practice is 3 days."

Architectural Robustness & Microservices Alignment

The proposed implementation follows a Modular Monolith pattern, which balances immediate performance with long-term scalability. This approach ensures that budolShap is well-positioned for a full microservices migration should the platform's traffic require it.

🛡️ Technical Safeguards

  • Service Decoupling: Business logic is strictly encapsulated within the lib/services/ layer, separating physical data access from the UI and external API handlers.
  • Atomic Transactions: All order status changes (cancellations, payment updates) utilize database-level transactions (Prisma $transaction) to guarantee data integrity and prevent race conditions.
  • Idempotency: Webhook handlers and status sync processes include checks to prevent duplicate processing of the same payment or status update event.
  • Eventual Consistency: Background processes (Cron Jobs) handle non-time-critical maintenance, ensuring the core system remains responsive even during heavy loads.
  • Audit Traceability: Every automated action is logged via the AuditLog service, providing a transparent history for dispute resolution and system monitoring.

UI/UX Enhancements (Visual Mockups)

Based on the current platform design, the following visual updates will be implemented to support the new Shopee-aligned flows:

🏢 Seller Dashboard (Orders Table)

  • "Verifying" Badge: Similar to the "ORDER_PLACED" badge shown in screenshots, a new VERIFYING state will use a distinct color (Indigo) to flag orders where the buyer has submitted proof.
  • Conditional Buttons: The "Review & Book" button will remain inactive or show a "Wait" cursor until payment is officially verified via webhook or admin override.
  • Returns Tab: A new top-level "Returns" tab will be added immediately following the "Cancelled" status tab for centralized management of R&R requests.
  • Auto-Cancellation Feedback: Orders cancelled by the system will populate the red CANCELLED card in the dashboard summary.

🛍️ Buyer Order Tracking Page

  • Modular Stepper: The horizontal progress tracker will dynamically insert a "Payment Verification" node between "Order Placed" and "Processing".
  • Return/Refund Tab: In the "My Orders" screen, a new "Return/Refund" tab will be added after the "Cancelled" tab, mirroring the Shopee layout shown in your screenshots.
  • Detailed History: The vertical "Tracking History" list will include refined time-stamped logs for proof submission and verification start.
  • Actionable UI: A "Upload Payment Proof" button will appear for offline/QRPH methods, ensuring buyers know exactly what next step is required to move their order forward.

Tracking & Synchronization Impact

System Integrity: The proposed changes are designed to complement, not conflict with, existing Lalamove synchronization.

  • No Interference with Lalamove Sync: Auto-cancellation only affects orders in ORDER_PLACED (unpaid) status. The shippingOrderUpdater.js only syncs PROCESSING or SHIPPED orders.
  • Tracking Component Compatibility: The LalamoveTracking.jsx component is only rendered for orders with a provider and booking ID. New pre-verification states will not trigger or break this component.
  • Improved Transparency: Buyers will transition from "Order Placed" to "Verification Requested" (if proof is uploaded) and then to "Paid," ensuring they understand exactly when fulfillment (and tracking) will begin.
