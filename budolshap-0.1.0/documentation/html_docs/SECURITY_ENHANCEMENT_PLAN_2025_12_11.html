Security Enhancement Plan

Application: Budolshap V3 | Date: December 11, 2025 | Author: Antigravity (Google DeepMind)
Phase 1: Cookie Security Standardization High Priority
Timeframe: 1-2 Hours Status: Partially Implemented

Objective: Ensure all authentication cookies are tamper-proof and only transmitted over secure connections (HTTPS) to prevent session hijacking.

Implementation Steps:

  • Audit app/api/auth/login/route.js (Done: Secure flag added).
  • Centralize cookie configuration options into a helper function in lib/auth.js to ensure consistency across the app.
  • Apply strict cookie policies:
    • httpOnly: true - Prevents JavaScript access (mitigates XSS).
    • secure: true - Ensures transmission only over HTTPS in production.
    • sameSite: 'lax' - Protects against CSRF attacks.
  • Verify correct NODE_ENV configuration on Vercel Dashboard.
Phase 2: Automatic Idle Logout Medium Priority
Timeframe: 3-4 Hours Status: Planned

Objective: Automatically log out users with sensitive access (Admins, Sellers) after a period of inactivity to prevent unauthorized access from unattended devices.

Implementation Steps:

  • Develop a custom React hook useIdleTimeout(timeoutMs, warningMs).
  • Implement event listeners for user activity: mousemove, keydown, click, scroll.
  • Create a "Session Expiring" Modal that warns the user 1 minute before logout.
  • Integrate into AdminLayout and StoreLayout components.
  • Ensure the Buyer/Shopper experience remains unaffected (no auto-logout for regular users).
  • Implement secure redirect logic to return the user to the login screen with a specific "Session Expired" message.
Phase 3: Rate Limiting & Brute Force Protection Critical Priority
Timeframe: 4-5 Hours Status: Planned

Objective: Protect authentication endpoints against brute-force password guessing and abuse.

Implementation Steps:

  • Select Rate Limiting Strategy:
    • Preferred: Vercel KV (Redis) or Upstash for distributed state management.
    • Alternative: Database-backed token bucket (if KV is unavailable).
  • Create generic Rate Limiter utility / middleware.
  • Apply limits to critical endpoints:
    • POST /api/auth/login - e.g., 5 attempts / 15 mins.
    • POST /api/auth/register - e.g., 3 attempts / hour per IP.
  • Implement graceful error handling (HTTP 429) with "Retry After" messaging for the user.
