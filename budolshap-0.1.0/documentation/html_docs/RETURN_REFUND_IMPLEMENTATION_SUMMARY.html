Return & Refund System

Implementation Summary | Date: December 20, 2025 | Phase: 3

🚀 Executive Summary

The Return & Refund System has been fully implemented, aligning BudolShap's order management flow with Shopee's best practices. This update introduces a new lifecycle for disputed orders, giving buyers a way to request refunds for missing or damaged items, and providing sellers with a dedicated dashboard to manage these requests.

Additionally, the default order status display has been refined to show "PROCESSING" for active orders, cleaner and more professional than "Order Placed".

🛠️ Processes Implemented

1. Buyer: Requesting a Return

1
Initiation: Buyers can click the "Return / Refund" button on any eligible order (Pending/Delivered).
2
Evidence Submission: A new Modal UI allows buyers to:
  • Select a Reason (e.g., "Did not receive order", "Damaged item").
  • Choose Type: "Refund Only" or "Return & Refund".
  • Upload Photo Evidence: Supports up to 5 images with live previews.
3
Status Update: The order status updates to RETURN_REQUESTED, and the tracking flow pauses.

2. Seller: Managing Returns

1
New Dashboard Tab: Added a dedicated "Return & Refund" tab in the Seller Center.
2
Review Process: Sellers can view details of the request, including the buyer's reason and uploaded proof.
New Feature: Clickable image previews with a large-screen modal for detailed inspection.
3
Action:
  • Accept & Refund: Immediately processes the refund. Deducts from Seller Wallet.
  • Decline: Sets status to DISPUTED (for Admin intervention in future phases).

3. System: Status & Wallet Logic

1
Automated Wallet Deduction: When RETURN_APPROVED or REFUNDED occurs, the system automatically creates a DEBIT transaction in the seller's wallet.
2
Order Status Refinement:
  • ORDER_PLACED & PAID → Displayed as PROCESSING
  • RETURN_REQUESTED → Displayed as RETURN REQUESTED
  • REFUNDED → Displayed as REFUNDED

🐛 Issues Encountered & Fixes

1. "Unique Key" Prop Warning

Issue: React warning Each child in a list should have a unique "key" prop in the Return Request card.

Cause: The code was using item.id, but OrderItem uses a composite key (OrderId + ProductId) and has no single ID field.

Fix: Updated the loop to use item.productId as the key.

2. Build Error: Syntax in JSX

Issue: Parsing ecmascript source code failed. Expected ',' got '{'

Cause: A premature closing </div> tag broke the nesting of the new Image Modal component.

Fix: Removed the extra closing tag and properly nested the Modal conditional rendering.

3. Browser Automation Rate Limiting

Issue: The E2E browser test failed with 429 Too Many Requests.

Mitigation: Verified the backend logic using a custom simulation script (scripts/simulate-return-flow.js) which successfully proved the "Accept & Refund" button updates the database and wallet correctly.

🌟 Recommendations for Future Enhancements

1. Admin Dispute Portal

Currently, if a seller declines a return, it goes to DISPUTED. An Admin interface is needed to review these cases and make a final ruling.

2. Logistics Integration for Returns

For "Return & Refund" (where item is shipped back), integrate with Lalamove/J&T to generate a return waybill automatically.

3. Email/SMS Notifications

Trigger alerts to the buyer when a refund is approved, and to the seller when a new return request is filed.

4. Partial Refunds

Allow sellers to offer a partial refund (e.g., "Keep the item, here's 50% back") to resolve disputes faster.
