📊 Executive Summary

Phase 2 of the Canceled Delivery Workflow has been successfully implemented. This phase focused on "Industry Standard Features" to ensure resilience and a seamless user experience when deliveries go wrong. Key achievements include a robust notification system, webhook reliance mechanisms, and a flexible "Manual Booking" fallback that effectively enables **Courier Switching**.

Key Outcome: Sellers can now recover from failed Lalamove bookings by either re-booking automatically or switching to a manual provider (e.g., J&T, Grab) without losing the order's history.

✅ Features Implemented

  • Buyer & Seller Notifications: Automated emails triggered via generic webhook handler (`lib/email.js`).
  • Resilient Rebooking Logic: Fixed API to allow re-booking on top of failed attempts while preserving history.
  • Manual Booking / Courier Switching: New API and UI to support non-integrated couriers.
  • Webhook Logging & Polling: implemented redundant checks via Cron and database logging (`WebhookEvent` table).
  • Manual Sync Button: UI control for sellers to force-refresh order status.

🛠️ Critical Issues Resolved

1. History Overwrite Bug CRITICAL

Issue: Upon rebooking, the system was completely replacing the `shipping` JSON object in the database. This caused the record of the previous failed attempt (and its reason) to be lost forever.

Fix: Modified app/api/shipping/lalamove/book/route.js and the new manual route to explicitly copy order.shipping.previousAttempts into the new payload.

// ✅ Preserve history fix
previousAttempts: order.shipping?.previousAttempts || []

2. "Order Already Has Booking" Lockout HIGH

Issue: The booking API blocked any request if a `bookingId` already existed, even if that booking was cancelled or failed.

Fix: Updated validation logic to check the status of the existing booking.

// Allow rebooking if existing booking is failed/cancelled
const isShippingActive = order.shipping?.bookingId && 
    !order.shipping?.failureReason && 
    !['CANCELLED', 'EXPIRED', 'REJECTED', 'COMPLETED'].includes(order.shipping?.status);

3. Lalamove-Only Lock-in

Issue: The UI only offered "Book Lalamove", leaving sellers stuck if Lalamove was unavailable or too expensive.

Fix: Implemented a Courier Switching Toggle in the UI that defaults to Lalamove but allows "Manual / Other" input.

💻 Technical Implementation Details

Backend Architecture

We introduced a generic "Manual Booking" endpoint that mimics the structure of the Lalamove booking but for offline/manual inputs.

Endpoint Method Purpose
/api/shipping/lalamove/book POST Automated booking integration. Now history-aware.
/api/shipping/manual/book POST NEW: Accepting Provider Name, Tracking #, and Link. Updates order status to PROCESSING.

Frontend State Management

The StoreOrders component (app/store/orders/page.jsx) now manages a local dual-state for booking:

  • bookingType: 'lalamove' | 'manual'
  • manualDetails: Object storing provider, trackingNumber, trackingUrl
