Strategic Phased Roadmap

Full platform alignment with Shopee Philippines Standards

Est. Total Duration: 8-10 Working Days
Phase 1: Cancellation Phase 2: QRPH & Proof Phase 3: R&R Ecosystem
Days 1-2

Automated Cancellation & Admin Control

Core Backend Stability
⚙️ Backend Processes
  • Prisma Migrations: Add orderCancellationHours and orderCancellationEnabled to SystemSettings. DB Schema
  • Service Logic: Implement cancelExpiredUnpaidOrders in ordersService.js using atomic transactions.
  • Cron Endpoint: Create secured API route for Vercel/External Cron triggers.
🖥️ Frontend & UI
  • Admin Submenu: Create /admin/settings/unpaid-cancellation page for control.
  • Config Inputs: Implement toggle switches and hour/day inputs with real-time validation.
  • Instructional Hints: Add tooltips explaining Shopee's 48-hour logic.
Technical Safeguard: The cancellation service will issue a "System Cancellation" Audit Log and notify the buyer automatically to prevent double-payment confusion.
🧪 Testing & Validation
  • Automation: Script-based trigger of /api/cron/cancel-unpaid-orders with time-warped mock data.
  • Admin Verification: Verify that toggling "Cancellation Enabled" off prevents the cron job from processing orders.
Dependency: Requires the Cron Job Secret to be shared with the deployment environment.
Days 3-4

QRPH Verification & Payment Proof UI

User Experience & Trust
⚙️ Backend Processes
  • Metadata Expansion: Enable paymentProof field in Order model to store transaction IDs or image URLs.
  • Webhook Sync: Ensure PayMongo QRPH success events auto-clear "Verification In Progress" status. PayMongo API
🖥️ Frontend & UI
  • Buyer Proof Modal: "Upload Receipt" button for QRPH/Bank orders.
  • Dynamic Stepper: Add "Verifying Payment" step in the Order Details tracker.
  • Seller Verification Tab: New dashboard badge for "Pending Verification" orders.
Technical Safeguard: Implementation of Cloudinary/S3 hooks for secure, temporary storage of payment receipts to avoid DB bloat.
🧪 Testing & Validation
  • Proof integrity: Verify uploaded images are correctly linked to the Order and visible in the Admin Verification view.
  • Stepper Logic: Verify "Verifying Payment" node is only visible for non-COD orders that have an uploaded receipt.
Days 5-9

Advanced Return & Refund (R&R) Ecosystem

Escrow & Legal Compliance
⚙️ Backend Processes
  • Escrow Holding Logic: Modifiy payout services to honor the protectionWindowDays config. Wallet System
  • Dispute Arbitration: Create multi-actor API (Buyer, Seller, Admin) to handle R&R threads.
  • Shipping Fee Rebates: Logic to determine who pays for return shipping based on "Seller Fault" flags.
🖥️ Frontend & UI
  • Returns Tab: Add dedicated "Return/Refund" tab in Buyer/Seller dashboards after the "Cancelled" tab.
  • R&R Control Center: Detailed dispute view with Accept/Dispute/Chat options for sellers.
  • Admin Arbitration View: Final verdict interface for platform moderators.
Technical Safeguard: Atomic balance transfers; refunds are ONLY processed from pendingBalance to ensure the platform doesn't "over-refund" if funds were already withdrawn.
🧪 Testing & Validation
  • Escrow Integrity: Ensure funds for delivered orders are locked for the exact protectionWindowDays count.
  • Resolution Logic: Manual E2E test of Buyer Request -> Seller Dispute -> Admin Resolution flow.
Dependency: Requires the new SystemSettings protection window value implemented in Phase 1.
