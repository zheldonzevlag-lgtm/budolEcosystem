Virtual Escrow & Funds Separation Plan

Solving Immediate Payout Risks via the Aggregator Model

DATE: 2025-12-08 VERSION: 1.2 (Architecture Analysis)

Architecture Analysis

Is it Robust? YES.

This system relies on ACID Database Transactions.

  • Atomic: Funds are moved (Pending → Available) in a single database step. It's impossible for money to be "lost" in between statuses.
  • Idempotent: Transactions are logged with unique IDs. We can prevent double-releasing funds by checking if a release transaction already exists for an Order ID.

Is it Gateway Agnostic? YES.

The logic is decoupled from PayMongo.

  • Internal Ledger: The Wallet table is your own internal ledger. It doesn't care source the money came from.
  • Plug & Play: If you add PayPal or Xendit later, you simply point their webhook to the creditPendingBalance() function. The core "Pending vs. Available" logic remains 100% unchanged.

Problem Statement Recap

Current Behavior: Funds are credited immediately.
New Behavior: Funds are held in pendingBalance until COMPLETED.

Technical Implementation Plan

Phase 1: Database & Webhook Logic

1. Modify Database Schema (Prisma)

File: prisma/schema.prisma

model Wallet {
  ...
  balance Float @default(0) // Available for withdrawal
  + pendingBalance Float @default(0) // Held in Escrow
}

2. Update Payment Webhook

File: app/api/webhooks/paymongo/route.js

Change the logic to increment pendingBalance instead of balance when a payment is confirmed.

Phase 2: The "Release" Trigger

Order Completion Logic

File: app/api/orders/[orderId]/status/route.js

When an Order status transitions to COMPLETED:

  • Decrease pendingBalance by the net earnings.
  • Increase balance (Available) by the net earnings.
  • Create a Transaction record: "Funds Released from Escrow".
