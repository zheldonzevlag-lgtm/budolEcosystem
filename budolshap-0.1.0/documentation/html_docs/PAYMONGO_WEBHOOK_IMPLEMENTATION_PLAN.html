PayMongo Webhook Implementation Plan

Date: December 15, 2025

Status: PLANNED

Target: Future Enhancement

1. Overview

This document outlines the plan to implement full server-side webhooks for PayMongo integration. This enhancement aims to improve the reliability of order status updates by decoupling them from client-side interactions.

2. Current System & Limitations

Current Behavior

Currently, the system relies on the client (browser) redirecting to /payment/return to update the order status to "Paid".

Limitations

  • Fragility: If the user closes the browser immediately after payment but before the redirect completes, the order remains "Unpaid" in the database despite the payment being successful.
  • Security: Relying on client-side redirects can be spoofed or manipulated, although signature verification mitigates this, server-to-server communication is more secure.
  • User Experience: Users may be confused if they pay but don't see the order update immediately if the network is slow during redirect.

3. Proposed Solution: Server-Side Webhooks

Implementing a dedicated webhook endpoint to listen for PayMongo events guarantees that the system is notified of payment status changes regardless of the user's browser state.

4. Implementation Details

4.1 New API Endpoint

Create a new route: app/api/webhooks/paymongo/route.js

POST /api/webhooks/paymongo

This endpoint will handle incoming events from PayMongo.

// Pseudocode for the webhook handler
export async function POST(req) {
    // 1. Verify Webhook Signature (Security Critical)
    const signature = req.headers.get('paymongo-signature');
    if (!isValidSignature(signature, req.body)) {
        return Response.json({ error: 'Invalid signature' }, { status: 401 });
    }

    // 2. Parse Event
    const event = await req.json();
    const type = event.data.attributes.type;

    // 3. Handle Events
    switch (type) {
        case 'source.chargeable':
            // Handle GCash/GrabPay source becoming chargeable
            await createPaymentFromSource(event.data);
            break;
        case 'payment_intent.succeeded':
            // Handle Card/success
            await markOrderAsPaid(event.data);
            break;
    }

    return Response.json({ status: 'ok' });
}

4.2 Event Handling Logic

A. source.chargeable (GCash / PayMaya / GrabPay)

When a source becomes chargeable:

  1. Retrieve the amount and id (source ID) from the event.
  2. Action: Immediately create a Payment resource using the PayMongo API.
  3. This automatically charges the user's wallet.

B. payment_intent.succeeded (Credit/Debit Cards)

When a payment intent succeeds:

  1. Extract the metadata attached to the payment intent (which should contain orderId).
  2. Action: Update the corresponding Order in the database to PAID status.
  3. Trigger any post-payment workflows (e.g., email notification, inventory deduction).

5. Migration & Deployment Strategy

Important: Webhooks must be registered in the PayMongo Dashboard or via API.
  1. Develop: Implement the API route and verification logic locally.
  2. Test: Use tools like ngrok to expose the local server and test webhook delivery from PayMongo's test mode.
  3. Deploy: Push changes to production.
  4. Register: Add the production URL (e.g., https://budolshap.com/api/webhooks/paymongo) to the PayMongo Webhook settings for live events.

6. Implementation Timeframe

Estimated Duration: 3-5 Days

Phase 1: Development (1-2 Days)

  • Implement webhook endpoint with signature verification
  • Add event handling logic for source.chargeable and payment_intent.succeeded
  • Create comprehensive error handling and logging
  • Write unit tests for webhook verification and event processing

Phase 2: Local Testing (1 Day)

  • Set up ngrok or similar tunneling for local webhook testing
  • Register test webhook URL in PayMongo test mode
  • Simulate payment events and verify order status updates
  • Test edge cases (duplicate events, out-of-order delivery, failed payments)

Phase 3: Production Deployment (1-2 Days)

  • Deploy webhook endpoint to production
  • Register production webhook URL in PayMongo dashboard
  • Monitor initial webhook deliveries for errors
  • Gradually phase out client-side status updates in favor of webhooks
  • Document webhook behavior and monitoring procedures

7. Benefits

  • 100% Reliability: Orders are marked paid even if the user's battery dies or internet fails immediately after paying.
  • Better UX: No need to force users to wait for a redirect; the system updates asynchronously.
  • Audit Trail: Webhook logs provide a clear history of payment events from the provider.
