Phase 2: Exception & Failure Management

Implementation Status: COMPLETED

This phase introduced robust handling for "Non-Happy Path" scenarios in the delivery process, ensuring that sellers can recover from courier failures without losing order momentum.

1. Failure Status Handling

The system now explicitly handles various courier failure states received via webhooks (e.g., Lalamove):

Courier Status System Response Seller Action
CANCELED Marks shipping as failed; preserves history. Re-book or Switch Courier
EXPIRED Alerts seller that no driver was found within the timeout period. Rebook Delivery link in order list.
REJECTED Logs reason for rejection. Update details and Re-book

2. Key Features

A. Courier Switching (Manual Booking)

Sellers are no longer locked into Lalamove. If a booking fails, they can switch to a Manual Provider (e.g., J&T, Grab, Personal Delivery).

  • New endpoint: /api/shipping/manual/book
  • Allows input of Provider Name, Tracking Number, and Tracking URL.
  • Updates order status to PROCESSING / SHIPPED while maintaining the Shopee-style UI.

B. History Preservation & UI Display

Re-booking no longer overwrites previous shipping data. All attempts are archived in order.shipping.previousAttempts, allowing for audit trails.

  • UI Integration: The "Tracking History" sidebar now displays a chronological list of all events, including failures.
  • Exception Highlighting: Failed attempts are marked with a red "✕" icon in the vertical history list.
  • Reason Logging: Detailed failure reasons (e.g., "No driver found", "Cancelled by sender") are displayed directly under the status label.

C. Manual Sync & Recovery

Added a "Manual Sync" button in the Seller Dashboard to force-refresh status from the courier API in case webhooks are delayed.

3. Success Criteria Validation

  • ✅ Sellers can re-book failed Lalamove orders without error.
  • ✅ "Switch to Manual Courier" option is functional in the booking modal.
  • ✅ Webhook failures are mitigated by a redundant polling mechanism.
  • ✅ Order history correctly logs all failed delivery attempts.
