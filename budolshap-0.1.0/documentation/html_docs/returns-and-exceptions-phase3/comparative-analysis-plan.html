Phase 3: Comparative Analysis & Implementation Plan

1. Comparative Analysis: Returns & Refunds

Current Implementation

UI Flow: Isolated modals and a standalone "Return Center" for sellers.

  • Returns exist as a separate entity from order tracking.
  • Status updates are text-based in the order list.
  • No clear distinction in the UI between returning an item vs. just asking for a refund.
  • Wallet deduction is manual upon seller approval.
Shopee-Style Integration (Phase 3)

UI Flow: Integrated 5-stage tracking bar with dual-path logic.

  • Dual Path Support: Explicitly handles REFUND_ONLY (instant funds) and RETURN_AND_REFUND (parcel tracking).
  • Unified Progress Bar: Status bar switches to Return icons (Requested → Review → Shipped → Refunded).
  • Seller Dashboard Cards: New "Return / Refund" summary card alongside Needs Booking, Booked, and In Transit.
  • Auto-Reconciliation: Integrated wallet transactions (Escrow DEBIT) with Pusher notifications.

2. Comparative UI (Mockup Descriptions)

Current Tracking UI

A simple horizontal bar with "Placed", "Paid", "Shipped", "Delivered". If a return happens, the bar either stays at "Delivered" or disappears, replaced by a text status like "RETURN_REQUESTED".

Phase 3 Tracking UI

When a return is active, the bar morphs:
[Placed] --- [Delivered] --- [↩️ Return Requested] --- [✅ Approved] --- [💰 Refunded]
The icons change to orange/red to indicate an exception state.

4. Implementation Plan

Step 1: UI Morphing & Dual-Path Logic

Update OrderProgressTracker.jsx to detect return states and switch icon sets. Implement specific UI branches for "Refund Only" (1-step) vs "Return & Refund" (3-step with parcel tracking).

Step 2: Courier Failure Mapping

Update Lalamove webhook to handle EXPIRED/CANCELED by triggering "Action Required" status for sellers.

Step 3: Evidence Integration

Add a new "Proof of Delivery" component to display courier photos alongside buyer's evidence.

Step 4: Real-time Pusher Sync

Standardize event names (e.g., order.exception) for instant UI updates.

Step 5: Platform Mediation & Risk Controls

Establish Admin Dispute Arbitration flows and implement technical safeguards (Escrow locking, atomic wallet transactions) to mitigate "Double Refund" and "Empty Box" fraud risks.

5. Testing & Timeframe

Testing Protocol

  • Unit Tests: Verify status mapping logic for all 12+ combined order/return states.
  • Integration Tests: Simulate a Lalamove "Expired" webhook and verify the "Retry Booking" button appears for the seller.
  • E2E Tests: Full flow from "Delivered" → "Return Requested" → "Accept & Refund" → "Wallet Balance Check".

Timeframe Estimate

Total Estimated Time: 4-5 Days

  • Day 1: UI Morphing & Status Bar Updates
  • Day 2: Lalamove Webhook Exception Mapping
  • Day 3: Evidence Hub & Proof Integration
  • Day 4: Real-time Sync & Wallet Reconciliation
  • Day 5: Final QA & Documentation
