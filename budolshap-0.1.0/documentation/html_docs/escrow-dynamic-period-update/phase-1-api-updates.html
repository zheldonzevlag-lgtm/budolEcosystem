Phase 1: API Updates

The first phase focused on extending existing API endpoints to include the dynamic protectionWindowDays value from the centralized system settings.

Overview

Previously, many UI components had hardcoded escrow holding periods (typically 7 days). This phase ensures that the backend provides the actual configured value to all relevant frontend interfaces.

Modified API Endpoints

Endpoint Service Used Purpose
GET /api/dashboard/store systemSettingsService Provide protectionWindowDays to the Seller Dashboard.
GET /api/store/wallet systemSettingsService Provide protectionWindowDays to the Seller Wallet page.
GET /api/orders/[orderId] systemSettingsService Provide protectionWindowDays to the Public Order Detail page.
GET /api/dashboard/admin/escrow systemSettingsService Provide protectionWindowDays to the Admin Dashboard stats.

Technical Implementation Details

Each API was updated to import getSystemSettings and inject the value into the JSON response:


import { getSystemSettings } from '@/lib/services/systemSettingsService';

// Inside GET handler
const settings = await getSystemSettings();
const protectionWindowDays = settings?.protectionWindowDays || 7;

return NextResponse.json({
    // ... other data
    protectionWindowDays
});

Validation & Testing

  • Verified that getSystemSettings() correctly falls back to 7 if settings are missing.
  • Confirmed that the protectionWindowDays property is present in all updated API responses.
  • Ensured no breaking changes were introduced to existing data structures.
