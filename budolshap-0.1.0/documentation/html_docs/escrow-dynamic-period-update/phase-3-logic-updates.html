Phase 3: Logic Updates

The final phase ensured that the underlying business logic for order auto-completion and fund release respects the dynamic settings.

Order Auto-Completion Utility

Updated lib/orderAutoComplete.js to fetch protectionWindowDays directly from system settings when scheduling a new completion date.


export async function scheduleAutoComplete(orderId, days) {
    let thresholdDays = days;
    if (thresholdDays === undefined || thresholdDays === null) {
        const { getSystemSettings } = await import('./services/systemSettingsService');
        const settings = await getSystemSettings();
        thresholdDays = settings?.protectionWindowDays || 7;
    }
    // ... calculate date based on thresholdDays
}

Lalamove Sync Integration

The Lalamove synchronization route (/api/orders/[orderId]/sync-lalamove) was updated to use the dynamic scheduleAutoComplete function instead of a hardcoded 7-day calculation.

Code Cleanup

  • Updated comments in lib/escrow.js and lib/orderAutoComplete.js to remove references to specific day counts (e.g., "7 days").
  • Replaced direct imports of lib/settings.js with the centralized lib/services/systemSettingsService.js where appropriate.

System Consistency

By centralizing the logic in scheduleAutoComplete, any future status changes to DELIVERED (via webhooks, manual updates, or provider syncs) will automatically use the correct configured window.
