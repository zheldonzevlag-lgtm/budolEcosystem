1. Goal Description

Display star ratings for products that have been reviewed by buyers in the "My Orders" page. Currently, ratings are stored in the database but are not being fetched or displayed when orders are loaded. The implementation will also include real-time updates so that when a user submits a rating, it appears immediately across all open browser tabs.

Key Features:
  • Fetch and display existing ratings from the database
  • Show star ratings for reviewed products in the orders list
  • Real-time updates when ratings are submitted (via Pusher/Socket.io/Polling)
  • Optimistic UI updates for immediate feedback

2. Current Problem

The rating system has the following issues:

  • Ratings are stored in the database but not fetched when orders are retrieved
  • The OrderItem component relies solely on Redux state for displaying ratings
  • Redux state is cleared on page refresh, causing ratings to disappear
  • No real-time synchronization when ratings are submitted in other tabs/sessions

Frontend Changes

NEW hooks/useRealtimeBuyerOrders.js

Change: Create a new custom hook for buyer orders with real-time support

Details:

  • Use SWR for data fetching with polling fallback
  • Support Pusher, Socket.io, and polling modes
  • Subscribe to user-specific channel for rating updates (user-{userId})
  • Listen for rating-created events
  • Automatically revalidate when ratings are created
  • Return orders, pagination, loading state, and mutate function
import useSWR from 'swr';
import { useEffect } from 'react';
import Pusher from 'pusher-js';
import { io } from 'socket.io-client';

const fetcher = url => fetch(url).then(r => r.json());

export function useRealtimeBuyerOrders({ userId, page = 1, limit = 10 }) {
    // Fetch realtime config
    const { data: config } = useSWR('/api/system/realtime', fetcher, {
        revalidateOnFocus: false,
        dedupingInterval: 60000
    });

    const isPolling = config?.provider === 'POLLING';
    const isPusher = config?.provider === 'PUSHER';
    const isSocket = config?.provider === 'SOCKET_IO';

    // Fetch orders
    const ordersUrl = userId 
        ? `/api/orders?userId=${userId}&page=${page}&limit=${limit}` 
        : null;

    const { data, error, mutate, isLoading } = useSWR(ordersUrl, fetcher, {
        refreshInterval: isPolling ? 5000 : 0,
        revalidateOnFocus: true
    });

    // Pusher subscription
    useEffect(() => {
        if (!isPusher || !userId || !config?.pusherKey) return;

        const pusher = new Pusher(config.pusherKey, {
            cluster: config.pusherCluster,
        });

        const channel = pusher.subscribe(`user-${userId}`);
        
        channel.bind('rating-created', () => {
            mutate();
        });

        return () => {
            channel.unbind_all();
            channel.unsubscribe();
            pusher.disconnect();
        };
    }, [isPusher, userId, config, mutate]);

    // Socket.io subscription
    useEffect(() => {
        if (!isSocket || !userId || !config?.socketUrl) return;

        const socket = io(config.socketUrl);

        socket.on('connect', () => {
            socket.emit('subscribe', `user-${userId}`);
        });

        socket.on('rating-created', () => {
            mutate();
        });

        return () => {
            socket.disconnect();
        };
    }, [isSocket, userId, config, mutate]);

    return {
        orders: data?.orders || [],
        pagination: data?.pagination || {},
        isLoading,
        error,
        mutate
    };
}

6. Technical Notes

Database Schema:

The Rating model already exists in the database with the following structure:

model Rating {
  id        String   @id @default(cuid())
  rating    Int
  review    String
  userId    String
  productId String
  orderId   String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  product   Product  @relation(...)
  user      User     @relation(...)

  @@unique([userId, productId, orderId])
}
Real-time Infrastructure:

The application already has a real-time system in place that supports three modes:

  • POLLING: Client polls every 5 seconds (default, no setup required)
  • PUSHER: Uses Pusher service for instant updates (requires API keys)
  • SOCKET_IO: Uses Socket.io microservice (requires server URL)

The implementation will work with all three modes automatically.

7. Solution Robustness & Provider Switching

✅ This is a Robust, Provider-Agnostic Solution

The implementation is designed to seamlessly work with any real-time provider (Polling, Pusher, or Socket.io) and allows runtime switching without code changes or redeployment.

How Provider Switching Works

1. Centralized Configuration

The real-time provider is stored in the database (SystemSettings table), not hardcoded in the application:

// Database determines which provider to use
{
  realtimeProvider: "POLLING" | "PUSHER" | "SOCKET_IO",
  pusherKey: "...",        // Only needed for Pusher
  pusherCluster: "...",    // Only needed for Pusher
  socketUrl: "..."         // Only needed for Socket.io
}

2. Dynamic Hook Behavior

The useRealtimeBuyerOrders hook automatically adapts based on the database configuration: 

// Fetches config from database
const { data: config } = useSWR('/api/system/realtime', fetcher);

// Determines behavior based on config
const isPolling = config?.provider === 'POLLING';
const isPusher = config?.provider === 'PUSHER';
const isSocket = config?.provider === 'SOCKET_IO';

// SWR refreshes every 5 seconds ONLY if polling mode
const { data, error, mutate, isLoading } = useSWR(ordersUrl, fetcher, {
    refreshInterval: isPolling ? 5000 : 0,  // Conditional polling
    revalidateOnFocus: true
});

3. Conditional Subscriptions

Each provider's subscription only activates when that provider is selected:

  • Pusher: Only connects if isPusher === true
  • Socket.io: Only connects if isSocket === true
  • Polling: Only polls if isPolling === true

4. Zero Code Changes Required

To switch providers, simply update the database:

  1. Open Prisma Studio (npx prisma studio)
  2. Navigate to SystemSettings table
  3. Change the realtimeProvider field
  4. Optionally add credentials (for Pusher/Socket.io)

That's it! The application automatically:

  • Disconnects from the old provider
  • Connects to the new provider
  • Continues working seamlessly
