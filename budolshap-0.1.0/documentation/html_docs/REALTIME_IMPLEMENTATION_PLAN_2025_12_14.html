Realtime Architecture Implementation Plan

Documenting the strategy, phases, and future roadmap for integrating robust realtime capabilities into BudolShap v3.

Executive Summary

The goal is to transition from a static request-response model to a reactive, realtime architecture. This allows specific events (like Order Placed, Payment Received, User Registered) to instantly update the Admin and Seller dashboards without manual page refreshes. The solution is designed to be agnostic, allowing the administrator to switch between SWR Polling (low cost/effort), Pusher (managed service), and Socket.io (self-hosted) seamlessly.

PHASE 1: COMPLETED

Foundation & Core Infrastructure (Dec 14, 2025 - Completed)

Establish the base architecture to support dynamic provider switching and basic event broadcasting.

Key Deliverables:

  • Configuration System: Database schema and Admin UI to store and toggle realtime settings (Polling/Pusher/Socket).
  • RealtimeProvider Component: A global context provider wrapping the app layout that:
    • Fetches system configuration on load.
    • Configures global SWR settings (Polling vs. Push).
    • Initializes WebSocket connections (Pusher/Socket.io) if enabled.
  • Backend Trigger Utility: lib/realtime.js helper function to abstract the sending logic for different providers.

Affected Areas:

  • app/layout.jsx (Root Layout)
  • components/RealtimeProvider.jsx
  • app/api/system/settings/*
PHASE 2: EVENT INTEGRATION

Core Business Events (Current Focus - 1-2 Days)

Hooking up specific business logic points to the realtime triggers.

Key Deliverables:

  • User Registration: Broadcast user-registered event to Admin. (Implemented)
  • Order Management:
    • Broadcast order-created event to relevant Store channel.
    • Broadcast order-updated (Status changes, Delivery) to Store. (Implemented)
  • SWR Integration (Hybrid Strategy):
    • Smart Revalidation: When a Pusher/Socket event is received, trigger SWR's mutate() to instantly refresh the data without a full page reload.
    • Fallback Polling: If realtime settings are disabled, automatically revert SWR to periodic polling (every 5s) to ensure data consistency.
  • Inventory Updates: Realtime stock decrement when orders are placed.

Dependencies:

  • swr (for Data Fetching & Mutation)
  • pusher (Backend SDK)
  • pusher-js (Client SDK)
  • socket.io & socket.io-client
