Escrow System & Seller Payouts: Complete Documentation

Date: December 9, 2025

Status: COMPLETED

Scope: Phase 1, Phase 2, Phase 3 (End-to-End Implementation)

1. Executive Summary

The Escrow System separates buyer funds into a "Pending" state until order delivery is confirmed + 7 days. This protects buyers from fraud and ensures sellers are paid only for successful deliveries. The system includes an Admin Dashboard for monitoring escrow funds, a Seller Wallet for viewing balances, and a Payout Request workflow.

2. Processes Implemented

Phase 1: Core Logic & Database

Designed and implemented the core database schema changes:

  • Wallet Model: Tracks balance (Available) and pendingBalance (Escrow).
  • PayoutRequest Model: Handles withdrawal requests states (PENDING, APPROVED, REJECTED).
  • Separation of Funds Logic: Money enters Escrow on payment, moves to Wallet on release.

Phase 2: Admin Dashboard & Controls

Enhanced the Admin interface to manage the financial flow:

  • Escrow Stats: Visual cards showing Total Pending, Ready for Payout, and Queue.
  • Payout Management: Dedicated page /admin/payouts to Approve/Reject requests.
  • UI Polish: Added hover tooltips (`cursor-help`) for better UX.

Phase 3: Seller Wallet & Integration

Completed the loop with Seller-side features:

  • Wallet Page: /store/wallet showing real-time balances.
  • Withdrawal: Mechanics for sellers to request payouts from Available Balance.
  • Transaction History: Logs of all credits, debits, and requests.

3. Issues Encountered & Fixes

Issue Root Cause Resolution
API Error Failed to load payouts Missing relation in prisma/schema.prisma between Store and PayoutRequest models. Updated Schema to include bidirectional relations (Store.payoutRequests and PayoutRequest.store). Ran prisma generate.
Auth Error Invalid or expired token (403) Token payload mismatch. Login generated token with userId, but Admin/Store middleware checked for id. Updated lib/adminAuth.js and API routes to check decoded.userId || decoded.id. Added detailed error logging to identify the mismatch.

4. Technical Details

Authentication Architecture

Token Payload: { userId, email, name, role }

Verification: lib/adminAuth.js uses verifyToken then queries DB to confirm isAdmin or `accountType === 'ADMIN'`. Seller Auth: Checks userId against Store.userId relation.

Atomic Transactions

Payout requests use prisma.$transaction to ensure data integrity:

  1. Check Wallet Balance (Must be > Amount).
  2. Decrement Wallet Balance.
  3. Create Transaction Record (DEBIT).
  4. Create PayoutRequest Record (PENDING).

If any step fails, the entire operation rolls back, preventing money loss.
