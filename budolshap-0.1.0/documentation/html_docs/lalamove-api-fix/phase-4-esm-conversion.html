Phase 4: ESM Conversion & Module Compatibility

Status: Completed

Issue Identification

A secondary error was identified: Class extends value #<Object> is not a constructor or null. This occurred because the project is configured with "type": "module" in package.json, but several service files were using CommonJS require and module.exports.

Root Cause Analysis

  • Module Type Conflict: In a "type": "module" project, .js files are treated as ES Modules. Mixing require and import can lead to incorrect resolution of exported values, especially for classes.
  • Turbopack Strictness: Next.js with Turbopack is more strict about module boundaries and expected export formats compared to standard Webpack or Node.js.

Implementation Details

All files in the services/ directory were converted to proper ES Modules:

  • shippingProvider.js: Converted to export default class.
  • lalamove.js: Updated to import dependencies with .js extensions and export default.
  • shippingFactory.js: Converted to use import and named exports for helper functions.
  • shippingService.js: Replaced internal require calls with top-level import statements.
// Example of ESM Conversion in lalamove.js

Key Improvements

  • Standard Compliance: Aligning the codebase with the "type": "module" configuration.
  • Better Performance: Turbopack and other modern tools can optimize ES Modules more effectively.
  • Fixed Constructor Error: Classes are now correctly resolved during inheritance, eliminating the "not a constructor" error.
