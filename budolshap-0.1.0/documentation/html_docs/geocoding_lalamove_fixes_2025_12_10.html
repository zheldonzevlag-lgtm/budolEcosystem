Geocoding & Lalamove Integration Fixes

Date: December 10, 2025

Author: Jon Galvez

Context: Resolving critical issues with store address geocoding, Lalamove pickup location accuracy, and map tracking UI display.

Table of Contents

  • Issues & Blockers

  • Recommendations

1. Issues & Blockers Identified

🚨 Issue 1: Lalamove Using Legacy Store Address

Description: Even when a seller updated their store address via the new "Address Management" modal, the Lalamove shipping calculator and booking system continued to use the old, legacy store.address string (e.g., "0334, Malate...").

Impact: Incorrect pickup locations, potentially wrong shipping fees, and drivers arriving at old addresses.

🚨 Issue 2: Geocoding Failures with "NCR" Prefix

Description: Addresses containing "NCR - National Capital Region" (generated by the new address selector) were causing the Nominatim geocoding service to fail or return zero results.

Impact: Stores were falling back to the default "Glorietta 4" pickup location, triggering "Using Default Pickup" warnings.

⚠️ Issue 3: Tracking Map UI Issues

Description: The embedded Lalamove tracking map was too small (400px), cramped, and often showed a "Delivery Completed" summary page instead of the map. Users had no way to view the map in full screen.

Impact: Poor user experience for buyers tracking their orders.

2. Implemented Solutions

✅ Fix 1: Prioritize Structured Addresses in Lalamove Logic

Action: Updated both the Cart Widget (`LalamoveWidget.jsx`) and the Booking API (`/api/shipping/lalamove/book/route.js`) to prioritize the new structured StoreAddress model.

Logic:

  1. Check if a Default address exists in the StoreAddress table.
  2. If found, construct the full address from its components (Street, Barangay, City, etc.) and use its specific coordinates.
  3. Only fallback to the legacy `store.address` string if no structured address is found.

✅ Fix 2: Robust Geocoding Fallback

Action: Enhanced the geocoding API to handle "NCR" prefixes intelligently.

Logic: If the initial geocode attempt fails and the address contains "NCR - National Capital Region", the system automatically retries the geocode with that prefix stripped out. This significantly increases success rates.

✅ Fix 3: Map UI Enhancements

Action: Overhauled the LalamoveTracking.jsx component.

  • Height Increased: Map container height increased from 400px to 600px.
  • "View Full Screen" Button: Added a prominent button to open the tracking link in a new tab, bypassing iframe limitations.
  • Improved Overlay: The "Package Delivered!" overlay now appears reliably for both `DELIVERED` and `COMPLETED` statuses.

3. Technical Details

Refactored Files

  • components/LalamoveWidget.jsx: Added logic to fetch and use `store.addresses[0]` (default) for pickup points. Added warning UI for fallback usage.
  • app/api/products/[productId]/route.js: Added `address: true` to store selection to ensure legacy fallback relies on actual data.
  • app/api/stores/[storeId]/geocode/route.js: Added retry mechanism for strings with "NCR" prefix.
  • app/api/shipping/lalamove/book/route.js: Complete logic overhaul to fetch `StoreAddress` before creating Lalamove orders.
  • components/LalamoveTracking.jsx: UI redesign, height adjustment, and external link addition.

// Logic to prioritize new structured address const storeWithAddresses = await prisma.store.findUnique({ where: { id: order.storeId }, include: { addresses: { where: { isDefault: true }, take: 1 } } }); const defaultAddress = storeWithAddresses?.addresses?.[0]; if (defaultAddress) { // Construct address from structured fields pickupAddress = [defaultAddress.street, defaultAddress.city, ...].join(', '); pickupCoordinates = { lat: defaultAddress.latitude, lng: defaultAddress.longitude }; } else { // Fallback to legacy pickupAddress = order.store.address; }

4. Recommendations for Future Enhancement

  • Migrate Legacy Data: Create a migration script to move all existing stores with only legacy `address` strings to the new `StoreAddress` table. This will eliminate the need for complex fallback logic in the future.
  • Validate Coordinates on Save: When a user adds/edits an address, trigger the geocoding validation immediately in the UI (or API) and prevent saving if coordinates cannot be found. This prevents "silent failures" later during booking.
  • Store "Verified" Flag: Add a `isVerified` boolean to the `StoreAddress` model. Only allow usage of verified addresses for shipping pickup to ensure driver success.
  • Lalamove Webhook Improvements: Ensure the webhook handler (`/api/webhooks/lalamove`) strictly updates the `shipping.status` field and logs every state change for better debugging of delivery flows.
