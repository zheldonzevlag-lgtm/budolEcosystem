Implementation Summary

Prisma Schema Synchronization & Return Request Fix

Overview

The PrismaClientValidationError encountered when creating a return request was due to the isEscrowLocked field being missing in the schema.local.prisma file used for local development. This caused the generated Prisma client to be unaware of this field.

Actions Taken

  • Schema Synchronization: Updated schema.local.prisma and schema.mysql.prisma to match the latest Return, Order, User, Store, and Wallet models from schema.prisma.
  • Fields Added:
    • Return: isEscrowLocked, type, refundAmount, sellerAction, sellerReason, trackingNumber, deadline, arbitrationNotes.
    • Order: isGuaranteeExtended, paymentProof.
    • User: buyerReturnRate, codUnpaidCount.
    • Store: nonFulfilmentRate, penaltyPoints.
    • Wallet: lockedBalance.
  • Enum Synchronization: Added missing enums (ReturnType, ReturnSellerAction) and updated existing ones (OrderStatus, ReturnStatus, PaymentMethod).
  • Database Sync: Successfully pushed the updated schema to the local database using npx prisma db push.

Issues Encountered

EPERM Error during Client Generation

When running npx prisma generate, an EPERM error occurred. This is because the Next.js development server is currently using the Prisma client binary, preventing it from being replaced.

Error: EPERM: operation not permitted, rename ... -> ...query_engine-windows.dll.node

Required Next Steps

To fully resolve the issue, please follow these steps:

  1. Stop the Development Server: Close the terminal running npm run dev or next dev.
  2. Regenerate Prisma Client: Run the following command: 
    npx prisma generate --schema=prisma/schema.local.prisma
  3. Restart the Server: Run npm run dev again.

Verification

Once the client is regenerated, the isEscrowLocked argument will be recognized by Prisma, and return requests will be created successfully with escrow funds locked.
