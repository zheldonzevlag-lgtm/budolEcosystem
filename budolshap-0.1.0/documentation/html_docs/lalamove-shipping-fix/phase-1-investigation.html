Lalamove Shipping Fee Discrepancy Fix

Implementation Plan - Phase 1: Investigation & Root Cause Analysis

Summary Phase 1: Investigation

Executive Summary

This phase focuses on identifying the root cause of the reported discrepancy where shipping fees show a difference (specifically 73 vs 102). The investigation covers geocoding precision, API quote handling, and data persistence layers.

Identified Issues & Hypotheses

Issue Area Description Hypothesis
Geocoding Precision Address to coordinate conversion using Nominatim OSM. Low-precision coordinates or address mismatches lead to different distance calculations in Lalamove v3 API.
Quote Storage Saving quotationId and priceBreakdown in Prisma Order model. The stored price might be excluding certain dynamic surcharges (e.g., peak hour, weight) that are present in the final booking.
UI vs Backend Logic LalamoveWidget.jsx (frontend) vs shippingService.js (backend). Discrepancies in how stops are formatted or how the serviceType is requested between the widget and the final order creation.

Technical Analysis & Findings

1. Geocoding Precision & Consistency

The system relies on Nominatim (OpenStreetMap) for geocoding. While functional, Nominatim's coordinates may vary from Lalamove's internal geocoding system, leading to distance calculation discrepancies.

  • Finding: Store addresses are geocoded once and cached in the database (prisma.store.update in /api/stores/[storeId]/geocode).
  • Finding: Delivery addresses use a multi-step geocoding process in LalamoveWidget.jsx with a 500ms debounce.
  • Risk: Small variations in coordinates (even 4-5 decimal places) can trigger different pricing tiers in Lalamove's distance-based calculation.

2. Price Breakdown & Surcharges

Analysis of services/lalamove.js and shippingService.js shows how the price is extracted from the Lalamove API response.

// Current extraction logic in lalamove.js
return {
    quotationId: result.data.quotationId,
    priceBreakdown: result.data.priceBreakdown, // Contains total, totalExcludeTax, totalIncludeTax
    distance: result.data.distance,
    // ...
};

The discrepancy (73 vs 102) strongly suggests the exclusion of certain surcharges or taxes in the initial display that are later added during the final booking process.

  • Hypothesis: The UI might be displaying totalExcludeTax while the booking uses totalIncludeTax, or a dynamic surcharge (like "Priority Delivery" or "Peak Surcharge") is missing from the initial quote.

3. Quote Refresh Logic

In lib/services/shippingService.js, there is logic to refresh quotes if they are considered "stale".

const isQuoteExpired = storedQuote?.expiresAt && new Date(storedQuote.expiresAt) < now;
const isQuoteTooOld = quoteAge > 30; // 30 minutes
const needsRefresh = !storedQuote || isQuoteExpired || isQuoteTooOld;

If a user waits on the checkout page for more than 30 minutes, the backend will request a NEW quote during the booking process. This new quote might have different pricing due to changing demand (Peak Surcharge).

Root Cause Candidates

  1. Coordinate Mismatch: Slight differences between coordinates used in the Widget vs. those stored in the Order.
  2. Tax/Surcharge Omission: Displaying totalExcludeTax instead of totalIncludeTax in the UI.
  3. Dynamic Pricing: Quote expiration leading to a fresh (and more expensive) quote at the time of booking.

Validation Criteria for Phase 2

  • PASS: Quote price in UI matches the final Lalamove booking amount exactly.
  • PASS: Coordinates sent to Lalamove are consistent between the Widget and the Order.
  • PASS: All surcharges and taxes are explicitly accounted for in the priceBreakdown.
