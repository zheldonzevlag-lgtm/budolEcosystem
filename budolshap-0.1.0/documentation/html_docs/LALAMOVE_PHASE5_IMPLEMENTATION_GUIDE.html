2 Issue: Generic Error Messages

Problem Description

Lalamove V3 API errors were being displayed as generic "Shipping service error. Please try again." messages instead of specific, actionable feedback.

Root Cause

The handleApiError() function in services/lalamove.js was only parsing legacy error formats:

Original Code (Incorrect)
handleApiError(error) {
    const errorCode = error.response?.data?.error?.code;
    const errorMessage = error.response?.data?.error?.message;
    // This only works for legacy format, not V3
}

Resolution

Updated the error handler to support multiple error formats with proper fallback logic:

services/lalamove.js - Updated Error Handler
handleApiError(error) {
    const responseData = error.response?.data;
    let errorCode, errorMessage;

    // Handle Lalamove V3 error format: { errors: [{ id: 'ERR_CODE', message: '...' }] }
    if (responseData?.errors && Array.isArray(responseData.errors) && responseData.errors.length > 0) {
        errorCode = responseData.errors[0].id;
        errorMessage = responseData.errors[0].message;
    } else if (responseData?.error) {
        // Handle legacy or alternative format
        errorCode = responseData.error.code;
        errorMessage = responseData.error.message;
    } else {
        // Fallback
        errorCode = responseData?.message || 'UNKNOWN_ERROR';
        errorMessage = responseData?.message || error.message;
    }

    const errorMap = {
        'ERR_NO_DRIVER': 'No drivers available at this time. Please try again later.',
        'ERR_INVALID_ADDRESS': 'Invalid delivery address. Please check and try again.',
        'ERR_OUTSIDE_SERVICE_AREA': 'Delivery address is outside our service area.',
        'ERR_QUOTA_EXCEEDED': 'Service temporarily unavailable. Please try again in a few minutes.',
        'ERR_INVALID_QUOTE': 'Quote has expired. Please request a new quote.',
        'ERR_CANCELLATION_NOT_ALLOWED': 'Order cannot be cancelled at this stage.',
        'ERR_INSUFFICIENT_CREDIT': 'Insufficient credit in shipping account. Please contact support.',
        'UNAUTHORIZED': 'Invalid API credentials. Please check your API key and secret.'
    };

    const friendlyMessage = errorMap[errorCode] || errorMessage || 'Shipping service error. Please try again.';

    const err = new Error(friendlyMessage);
    err.code = errorCode;
    err.originalError = responseData;
    return err;
}
✅ Result: Users now see specific, actionable error messages like "Insufficient credit in shipping account. Please contact support."
3 Issue: API Route Error Handling

Problem Description

The booking API route (app/api/shipping/lalamove/book/route.js) was not properly catching and returning custom service errors, resulting in generic 500 responses.

Resolution

Enhanced the catch block to handle custom service errors before falling back to generic error handling:

app/api/shipping/lalamove/book/route.js - Enhanced Error Handling
} catch (error) {
    console.error('Lalamove booking error:', error);

    // Handle custom Lalamove service errors (thrown by services/lalamove.js)
    if (error.code) {
        return NextResponse.json(
            {
                error: 'Failed to create booking',
                message: error.message,
                code: error.code,
                details: error.originalError
            },
            { status: 400 }
        );
    }

    // Handle raw Axios errors (fallback)
    if (error.response?.data) {
        const lalamoveError = error.response.data;
        return NextResponse.json(
            {
                error: 'Failed to create booking',
                message: lalamoveError.message || 'Unknown error from Lalamove',
                details: lalamoveError
            },
            { status: error.response.status || 500 }
        );
    }

    // Generic error
    return NextResponse.json(
        {
            error: 'Failed to book delivery',
            message: error.message
        },
        { status: 500 }
    );
}
✅ Result: API now returns proper 400 status codes with detailed error information for client-side handling.
4 Understanding: Sandbox Driver Assignment Behavior

Observation

Orders remain in ASSIGNING_DRIVER status indefinitely with "Finding you nearby drivers..." message.

⚠️ This is NOT a bug! This is expected behavior in the Sandbox environment.

Explanation

The Lalamove Sandbox environment does not have real drivers. Orders will remain in ASSIGNING_DRIVER status because:

  • No actual drivers are available in sandbox
  • The environment is designed to test order creation, not full delivery lifecycle
  • This status confirms the order was successfully created

Production Behavior

In production with real API credentials, the order lifecycle will be:

  1. ASSIGNING_DRIVER - System searching for available driver
  2. ON_GOING - Driver accepted and heading to pickup
  3. PICKED_UP - Driver collected the package
  4. COMPLETED - Package delivered successfully
💡 Verification: The fact that orders reach ASSIGNING_DRIVER status proves the integration is working correctly!

require('dotenv').config({ path: '.env.local' });
const { getShippingProvider } = require('./services/shippingFactory');

const LOG_FILE = path.join(__dirname, 'lalamove-test-log.txt');

function log(message, ...args) {
    const timestamp = new Date().toISOString();
    const formattedMessage = `[${timestamp}] ${message} ${args.map(a => typeof a === 'object' ? JSON.stringify(a, null, 2) : a).join(' ')}\n`;
    console.log(message, ...args);
    fs.appendFileSync(LOG_FILE, formattedMessage);
}

// Clear log file

🔑 Key Learnings

1. Lalamove V3 API Error Format

Lalamove V3 uses a different error structure than many APIs:

  • V3 Format: { errors: [{ id: 'ERR_CODE', message: '...' }] }
  • Legacy Format: { error: { code: 'ERR_CODE', message: '...' } }

Always implement parsers that handle multiple formats with proper fallbacks.

2. Sandbox Wallet Requirements

Unlike some testing environments that use mock credits, Lalamove Sandbox requires:

  • Real wallet balance for order creation
  • Actual deductions from balance (₱82 per test order)
  • Manual top-up via Partner Portal

Implication: Budget for sandbox testing costs when planning integration timeline.

3. Sandbox Driver Assignment Behavior

Orders in sandbox will remain in ASSIGNING_DRIVER status indefinitely because:

  • No real drivers exist in sandbox environment
  • This is expected and correct behavior
  • Reaching this status proves integration is working

Implication: Don't wait for driver assignment in sandbox tests. Focus on verifying order creation success.

4. StopId Importance

The stopId values from the quotation response are critical:

  • Must be stored when quote is received
  • Must be passed to order creation exactly as received
  • Cannot be regenerated or modified

Implication: Always store the complete quote response, not just the quotationId.
