Synchronization Report

Lalamove Integration, Order Rebooking & Microservice Architecture Prep

Executive Summary

This session focused on hardening the integration between the BudolShap Order System and the Lalamove Delivery Service. Crucial synchronization gaps were identified where local order states became desynchronized from the actual courier status. Additionally, the user interface for handling "Delivery Canceled" events and subsequent "Rebooking" flows was refined to provide a seamless experience for buyers.

  • Status Sync: Implemented deep inspection of Lalamove stop data to detect deliveries even when the main API status lags.
  • Rebooking Flow: Corrected database logic to ensure rebooked orders instantly exit the "Cancelled" tab and return to "To Receive".
  • Stability: Patched client-side crashes caused by malformed coordinate data from the tracking API.

Issue Resolution Log

Issue #1

Lalamove Status Lag

Problem: The internal `order.status` was updating slower than the actual Lalamove delivery, causing the UI to show "In Transit" when the package was already delivered.

Implementation Fixes:

  • Frontend (OrderProgressTracker): Prioritized Lalamove specific statuses (COMPLETED, DELIVERED, SUCCEEDED) over internal status for the progress bar.
  • Deep Check (lalamove.js): Implemented logic to inspect the `stops` array in the API response. If the final stop has a POD (Proof of Delivery) or 'COMPLETED' status, we override the top-level status to 'COMPLETED'.
Issue #2

Rebooked Orders Stuck in "Cancelled"

Problem: After a delivery failed and was successfully rebooked, the order remained in the "Cancelled" tab instead of moving back to "To Receive".

Implementation Fixes:

  • Service Layer (shippingService.js): Updated `bookShipping` and `trackShipping` to explicitly clear `failureReason`, `failedAt`, `cancellationReason`, and `cancelledAt` when a new valid booking or status is confirmed.
  • Database Query (ordersService.js): The initial fix to clear flags wasn't enough because the database filter not: Prisma.DbNull still matched explicit null (JSON null) values.
  • Final Fix: Updated the Prisma query to explicitly exclude both `DbNull` (database null) AND `JsonNull` (JSON null).

Enhancement

"Delivery Canceled" Visibility

Requirement: Users couldn't see orders with failed deliveries in the "Cancelled" tab if the payment status was still valid.

Implementation Fixes:

  • Frontend: Updated `orders/page.jsx` to send a new flag `isCancelledTab=true` when that tab is active.
  • Backend: Updated `ordersService.js` to use an `OR` condition when this flag is present: (Status=CANCELLED) OR (Payment=CANCELLED) OR (Shipping=FAILED).
Critical

Client-Side Crash (Order Details)

Problem: The Order Details page crashed with an Application Error due to a TypeError when formatting coordinates.

Implementation Fixes:

  • Component (LalamoveTracking.jsx): The API sometimes returned latitude/longitude as strings. Calling .toFixed() on a string threw an error.
  • Fix: Added safety casting Number(lat).toFixed(4) to ensure robustness.
Issue #5

Hook Reference Error

Problem: A `ReferenceError: filters is not defined` occurred in the `useRealtimeBuyerOrders` hook.

Implementation Fixes:

  • Hook: Corrected the variable scope issues. Passed `isCancelledTab` as a top-level argument to the hook instead of trying to access it from an undefined object.

Microservice Architecture Readiness

The changes made in this session move the system closer to a microservices-ready state by enforcing stricter separation of concerns:

  1. Service Isolation: `shippingService` now autonomously manages the shipping state lifecycle, including error recovery (rebooking), without the main order logic needing to know the specifics.
  2. Robust Data Handling: The introduction of "Deep Checks" and JSON null safety means the system is less reliant on perfect data synchronization and can handle eventual consistency better.
  3. Flag-Based Filtering: Moving the filtering logic to specific flags (`isCancelledTab`) allows for more flexible API contracts that can be easily ported to a standalone Order Service in the future.
