Testing Protocol

To ensure the fix works across all scenarios, the following testing protocol was established:

1. Webhook Simulation

Simulate Lalamove webhook POST requests using the booked ID in the payload and metadata: { type: 'return' }. Verify Return status in DB changes from APPROVED to SHIPPED.

2. Manual Sync Verification

Trigger the /api/orders/[orderId]/sync endpoint manually via the UI "Sync Status" button. Confirm that the system correctly identifies the return booking and updates the UI via the return-updated event.

3. Background Service Audit

Monitor logs for shippingOrderUpdater.js to ensure periodic syncs are picking up return statuses without manual intervention.

Success Criteria

  • UI Consistency: "BOOKED" returns transition to "SHIPPED" and "RECEIVED" in both Store and User dashboards without page refresh.
  • Data Integrity: Main Order.status only updates to IN_TRANSIT or COMPLETED when appropriate for return flows, without overwriting original delivery data.
  • Error Handling: Failed Lalamove bookings (CANCELLED) reset the return status to APPROVED allowing for re-booking.

Rollback Plan

In case of critical failures:

  1. Revert changes to webhook/route.js and sync/route.js.
  2. The system will default back to tracking the main order shipping, which will prevent return updates but maintain core order stability.
