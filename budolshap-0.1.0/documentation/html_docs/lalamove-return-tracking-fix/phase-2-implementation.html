Core Logic: Return-Shipping Isolation

The fundamental fix involved creating a reliable way to detect if a Lalamove update belongs to a return shipment. This was achieved using the isReturnSync pattern.

services/shippingOrderUpdater.js 
// Detecting return sync context
const activeReturn = order.returns?.[0];
const isReturnSync = !!(activeReturn && activeReturn.returnShipping?.bookingId);

// Selecting the correct data source
const shippingData = isReturnSync ? activeReturn.returnShipping : order.shipping;

Webhook Enhancement

The webhook route was modified to prioritize metadata and check both the trackingNumber and booked ID (bookingId).

app/api/shipping/lalamove/webhook/route.js 
const isReturnBooking = (data.metadata?.type === 'return') || 
                       (activeReturn && (
                           activeReturn.trackingNumber === lalamoveOrderId ||
                           activeReturn.returnShipping?.bookingId === lalamoveOrderId
                       ));

if (isReturnBooking && activeReturn) {
    // Update Return record directly
    await prisma.return.update({
        where: { id: activeReturn.id },
        data: {
            status: mappedStatus,
            returnShipping: updatedShipping
        }
    });
    
    // Trigger Realtime Events
    await triggerRealtimeEvent(`store-${storeId}`, 'return-updated', eventData);
}

Status Mapping Strategy

We implemented a robust mapping between Lalamove statuses and Budolshap return statuses:

  • PICKED_UP → SHIPPED
  • COMPLETED → RECEIVED
  • CANCELLED/EXPIRED → APPROVED (reset to previous valid state)

Realtime Update Propagation

To ensure the UI updates without refresh, we added dual-channel event triggering in both the webhook and manual sync routes, targeting both the store-{id} and user-{id} channels.
