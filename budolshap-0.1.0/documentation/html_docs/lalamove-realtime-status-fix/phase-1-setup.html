Phase 1: Investigation & Environment Setup

Completed

Executive Summary

This phase focused on analyzing the existing Lalamove webhook integration and real-time event system to identify why automatic status updates were failing while manual sync remained functional.

Technical Objectives

  • Map the data flow from Lalamove Webhook to UI Update.
  • Analyze triggerRealtimeEvent logic and its dependency on SystemSettings.
  • Inspect environment-specific behaviors for webhook execution.

Key Components Analyzed

The following files were identified as critical to the synchronization flow:

  • app/api/webhooks/lalamove/route.js: Webhook entry point.
  • lib/realtime.js: Server-side event triggering logic.
  • hooks/useRealtimeOrders.js: Client-side subscription logic.

Diagnostic Findings

1. Webhook Lookup Logic

The webhook finds orders using the bookingId stored in a JSON column:

const order = await prisma.order.findFirst({
    where: {
        shipping: {
            path: ['bookingId'],
            equals: lalamoveOrderId
        }
    }
});

Verification confirms this logic is correct, but relies on lalamoveOrderId being present in the webhook payload.

2. Real-time Trigger Fallback

The getSettings() function in lib/realtime.js uses a 1-minute cache and falls back to POLLING mode if the database is unreachable. This could cause webhooks to silently skip Pusher events during high-traffic bursts.

Risk Assessment

Risk Impact Mitigation
DB Connection Limits High Implement connection pooling or retries.
Inconsistent Status Mapping Medium Normalize status labels across all triggers.
