Testing Strategy

To validate the fix, we will use a combination of automated webhook simulations and manual UI observation.

1. Webhook Simulation Script

A script will be used to send POST requests to the webhook endpoint with mock Lalamove payloads.

// Example Simulation Command
curl -X POST http://localhost:3000/api/webhooks/lalamove \
-H "Content-Type: application/json" \
-d '{
    "eventType": "ORDER_STATUS_CHANGED",
    "orderId": "LALA-123456",
    "status": "PICKED_UP",
    "data": {
        "order": { "status": "PICKED_UP" }
    }
}'

2. Test Cases

TC-01: Status Change Notification

Input: Webhook with PICKED_UP status.

Expected: UI orders list refreshes, order status changes to IN_TRANSIT.

TC-02: Driver Assignment Update

Input: Webhook with DRIVER_ASSIGNED event.

Expected: Driver info appears in order details, status remains or moves to PROCESSING.

TC-03: Failed Real-time Fallback

Input: Webhook received while Pusher is misconfigured.

Expected: Order status updates in DB, webhook returns 200, error is logged in WebhookEvent table.

Success Criteria

  • UI refreshes within 2 seconds of webhook receipt.
  • Webhook status in WebhookEvent table shows SUCCESS.
  • Pusher debug console shows events reaching the correct channel.
