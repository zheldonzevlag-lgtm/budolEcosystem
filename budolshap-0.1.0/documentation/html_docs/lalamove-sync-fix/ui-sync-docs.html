UI Synchronization: Auto-Sync & Polling

Problem: The "Passive UI" Limitation

Previously, the Seller UI only updated its status if a webhook was received or if the Buyer specifically viewed the order detail page. If the webhook failed or was delayed, the Seller UI appeared "stuck".

Solution: Proactive Client-Side Sync

We implemented a useEffect hook in both the Seller's Store Orders page and the Buyer's Order List page to proactively poll the synchronization endpoint for active Lalamove orders.

Implementation (Seller UI)

useEffect(() => {
  if (!orders || orders.length === 0) return;

  // 1. Filter for orders that require active tracking
  const activeLalamoveOrders = orders.filter(order => 
    order.shipping?.provider === 'lalamove' && 
    order.shipping?.bookingId &&
    ['ORDER_PLACED', 'PROCESSING', 'SHIPPED', 'PICKED_UP', 'IN_TRANSIT'].includes(order.status)
  );

  if (activeLalamoveOrders.length === 0) return;

  // 2. Define sync function
  const syncAll = async () => {
    for (const order of activeLalamoveOrders) {
      await fetch(`/api/orders/${order.id}/sync-lalamove`, { method: 'POST' });
    }
    mutate(); // Refresh SWR cache
  };

  // 3. Set 15-second interval
  const interval = setInterval(syncAll, 15000);
  return () => clearInterval(interval);
}, [orders, mutate]);

Polling Strategy

  • Interval: 15 seconds. This provides a balance between responsiveness and server load.
  • Targeting: Only polls orders in transient statuses (e.g., IN_TRANSIT). Does not poll DELIVERED or CANCELLED orders.
  • Refresh: Calls mutate() from SWR to update the local UI state immediately after the server-side sync completes.

Benefits

  • Redundancy: Status sync works even if Lalamove webhooks are down.
  • Independence: Seller UI updates correctly regardless of what page the Buyer is on.
  • Consistency: Both Buyer and Seller see identical status information within a 15-second window.
