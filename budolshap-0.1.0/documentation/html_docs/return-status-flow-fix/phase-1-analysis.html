Phase 1: Analysis & Requirements

Identifying the root cause and defining the solution for premature status transitions.

Problem Identification

The user reported that upon booking a return courier via Lalamove, the return status immediately transitioned to SHIPPED. This was inaccurate because the package had not yet been picked up by the driver, let alone transitioned through the intermediate stages of the Lalamove workflow.

"The status immediately shows shipped even if these status haven't been happening. I still haven't changed the status in Lalamove portal to picked up..."

Root Cause Analysis

Auditing the shippingService.js file revealed that the status update logic was hardcoded to set the return status to SHIPPED immediately after a successful API call to Lalamove's booking endpoint.

Location: lib/services/shippingService.js (Line 167)

Existing logic:

status: 'SHIPPED', // Incorrectly setting to SHIPPED immediately

Proposed Solution

To fix this, we proposed adding a new status: BOOKED. This status represents the state where the courier has been requested but pickup has not occurred.

  • New Status: BOOKED (Waiting for Pickup)
  • Initial State: APPROVED
  • Booking Completion: Transition to BOOKED
  • Lalamove Webhook (PICKED_UP): Transition to SHIPPED
  • Lalamove Webhook (COMPLETED): Transition to RECEIVED

Risk Assessment

Risk Impact Mitigation
Database Enum Mismatch High Ensure schema.prisma is updated and client is regenerated.
UI Filtering Issues Medium Update all seller and admin return management pages to include the BOOKED filter.
Webhook Logic Gaps High Ensure the webhook correctly identifies BOOKED returns for status updates.
