Implementation Plan: Provider-Agnostic Delivery Logic

Overview

This plan outlines the steps to refactor the frontend delivery status logic. The goal is to make the "Booked", "In Transit", and "Delivered" tabs robust enough to handle any third-party delivery provider (e.g., Lalamove, J&T, Grab), not just Lalamove.

1. Current Limitation CURRENT

The current getDeliveryStatus function explicitly checks for the string 'lalamove'. If a new provider is added, the system will treat it as 'other' or 'needs_booking' even if it is valid.

const getDeliveryStatus = (order) => { // ... checks for paid, delivered, cancelled ... // ❌ Hardcoded Dependency if (order.shipping?.provider === 'lalamove') { if (order.shipping?.bookingId) { /* ... */ } } // ... }

2. Proposed Solution PROPOSED

We will abstract the logic to check for the presence of shipping details rather than a specific provider name. This makes the system "Provider Agnostic".

const getDeliveryStatus = (order) => { if (!order.isPaid) return 'unpaid'; if (order.status === 'DELIVERED') return 'delivered'; // ✅ GENERIC CHECK: Does it have a valid provider and booking? if (order.shipping?.provider && order.shipping?.bookingId) { if (order.status === 'SHIPPED') return 'in_transit'; return 'booked'; } // Needs booking if paid but no shipping info if (order.isPaid && !order.shipping?.bookingId) return 'needs_booking'; return 'other'; }

3. Implementation Steps

1 Refactor `getDeliveryStatus`

Modify app/store/orders/page.jsx to remove the `lalamove` specific check unless necessary for provider-specific API calls. All status determination will be based on data structure (existence of `bookingId`).

2 Dynamic UI Labels

Update the "Review & Book" modal and button labels. Instead of hardcoding "Lalamove", we will display the provider name dynamically if available, or default to "Delivery".

  • Change: "Book Lalamove Delivery" → "Book {Provider} Delivery"
  • Change: "Lalamove Details" → "Delivery Details"

3 Verify Backend Compatibility

Ensure that any new provider we add (in the backend factory) writes the provider and bookingId fields to the shipping JSON column in the database. No backend code changes are needed right now, as the current structure already supports this.

4. Benefits

  • Future Proof: Add J&T, Grab, or generic "Rider" later without touching the frontend logic.
  • Consistency: "Booked", "In Transit", and "Delivered" tabs will work automatically for any provider that updates the order status.
  • Scalability: Reduce maintenance overhead when expanding shipping options.
