Phase 2: Implementation

Overview

The implementation involved identifying every instance where Lalamove API data was used in a Prisma update call and ensuring that no field could be undefined.

Key Changes

1. Shipping Service (Core Logic)

Modified lib/services/shippingService.js to fix return booking and tracking updates.

// Fixed returnShipping update
customerOrderId: booking.customerOrderId || null,
shareLink: booking.shareLink || null,

// Fixed order shipping update
trackingUrl: booking.trackingUrl || null,
shareLink: booking.shareLink || null,

// Fixed driver info property mismatch
driver: trackingInfo.driverInfo || null,

2. Webhook Handlers

Modified app/api/shipping/lalamove/webhook/route.js and app/api/webhooks/lalamove/route.js to handle driver and location data safely.

driverInfo = {
    name: driverData.name || driverInfo.name || null,
    phone: driverData.phone || driverInfo.phone || null,
    plateNumber: driverData.plateNumber || driverInfo.plateNumber || null,
    photo: driverData.photoUrl || driverData.photo || driverInfo.photo || null,
    vehicleType: driverData.vehicleType || driverInfo.vehicleType || null,
    rating: driverData.rating || driverInfo.rating || null
};

3. Manual Sync & Background Updaters

Updated app/api/orders/[orderId]/sync-lalamove/route.js and services/shippingOrderUpdater.js to ensure consistent behavior across all sync methods.

Technical Decisions

  • Null Coalescing: Used || null instead of ?? null because Lalamove sometimes returns empty strings which we also want to treat as null for database consistency in these specific JSON fields.
  • JSON Field Preservation: When updating partial JSON data, we now consistently spread existing data (...currentShipping) and then overwrite with sanitized new values.
  • Driver Info Mapping: Fixed a property name mismatch where some code expected driver and others driverInfo. Standardized on driverInfo while maintaining driver as a fallback for legacy support.

Implementation complete across 5 critical files.
