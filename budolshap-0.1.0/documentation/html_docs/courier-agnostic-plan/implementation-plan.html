🏗️ Architecture Overview

The BudolShap platform is evolving towards a Centralized Configuration Management system. By migrating environment variables (.env) to a secure Admin UI, we enable real-time platform adjustments without requiring code redeployments or server restarts.

  • Dynamic Configuration: Switch payment gateways (Paymongo/Xendit) or tracking providers instantly.
  • Environment Agnostic: Manage local, staging, and Vercel-specific settings from a single dashboard.
  • Secure Storage: Secrets are encrypted at rest using industry-standard AES-256 before being saved to the database.

🔔 Multi-Channel Notification System

A comprehensive notification strategy for Buyers, Sellers, and Admins to ensure transparency at every stage of the shipment.

1. Email Notifications (Nodemailer + SMTP / Brevo)

  • Buyer: Tracking number issued, In Transit alert, Out for Delivery, and Successful Delivery.
  • Seller: Order booked confirmation, Pickup successful, and Delivery failure alerts (for rebooking).
  • Admin: Aggregator API credit warnings, failed webhook processing alerts.
  • Brevo Integration: Future-proof integration for transactional and marketing emails with advanced tracking and templates.

2. Real-time Notifications (Pusher)

Note: Pusher is used for Real-time UI Updates (toasts, live maps, badge increments) rather than sending actual Emails or SMS.

  • UI Toasts: Immediate popup when a courier status changes while the user is on the site.
  • Dashboard Sync: Update the "In Transit" and "Delivered" counts in real-time.

3. SMS Notifications (Semaphore / Zerix / Twilio)

  • Buyer: "Out for Delivery" SMS with the courier's name and contact number.
  • Zerix Text Integration: Leveraging Zerix Text for reliable PH-local messaging, bulk updates, and structured two-way threads.
  • Reliability: SMS is the fallback for buyers without consistent internet access.

🔌 Aggregator Integration Strategy

Instead of building separate integrations for 50+ local couriers, we will leverage tracking aggregators:

1. Ship24 Integration

  • Role: Global tracking and logistics aggregator.
  • Capability: Supports J&T Express, NinjaVan, and LBC natively.
  • Feature: Webhook support for real-time status updates from couriers.

2. TrackingMore Integration

  • Role: Specialized e-commerce tracking API.
  • Capability: Strong support for Southeast Asian couriers including Shopee and Lazada express services.
  • Feature: Auto-detection of couriers based on tracking number format.

⚙️ Centralized Platform Settings

The Admin Settings will be reorganized into logical categories to manage all platform dependencies currently residing in .env files.

💳 Payments

  • Paymongo: Public/Secret Keys.
  • Xendit: API Key, Callback Tokens.

📦 Shipping

  • Lalamove: API Key, Secret, Region.
  • Aggregators: Ship24 / TrackingMore.

🖼️ Media & Storage

  • Cloudinary: Cloud Name, API Key/Secret.

📧 Email (SMTP/API)

  • Gmail: SMTP User, App Password.
  • Brevo: API Key, Transactional templates.

🗄️ Infrastructure

  • Database: Local DB strings, Vercel Postgres URLs.
  • Vercel: Project ID, Team ID, Env Vars sync.

🔒 Security & Secret Management

Moving secrets from .env to the database requires a robust security layer:

  • AES-256 Encryption: All sensitive keys (Secret Keys, Passwords) will be encrypted before being stored in the database.
  • Master Key: A single ENCRYPTION_KEY will remain in the server's environment to decrypt values at runtime.
  • Audit Logs: Track which admin user modified specific configuration values.

📅 Implementation Phases & Timeframe

Phase 1: Core Service Layer

Duration: 3-5 Days

  • Implement Ship24Provider and TrackingMoreProvider classes.
  • Update ShippingFactory to register new providers.
  • Create unified data normalization layer.
Phase 2: Admin Settings UI

Duration: 2-3 Days

  • Develop /admin/settings/shipping page.
  • Add secure API key management in backend.
  • Implement provider switching logic.
Phase 3: Webhook Handlers

Duration: 4-5 Days

  • Create /api/webhooks/ship24 and /api/webhooks/trackingmore endpoints.
  • Implement signature verification for security.
  • Map external status updates to real-time UI events.
Phase 4: Testing & UAT

Duration: 3 Days

  • End-to-end testing with sandbox tracking numbers.
  • Verify mobile/desktop timeline responsiveness.
  • Load testing for high-volume webhook updates.

Total Estimated Time: 12-16 Working Days
