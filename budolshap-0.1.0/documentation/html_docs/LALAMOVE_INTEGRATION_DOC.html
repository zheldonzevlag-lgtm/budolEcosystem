Lalamove Integration – Full Documentation

1. Overview

This document describes the end‑to‑end process for adding Lalamove as the primary shipping partner in the Budolshap application and provides a pattern for adding other Philippine carriers later.

The integration consists of four major layers:

  1. Core architecture (service layer, DB schema, provider interface)
  2. Frontend checkout flow
  3. Backend API endpoints & webhook handling
  4. Testing, security, and deployment

2. Core Architecture Setup

2.1 Provider‑agnostic Interface

Create services/shippingProvider.js that defines the contract each carrier must implement:

class ShippingProvider {
  async getQuote(payload) {}
  async createOrder(payload) {}
  async trackOrder(orderId) {}
  async cancelOrder(orderId) {}
}
module.exports = ShippingProvider;

2.2 Lalamove Service Wrapper

File: services/lalamove.js. It extends the interface and handles:

  • OAuth2 client‑credentials token acquisition
  • Quote request (/v2/quotes)
  • Order creation (/v2/orders)
  • Tracking (/v2/orders/:id)
  • Cancellation (/v2/orders/:id/cancel)

All HTTP calls should use axios with a retry interceptor and respect Lalamove's rate limits.

2.3 Database Extension

Update the Prisma Order model (or your ORM) to store shipping details:

model Order {
  id          Int      @id @default(autoincrement())
  // … existing fields …
  shipping    Json?
}

The JSON structure will be:

{
  "provider": "lalamove",
  "quoteId": "string",
  "bookingId": "string",
  "status": "pending|assigned|picked_up|delivered|canceled",
  "trackingUrl": "https://api.lalamove.com/...",
  "driverInfo": {"name":"","phone":""}
}

2.4 Service Factory

File: services/shippingFactory.js. Returns the correct provider implementation based on a string key.

const Lalamove = require('./lalamove');
function getProvider(name) {
  switch(name) {
    case 'lalamove': return new Lalamove();
    // future: case 'jnt': return new JNT();
    default: throw new Error('Unsupported shipping provider');
  }
}
module.exports = { getProvider };

3. Front‑end Checkout Flow

Step UI Action Backend Call
3.1 Add "Lalamove" radio button to the shipping selector. –
3.2 Open quote modal – collect address, weight, dimensions, pickup window. GET /api/shipping/lalamove/quote
3.3 Show price, ETA, any surcharges. –
3.4 User confirms – click "Confirm Shipping". POST /api/shipping/lalamove/book
3.5 Store bookingId & trackingUrl in order (hidden field). –
3.6 Display order summary with tracking link and optional cancel button. –
3.7 In "My Orders" page, embed live tracking widget (poll /track/:orderId). GET /api/shipping/lalamove/track/:orderId

4. Backend API Endpoints

4.1 Quote Endpoint

GET /api/shipping/lalamove/quote
Query Params: address, lat, lng, weight, dimensions, pickup_time
Response: {price, eta, currency, quoteId}

4.2 Booking Endpoint

POST /api/shipping/lalamove/book
Body: {address, weight, dimensions, pickup_time, quoteId}
Response: {bookingId, trackingUrl, status}

4.3 Tracking Endpoint

GET /api/shipping/lalamove/track/:orderId
Response: {status, driver:{name,phone}, location:{lat,lng}}

4.4 Cancel Endpoint

POST /api/shipping/lalamove/cancel/:orderId
Response: {canceled:true}

4.5 Webhook Receiver

POST /api/webhooks/lalamove
Headers: X-Lalamove-Signature (HMAC)
Body: {order_id, status, driver, location}
Action: Verify signature → update Order.shipping.status & driverInfo.

All endpoints should return proper HTTP status codes (200, 400, 401, 500) and JSON error messages.

5. Adding Additional Philippine Carriers (Future‑Proofing)

  • Create a new service file (e.g., services/jnt.js) that implements the same methods.
  • Register the provider in shippingFactory.js.
  • Extend the checkout UI to load carrier‑specific fields dynamically.
  • Add a new webhook route (or reuse the generic one) that distinguishes providers via a payload field.

6. Security, Rate‑Limiting & Idempotency

  • OAuth2 token: store LALAMOVE_CLIENT_ID and LALAMOVE_CLIENT_SECRET in .env. Refresh automatically before expiry.
  • Rate limiting: use a job queue (Bull/Redis) or a simple in‑memory token bucket to keep calls < 10 req/s.
  • Idempotency key: generate a UUID on the client side (client_reference_id) and pass it to Lalamove; repeat calls with the same key will not duplicate orders.
  • Webhook verification: compute HMAC‑SHA256 of the raw request body using LALAMOVE_WEBHOOK_SECRET and compare with the header.

7. Testing & Deployment Checklist

  1. Set LALAMOVE_ENV=sandbox in .env and obtain sandbox credentials.
  2. Write unit tests for each method in services/lalamove.js (mock HTTP responses).
  3. Create integration tests that simulate the full checkout flow (quote → book → webhook → track).
  4. Run the flow locally on http://localhost:3000 and verify DB updates.
  5. Add the required environment variables to Vercel (client id, secret, webhook secret).
  6. Deploy with a feature flag ENABLE_LALAMOVE=true and monitor logs.
  7. After successful sandbox testing, switch LALAMOVE_ENV=production and repeat a limited live order.
