🚚 Lalamove Phase 3 Integration - Complete Documentation

This document provides a comprehensive summary of Phase 3 (Frontend Integration & User Experience), detailing all features implemented, technical challenges encountered, solutions applied, and UI/UX refinements made.

1. Overview - What Was Accomplished

  • End-to-End Checkout Flow: Users can select "Same Day Delivery (Lalamove)" at checkout, view real-time shipping costs, and complete orders seamlessly.
  • Real-Time Quote Integration: Live delivery quotes fetched from Lalamove API based on pickup and delivery locations.
  • Smart Address Handling: Automatic geocoding for addresses missing coordinates using Nominatim API.
  • Store Dashboard Booking: Manual booking interface for store owners to confirm Lalamove deliveries.
  • Order Tracking: Real-time tracking links embedded in user orders, store dashboard, and admin panel.
  • Professional Branding: Lalamove logo integration with optimized sizing for maximum visibility.

2. Database Schema Updates

🔴 Issue: Missing GPS Coordinates in Address Model

The existing Address model only stored text fields (street, city, district), but Lalamove's API strictly requires GPS coordinates (latitude/longitude) for accurate delivery quotes and bookings.

🟢 Fix: Schema Enhancement

File Modified: prisma/schema.prisma

Changes:

  • Added latitude Float? field to Address model
  • Added longitude Float? field to Address model
  • Ran npx prisma db push to apply schema changes

Impact: Enables storage of precise GPS coordinates for accurate Lalamove delivery routing.

3. Frontend Components Implementation

3.1 LalamoveWidget Component

File: components/LalamoveWidget.jsx
Purpose: Handles quote fetching, geocoding, and UI display for Lalamove shipping
🔴 Issue: Missing Address Coordinates

User addresses in the database often lack latitude/longitude coordinates, causing quote API calls to fail.

🟢 Fix: Client-Side Geocoding Fallback

Implemented automatic geocoding using Nominatim (OpenStreetMap) API:

  • Checks if delivery address has coordinates
  • If missing, constructs full address string and geocodes it
  • Caches coordinates for subsequent requests
  • Provides user feedback during geocoding process
🔴 Issue: Unknown Pickup Location

The widget didn't know which store the items were coming from, leading to inaccurate quotes based on placeholder addresses.

🟢 Fix: Dynamic Store Address Fetching

Implementation:

  • Widget now accepts storeId prop from parent component
  • Fetches store profile via /api/stores/:storeId
  • Geocodes store address if coordinates are missing
  • Uses accurate pickup location for quote calculation

3.2 OrderSummary Component Updates

File: components/OrderSummary.jsx
Changes: Added shipping method selection and Lalamove integration
🔴 Issue: Multi-Store Order Complexity

A single cart can contain items from multiple stores. Creating a single Lalamove order for multiple pickups is complex and doesn't match user expectations for same-day delivery speed/cost.

🟢 Fix: Multi-Store Detection & Restriction

Logic Implemented:

  • Detects if cart items belong to different stores
  • Automatically disables Lalamove option for multi-store carts
  • Displays helpful message: "Lalamove same-day delivery is only available for single-store orders"
  • Guides users to checkout per store or use standard shipping
🔴 Issue: Total Price Calculation Mismatch

The checkout total didn't include Lalamove shipping costs, causing payment amount discrepancies.

🟢 Fix: Dynamic Total Calculation

Updated Calculation:

const shipping = shippingMethod === 'LALAMOVE' && lalamoveQuote ? parseFloat(lalamoveQuote.price.amount) : 0; const subtotal = coupon ? (totalPrice - (coupon.discount / 100 * totalPrice)) : totalPrice; const finalTotal = subtotal + shipping;

Applied to both display total and GCash payment amount calculation.

3.3 Backend Order API Enhancement

File: app/api/orders/route.js
Method: POST - Order Creation
🔴 Issue: Backend Ignoring Lalamove Shipping Costs

The order creation endpoint calculated shipping based solely on store profiles (Flat Rate/Free), completely ignoring the dynamic Lalamove quote from the frontend.

🟢 Fix: Shipping Cost Override Logic

Implementation:

  • Accept shipping object and shippingCost in request body
  • Check if shipping.provider === 'lalamove'
  • Override default shipping calculation with provided cost
  • Save shipping metadata (Quote ID, Service Type, ETA) to order
if (shipping && shipping.provider === 'lalamove') { shippingCost = parseFloat(body.shippingCost) || 0; } await prisma.order.create({ data: { // ... other fields shippingCost: shippingCost, shipping: shipping || {} } });

3.4 Order Tracking UI

File: components/OrderItem.jsx
Purpose: Display Lalamove shipping status and tracking link to users

Features Added:

  • Detects Lalamove orders via order.shipping.provider === 'lalamove'
  • Displays Lalamove logo with "Delivery" text
  • Shows clickable tracking link if order.shipping.shareLink exists
  • Shows "Pending Booking" status if store hasn't booked yet

3.5 Store Dashboard - Booking Interface

File: app/store/orders/page.jsx
Feature: Manual Lalamove delivery booking by store owners
🔴 Issue: No Booking Mechanism

After a user pays for an order with Lalamove shipping, the order exists in the system, but no actual driver is requested from Lalamove.

🟢 Fix: Store Dashboard Booking Action

Implementation:

  • Added "Book Delivery Now" button in order details modal
  • Button only appears for Lalamove orders without shareLink
  • Calls /api/shipping/lalamove/book with order details
  • Updates order with booking ID and tracking link
  • Provides real-time feedback with loading states

3.6 Admin Dashboard - Monitoring Interface

File: app/admin/orders/page.jsx
Purpose: Read-only Lalamove order monitoring for administrators

Features:

  • Displays Lalamove logo and "Delivery" text
  • Shows Quote ID and Service Type
  • Provides tracking link for administrative oversight
  • No booking functionality (handled by store owners)

🟢 Fix: Official Lalamove Logo Integration

6. Known Limitations & Future Enhancements

Current Limitations:
  • Lalamove only available for single-store orders
  • Geocoding relies on third-party Nominatim API (rate limits apply)
  • Manual booking required by store owners (not automatic)
  • Real-time status updates depend on webhook reliability
Recommended Future Enhancements:
  • Implement map-based address selection with coordinate pinning
  • Add automatic booking option for stores (configurable)
  • Support multi-store Lalamove orders with route optimization
  • Cache geocoding results in database to reduce API calls
  • Add delivery time slot selection for scheduled deliveries
  • Implement driver location tracking on order page
