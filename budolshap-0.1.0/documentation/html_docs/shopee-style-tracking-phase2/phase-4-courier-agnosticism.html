Phase 4: Courier Agnosticism & Provider Support

Architectural Strategy

The system was designed using a factory pattern to be completely courier-agnostic. All provider-specific statuses are mapped to a central "Shopee-style" lifecycle in lib/shipping/config.js.

Provider Support: Ship24 & TrackingMore

Added support for Ship24 and TrackingMore by defining their specific status triggers for the "In Transit" (To Receive) stage.

// lib/shipping/config.js
export const PROVIDER_TIMELINE_EVENTS = {
    [SHIPPING_PROVIDERS.SHIP24]: {
        // ... mappings
        IN_TRANSIT: {
            title: 'In Transit',
            triggerStatuses: ['IN_TRANSIT', 'OUT_FOR_DELIVERY']
        }
    },
    [SHIPPING_PROVIDERS.TRACKINGMORE]: {
        // ... mappings
        IN_TRANSIT: {
            title: 'In Transit',
            triggerStatuses: ['Transit', 'Out for Delivery']
        }
    }
};

Validation via Test Script

A specialized test script (scripts/test-status-mappings.mjs) was created to simulate various provider payloads and verify they render correctly in the 5-stage UI.

node scripts/test-status-mappings.mjs
# Output verifies that 'Transit' from TrackingMore correctly triggers 'To Receive'
