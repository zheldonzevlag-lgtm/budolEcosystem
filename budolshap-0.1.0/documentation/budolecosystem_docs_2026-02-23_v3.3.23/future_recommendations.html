Future Recommendations: Geocoding System

1. Caching Geocode Results

Recommendation: Implement a Redis or database-backed cache for geocoding results.

Rationale: Reduces API usage (costs and rate limits) and improves response time for frequently accessed locations (e.g., city centers, popular landmarks).

Proposed Implementation: Cache key based on rounded coordinates (e.g., 4 decimal places) for reverse geocoding.

2. Auto-Failover Provider

Recommendation: Automatically switch to Geoapify if Nominatim fails.

Rationale: Increases reliability. Currently, switching providers requires manual admin intervention.

Proposed Implementation: In app/api/system/geocode/route.js, if the primary provider fails, catch the error and try the secondary provider.

3. Enhanced Rate Limiting

Recommendation: Move rate limiting logic to the backend (e.g., Redis-based token bucket).

Rationale: Current client-side rate limiting prevents spam from a single user but doesn't protect the API from aggregate usage spikes.
