API Gateway & Database Strategy

Implementation Guide for BudolShap on Vercel

🏆 Core Strategy

API Gateway: Use Next.js Middleware as your Edge Gateway.
Database: Use logic-separated databases (Database-per-Service) pattern.

  • Gateway: Handles Auth & Routing at the Edge.
  • Microservices: Separate concerns (Orders, Products, Users).
  • Databases: Dedicated DBs for critical services to ensure scalability.

Microservices Database Strategy

You asked: "Can we have separate databases on each service?"

YES. This is the recommended "Database-per-Service" pattern. It prevents a single monolithic database from becoming the bottleneck and ensures that an issue in the "Products" DB doesn't bring down "User Auth".

Recommended Database Split for BudolShap

Based on your schema.prisma, here is the optimal separation:

🆔 Identity DB
Entities: User, Auth Tables
Why: High reliability required. If this goes down, no one can login. Should be isolated.
🛒 Catalog DB
Entities: Product, Store, Category, Rating
Why: High read volume (browsing). Needs aggressive caching and read-replicas.
📦 Order DB
Entities: Order, Cart, Shipping
Why: High write volume (transactions). Needs strict ACID compliance and row locking.
💰 Payment DB
Entities: Wallet, Payouts, AuditLog
Why: Financial data requires maximum security and separate backups for compliance.

How to Implement on Vercel

You can achieve this easily using Vercel Postgres (powered by Neon):

  1. Create Multiple Stores: In your Vercel dashboard, create separate Postgres stores (e.g., budolshap-auth, budolshap-order).
  2. Environment Variables: Connect each service to its specific DB. 
    # Service A (Auth)
DATABASE_URL=postgres://user:pass@auth-db.neon.tech/main

# Service B (Orders)
DATABASE_URL=postgres://user:pass@order-db.neon.tech/main

Tip: On the Free Tier, you are limited to 1 Postgres store per project usually. To bypass this, simply create multiple "Projects" in Vercel (e.g., `budolshap-auth-service`, `budolshap-order-service`) or use an external provider like Neon directly which allows multiple branches/DBs.

Gateway Implementation Guide

Code pattern for your middleware.js:

import { NextResponse } from 'next/server'
import { verifyTokenEdge } from './lib/token-edge'

export async function middleware(request) {
    const { pathname } = request.nextUrl
    
    // 1. API Gateway Logic - Routing to Microservices
    if (pathname.startsWith('/api/orders')) {
        // Authenticate
        const token = request.cookies.get('token')?.value
        if (!token) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })

        // Rewrite to Order Service (internal or external URL)
        // Ensure you have ORDER_SERVICE_URL in your .env
        const targetUrl = new URL(request.nextUrl.pathname, process.env.ORDER_SERVICE_URL)
        return NextResponse.rewrite(targetUrl)
    }

    return NextResponse.next()
}
