1. What is Pusher?

Pusher is a hosted service that provides real-time messaging infrastructure. It's perfect for microservices because:

  • ✅ No server to manage - Fully hosted by Pusher
  • ✅ Vercel compatible - Works perfectly with serverless functions
  • ✅ Automatic scaling - Handles millions of connections
  • ✅ Easy integration - Simple API, well-documented
  • ✅ Free tier available - Good for getting started

2. What You Need: Complete Checklist

  • Test connection

3. Step-by-Step Setup

Step 1: Create Pusher Account

  1. Go to https://pusher.com
  2. Click "Sign Up" (top right)
  3. Sign up with email or GitHub
  4. Verify your email address
💡 Free Tier: Pusher offers a free "Sandbox" plan with:
  • 200,000 messages/day
  • 100 concurrent connections
  • Unlimited channels
  • Perfect for development and small-scale production

Step 2: Create a Pusher App

  1. After logging in, you'll see the Pusher dashboard
  2. Click "Create app" or "Channels" → "Create app"
  3. Fill in the form:
    • App name: BudolShap (or any name)
    • Cluster: Choose closest to your users (e.g., ap1 for Asia Pacific)
    • Front-end tech: React
    • Back-end tech: Node.js
  4. Click "Create app"
⚠️ Important: Choose your cluster carefully! This affects latency. For Philippines users, ap1 (Singapore) or ap2 (Mumbai) are good choices.

Step 3: Get Your Credentials

After creating the app, you'll see the "App Keys" page with 4 values:

Credential Where to Find Example
App ID Under "App ID" section 1234567
Key Under "Key" section a1b2c3d4e5f6g7h8
Secret Under "Secret" section (click "Reveal") 1a2b3c4d5e6f7g8h9i0j
Cluster Shown at top of page or in app settings ap1 or us2
🔒 Security: Keep your Secret secure! Never expose it in client-side code. It's only used server-side in lib/realtime.js.

Step 4: Configure in BudolShap Admin Panel

  1. Log into your BudolShap admin panel
  2. Navigate to Admin → Settings → Realtime Updates
  3. Select "⚡ Pusher (Hosted, Paid)" from the dropdown
  4. Fill in the 4 fields:
    • App ID: Paste your Pusher App ID
    • Key: Paste your Pusher Key
    • Cluster: Paste your cluster (e.g., ap1)
    • Secret: Paste your Pusher Secret
  5. Click "Save Configuration"
  6. The page will reload automatically

What Happens Behind the Scenes:

// Your credentials are saved to SystemSettings table:
{
  realtimeProvider: 'PUSHER',
  pusherAppId: '1234567',
  pusherKey: 'a1b2c3d4e5f6g7h8',
  pusherSecret: '1a2b3c4d5e6f7g8h9i0j',
  pusherCluster: 'ap1'
}

Step 5: Test the Connection

After saving, test that Pusher is working:

  1. Open your app in a browser
  2. Open browser console (F12)
  3. Look for messages like: 
    🔌 Connecting to Pusher... ap1
[Realtime] Mode: PUSHER
  4. Trigger a realtime event (e.g., create an order)
  5. You should see Pusher events in the console
✅ Success Indicators:
  • No errors in browser console
  • See "Connecting to Pusher..." message
  • Realtime updates work (orders update without page refresh)
  • Pusher dashboard shows connection activity

5. How Pusher Works in Your Microservices Architecture

5.1 Current Architecture

┌─────────────────────────────────────────┐
│   Backend Service (Any Microservice)    │
│                                          │
│   triggerRealtimeEvent(channel, event)   │
│           ↓                              │
│   lib/realtime.js                        │
│           ↓                              │
│   Pusher.trigger(channel, event, data)   │
└─────────────────────────────────────────┘
                    ↓
        ┌───────────────────────┐
        │   Pusher Cloud       │
        │   (Hosted Service)    │
        └───────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Frontend (Next.js App)                │
│                                          │
│   RealtimeProvider.jsx                  │
│           ↓                              │
│   Pusher.subscribe(channel)              │
│           ↓                              │
│   channel.bind(event, callback)          │
│           ↓                              │
│   mutate() - Refresh SWR data            │
└─────────────────────────────────────────┘

5.2 How It Works with Microservices

When you extract services, Pusher works the same way:

  1. Any microservice calls triggerRealtimeEvent()
  2. lib/realtime.js sends event to Pusher
  3. Pusher broadcasts to all connected clients
  4. Frontend receives event and updates UI via SWR
💡 Key Benefit:
Each microservice doesn't need to know about other services. They just trigger events, and Pusher handles the distribution. This is perfect for microservices!

5.3 Channel Naming Convention

Your codebase uses these channel patterns:

// Store-specific channels
`store-${storeId}`           // e.g., "store-abc123"

// User-specific channels  
`user-${userId}`             // e.g., "user-xyz789"

// Global channels
`admin-notifications`       // Admin events
`marketplace-updates`        // Product updates

5.4 Event Types

Your app currently uses these events:

  • order-updated - Order status changed
  • order-created - New order placed
  • user-registered - New user signed up
  • product-added - New product created
  • product-updated - Product details changed
    • rating-created - New rating submitted

6. Environment Variables (Optional)

Currently, Pusher credentials are stored in the database (SystemSettings table). This is fine, but you can also use environment variables for better security:

Option A: Database Storage (Current)

// Credentials stored in SystemSettings table
// Configured via admin panel
// ✅ Easy to change without redeploying
// ⚠️ Less secure (stored in database)

Option B: Environment Variables (Recommended for Production)

// .env.local or Vercel environment variables
PUSHER_APP_ID=1234567
PUSHER_KEY=a1b2c3d4e5f6g7h8
PUSHER_SECRET=1a2b3c4d5e6f7g8h9i0j
PUSHER_CLUSTER=ap1

// Then modify lib/realtime.js to check env vars first:
const pusherAppId = process.env.PUSHER_APP_ID || settings.pusherAppId;
const pusherKey = process.env.PUSHER_KEY || settings.pusherKey;
// ... etc
💡 Recommendation:
For microservices, use environment variables in Vercel. This way, each service can have its own Pusher credentials if needed, and secrets are more secure.

7. Troubleshooting

7.1 Common Issues

❌ "Missing Pusher credentials"

Cause: Credentials not saved in database

Solution: Go to admin panel and save all 4 credentials

❌ "Failed to connect to Pusher"

Cause: Wrong cluster or network issues

Solution:

  • Verify cluster matches your Pusher app settings
  • Check browser console for specific error
  • Verify Key is correct (not Secret)

❌ "Events not received"

Cause: Wrong channel name or not subscribed

Solution:

  • Check channel names match between backend and frontend
  • Verify subscription in browser console
  • Check Pusher dashboard for event activity

7.2 Debugging Tips

  1. Check Pusher Dashboard:
    • Go to your Pusher app dashboard
    • Click "Debug Console" tab
    • See real-time events and connections
  2. Browser Console:
    • Open browser DevTools (F12)
    • Look for Pusher connection messages
    • Check for subscription confirmations
  3. Server Logs:
    • Check Vercel logs for triggerRealtimeEvent calls
    • Look for Pusher errors

8. Best Practices for Microservices

8.1 Centralized Realtime Service

Keep lib/realtime.js as a shared module that all microservices can use: 

// In any microservice (Payment, Shipping, Orders, etc.)
import { triggerRealtimeEvent } from '@/lib/realtime';

// Trigger event when something happens
await triggerRealtimeEvent(
    `store-${storeId}`,      // Channel
    'order-updated',          // Event name
    { orderId, status }        // Data
);

8.2 Consistent Channel Naming

Use consistent channel naming across all services:

  • store-{storeId} - Store-specific events
  • user-{userId} - User-specific events
  • order-{orderId} - Order-specific events (optional)
  • admin-* - Admin events
  • marketplace-* - Public marketplace events

8.3 Event Naming Convention

Use consistent event names:

  • {entity}-created - e.g., order-created
  • {entity}-updated - e.g., order-updated
  • {entity}-deleted - e.g., product-deleted
  • {entity}-{action} - e.g., payment-completed

8.4 Error Handling

Always handle Pusher errors gracefully:

const result = await triggerRealtimeEvent(channel, event, data);
if (!result.success) {
    console.error('Realtime event failed:', result.error);
    // Fallback to polling or queue for retry
}

9. Migration from Socket.io to Pusher

If you're currently using Socket.io and want to migrate to Pusher:

  1. Set up Pusher (follow steps 1-5 above)
  2. Configure Pusher in admin panel
  3. Test thoroughly - Verify all events work
  4. Decommission Socket.io server - Stop the websocket-server
  5. Remove Socket.io code (optional cleanup)
✅ Benefits of Migration:
  • No server to manage
  • Better scalability
  • Works perfectly with Vercel
  • Simpler architecture
