1. Executive Overview

This document outlines a comprehensive plan to migrate BudolShap from its current modular monolith architecture to a microservices architecture while maintaining deployment compatibility with Vercel and ensuring minimal disruption to existing functionality.

🎯 Key Objectives:
  • Improve scalability and independent deployment of services
  • Enable team autonomy and faster feature development
  • Maintain compatibility with Vercel serverless platform
  • Minimize risk through incremental migration
  • Preserve existing functionality during transition
📊 Current State: Next.js 16 monolith with 80+ API routes, Prisma ORM, separate WebSocket server, deployed on Vercel with external PostgreSQL database.

3.1 Architecture Principles

  • Single Responsibility: Each service owns one business domain
  • Independent Deployment: Services can be deployed separately
  • API-First: Services communicate via well-defined REST APIs
  • Database per Service: Eventually, but start with logical separation
  • Fail-Safe: Services degrade gracefully when dependencies fail

5. Communication Patterns

5.1 Synchronous Communication (REST APIs)

Services communicate via REST over HTTPS:

  • API Gateway → Services: Direct HTTP calls to service endpoints
  • Service → Service: Direct HTTP calls (minimize cross-service calls)
  • External → Services: Webhooks (PayMongo, Lalamove) hit service endpoints directly
Example Flow:
User clicks "Checkout" → API Gateway → Order Service (creates order) → Payment Service (initiates payment) → Returns redirect URL to client

5.2 Asynchronous Communication (Events)

For real-time updates and cross-service notifications:

Current Approach:

  • lib/realtime.js triggers events via Pusher or Socket.io
  • Clients subscribe to channels (e.g., store-{storeId}, user-{userId})

Future Enhancement (Optional):

  • Introduce event bus (Redis Pub/Sub, RabbitMQ, or AWS EventBridge)
  • Services publish events when state changes
  • Other services subscribe to relevant events
  • Note: Not required initially; direct REST calls are sufficient

5.3 Database Strategy

Phase 1-2: Shared Database (Logical Separation)

  • Keep single PostgreSQL database
  • Each service only accesses its "owned" tables (enforced by code)
  • Use Prisma with separate schema files per service
  • Simpler to start, easier to migrate

Phase 3+: Database per Service (Strict Microservices)

  • Each service has its own database schema
  • Services communicate only via APIs
  • Event-driven consistency for cross-service data
  • Note: This is optional and adds significant complexity
⚠️ Recommendation: Start with shared database (logical separation). Only split databases if you have strong organizational or scaling reasons.

6. Step-by-Step Migration Plan

Phase 1: Modularize Inside Monolith (Weeks 1-2)

Goal: Clean separation of domains within existing Next.js app

Actions:

  1. Create Service Layer:
    • Create lib/services/ directory
    • Extract business logic from API routes into service modules:
      • lib/services/ordersService.js
      • lib/services/paymentService.js
      • lib/services/shippingService.js
      • lib/services/authService.js
      • lib/services/catalogService.js
    • API routes become thin controllers that call services
  2. Standardize Responses:
    • Create common response format
    • Standardize error handling
    • Add request/response logging
  3. Enforce Domain Boundaries:
    • Each service module only touches its own Prisma models
    • No direct cross-domain database access
    • Use service methods for cross-domain operations
  4. Decide on Realtime:
    • Choose Pusher (recommended) or keep Socket.io server
    • Update system settings accordingly

Deliverables:

  • ✅ Service layer modules created
  • ✅ API routes refactored to use services
  • ✅ Domain boundaries enforced
  • ✅ All tests passing

Phase 2: Introduce Internal Service Boundaries (Weeks 3-4)

Goal: Treat domains as separate services, but still in one codebase

Actions:

  1. Rename API Routes:
    • Create app/api/internal/ directory structure
    • Move domain routes to app/api/internal/{service}/*
    • Keep public-facing routes in app/api/public/* or root app/api/*
  2. Define Service Interfaces:
    • Document expected request/response formats
    • Create TypeScript types or JSDoc comments
    • Define error codes and status codes
  3. Implement Service-to-Service Calls:
    • Services call each other via HTTP (even if internal)
    • Use environment variables for service URLs
    • In development, use http://localhost:3000/api/internal/...
    • In production, use same URLs (still internal to Vercel)
  4. Add Service Health Checks:
    • Create GET /api/internal/{service}/health endpoints
    • Monitor service availability

Deliverables:

  • ✅ Internal service structure defined
  • ✅ Service interfaces documented
  • ✅ Cross-service calls implemented via HTTP
  • ✅ Health checks in place

Phase 3: Extract First Microservice - Payment Service (Weeks 5-6)

Goal: Extract Payment Service as separate deployable unit

Why Payment First?

  • High isolation value (security, compliance)
  • Clear boundaries (webhooks, external APIs)
  • Independent scaling needs
  • Minimal dependencies on other services

Actions:

  1. Create New Vercel Project:
    • Create budolshap-payment-service repository
    • Copy payment-related code
    • Set up Prisma (shared DB or separate schema)
    • Configure environment variables
  2. Deploy Payment Service:
    • Deploy to Vercel: payment.budolshap.com
    • Set up webhook endpoints
    • Configure PayMongo webhook URL
  3. Update Main App:
    • Replace local payment code with HTTP calls to payment service
    • Update app/api/payment/* to proxy to payment service
    • Update lib/payment/service.js to call external API
  4. Test Thoroughly:
    • Test payment flow end-to-end
    • Verify webhooks work
    • Test error scenarios

Deliverables:

  • ✅ Payment service deployed independently
  • ✅ Main app calls payment service via HTTP
  • ✅ All payment tests passing
  • ✅ Webhooks functioning correctly

Phase 4: Extract Shipping Service (Weeks 7-8)

Goal: Extract Shipping Service as separate deployable unit

Actions:

  1. Create Shipping Service:
    • Create budolshap-shipping-service repository
    • Copy shipping-related code
    • Set up Lalamove API client
    • Configure webhook endpoints
  2. Deploy Shipping Service:
    • Deploy to Vercel: shipping.budolshap.com
    • Set up Lalamove webhook URL
    • Configure cron jobs for order sync
  3. Update Main App:
    • Replace local shipping code with HTTP calls
    • Update Order Service to call Shipping Service
  4. Update Order Service:
    • Shipping Service calls Order Service to update order status
    • Implement order update endpoint in Order Service

Deliverables:

  • ✅ Shipping service deployed independently
  • ✅ Order-to-Shipping integration working
  • ✅ Webhooks and cron jobs functioning

Phase 5: Extract Remaining Services (Weeks 9-12)

Goal: Extract Auth, Orders, Catalog, Admin services

Extraction Order:

  1. Auth Service (Week 9)
  2. Order Service (Week 10)
  3. Catalog Service (Week 11)
  4. Admin Service (Week 12, or keep in API Gateway)

Pattern for Each Service:

  1. Create new Vercel project
  2. Copy service code
  3. Deploy to subdomain
  4. Update main app to call service
  5. Test thoroughly
✅ Migration Complete: All services extracted, main app is now API Gateway/BFF

7. Vercel Deployment Strategy

7.1 What Works on Vercel

Service Type Vercel Compatibility Notes
Next.js BFF/API Gateway ✅ Perfect Native Next.js deployment
REST API Services ✅ Excellent Serverless functions, separate Vercel projects
Webhook Handlers ✅ Excellent Stable HTTPS URLs, perfect for webhooks
Cron Jobs ✅ Supported Vercel Cron (already using)
Payment Service ✅ Excellent Serverless, scales automatically
Shipping Service ✅ Excellent Webhooks + cron jobs supported
Auth Service ✅ Excellent Stateless JWT, perfect for serverless

7.2 What Doesn't Work on Vercel

Component Issue Solution
Long-lived WebSocket Server ❌ Not Supported Use Pusher (recommended) or deploy Socket.io server to Railway/Fly.io/Render
Database ⚠️ External Required Already using external DB (PostgreSQL), continue with same approach

7.3 Deployment Architecture on Vercel

Option A: Multiple Vercel Projects (Recommended)

  • Main App: budolshap-v3 → budolshap.com
  • Payment Service: budolshap-payment-service → payment.budolshap.com
  • Shipping Service: budolshap-shipping-service → shipping.budolshap.com
  • Auth Service: budolshap-auth-service → auth.budolshap.com
  • Order Service: budolshap-order-service → orders.budolshap.com
  • Catalog Service: budolshap-catalog-service → catalog.budolshap.com

Benefits: Independent deployments, separate scaling, team autonomy

Option B: Monorepo with Vercel Workspaces

  • Single repository with multiple Next.js apps
  • Vercel detects and deploys each app separately
  • Shared code in packages/ directory

Benefits: Code sharing, unified versioning, easier local development

7.4 Environment Variables Strategy

  • Per-Service Variables: Each Vercel project has its own environment variables
  • Shared Secrets: Use Vercel's shared environment variables or secrets manager
  • Database Connection: Each service connects to same DB (Phase 1-2) or separate DBs (Phase 3+)
  • Service URLs: Store service URLs in environment variables for service-to-service calls

7.5 Realtime Strategy for Vercel

✅ Recommended: Use Pusher
  • Set realtimeProvider = 'PUSHER' in system settings
  • Configure Pusher credentials in Vercel environment variables
  • Remove need for separate WebSocket server
  • 100% compatible with Vercel serverless architecture
  • Scales automatically, no infrastructure management
Alternative: External Socket.io Server
  • Deploy websocket-server/ to Railway, Fly.io, or Render
  • Set socketUrl environment variable
  • More infrastructure to manage, but gives you full control

8. Practical Recommendations

8.1 Start Small, Iterate

  • Don't try to extract all services at once
  • Start with Phase 1 (modularization) - this alone provides significant value
  • Extract services one at a time, starting with Payment Service
  • Test thoroughly after each extraction

8.2 Keep Database Shared Initially

  • Start with logical separation (services own specific tables)
  • Only split databases if you have strong reasons (compliance, scaling, team boundaries)
  • Splitting databases adds significant complexity (distributed transactions, eventual consistency)

8.3 Use Pusher for Realtime

  • Simplest path for Vercel deployment
  • No infrastructure to manage
  • Already integrated in your codebase
  • Scales automatically

8.4 Implement Service Health Checks

  • Each service should expose GET /health endpoint
  • Monitor service availability
  • Implement circuit breakers for service-to-service calls
  • Add retry logic with exponential backoff

8.5 Add Observability

  • Centralized logging (e.g., Vercel Logs, Datadog, Sentry)
  • Request tracing across services
  • Performance monitoring
  • Error tracking and alerting

8.6 API Versioning

  • Version your service APIs from the start: /api/v1/orders
  • Allows backward compatibility during migrations
  • Easier to deprecate old endpoints

8.7 Documentation

  • Document each service's API (OpenAPI/Swagger)
  • Maintain service dependency diagram
  • Document deployment procedures
  • Keep runbooks for common operations

9. Risks & Mitigation

Risk Impact Mitigation
Service communication failures High Implement retries, circuit breakers, graceful degradation
Database connection limits Medium Use connection pooling, consider separate DBs if needed
Increased latency Medium Optimize service calls, use caching, minimize cross-service calls
Deployment complexity Medium Use CI/CD pipelines, automate deployments, test thoroughly
Data consistency issues High Start with shared DB, implement idempotency, use transactions where possible
Debugging complexity Medium Implement distributed tracing, centralized logging, request IDs
Cost increase Low Vercel serverless scales efficiently, monitor usage
