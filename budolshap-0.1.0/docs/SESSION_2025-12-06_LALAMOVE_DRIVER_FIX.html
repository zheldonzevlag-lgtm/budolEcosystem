💡 Key Learnings

1. Always Verify the Actual Endpoint Being Used

Don't assume which endpoint is being called. Check:

  • Webhook configuration in third-party portal
  • Database to see if data is being saved
  • Server logs to see which endpoint receives requests

2. Use Defensive Programming

Using optional chaining (?.) and fallback operators (||) makes code resilient to API structure variations.

3. Comprehensive Logging is Essential

Detailed logging helped identify exactly what data was being received and where the extraction was failing.

4. Test with Real Data

The breakthrough came when the user shared actual webhook payloads from Lalamove Partner Portal. Documentation may not always match reality.

5. Database Verification is Critical

Checking the database directly revealed that the webhook wasn't saving data, which led to discovering the wrong endpoint was fixed.

🚑 Phase 2: Frontend Crash Investigation (02:00 PHT)

Background

After fixing the webhook and manually injecting data, we confirmed driver data was in the database. But the page started crashing ("Application Error")!

The Crash

Error: TypeError: .toFixed is not a function

Cause: Lalamove API returns coordinates as Strings (e.g. "14.5505"), but .toFixed() is a Number method.

Impact: Any order with valid driver data caused the page to crash immediately.

The Fix (Commit fac7859)

// ❌ OLD (Crashes on string) {location.lat?.toFixed(4)} // ✅ NEW (Safe cast) {Number(location.lat).toFixed(4)}

🐛 Phase 3: Webhook Runtime Bug (02:35 PHT)

Self-Inflicted Error during Debugging

To debug why some new orders were still missing data, I added logic to save the raw webhook payload to the database.

The Mistake: I referenced a variable named payload but the defined variable was body.

// ❌ Wrong variable name lastWebhookPayload: payload // ✅ Correct variable name lastWebhookPayload: body

Impact: This caused a 500 Internal Server Error for new orders placed between 02:40 and 02:50 AM, causing the webhook to fail silently.

Final Resolution (Commit af03269)

  • Fixed the variable name in `route.js`
