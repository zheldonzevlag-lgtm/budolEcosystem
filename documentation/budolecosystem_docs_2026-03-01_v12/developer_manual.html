Developer Manual: Maker-Checker Integration

Technical guidance for maintaining and extending the compliance-grade workforce management system.

1. Data Schema Evolution

We introduced the ChangeRequest model to facilitate asynchronous approvals. This ensures the User table remains the "Golden Record" while ChangeRequest serves as the temporary staging area.

packages/database/prisma/schema.prisma Open File
model ChangeRequest { id String @id @default(uuid()) entity String // e.g., "User" entityId String details Json // Stores { field: newValue } makerId String checkerId String? status ChangeRequestStatus @default(PENDING) maker User @relation("MakerRequests", fields: [makerId], references: [id]) checker User? @relation("CheckerRequests", fields: [checkerId], references: [id]) }

2. API Workflow Orchestration

The /api/employees endpoint handles the transition between states. It now detects if a modification requires Maker-Checker logic.

apps/admin/app/api/employees/route.ts Open File
// Maker Phase: Intercepting UPDATE_PROFILE if (action === "UPDATE_PROFILE") { const changeRequest = await prisma.changeRequest.create({ data: { entity: "User", entityId: userId, details: changes, makerId: adminId } }); return NextResponse.json({ success: true, isPending: true }); } // Checker Phase: APPROVE_CHANGE_REQUEST if (action === "APPROVE_CHANGE_REQUEST") { await prisma.user.update({ where: { id: req.entityId }, data: req.details }); await prisma.changeRequest.update({ where: { id: req.id }, data: { status: "APPROVED" } }); }

3. Frontend Integration

The UI reactively updates based on the isPending flag. The EmployeesPage component polls or fetches the GET_CHANGE_REQUESTS action to populate the Approval Queue.

4. Dual-Channel Notification Service

To ensure high availability of verification codes and compliance with redundant alert standards (BSP Circular 808), we implemented a centralized notification service that broadcasts via both SMS and Email simultaneously.

apps/admin/lib/notifications.ts Open File
export async function sendDualChannelNotification(user, type, value) { // Logic to route both to SMS Gateway and SMTP Relay // This ensures that even if one channel is delayed, the user receives the code. console.log(`[SIMULATION] SENDING ${type} TO SMS: ${user.phoneNumber}`); console.log(`[SIMULATION] SENDING ${type} TO EMAIL: ${user.email}`); return { sms: true, email: true }; }

6. Compliance & Governance

The Maker-Checker system is designed to meet specific Philippine and International regulatory requirements:

Requirement Legal Framework Technical Implementation
Segregation of Duties (SoD) BSP Circular 808 Enforced via makerId !== checkerId validation in route.ts.
Forensic Audit Logs PCI DSS v4.0 (Req 10) Immutable JSON snapshots of oldValue and newValue stored in PG.
Accountability NPC Data Privacy Act IP mapping, User Agent tracking, and timestamping on every Approval/Rejection.
High Availability Alerts BSP Circular 943 Dual-Channel broadcast (SMS + Email) for all security-sensitive events.
