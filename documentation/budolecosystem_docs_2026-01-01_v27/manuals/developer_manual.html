Developer Manual Update (v27)

Topic: Full LAN Orchestration & Token Separation

Problem: Redirect Loop on LAN

Even after setting secure: false, users were still seeing redirect loops when accessing the apps via LAN IP. This was caused by two main issues:

  • Port Binding: budolPay was still bound to localhost, preventing external access on the LAN IP.
  • Token Collision: Both apps used a generic token cookie name. Since they share the same IP/domain on the LAN, they were overwriting each other's session tokens.

Solutions Implemented

1. Global LAN Access

Updated budolPay's package.json to bind to 0.0.0.0, allowing it to be reached from any device on the network.

2. App-Specific Token Names

Renamed session cookies to avoid collisions:

  • budolPay now uses budolpay_token
  • budolShap now uses budolshap_token

3. Dynamic Environment Sync

Enhanced sync-network.js to automatically generate and update .env files for all apps in the ecosystem. This ensures SSO_URL and NEXT_PUBLIC_APP_URL always point to the current machine's IP.

4. SSO Login Button Refinement

Updated login pages in budolPay and budolShap to prioritize NEXT_PUBLIC_SSO_URL from environment variables. This ensures the "Sign in with budolID" button always points to the correct dynamic IP, even if the user is accessing via a mobile device or another PC on the LAN.

5. 0.0.0.0 Redirect Protection

While 0.0.0.0 is used for binding the server to all interfaces (allowing LAN access), it is not a valid address to visit in a browser. To prevent ERR_ADDRESS_INVALID errors after login, we updated the SSO callback routes to use the explicit NEXT_PUBLIC_APP_URL for the final redirect, ensuring users always end up on a valid IP (like localhost or 192.168.x.x) instead of the generic binding address.

How to Sync for a New Network

Whenever you change networks, simply run:

npm run sync

Then restart the ecosystem with npm start.
