1. Executive Summary
📌 Overview
This document outlines a comprehensive plan to deploy the budolEcosystem on Vercel's free tier (Hobby plan) while staying within its limitations. The plan leverages external services for stateful components and focuses the Vercel deployment on the Next.js frontend and serverless functions.

Target Subdomains (duckdns.org)

  • budolshap.duckdns.org - E-commerce storefront
  • budolpay.duckdns.org - Android app backend
  • budolid.duckdns.org - Identity/SSO service
  • budolws.duckdns.org - WebSocket server
  • budolaccounting.duckdns.org - Accounting module
  • budoladmin.duckdns.org - Admin dashboard (budolPay)

Key Challenges

  • Vercel only supports serverless functions (no long-running Node.js services)
  • WebSocket requires external service (Pusher or separate server)
  • PostgreSQL and Redis need external managed services
  • Multiple backend services need conversion or alternative hosting

4. Microservices Strategy
⚠️ Key Challenge: Microservices on Vercel
Vercel doesn't support long-running Node.js processes. Each microservice must be converted to serverless functions or hosted on alternative infrastructure.

Current Microservices Architecture

The budolEcosystem consists of 10 microservices that need to be rearchitected for Vercel:

Service Port Strategy Conversion Effort
budolID 8000 Use NextAuth.js for authentication Medium
budolAccounting 8005 Convert to serverless functions + external API High
budolpay-gateway 8080 Use Vercel API Routes as gateway Medium
budolpay-auth 8001 Replace with NextAuth.js providers Low
budolpay-wallet 8002 Convert to serverless functions Medium
budolpay-transaction 8003 Convert to serverless functions Medium
budolpay-payment 8004 Use external payment APIs (PayMongo) Low
budolpay-kyc 8006 Convert to serverless functions Medium
budolpay-settlement 8007 Convert to serverless functions Medium

Migration Strategy: 3-Tier Approach

Tier 1 Vercel-Native (Immediate)
  • Authentication: NextAuth.js (already in use)
  • API Gateway: Vercel API Routes
  • Real-time: Pusher integration
  • Payments: Direct PayMongo integration
Tier 2 Serverless Conversion (Weeks 2-3)
  • Wallet: Convert to Vercel serverless functions
  • Transaction: Convert to Vercel serverless functions
  • KYC: Convert to Vercel serverless functions
  • Settlement: Convert to Vercel serverless functions
Tier 3 External Services (Long-term)
  • Accounting: Keep on ECS or use external accounting API
  • SSO: Keep on ECS or use Auth0/Clerk
  • Complex Business Logic: AWS Lambda functions

4 Set Up duckdns.org
  1. Register at duckdns.org
  2. Create your domain (e.g., yourname.duckdns.org)
  3. Add subdomains: budolshap, budolpay, budolid, budolaccounting, budoladmin, budolws
  4. For each subdomain, add a CNAME record pointing to your target
  5. For Vercel: cname.vercel-dns.com
  6. For AWS: Your EC2 elastic IP or ALB DNS name
5 Add Custom Domains in Vercel
  1. Go to Vercel Dashboard → Project → Settings → Domains
  2. Add each subdomain
  3. Update DNS records in duckdns.org dashboard
  4. Wait for SSL certificate generation (automatic)

4 Code Restructuring Example 
# Convert Express microservice to Vercel API Route

# Before: budolpay-wallet/src/index.js
app.get('/balance', async (req, res) => {
  const balance = await walletService.getBalance(req.user.id);
  res.json({ balance });
});

# After: app/api/wallet/balance/route.ts
import { NextResponse } from 'next/server';
import { getServerSession } from 'next-auth';
import { walletService } from '@/lib/walletService';

export async function GET() {
  const session = await getServerSession();
  if (!session) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
  const balance = await walletService.getBalance(session.user.id);
  return NextResponse.json({ balance });
}

Local Development Compatibility
📌 Overview
The budolEcosystem has a built-in environment switching system that allows seamless switching between local development and Vercel/production environments without breaking the local setup.

How the Local Environment System Works

Command Purpose Effect
npm start Central runner Prompts for environment choice, then starts all 13 services
npm run db:local Switch to local DB Uses .env.connection.local for local PostgreSQL
npm run db:vercel Switch to Vercel DB Uses .env.vercel for Supabase/Neon PostgreSQL
RUN_ENV=1 npm start Auto-select env Skips prompt, uses saved environment

Environment Switching Mechanism

# The env-manager.cjs script handles switching:
# 1. Reads from .env.connection.local or .env.vercel
# 2. Maps service-specific DATABASE_URLs to each service
# 3. Sets VERCEL=1 and NODE_ENV=production for Vercel mode
# 4. Propagates WebSocket URLs (INTERNAL_WS_URL, NEXT_PUBLIC_SOCKET_URL)

# Environment files:
# - .env.connection.local = Local PostgreSQL URLs
# - .env.connection.prod = Production AWS PostgreSQL URLs
# - .env.vercel = Supabase/Neon PostgreSQL URLs

Best Practices for Vercel-Compatible Development

1 Use Environment Variables for All Configurations

Never hardcode URLs, API keys, or service endpoints. Use process.env variables.

2 Check VERCEL Environment Variable 
// Detect Vercel environment
const isVercel = process.env.VERCEL === '1';

// Different config for Vercel vs local
const dbUrl = isVercel 
  ? process.env.DATABASE_URL  // Supabase/Neon
  : process.env.BUDOLSHAP_DATABASE_URL;  // Local
3 Keep Microservices Running Locally

Don't remove the local microservices. They are needed for:

  • Local development and testing
  • Development of new features
  • Debugging without affecting production
4 Use Next.js API Routes as Bridge

When migrating to Vercel, create API routes in Next.js that can call either:

  • Local microservices during development
  • Vercel serverless functions in production
⚠️ Important: Don't Break Local Development
  • Always test with npm run db:local before committing
  • Don't remove or modify the env-manager.cjs script
  • Keep .env.connection.local and .env.vercel files updated
  • Make sure Prisma generates clients for both environments
✅ Easy Environment Switching

With this system, you can easily switch between environments:

# Switch to local (for development)
npm run db:local

# Switch to Vercel (for production testing)
npm run db:vercel

# Or use the central runner
npm start
# Then choose option 1 (LOCAL) or 2 (PRODUCTION/VERCEL)
