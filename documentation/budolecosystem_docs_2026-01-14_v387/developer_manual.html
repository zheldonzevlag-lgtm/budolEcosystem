Developer Manual - Budol Ecosystem v387

Purpose

This manual documents the implementation of robust biometric security using SHA256 model validation and enhanced device-level authentication (local_auth) for the BudolPay mobile application.

Rationale

The previous version (v386) used a simple checksum for TFLite model validation. To comply with BSP Circular 808 and NPC Data Privacy Act, a cryptographically secure SHA256 hash is required to prevent model tampering. Additionally, user feedback indicated the need for better biometric dialogs, addressed via local_auth 2.3.0 enhancements.

Technical Explanation

The system now utilizes the crypto package to calculate the SHA256 hash of the 93MB FaceNet TFLite model during initialization. The hash is calculated on a background thread using compute() to prevent UI freezing.

Changes

  • Package Update: local_auth updated to ^2.3.0 for better OS compatibility.
  • New Dependency: Added crypto: ^3.0.3 for SHA256 support.
  • FaceEmbeddingService: Switched from checksum to SHA256 validation.
  • BiometricService: Added hasEnrolledFace() to detect OS-level Face ID and AndroidAuthMessages for custom UI strings.
  • LoginScreen: Fixed "Face icon" discrepancy by redirecting to native Face ID if enrolled in the OS.
  • Version Sync: Incremented to 1.2.83 across all services.

Issues & Fixes

Issue Fix

Compliance

  • BSP Circular 808: Cryptographic integrity for biometric templates.
  • NPC Data Privacy Act: Secure handling of face embeddings.
  • Cybersecurity Law: Multi-layered authentication (Biometric + PIN fallback).

Future Enhancements

  • Implement Git LFS properly for TFLite models in CI/CD.
  • Add face liveness detection (blink/smile) to registration flow.

Recommendation

Ensure that all developer machines have the facenet.tflite file with the matching SHA256 hash to avoid initialization failures.
