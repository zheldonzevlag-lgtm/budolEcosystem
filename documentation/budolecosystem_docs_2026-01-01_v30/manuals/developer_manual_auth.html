Developer Manual: SSO & Session Management Fixes

Technical details for the ecosystem-wide Single Sign-On (SSO) implementation and recent session persistence fixes.

Version: 2026-01-01_v30
Focus: Resolving SSO redirect session persistence and cookie security on LAN environments.

SSO Architecture

The ecosystem uses budolID as the central identity provider. Apps like budolShap integrate with budolID using a token-based redirect flow.

Critical Fixes UPDATED

1. Token Secret Mismatch Resolution

Previously, budolShap was attempting to use the JWT token issued by budolID directly. This failed because each application has its own JWT_SECRET. The SSO callback now generates a local token after verification.

// Location: budolshap-0.1.0/app/api/auth/sso/callback/route.js

// 1. Verify budolID token via back-channel API
const verifyResponse = await fetch(`${ssoUrl}/auth/verify`, { ... });

// 2. Generate local budolShap token
const localToken = generateToken({
    userId: localUser.id,
    email: localUser.email,
    role: localUser.accountType
});

// 3. Set local cookie
response.cookies.set('budolshap_token', localToken, { ... });

2. LAN IP Cookie Security (Non-HTTPS)

In development and LAN environments (e.g., 192.168.1.24), secure: true cookies are often rejected by browsers because they are not served over HTTPS. We have updated COOKIE_OPTIONS to allow cookies on non-HTTPS connections during testing.

// Location: budolshap-0.1.0/lib/auth.js
export const COOKIE_OPTIONS = {
    httpOnly: true,
    secure: false, // Updated to allow LAN/HTTP access
    sameSite: 'lax',
    maxAge: 60 * 60 * 24 * 7,
    path: '/'
}

3. Automated Session Handshake

The AuthContext now proactively performs a handshake with /api/auth/me on mount to synchronize the server-side cookie with the client-side state (LocalStorage).

// Location: budolshap-0.1.0/context/AuthContext.jsx
useEffect(() => {
    const loadUser = async () => {
        const result = await getCurrentUser(); // Handshake with server
        if (result && result.user) {
            setUser(result.user);
            setToken(result.token);
            localStorage.setItem('token', result.token);
        }
    };
    loadUser();
}, []);

Troubleshooting

  • Redirect Loop: Check if NEXT_PUBLIC_APP_URL matches the requesting host (e.g., http://192.168.1.24:3001).
  • Not Signed In After Redirect: Open Browser Console and look for [AuthContext] logs. If "No active session found" appears, check the budolshap_token cookie in the Application tab.
