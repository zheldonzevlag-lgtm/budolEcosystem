1. BudolPay QR Integration Fixes

Problem

Users reported that selecting BudolPay redirected them to the orders page immediately without showing the QR code, or the QR code modal would appear but fail to update the payment status.

Root Cause Analysis

  • Redirect Logic: OrderSummary.jsx was correctly redirecting to /payment/qr, but if data was missing or invalid, QRPaymentPage.jsx would redirect back to /orders.
  • ID Mismatch: BudolPayAdapter was using the internal transaction.id (UUID) as the paymentIntentId, but the Payment Gateway Service expects referenceId (starts with JON-) for status lookups.
  • Polling Endpoint: QRCodeModal.jsx was hardcoded to poll /api/paymongo/status, which strictly checked for IDs starting with INTENT-, causing BudolPay status checks to fail.

Implementation Details

1. Adapter Update (lib/payment/adapters/budolpay-adapter.js)

Modified the adapter to prioritize referenceId when returning the payment intent ID. This ensures consistency with the gateway's status endpoint.

// Old
const paymentIntentId = data.id || data.paymentIntentId || data.reference;

// New
const paymentIntentId = data.referenceId || data.id || data.paymentIntentId || data.reference;

2. Status API Enhancement (app/api/paymongo/status/route.js)

Updated the status API to accept a provider query parameter and recognize BudolPay-specific ID prefixes (JON-).

const provider = searchParams.get('provider');

if (intent_id.startsWith('INTENT-') || 
    intent_id.startsWith('JON-') || 
    provider === 'budolpay' || 
    provider === 'BUDOL_PAY') {
    // Use BudolPay Adapter
    // ...
}

3. Frontend Polling Update (components/payment/QRCodeModal.jsx)

Updated the polling logic to pass the provider parameter when the payment method is BudolPay.

const providerParam = paymentMethod === 'BUDOL_PAY' ? '&provider=budolpay' : '';
const response = await fetch(`/api/paymongo/status?intent_id=${paymentIntentId}${providerParam}`);

2. Mock Gateway Script Update

Updated scripts/mock-budolpay-gateway.cjs to redirect to /payment/return instead of a generic orders page, ensuring the correct post-payment flow is triggered even in mock scenarios.

3. Dependencies & Risks

  • Risk: If the Payment Gateway Service changes its ID format (e.g., stops using JON-), the regex check in the status API might need updating.
  • Mitigation: The provider=budolpay parameter serves as a robust fallback regardless of ID format.

4. Verification

Test: debug-adapter.mjs

Result: Successfully retrieved paymentIntentId starting with JON- and generated valid QR data.
