Vercel Deployment Plan: budolPay Admin (App)

Version: 27.0.0 Component: budolPay Admin Dashboard Target Platform: Vercel Serverless
Architecture Validation: A deep-dive review of `budolpay-0.1.0/apps/admin` reveals a robust Next.js 14 architecture ready for zero-config Vercel deployment. The app uses the `budolpay-monorepo` structural standards, resolving Prisma and Realtime limits naturally in serverless execution environments.

1. Vercel Project Configuration

Because the Admin app exists within a monorepo (`@budolpay/admin`), deploying requires properly mapping Vercel's build context to the workspace root to ensure shared packages (`@budolpay/database`) compile correctly.

Setting Recommended Configuration Rationale
Framework Preset Next.js Auto-detects the built-in routing layer and optimizes serverless endpoints.
Root Directory budolpay-0.1.0/apps/admin Tells Vercel where the application code resides while utilizing the Turborepo context.
Build Command (Leave Empty) Vercel naturally inherits the script npm run build which executes Next.js build natively.
Install Command (Leave Empty) Vercel detects the root package.json and automatically installs monorepo workspaces and triggers Prisma generation via the `postinstall` script.

2. Environment Mapping (Production)

Vercel relies strictly on environment injection for the deployment's secrets. Map the local `.env` architecture to production equivalents via Vercel's Project Dashboard:

Database Integrity

  • DATABASE_URL: Map to the production connection string (e.g., postgresql://.../budolshap_1db?schema=budolpay). Important: Ensure there is no trailing whitespace.
  • DIRECT_URL: Map to the direct non-pooled database reference for Prisma migrations or deep queries.

External Services Integration

  • NEXT_PUBLIC_SSO_URL: The official Public Endpoint for budolID (e.g., https://budol-id-sso.onrender.com).
  • BUDOLACCOUNTING_URL: Endpoint to the Ecosystem's accounting ledger (e.g., AWS ECS or Render instance).
  • JWT_SECRET: Match exactly the securely generated token signature from the rest of the ecosystem.

Media Storage (Cloudinary)

For KYC and merchant image assets to resolve correctly in the admin dashboard:

  • CLOUDINARY_API_KEY
  • CLOUDINARY_API_SECRET
  • NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME

3. Real-Time Serverless Abstraction

A major obstacle transitioning to Vercel is the persistence of long-running WebSocket connections. Our code review confirms the `lib/realtime-server.ts` handles this natively:

Socket.io Limitation: Vercel's 10-second serverless timeouts break persistent Socket.io connections.

Proposed Solution: The existing implementation gracefully falls back to SWR (Revalidation Polling), which is incredibly stable on Vercel. For advanced realtime requirements, the database SystemSetting for REALTIME_METHOD can be switched to PUSHER via external provider configuration.

4. Ecosystem Integration Architecture (Deep Dive)

During the deep dive study of the codebase, we analyzed exactly how budolPay Admin is glued together with budolShap and the rest of the ecosystem. Understanding this architecture is critical for a valid Vercel deployment:

1. Database Cross-Schema Aggregation

The Admin dashboard does not have its own isolated database. Instead, it relies on the turborepo package @budolpay/database. By connecting to the same DATABASE_URL as the frontend (which connects to the centralized budolshap_1db), the Admin app runs direct Prisma queries (e.g., prisma.user.findMany and prisma.transaction.findMany) against the live tables generated by budolShap and budolPay. There are no webhooks traversing data—the Admin app reads and writes directly to the golden source.

2. Gateway Real-Time Sync (Socket.io Bridge)

To provide real-time global transaction monitoring, the Admin frontend uses Socket.io to subscribe to the payment-gateway-service (via the NEXT_PUBLIC_GATEWAY_URL environment variable). When a user completes a checkout on budolShap, the gateway emits a new_transaction socket event that the Admin dashboard immediately captures and displays.

3. Identity & Single Sign-On (budolID)

Unlike standard administrative panels, authentication is completely outsourced to budolID. The NEXT_PUBLIC_SSO_URL environment variable directs administrators to authenticate centrally. Valid sessions emit ecosystem-standard JWT signatures validated strictly by the Admin service.

4. KYC Document Resolution

KYC identity documents are stored either as Base64/BYTEA blob data or as remote files (managed by Cloudinary or local drives). If remote URLs are relative, the Admin app automatically prepends the NEXT_PUBLIC_GATEWAY_URL to resolve standard images stored safely on the gateway service, seamlessly rendering budolShap verified documents on the Vercel-hosted admin UI.

5. Deployment Steps

  1. Ensure all changes to the `budolpay-0.1.0/apps/admin` app are pushed to GitHub.
  2. In Vercel, select Add New Project and import the `budolEcosystem` repository.
  3. Click Edit next to Root Directory and select budolpay-0.1.0/apps/admin.
  4. Expand Environment Variables and paste all mapped production keys.
  5. Click Deploy.

Once deployed, bind the assigned vercel application URL to budoladmin.duckdns.org using standard CNAME records.
