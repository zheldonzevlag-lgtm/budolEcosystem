Consolidated Vercel Deployment Configuration

Deployment Strategy

Due to Vercel's 15,000 file limit per deployment, the ecosystem uses a Hybrid Consolidated Approach:

  1. Backend Services: Consolidated into the primary budolecosystem Vercel project using --archive=tgz.
  2. Frontend Apps: Deployed as standalone projects (budolpay-admin, budolshap-ecosystem) and linked via Vercel Routing.

vercel.json Routing

The root vercel.json acts as the central ingress controller, mirroring the local Central Runner's port logic:


{
  "routes": [
    { "src": "/api/id/(.*)", "dest": "budolID-0.1.0/index.js" },
    { "src": "/api/(.*)", "dest": "budolpay-0.1.0/services/api-gateway/index.js" },
    { "src": "/admin/(.*)", "dest": "https://budolpay-admin.vercel.app/$1" },
    { "src": "/shap/(.*)", "dest": "https://budolshap-ecosystem.vercel.app/$1" },
    { "src": "/(.*)", "dest": "https://budolpay-admin.vercel.app/$1" }
  ]
}

Operational Procedures

  • Deployment Command: Use vercel --prod --yes --archive=tgz to bypass file count limits.
  • Health Monitoring: Run node scripts/test_scripts/verify-vercel-deployment.js after every deployment.
  • Prisma: Ensure DATABASE_URL is set in Vercel environment variables for budolAccounting and budolID.

Risk & Mitigation

  • Risk: File count exceeds 15k even with tarball.
  • Mitigation: Exclude node_modules from the tarball or move more services to standalone deployments.
