Developer Manual: Interactive Environment Selector

Purpose

To provide a streamlined, interactive way for developers to configure the ecosystem environment (Local vs. Vercel/Production) at the moment of execution, ensuring consistency across all sub-services.

Technical Details

1. Interactive Prompt Implementation

Using Node.js built-in readline module to create an asynchronous CLI interface:

const readline = require('readline');
const rl = readline.createInterface({
    input: process.stdin,
    output: process.stdout
});

function askQuestion(query) {
    return new Promise(resolve => rl.question(query, resolve));
}

2. Environment Switching Logic

The selector triggers the root package.json scripts using execSync to ensure the environment is set before the apps are spawned:

if (choice === '1') {
    execSync('npm run db:local', { stdio: 'inherit' });
} else if (choice === '2') {
    execSync('npm run db:vercel', { stdio: 'inherit' });
}

3. Pre-flight Validation Enhancement

The validation logic was updated to read NODE_ENV and VERCEL flags from .env files to correctly categorize the database connection:

const nodeEnvMatch = envContent.match(/^NODE_ENV=["']?(.+?)["']?$/m);
const vercelMatch = envContent.match(/^VERCEL=["']?(.+?)["']?$/m);

process.env.NODE_ENV = nodeEnvMatch ? nodeEnvMatch[1] : 'development';
process.env.VERCEL = vercelMatch ? vercelMatch[1] : undefined;

Test Results

  • Interactive Flow: Verified that the prompt appears and correctly maps inputs (1, 2, 3).
  • DB Switching: Verified that npm run db:local and npm run db:vercel are called with stdio: inherit.
  • Safety Barrier: Verified that if a developer selects "Local" but the .env still contains a production URL (and vice-versa), the ecosystem blocks startup with a clear error message.

Recommendations for Future Enhancement

  • Persistence: Allow saving the last choice as a default for subsequent runs.
  • Health Checks: Add a ping test to the database after switching to ensure the connection is actually valid.
