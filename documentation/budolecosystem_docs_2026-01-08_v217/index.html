1. Purpose

The purpose of this update is to resolve a critical registration failure in the BudolEcosystem where users were unable to register due to unhandled duplicate phone number formats and generic error propagation from the BudolID centralized identity service.

2. Explanation & Reason

The system previously allowed duplicate registrations if the phone numbers were formatted differently (e.g., "9484099300" vs "+639484099300"). Additionally, when a registration failed in the BudolID microservice, the BudolShap frontend would often display a generic "Centralized Identity service is currently unavailable" message, even for specific validation errors like duplicate phone numbers.

3. Issues Identified

  • Phone Number Formatting: Lack of normalization allowed bypass of unique constraints.
  • Error Propagation: BudolShap API routes were swallowing specific error messages from BudolID.
  • Service Stability: BudolID service was crashing due to path resolution issues for new utilities.

4. Fixes Implemented

  • Phone Normalization: Created a utility in budolID-0.1.0/api/lib/utils.js to enforce E.164 format for Philippines phone numbers (+63...).
  • BudolID Validation: Updated budolID-0.1.0/api/index.js to perform explicit checks using normalized phone numbers before user creation.
  • Error Propagation: Modified budolshap-0.1.0/app/api/auth/register/route.js to propagate specific error messages (e.g., "This phone number is already registered") back to the client.
  • Path Fix: Moved the utility library into the api/ folder of BudolID to ensure correct relative path resolution during execution.

5. Technical Details

Code Changes

budolID-0.1.0/api/lib/utils.js:

function normalizePhone(phone) {
    if (!phone) return null;
    let cleaned = phone.replace(/\D/g, '');
    if (cleaned.startsWith('63')) return '+' + cleaned;
    if (cleaned.startsWith('0')) return '+63' + cleaned.substring(1);
    if (cleaned.length === 10) return '+63' + cleaned;
    if (phone.startsWith('+')) return phone;
    return '+' + cleaned;
}

7. Risks & Mitigation

  • Risk: Normalization might incorrectly format international numbers.
  • Mitigation: Currently focused on Philippines (+63). Future updates should use a robust library like libphonenumber-js for international support.

8. Dependencies

  • BudolID Service (Port 8000)
  • BudolShap Service (Port 3000)
  • Prisma Database Client

9. Recommendations for Future Enhancement

  • Implement international phone number validation using libphonenumber-js.
  • Add more granular error logging for microservice communication failures.
  • Automate end-to-end registration tests in the CI/CD pipeline.
