Initialization Flow Refactoring

The primary goal of v346 was to prevent the app from getting stuck during the loading phase. This was achieved by decoupling the core app initialization from network-dependent tasks.

1. ApiService Background Initialization

The _initialize method in api_service.dart now triggers a background initialization process that doesn't block the main init() future.


// lib/services/api_service.dart
Future<void> _initialize() async {
  try {
    // ... load local settings (prefs, host, deviceId) ...
    
    // Non-blocking background initialization for network-dependent tasks
    _backgroundInit();
    
    if (kDebugMode) {
      print('ApiService: Basic initialization complete...');
    }
  } catch (e) {
    if (kDebugMode) print('ApiService: Critical error during initialization: $e');
  }
}

Future<void> _backgroundInit() async {
  await fetchSystemSettings();
  if (isAuthenticated) {
    await _fetchUserProfile();
  }
  notifyListeners();
}

2. Splash Screen Navigation Logic

In splash_screen.dart, we removed the redundant apiService.init() call (as it's already handled in main.dart) and implemented smarter navigation based on the authentication state.


// lib/screens/splash_screen.dart
Future<void> _navigateToNext() async {
  await Future.delayed(const Duration(seconds: 2));
  if (!mounted) return;
  
  final apiService = context.read<ApiService>();
  if (apiService.isAuthenticated) {
    Navigator.pushReplacementNamed(context, Routes.home);
  } else {
    Navigator.pushReplacementNamed(context, Routes.login);
  }
}

3. Host Normalization for Web

The host getter in ApiService remains optimized for web by utilizing Uri.base.host, ensuring the app correctly identifies the backend server when accessed over a LAN.

Compliance Check
