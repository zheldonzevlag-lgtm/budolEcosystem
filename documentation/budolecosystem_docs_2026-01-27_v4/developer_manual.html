Developer Manual - v541

Phase 8: Dependencies & Software Rationale

Core Technologies

  • Prisma ORM (v6.2.1): Chosen for its type-safe database access and automated migrations. It ensures that our complex relational data (Orders, Items, Payments) remains consistent across the ecosystem.
  • Next.js (v14.2.3): Utilized for its robust server-side rendering (SSR) and API routes, enabling high-performance frontend delivery and seamless backend integration.
  • SWR (Vercel): Selected for realtime data fetching. It handles caching, revalidation, and optimistic UI updates, ensuring users see the most up-to-date order statuses without manual refreshes.
  • Pusher Channels: Implementation choice for ecosystem-wide realtime notifications. It provides low-latency websocket communication for instant order updates across mobile and web platforms.

Compliance Libraries

  • dotenv: Ensures secure management of environment variables (API keys, database credentials), essential for NPC and PCI DSS compliance.
  • jsonwebtoken: Powers our secure SSO (BudolID), providing stateless and verifiable authentication across microservices.

Technical Overview

The core of the "Missing Orders" issue was located in the order retrieval logic within the ordersService.js. The service used a flag excludeAbandonedPayments which defaults to true.

Root Cause Analysis

In ordersService.js, the filter was structured to hide any non-COD order that was in awaiting_payment status for more than 30 minutes. This was intended to clean up the UI from abandoned carts but accidentally hid valid orders where the paymentStatus was not synchronized after a successful transaction.

Phase 7: Monitoring & Audit

To prevent future occurrences, we have implemented monitoring logs in the retrieval service:

// Monitoring log added in v541
console.log(`[Monitoring] Applying Abandoned Payment Filter. Threshold: ${thirtyMinutesAgo.toISOString()}`);

These logs allow administrators to verify when the filtering logic is active and identify if legitimate orders are being caught in the exclusion criteria.

Implementation Details

1. Webhook Update

Modified budolpay/route.js to ensure paymentStatus is set to 'paid' when a payment is successful.

await updateOrderStatus(orderId, 'processing', {
    paymentStatus: 'paid',
    isPaid: true
});

2. Legacy Data Migration

Created and executed fix-paid-orders-status.mjs to fix 30+ legacy orders that were hidden.

3. Dependency Resolution

Fixed monorepo imports in BudolPay services. Services now correctly reference @budolpay/database and @budolpay/notifications packages.
