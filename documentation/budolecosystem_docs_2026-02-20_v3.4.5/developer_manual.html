Developer Manual

This document details the critical changes made in v3.4.5 to support multi-store payments.

Purpose

When a user places an order containing items from multiple stores (e.g., Store A and Store B), the system creates multiple Order records but processes a single Payment transaction. The previous implementation only linked the payment to the first order ID, leaving subsequent orders in an "UNPAID" state.

Technical Implementation

1. Frontend: OrderSummary.jsx

Modified to capture the full array of created Order IDs.

// Old:
const orderId = Array.isArray(data) ? data[0]?.id : data.id;

// New:
const orderIds = Array.isArray(data) ? data.map(o => o.id) : [data.id];
// Sent to API:
body: JSON.stringify({ ..., orderIds, ... })

2. Backend: Payment Checkout API

Modified /api/payment/checkout to iterate through orderIds and link each one to the generated paymentIntentId in the database. This ensures that even if webhooks fail, the relationship is established.

3. Webhooks & Adapters

The Payment Adapters now include orderIds in the metadata sent to the gateway. The Webhook handlers (PayMongo & BudolPay) have been refactored to:

  • Parse orderIds from the webhook event metadata.
  • Iterate through each ID and update the corresponding Order status to PAID.
  • Fallback: If metadata is missing, query the database for all orders with the matching paymentIntentId (set in step 2).
