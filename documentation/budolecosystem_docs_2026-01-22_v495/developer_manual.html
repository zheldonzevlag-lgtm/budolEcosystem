Developer Manual: Realtime Architecture

Technical deep-dive into the multi-provider realtime system.

Architecture Overview

The Budol Ecosystem uses a hybrid realtime strategy that respects administrative configuration. It supports three modes:

  • POLLING: Client-side SWR periodic fetching (Default).
  • PUSHER: WebSockets via Pusher.com (Hosted).
  • SOCKET_IO: Self-hosted WebSockets via a dedicated microservice.

Core Component: RealtimeProvider

The RealtimeProvider.jsx is the heart of the system. It fetches system settings and initializes the appropriate client.

// RealtimeProvider.jsx
const { data: configData } = useSWR('/api/system/settings', fetcher, {
    dedupingInterval: 60000 // Cache for 1 min
})

useEffect(() => {
    if (!configData) return
    const provider = configData.realtimeProvider || 'POLLING'
    
    if (provider === 'PUSHER') {
        const pusherInstance = new Pusher(configData.pusherKey, { cluster: configData.pusherCluster })
        setPusher(pusherInstance)
    } else if (provider === 'SOCKET_IO') {
        const socketInstance = io(configData.socketUrl)
        setSocket(socketInstance)
    }
}, [configData])

Server-Side Triggering

Events are triggered via realtime.js, which dynamically routes the event based on the active provider.

// lib/realtime.js
export async function triggerRealtimeEvent(channel, event, data) {
    const settings = await getSettings();
    if (settings.realtimeProvider === 'PUSHER') {
        const pusher = new Pusher({...});
        await pusher.trigger(channel, event, data);
    } else if (settings.realtimeProvider === 'SOCKET_IO') {
        await fetch(`${settings.socketUrl}/trigger`, {...});
    }
    // POLLING requires no server-side trigger
}

API Integration

The client fetches configuration from route.js, which uses the systemSettingsService.js to retrieve the latest configuration.
