Developer Manual

Technical documentation for the SSO schema fix and system synchronization.

Purpose & Rationale

The primary objective of this update was to resolve a critical 500 Internal Server Error encountered by users attempting to log in via SSO. The error was traced to a desynchronization between the application's Prisma models and the physical database schema in the budolID service.

Technical Explanation

1. SSO Login Route Implementation

The budolID service handles SSO requests via the `/auth/sso/login` endpoint. It validates the API key, authenticates the user, and creates a session.

app.post('/auth/sso/login', async (req, res) => {
    const { email, password, apiKey, redirect_uri } = req.body;
    // ... validation and authentication logic
    const user = await prisma.user.findUnique({ where: { email } });
    // ... session creation
});
View budolID api/index.js

2. Database Schema Synchronization

The error occurred because the `User.faceTemplate` column was missing in the database but required by the Prisma client. We synchronized the schema using the ecosystem's master schema.

// Master Schema Fragment
model User {
  id                 String                 @id @default(uuid())
  email              String                 @unique
  // ... other fields
  faceTemplate       String?
}
View Master schema.prisma

Changes & Fixes

  • Identified missing faceTemplate column in budolid database.
  • Executed prisma db push to synchronize the database with the Prisma model.
  • Created seed-test-user.js to verify authentication flow with known credentials.
  • Updated check-db.js for improved diagnostic reporting.

Phase 8: Dependencies & Software Choices

Prisma ORM

Chosen for its type-safe database access and automated migration capabilities. It ensures that the database schema is always in sync with the application models, which was the primary fix in this release.

Express.js

A minimalist and flexible Node.js web application framework used for the budolID SSO service. Its middleware-based architecture allows for efficient handling of authentication flows and JWT issuance.

PostgreSQL

Our primary relational database for Identity and Session management. It provides robust ACID compliance and advanced features like JSONB support, which are essential for storing complex user profiles and session data.

JWT (JSON Web Tokens)

Used for secure, stateless session management across the ecosystem. JWTs allow us to share user identity between services (budolID, budolPay, budolShap) without constant database lookups.
