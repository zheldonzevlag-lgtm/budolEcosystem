Local DNS & Service Discovery Feasibility Report

Date: 2026-01-04 | Version: 1.0

1. Purpose & Explanation

The goal is to eliminate the need for users to manually enter IP addresses in the mobile app when moving between network environments (home, office, cafe). We analyzed two primary methods: Local DNS (Static) and mDNS/ZeroConf (Dynamic).

2. Reason for Consideration

Manual IP entry is error-prone and creates friction for non-technical users. Automated discovery provides a "magic" user experience where the app simply works as soon as it connects to the local network.

3. Technical Details

Option A: mDNS (Multicast DNS)

Uses the _http._tcp service type to broadcast the ecosystem's presence. Libraries like nsd for Flutter and bonjour-service for Node.js are the industry standard for this.

Difficulty: Moderate Requires app permission changes and code integration.

Option B: Local DNS Server

Uses a standard DNS server to map a hostname (e.g., budol.local) to the server's IP. No app changes are needed if the app is already configured to use hostnames.

Difficulty: Low Requires infrastructure setup outside the codebase.

4. Issues & Risks

  • Network Isolation: Some routers block multicast/mDNS traffic (especially in corporate or public Wi-Fi).
  • iOS Permissions: iOS 14+ requires explicit NSLocalNetworkUsageDescription and NSBonjourServices in Info.plist.
  • Android Permissions: Requires CHANGE_WIFI_MULTICAST_STATE.
  • Power Consumption: Continuous scanning for services can impact mobile battery life.

5. Mitigation Strategies

  • Fallback to Manual: Always keep the manual IP entry (added in v145) as a fallback if auto-discovery fails.
  • On-Demand Scanning: Only scan when the app opens or when the "Connect" button is pressed, rather than continuously in the background.
  • Clear Error Messaging: Inform the user if the network environment blocks discovery.
  • Environment Guard: Wrap discovery in a production check to prevent accidental local scans in the live app.

6. Environment Switching (Local vs. Vercel)

Switching between Local (mDNS) and Vercel (Production) is highly straightforward using Flutter's environment flags. We can use a constant flag to gate the discovery logic:

// Example implementation logic
bool get isProduction => const String.fromEnvironment('APP_ENV') == 'production';

void startDiscovery() {
  if (isProduction) {
    print("mDNS disabled in production.");
    return;
  }
  // Proceed with mDNS scan for local server...
}

When deploying to Vercel, the build command will include --dart-define=APP_ENV=production, which automatically disables all local discovery logic and forces the app to use the hardcoded Vercel API URLs.

7. Dependencies

  • Mobile: nsd: ^4.0.3
  • Server: bonjour-service: ^1.2.1

7. Test Results (Simulated)

Simulated discovery logic shows that mDNS is highly effective in standard home/office LANs. Discovery time is typically < 2 seconds.

8. Recommendation for Future Enhancement

Implement mDNS discovery as the primary connection method, with the manual IP configuration (v145) as a robust secondary option. This provides the best balance of user experience and reliability.

9. Roadmap

  2. Update Mobile App permissions and add nsd dependency (Phase 2)
  3. Implement DiscoveryService in Flutter (Phase 3)

10. Server-Side Integration (Ecosystem Services)

For the mobile app to discover the ecosystem, the server-side services must announce their presence. This will be integrated into the start-ecosystem.js runner.

Implementation Strategy:

  • Centralized Broadcast: The start-ecosystem.js script will initialize a bonjour-service instance and publish a primary service (e.g., budolecosystem-gateway) on port 8080.
  • Service-Specific Broadcast: Each microservice (Auth, Wallet, etc.) can optionally publish its own service record to allow direct discovery by other services in the future.
  • Package Requirement: bonjour-service is a pure JavaScript library, making it compatible across Windows, Mac, and Linux without native dependencies.
// Proposed start-ecosystem.js integration
const { Bonjour } = require('bonjour-service');
const bonjour = new Bonjour();

// After starting services
bonjour.publish({ 
  name: 'BudolPay API Gateway', 
  type: 'http', 
  port: 8080,
  txt: { version: '1.0.0', env: 'local' } 
});

Cross-Service Discovery:

While services currently use environment variables injected at startup, they can be updated to use mDNS for dynamic discovery if IP addresses change while the ecosystem is running. This improves robustness in dynamic network environments.

11. Philippine ISP Compatibility Analysis

A critical requirement is ensuring the solution works with major Philippine DSL/Fiber providers (PLDT, Globe, Converge). Since mDNS is the foundation for common consumer tech like Chromecast, AirPlay, and Spotify Connect, it is widely supported by ISP-provided routers.

Provider-Specific Notes:

  • PLDT Home Fibr (Huawei/FiberHome): Generally supports mDNS out of the box. Users should ensure "AP Isolation" is disabled in the 2.4G/5G settings to allow Wi-Fi devices to communicate.
  • Globe At Home (Huawei/ZTE): High compatibility. Standard multicast settings usually allow discovery between Wi-Fi and Ethernet clients without modification.
  • Converge ICT (ZTE/Huawei): Compatible, though some ZTE models may require "IGMP Snooping" to be enabled if discovery between different Wi-Fi bands (2.4GHz vs 5GHz) fails.

Common Troubleshooting for PH Users:

  • AP Isolation: If enabled, the mobile app cannot "see" the server even on the same Wi-Fi. This must be turned OFF.
  • Multicast/IGMP: Ensure these are not explicitly blocked in the "Security" or "Advanced" firewall settings of the router.
  • Guest Wi-Fi: Most Philippine routers isolate Guest networks. The app and server must be on the Main Wi-Fi network.
