Developer Manual - Transaction Fixes & Lookup Endpoints

Purpose

This update resolves the critical mismatch between SSO user identities and BudolPay internal transaction processing, ensuring that recipients can be correctly identified by email and that transactions adhere to database integrity constraints.

Technical Details

1. Recipient Lookup Endpoints

New endpoints were implemented to allow the mobile app and transaction service to resolve email addresses into internal UUIDs:

  • SSO (budolID): GET /auth/user/find?email=... - Searches the SSO database for a user and returns their public profile (ID, Name, Avatar).
  • Internal Auth (budolpay): GET /user/find?email=... - Searches the local BudolPay database for the same user, ensuring cross-service identity consistency.

2. Foreign Key Constraint Resolution

The Transaction model requires senderId and receiverId to exist in the local User table. We verified that the setup-test-users.js script correctly synchronizes IDs between the budolid and budolpay databases during development, preventing Foreign key constraint violated errors.

3. Socket.io Event Mapping

Real-time notifications were standardized across services:

  • new_transaction: Emitted to the admin room for global monitoring.
  • balance_update: Emitted to specific user:{userId} rooms to update mobile app UI instantly.
  • transaction_update: Emitted to user rooms with descriptive messages (e.g., "Wallet balance increased by PHP 1000").

Issues & Fixes

Issue: Send Money failed because the recipient's email couldn't be resolved to a valid ID in the Transaction Service.
Fix: Implemented /auth/user/find in budolID-0.1.0/index.js and updated api_service.dart to use this for recipient lookup.
Issue: Admin dashboard didn't show new transactions without a refresh.
Fix: Added notifyAdmin helper in Transaction Service to push events to the API Gateway's Socket.io admin room.

Dependencies

  • Prisma ORM: Used for all database queries and constraint management.
  • Socket.io-client: Used in verification scripts to monitor real-time event propagation.
  • Axios: Used for inter-service communication (Transaction -> Wallet -> Gateway).

Recommendations

Implement a User Sync Webhook: When a user is created in SSO, automatically trigger a creation event in BudolPay to ensure IDs are always synchronized without manual scripts.
