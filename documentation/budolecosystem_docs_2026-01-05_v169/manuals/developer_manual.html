Developer Manual: Wallet & KYC Architecture

Technical specifications for the v169 core upgrades including Mobile, Backend, and Database layers.

1. Purpose & Architectural Overview

Purpose: To implement a production-grade KYC verification pipeline and expand wallet functionality to include real-world transaction simulations.

Explanation: v169 introduces the Verification Service, a new microservice that handles document processing, image optimization, and state management for user identity verification.

Reason: Decoupling verification from the main Auth or Wallet services allows for independent scaling and easier integration of third-party ML/AI providers in the future.

Flutter 3.x Node.js Prisma ORM Google ML Kit Sharp (Image Processing)

2. Technical Details: Mobile (Flutter)

KYC Flow Implementation

The KYC flow is managed via kyc_verification_screen.dart, which utilizes a PageView for step-by-step navigation. Key technical components include:

  • On-Device ML Kit: Uses google_mlkit_text_recognition for OCR and google_mlkit_face_detection for liveness checks.
  • Image Picking: Integrated with image_picker to handle camera and gallery access.
  • State Management: Uses Provider (via ApiService) to handle authentication tokens and base URLs.

3. Technical Details: Backend (Verification Service)

The verification-service handles the heavy lifting of image optimization and database storage.

Image Optimization Pipeline

To reduce storage costs and bandwidth, all uploads are processed using sharp before being saved to the database.

// Image Optimization Logic
const optimizedBuffer = await sharp(file.buffer)
    .resize(1200, 1200, { fit: 'inside', withoutEnlargement: true })
    .jpeg({ quality: 80 })
    .toBuffer();

API Proxying

The API Gateway proxies /verification requests to the internal service running on port 8006.

4. Database Schema (Prisma)

The VerificationDocument model tracks all user-submitted identity documents.

model VerificationDocument {
  id           String   @id @default(uuid())
  userId       String
  type         String   // ID_FRONT, ID_BACK, SELFIE
  documentType String   // PASSPORT, DRIVERS_LICENSE, etc.
  status       String   @default("PENDING")
  blobData     Bytes?   // Local storage for images (BYTEA)
  ocrData      Json?    // Metadata extracted via ML
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
  user         User     @relation(fields: [userId], references: [id])
}

7. Impact Analysis & Risks

  • Performance: Heavy image processing on the backend may increase latency during peak times. Mitigation: Move optimization to serverless functions in Phase 4.
  • Security: Storing raw ID images in BYTEA is suitable for development but not production. Mitigation: Implement S3/Minio storage provider.
  • Dependency Risk: Reliance on Google Play Services for ML Kit. Mitigation: Provide fallback manual upload if Play Services are missing.
