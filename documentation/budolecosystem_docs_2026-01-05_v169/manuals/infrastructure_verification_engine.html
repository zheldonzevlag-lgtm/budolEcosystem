Infrastructure & Verification Engine

Evaluating storage options and the internal logic of the KYC verification system.

1. Document Storage: Vercel vs AWS S3

Storing identity documents requires extreme security, long-term durability, and compliance with Philippine data privacy laws (NPC).

Vercel Blob

Pros: Zero-config for Vercel users, built-in Edge network, easy API.

Cons: Primarily for public assets. Private blobs require signed URLs which can be complex to manage at scale for PII. Limited granular IAM policies.

RECOMMENDED

AWS S3 + CloudFront

Pros: Industry standard for PII. Granular IAM roles, Object Lock (for audit compliance), and native AES-256 encryption.

Cons: Requires separate AWS account management and configuration.

Decision Matrix

Feature Vercel Blob AWS S3
Encryption Server-side (standard) Server-side (KMS managed)
Access Control Token-based IAM, Bucket Policies, VPC Endpoints
Data Residency Global (Edge) Region-specific (e.g., ap-southeast-1)

2. The Verification Engine Architecture

The "Verification Engine" is the brain of the verification-service. It coordinates automated checks and human reviews.

[Client App] --(ID + Selfie)--> [API Gateway]
                              |
                        [Verification Engine]
                       /            |
         [Automated Logic]       [Manual Queue]
        (OCR, Face Match)      (Admin Dashboard)

How it works:

  1. Ingestion: The engine receives images, validates file types, and compresses them for storage.
  2. Automated Screening:
    • OCR Check: Extracts name/DOB and matches against database records.
    • Face Comparison: Compares the selfie against the ID photo using Amazon Rekognition or similar.
    • Liveness Score: Validates the liveness data sent from the Flutter app.
  3. Decision Matrix:
    • If confidence > 95% → Auto-Approve.
    • If confidence 70-95% → Flag for Manual Review.
    • If confidence < 70% → Auto-Reject (request recapture).
  4. Tier Update: Upon approval, the engine triggers a webhook to user-service to upgrade the user's KYC tier.

3. Security Considerations

  • Short-lived URLs: Never store permanent links to IDs. Use pre-signed URLs with a 5-minute expiry.
  • PII Masking: The engine should only store "Hashes" or masked versions of ID numbers unless specifically required for compliance.
  • Rate Limiting: Prevent brute-force verification attempts by limiting a user to 3 attempts per 24 hours.
