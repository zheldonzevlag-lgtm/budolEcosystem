System Admin Manual - v61

Testing Infrastructure & Rate Limiting

1. Rate Limit Management

During high-frequency automated testing, the login endpoint may trigger a 429 Too Many Requests error. This is managed via the RateLimit table in the database.

Manual Reset: To reset rate limits for a specific test user during development, you can run the following SQL or use the provided clear-rate-limit.js script: 
DELETE FROM RateLimit WHERE "key" LIKE '%user@email.com%';

2. Environment Configuration

Ensure the following environment variables are correctly set for the BudolPay and Lalamove integrations to function in the test environment:

  • LALAMOVE_CLIENT_ID: Sandbox API Key
  • LALAMOVE_CLIENT_SECRET: Sandbox API Secret
  • LALAMOVE_ENV: Set to sandbox
  • BUDOLPAY_API_KEY: Integration API Key
  • BASE_URL: Should be http://localhost:3001 for the Auth/Payment services.

3. Database Integrity

The test script creates real orders in the database. For clean test runs, ensure the Order, Payment, and Shipping tables are periodically pruned or that unique identifiers (like email) are used for each test cycle.

Operational Runbook

Lalamove Quote Failures

If the Lalamove API returns a 500 Missing pickup info error, verify that the shippingService.js is receiving the full contact object (name, phone) for both pickup and delivery stops. Coordinates alone are insufficient for booking.

BudolPay Intent Failures

If BudolPay fails to return an intent ID, check the gateway logs. The system now supports a fallback to the reference field if the id field is missing in the response payload.
