Mobile Resilience & Error Handling Architecture

← Back to v164 Index

Purpose & Explanation

The primary goal of v164 was to eliminate the "white screen" crash occurring during mobile app initialization. This crash was typically caused by unhandled exceptions in the build method or during main() initialization before the widget tree was established.

Technical Details: Global Error Boundary

We implemented a custom ErrorWidget.builder in main.dart. This catches any exception thrown during a widget's build phase and replaces the crash with a user-friendly error UI.

// lib/main.dart
ErrorWidget.builder = (FlutterErrorDetails details) {
  return Scaffold(
    body: Center(
      child: Column(
        children: [
          Icon(Icons.error_outline),
          Text('Oops! Something went wrong.'),
          Text(details.exceptionAsString()),
          ElevatedButton(
            onPressed: () => apiService.logout(),
            child: Text('Reset App'),
          ),
        ],
      ),
    ),
  );
};

Defensive Data Parsing

In HomeScreen, we added multiple layers of safety when parsing transaction data from the backend. This prevents TypeError crashes if the backend returns unexpected data types (e.g., Decimal as String vs Number).

// lib/screens/home_screen.dart
double amount = 0.0;
if (tx['amount'] != null) {
  if (tx['amount'] is num) {
    amount = (tx['amount'] as num).toDouble();
  } else {
    amount = double.tryParse(tx['amount'].toString()) ?? 0.0;
  }
}

Issues & Fixes

  • Issue: White screen on app start.
    Fix: Added 5-second timeout to apiService.init() in main.dart and wrapped in try-catch.
  • Issue: TypeError in transaction list.
    Fix: Added try-catch around the map function in HomeScreen build.
  • Issue: Socket data mismatch.
    Fix: Added type checks (is Map) before calling Map.from() in SocketService.

Test Results

Verified via test-transaction-history-v163.js. The backend correctly returns transaction data, and the mobile app (simulated logic) successfully parses various formats of the amount field without crashing.
