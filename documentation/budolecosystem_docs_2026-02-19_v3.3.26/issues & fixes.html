Issues & Fixes

Version: v3.3.26

Date: 2026-02-19

Resolved Issues

Issue 1: Persistent Masked Data in Edit Profile

Description: Even when toggling the visibility icon, the "First Name" and "Last Name" fields continued to show masked values (e.g., "M*****") instead of the real data.

Fix:
  • Backend: The /verify endpoint was updated to return unmasked firstName, lastName, and phoneNumber. Previously, the mobile login endpoints were returning masked data which was being cached.
  • Mobile: The ApiService.fetchUserProfile method now explicitly calls /verify to retrieve this unmasked data when the profile screen loads.

Issue 2: Potential Data Corruption (Saving Masked Data)

Description: There was a risk that if a user saved their profile while the fields contained asterisks, the masked string would overwrite the actual name in the database.

Fix:
  • Mobile: Added a validation check in edit_profile_screen.dart. The _handleUpdate method now scans for asterisks and aborts the save operation if found, prompting the user to wait for data to load.

Issue 3: Edit Profile Screen Stale Data

Description: The screen sometimes showed old or empty data because the UI didn't reliably trigger a refresh from the server upon entry.

Fix:
  • Mobile: Implemented WidgetsBinding.instance.addPostFrameCallback in the initState of EditProfileScreen. This forces a call to ApiService.fetchUserProfile() every time the screen is opened.

Issue 4: Duplicate /verify Endpoint

Description: A duplicate /verify endpoint existed in the auth-service code, causing unexpected response fields (e.g., ecosystem: 'budol') and potentially conflicting logic.

Fix:
  • Backend: Removed the duplicate code block and consolidated all verification logic into a single, robust /verify endpoint.
