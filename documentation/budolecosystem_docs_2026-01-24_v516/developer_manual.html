Developer Manual - Version v516

Purpose & Rationale

The primary objective of this release was to eliminate hardcoded merchant names in the payment flow. Previously, "budolShap Merchant" was hardcoded in several frontend files, causing confusion for users purchasing from specific stores like "Online Micro Sellers Multipurpose Cooperative".

Technical Explanation

The solution involved tracing the data flow from the cart state to the QR generation API. We implemented dynamic store name resolution in the OrderSummary component and ensured it propagated through the query parameters to the standalone QR display page.

Key Changes

  • Dynamic Store Name Calculation: Aggregated store names from cart items to create a display name (e.g., "Store A +1").
  • Payload Enrichment: Injected storeName into the QR code payload (type: budolpay_payment).
  • Mobile UI Refinement: Updated PaymentSummaryScreen to prioritize storeName and use compact typography.

Dependencies & Software Choice

  • Next.js 14: Chosen for server-side rendering of payment pages and API routes.
  • Flutter 3.x: Cross-platform mobile development for BudolPay.
  • SWR: For real-time data fetching and cache management in the web frontend.
  • Prisma: Type-safe database access for transaction logging.

Compliance

PCI DSS & NPC Compliance: This change ensures that the store name displayed matches the actual transaction intent, preventing phishing or merchant misidentification. No PII is stored in the QR payload.
