1. Initial Assessment & Readiness

The budolEcosystem consists of 7 core microservices (Auth, Gateway, Payment, Settlement, Transaction, Verification, Wallet). The primary objective is to move from a local Docker-based environment to a hybrid cloud environment (Vercel for Frontend/Functions, AWS for Storage/DB/Sockets).

Current Constraints

  • Socket.io requires persistent connections (Incompatible with Vercel Serverless).
  • Prisma ORM needs database connection pooling (AWS RDS + Prisma Accelerate recommended).
  • KYC Assets currently local (Migrating to AWS S3).

2. Phase-Based Project Decomposition

Phase 1: Foundation & Data Layer

Status: Pending

Objective: Establish the cloud database and storage infrastructure to support all microservices.

  • Tasks:
    • Provision AWS RDS (PostgreSQL) Free Tier.
    • Configure AWS S3 Bucket for KYC assets.
    • Set up IAM roles for service-to-service communication.
  • Entry Criteria: AWS Account verified and local database backup ready.
  • Exit Criteria: Database connection string verified from local environment.

Phase 2: Service Refactoring (Vercel Readiness)

Status: Pending

Objective: Modify microservices to run as Vercel Serverless Functions.

  • Tasks:
    • Refactor index.js to export handlers for Vercel.
    • Update package.json dependencies for cloud runtimes.
    • Align environment variables with Vercel naming conventions.
  • Dependency: Phase 1 (Database must be reachable).
  • Exit Criteria: vercel dev runs successfully for each service locally.

Phase 3: Persistent Layer (AWS EC2 / Sockets)

Status: Pending

Objective: Deploy the Socket-heavy services to a persistent AWS EC2 instance.

  • Tasks:
    • Deploy api-gateway with Socket.io to EC2.
    • Configure Nginx as a reverse proxy for Sockets.
    • Set up Domain CNAME for api.budolpay.com.
  • Dependency: Phase 2 completion for core business logic.

Phase 4: Global Deployment & Optimization

Status: Pending

Objective: Final domain mapping and performance tuning.

  • Tasks:
    • Deploy Frontend to Vercel production.
    • Implement AWS CloudFront for asset caching.
    • End-to-end security audit (JWT validation).

3. Technical Specifications

Service Target Platform Critical Env Vars Architectural Impact
API Gateway AWS EC2 (Persistent) PORT, SOCKET_AUTH_SECRET Handles all real-time traffic; acts as reverse proxy.
Auth Service Vercel (Serverless) DATABASE_URL, JWT_SECRET Moved to ephemeral functions; faster scaling.
Wallet Service Vercel (Serverless) DATABASE_URL, LEDGER_KEY Strict atomic transaction requirements.

4. Testing & Validation Protocols

Quality Gates

  • Unit Testing: Every refactored function must pass existing Jest/Mocha tests.
  • Integration Testing: Vercel functions must successfully query AWS RDS.
  • E2E Testing: Full transaction flow from Mobile App -> Gateway -> Transaction Service -> RDS.
  • Performance: Socket connection latency must be < 200ms globally.

5. Risk Management & Rollback

Risk Impact Mitigation Rollback Trigger
Database Latency High Use AWS RDS Read Replicas > 2s query time
Socket Drops Critical AWS NLB (Network Load Balancer) > 5% disconnect rate

6. Operational Guidelines

Post-deployment, all logs will be centralized in AWS CloudWatch. CI/CD will be handled via GitHub Actions for Vercel and AWS CodePipeline for EC2 components.
