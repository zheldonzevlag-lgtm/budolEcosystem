Consolidated Recommendations for Future Enhancement

1. Event-Driven Architecture (Message Broker)

Purpose: To decouple cross-service status updates and improve system reliability.

Explanation: Currently, services communicate via direct HTTP webhooks. If a receiving service is down, the update is lost. Implementing a message broker like RabbitMQ or Redis Pub/Sub would allow for "at-least-once" delivery guarantees.

Reason: Resolves the current "race condition" where the gateway service might process a webhook before the wallet service, or vice versa, leading to manual re-triggering logic.

2. Unified User Identification (BudolID)

Purpose: To eliminate the need for manual "User creation on-the-fly" in sub-services.

Explanation: A centralized identity service should propagate user creation events to all sub-services (Wallet, Shap, Loan) via a background worker or event bus.

Reason: Reduces the complexity of sub-service code and ensures database integrity across the entire ecosystem.

3. Standardized Currency & Precision Library

Purpose: To prevent floating-point errors and parsing issues across different platforms (Mobile vs Web vs Backend).

Explanation: Use a shared package that defines standard currency types and precision rules (e.g., always using centavos/integers for calculations and only converting to decimals for display).

Reason: Fixes the root cause of the "0.00 balance" parsing issues encountered in this version.

4. Centralized Webhook Registry

Purpose: To manage cross-service notifications dynamically.

Explanation: A service where sub-apps can register their webhook endpoints for specific event types (e.g., payment.success).

Reason: Simplifies the logic in payment-gateway-service, which currently has hardcoded logic for budolShap.
