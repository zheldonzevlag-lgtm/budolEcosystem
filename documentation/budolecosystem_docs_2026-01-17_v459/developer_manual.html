Technical Architecture & Changes

This release focuses on UI stability and diagnostic capabilities within the Flutter Web environment. The core issue addressed was a race condition/rendering failure during the transition from the splash screen to the marketing ads screen.

1. Rendering Guards in MarketingAdsScreen

We implemented explicit sizing constraints and a Material wrapper to ensure that the Flutter web engine correctly calculates the layout even if the initial window dimensions are reported incorrectly by the browser.

File: marketing_ads_screen.dart

// Robust size detection for Web
final double width = constraints.maxWidth.isFinite && constraints.maxWidth > 0 
    ? constraints.maxWidth 
    : 393;
final double height = constraints.maxHeight.isFinite && constraints.maxHeight > 0 
    ? constraints.maxHeight 
    : 780;

Rationale: On some web browsers, maxWidth or maxHeight might be reported as infinite or zero during the very first frame after a route change. Defaulting to standard mobile dimensions prevents the "black screen" caused by zero-size widgets.

2. Diagnostic Navigation Controls

To assist in debugging navigation hangs in production-like web environments, we added "FORCE ADS" and "FORCE LOGIN" buttons to the splash screen when in debug mode.

File: splash_screen.dart

if (kDebugMode) ...[
    const SizedBox(height: 10),
    Row(
        mainAxisAlignment: MainAxisAlignment.center,
        children: [
            TextButton(
                onPressed: () {
                    context.read().resetAdsStatus();
                    Navigator.pushReplacementNamed(context, Routes.marketing);
                },
                child: const Text('FORCE ADS', style: TextStyle(color: Colors.white24, fontSize: 10)),
            ),
            // ...
        ],
    ),
]
