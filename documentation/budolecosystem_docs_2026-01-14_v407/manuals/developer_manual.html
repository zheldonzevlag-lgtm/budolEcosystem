Developer Manual: Mobile Auth Audit Integration

1. Mobile Auth Audit Implementation (Dashboard Visibility Fix)

To ensure full traceability and visibility in the Admin Dashboard, the auth-service now explicitly creates AuditLog entries with aligned entity and action fields. This ensures they appear under the "Security" filter in the Forensic Audit Trail.

File: budolpay-0.1.0/services/auth-service/index.js
// Helper update to support entity filtering
const createAuditLog = async (req, userId, action, metadata = {}, entity = 'Security', entityId = null) => {
    // ...
    await prisma.auditLog.create({
        data: {
            userId,
            action,
            entity,
            entityId: entityId || userId,
            // ...
        }
    });
};

// Implementation in /login/mobile/verify-pin
await createAuditLog(req, user.id, 'SECURITY_MOBILE_LOGIN_PIN', {
  method: 'PIN',
  deviceId: deviceId,
  status: 'SUCCESS'
}, 'Security', user.id);

// Implementation in /biometric/login-verify
await createAuditLog(req, user.id, 'SECURITY_MOBILE_LOGIN_BIOMETRIC', {
  method: 'BIOMETRIC',
  deviceId: deviceId,
  status: 'SUCCESS'
}, 'Security', user.id);

2. Dashboard Filtering Logic Alignment

The Admin Dashboard filters logs based on the action prefix and the entity field. By prefixing actions with SECURITY_ and setting entity to Security, we guarantee these logs are captured in forensic reports.

3. Technical Rationale

The createAuditLog helper function captures the request context (IP address) and merges it with the provided metadata. The deviceId is passed from the mobile app (Flutter) via the deviceId parameter in the request body, which is then persisted in the audit log for forensic device fingerprinting.

4. Security Standards

All authentication logs must be stored in the AuditLog table which is designated as an "Immutable Log Store". Any attempt to modify these logs should be detected by database triggers (planned for v410).
