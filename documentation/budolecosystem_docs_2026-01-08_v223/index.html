v223: RBAC Compliance & Implementation

Date: 2026-01-08 | Status: Implemented

Purpose

To address compliance requirements (PCI DSS, BSP Circular 808, NPC DPA 2012) by implementing Role-Based Access Control (RBAC) across the BudolPay ecosystem. This ensures the principle of "Least Privilege" and "Segregation of Duties" (SoD).

Compliance Analysis & Justification

The implementation of Role-Based Access Control (RBAC) within the budolPay Admin and Gateway is a critical security mandate. Far from being a violation, it serves as the cornerstone for institutional-grade financial compliance in the Philippines.

1. PCI DSS v4.0 (Requirement 7)

Standard: Restrict access to system components and cardholder data by business need-to-know.

Justification: RBAC is the primary mechanism to ensure that only authorized roles (e.g., STAFF handling disputes) can access transaction details, while general users or unrelated roles are blocked from sensitive data exposure.

2. BSP Circular No. 808

Standard: IT Risk Management - Segregation of Duties (SoD).

Justification: In financial systems, a single individual should not have the power to both initiate and approve a high-value transaction or KYC status. RBAC enforces these boundaries, preventing internal fraud and operational errors.

3. NPC Data Privacy Act (DPA 2012)

Standard: Technical Security Measures - Access Control Policy.

Justification: The DPA requires that personal data (KYC documents, addresses, full names) be protected from unauthorized access. RBAC ensures that KYC_READ_ALL permissions are strictly limited to verified administrative roles.

4. BIR Tax Integration Compliance

Standard: Data Integrity and Auditability.

Justification: To maintain the integrity of tax-related reporting (VAT, withholding tax), the system must restrict access to invoicing and accounting adjustments. RBAC ensures that only authorized accounting personnel can modify financial records, providing a clear audit trail.

Compliance Mapping

Standard Requirement Implementation Detail
PCI DSS v4.0 Requirement 7: Restrict access by business need-to-know. Defined ROLE_PERMISSIONS mapping in rbac-config.js to restrict cardholder data access.
BSP Circular 808 Section 2.1.2: Segregation of Duties (SoD). Implemented authorize middleware to prevent standard users from performing admin actions.
NPC DPA 2012 Rule VII, Sec 28: Policy on access to personal data. Restricted KYC_READ_ALL permissions to STAFF and ADMIN roles only.
BIR Tax Integration Data Integrity & Reporting Controls. Restricted accounting and invoicing adjustments to authorized roles.

Technical Implementation

1. RBAC Configuration

Created packages/database/rbac-config.js to centralize role-permission mappings.

const ROLE_PERMISSIONS = {
  USER: ['wallet:read', 'wallet:transfer', ...],
  STAFF: ['kyc:approve', 'transaction:read_all', ...],
  ADMIN: ['*'] // All permissions
};

2. Authorization Middleware

Implemented packages/database/authorize.js as a reusable Express middleware for microservices.

3. Gateway Enforcement

Updated services/api-gateway/index.js to block unauthorized access to /admin routes with descriptive compliance alerts.

if (req.originalUrl.includes('/admin') && !hasPermission(role, PERMISSIONS.TRANSACTION_READ_ALL)) {
  return res.status(403).json({ error: 'Forbidden', message: 'Admin privileges required' });
}

Issues & Mitigations

  • Risk: Potential for "Privilege Escalation" if JWT is compromised.
  • Mitigation: Combined RBAC with existing PIN/Biometric verification (GoTyme style) for sensitive operations.
  • Risk: Microservice isolation bypassing Gateway checks.
  • Mitigation: Deployed authorize middleware directly within internal microservices as "Defense in Depth."
