Comparative Analysis & Implementation Plan - v204

Subject: Transitioning budolPay to a GCash-Like Login Model

Version: v204 (2026-01-05)

1. Comparative Analysis: budolPay vs GCash

This analysis compares the current budolPay authentication model with the industry-standard GCash mobile-first flow.

Feature budolPay (Current) GCash (Target) Gap / Improvement
Primary Identity Email Address Mobile Number Shift to Mobile-first identity for local PH market suitability.
Credential Password (Alphanumeric) 4-digit MPIN MPIN is faster for mobile users and reduces phishing risks.
MFA / Security Optional / Manual Mandatory OTP (New Device) Device binding & OTP-on-new-device adds critical security.
Session Lock Token-based (7 days) MPIN / Biometric Lock App-level locking for better privacy on shared/unlocked devices.
Biometrics Not Implemented Face ID / Fingerprint High convenience for repeat daily transactions.

2. Proposed GCash-Like Login Flow for budolPay

New Sign-In Flow:
1. Open App → 2. Enter Mobile Number (+63) → 3. Check Device Trust
    ├─ If NOT Trusted: Send OTP → Verify OTP → Register Device
    └─ If Trusted: Proceed to Lock Screen
4. Enter 4-digit MPIN / Biometric → 5. Access Wallet

3. Implementation Roadmap

Phase 1: Identity & Schema Migration PLANNED

  • Update User model to include mpinHash and trustedDevices.
  • Implement migration script to set default MPINs or force users to set one on next login.
  • Update auth-service to support POST /login/mobile.

Phase 2: OTP & Device Binding PLANNED

  • Integrate deviceFingerprint in the login payload.
  • Modify sendOTP logic to trigger automatically when a new fingerprint is detected.
  • Implement POST /verify-device to whitelist new fingerprints.

Phase 3: MPIN & Biometric Lock PLANNED

  • Develop MPIN entry UI for Mobile/Web clients.
  • Implement MPIN hashing and verification logic in auth-service.
  • Add biometricToken field for mapping FaceID/Fingerprint results to the user session.

Phase 4: Compliance & Session Management PLANNED

  • Implement automatic session timeout (e.g., lock after 5 mins of inactivity).
  • Add BSP-compliant login attempt limits (lock after 3 failed MPIN attempts).
  • Implement "Forgot MPIN" flow via OTP verification.

4. Technical Specifications

  • MPIN Storage: Hashed using bcrypt (consistent with current password storage).
  • Device Trust: JWT-based device tokens stored in localStorage or Secure Enclave.
  • API Security: Rate limiting on /login/mobile and /verify-otp endpoints.
