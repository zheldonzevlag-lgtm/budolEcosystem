Admin Menu & Auth Validation

Version 298 | 2026-01-10
← Main Index

Purpose & Explanation

This update focuses on standardizing administrative authentication across all API endpoints and validating full menu access for the newly elevated admin account (reynaldomgalvez@gmail.com). The goal was to ensure that any user with administrative privileges can seamlessly navigate the dashboard and perform management tasks without encountering "Unauthorized" errors.

Reason for Change

The user reported continued inability to access the admin dashboard despite previous elevation attempts. Investigation revealed that the AuthContext and various API routes were not consistently checking the isAdmin flag from the database, often relying on legacy accountType checks or hardcoded email lists that were out of sync.

Technical Details & Implementation

Issues Identified

  • Inconsistent Auth Helpers: Many API routes were manually verifying JWT tokens and roles instead of using the centralized requireAdmin helper.
  • Missing Database Flags: The /api/auth/me endpoint was not returning the isAdmin boolean, causing the frontend AuthContext to lose admin status on page refreshes.
  • Priority Logic: The isUserAdmin utility did not prioritize the database isAdmin flag, which is the most reliable source of truth.

Fixes Applied

  • Centralized Auth: Refactored over 10 critical admin API routes (Users, Products, Webhooks, Audit Logs, etc.) to use the requireAdmin helper from lib/adminAuth.js.
  • API Update: Modified /api/auth/me/route.js to include isAdmin: true in the Prisma select clause.
  • Logic Refinement: Updated lib/adminAccess.js to prioritize user.isAdmin === true as the primary check for administrative access.

Code Changes

// lib/adminAccess.js export function isUserAdmin(user, adminEmailSet = getConfiguredAdminEmailSet()) { if (!user) return false // Check database isAdmin flag first (highest priority) if (user.isAdmin === true) { return true } // Check account type if (isAccountTypeAdmin(user.accountType)) { return true } // Check configured admin emails return isEmailConfiguredAsAdmin(user.email, adminEmailSet) }

Risk & Mitigation

Risk Mitigation Strategy
Session Hijacking: Elevated privileges increase the impact of a compromised account. Enforced requireAdmin server-side verification on every request. Implemented session timeout monitoring.
Permission Drift: Users might retain admin flags in database but lose them in SSO. Logic prioritizes local database flags but allows for SSO/Config fallback if needed.

Recommendation for Future Enhancement

All recommendations have been consolidated into the global Future Enhancements document. Key items include:

  • Fine-grained RBAC: Transition from binary isAdmin to a permission-based system (e.g., 'manage_users', 'view_analytics').
  • Multi-Factor Authentication (MFA): Require MFA for all accounts with the isAdmin flag set to true.
  • Audit Logging Enhancement: Log every destructive action (DELETE/PUT) performed by an admin with the associated IP and User Agent.
