Developer Manual - Version v300

Technical Overview

This version implements a standardized RBAC (Role-Based Access Control) architecture and secures the administrative surface area of BudolShap. The implementation centralizes authentication logic and ensures that the database isAdmin flag is the primary source of truth for administrative elevation.

Architectural Decisions

1. Centralized Admin Middleware

Instead of disparate checks across routes, we now use a unified requireAdmin helper located in lib/adminAuth.js. This ensures consistent responses (403 Forbidden) and standardized error messages.

// Example Usage in API Routes
const { authorized, errorResponse } = await requireAdmin(request, PERMISSIONS.SETTINGS_EDIT);
if (!authorized) return errorResponse;

2. Three-Tier Role Resolution

Role resolution now follows a prioritized hierarchy:

  1. Database Flag: user.isAdmin (Boolean)
  2. Account Type: user.accountType === 'ADMIN' (Enum)
  3. Permissions JSON: Granular checks in user.permissions

Code Logic Explanations

Admin Access Resolution (lib/adminAccess.js)

The isUserAdmin function was updated to prioritize the database boolean flag over email configuration lists, allowing for dynamic admin management without code changes.

Fix: Updated isUserAdmin to check user.isAdmin first. This prevents access denial for users who are elevated in the database but not present in the hardcoded ADMIN_EMAILS list.

RBAC Permission Mapping (lib/rbac.js)

Standardized mappings for ADMIN, MANAGER, SELLER, and BUYER roles were enforced. Permissions are now stored as a JSON array in the User model.

Impact Analysis

  • Security: Significant reduction in attack surface by securing 18+ previously unprotected admin endpoints.
  • Performance: Minimal overhead due to indexed Prisma queries for user roles.
  • Scalability: The system now supports granular permissions (e.g., products:approve vs products:view), enabling future implementation of sub-admin roles.
