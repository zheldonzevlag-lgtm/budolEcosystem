System Admin Manual - Version v300

Infrastructure Changes

No direct infrastructure changes were required for this version. All modifications were implemented at the application and database level (Prisma/PostgreSQL).

Role Assignment Procedures

Admin users can now be managed via the isAdmin flag in the database. To elevate a user to Administrator status, the following fields must be set:

Field Value Reason
isAdmin true Primary elevation flag.
accountType ADMIN Legacy compatibility and UI grouping.
role ADMIN RBAC permission mapping.
permissions JSON array Explicit permission list (optional if role is ADMIN).
Administrative Elevation Script: The script update_admin.mjs remains available in the root directory for secure elevation and password resets of administrative accounts.

Security Monitoring

Admins should monitor the Audit Logs (once fully integrated) for any unauthorized access attempts to administrative endpoints. All 403 Forbidden responses are now logged on the server side with the attempted user ID.

Rollback Procedures

In the event of a system-wide access lockout:

  1. Revert changes to lib/adminAccess.js to restore reliance on hardcoded ADMIN_EMAILS.
  2. Ensure the database backup from 2026-01-10 is available to restore the User table roles if necessary.
WARNING: Modifying isAdmin flags directly in production databases should only be done by authorized personnel.
