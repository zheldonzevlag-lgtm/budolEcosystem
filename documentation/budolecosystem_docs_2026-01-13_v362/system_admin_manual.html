System Admin Manual: Biometric Infrastructure

Version: v362

Subject: Model deployment and backend synchronization for Face Recognition.

1. Model Deployment

The Face Recognition system relies on the facenet.tflite model. Admins must ensure this file is present in the following location before building the mobile application:

budolPayMobile/assets/models/facenet.tflite

Failure to include this file will result in the FaceEmbeddingService failing to initialize.

2. Backend Configuration

The backend /verification/upload endpoint must be prepared to handle the faceTemplate multipart field. This field contains a string of 128 comma-separated floating-point numbers.

Database Schema Update

Ensure the user_verifications table (or equivalent) has a column for face_template (TEXT or BLOB).

3. Troubleshooting

  • Model Loading Error: Check pubspec.yaml for asset inclusion and verify file path.
  • Low Similarity Scores: Ensure the user is in a well-lit environment. Verify that the model version on the client matches the expected version on the server.

4. Phase 8: Dependencies & Software Choice Rationale

The following libraries and technologies were chosen for the BudolPay Face Recognition system:

  • TFLite (tflite_flutter): Chosen for its high performance on mobile devices and small binary footprint. It allows us to run industry-standard models like FaceNet directly on the device.
  • Google ML Kit (Face Detection): Selected for its robust and fast face detection and classification (blink/smile) capabilities, which are essential for liveness detection.
  • FaceNet: Chosen as the base model for embedding generation due to its proven accuracy in face verification tasks and the availability of optimized TFLite versions.
  • FlutterSecureStorage: Used for storing sensitive biometric metadata (templates) and PINs locally, leveraging hardware-backed keystores on both Android and iOS.
  • Prisma & PostgreSQL: Used on the backend for reliable, structured storage of face templates, enabling future cross-device verification.

Rationale: This combination balances high security (liveness detection + local encryption) with a seamless user experience (fast on-device matching).
