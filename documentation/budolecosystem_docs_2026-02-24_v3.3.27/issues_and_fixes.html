Issues & Fixes Log: v3.3.27

Issue: Prisma Client Stale Schema (isDefault)

Description: After adding the isDefault field to the Address model, the application threw a PrismaClientValidationError: "Unknown argument `isDefault`". This occurred because the running development server was using a cached version of the Prisma Client generated before the schema update.

Reported By: User Feedback (Screenshot)

Date: 2026-02-24

Fix Implemented (v3.3.27)

Action:

  • Verification: Verified that the schema.prisma includes isDefault and that npx prisma migrate status confirms the database is up to date.
  • Resolution: The fix requires restarting the development server to reload the regenerated Prisma Client. The client on disk was confirmed correct by running verify_all_address_scenarios.test.js which successfully used isDefault.

Status: Resolved (Requires Server Restart)

Issue: Failed to Create Address (Phone Format)

Description: When creating a new address, users encountered a "Failed to create address" error. This was caused by the API strictly requiring phone numbers in E.164 format (e.g., +639xxxxxxxxx) without spaces or dashes, while the frontend allowed and submitted formatted numbers (e.g., +63 948-409-9389).

Reported By: User Feedback (Screenshot)

Date: 2026-02-24

Fix Implemented (v3.3.27)

Action:

  • Backend: Updated app/api/addresses/route.js (POST) and app/api/addresses/[addressId]/route.js (PUT) to automatically sanitize phone numbers by removing spaces and dashes before validation.
  • Validation: The E.164 regex check now runs on the sanitized number, allowing user-friendly formatting in the UI while maintaining strict data integrity in the database.
  • Verification: Verified with new test script verify_phone_fix.js.

Status: Resolved

Issue: Missing Required Fields (Barangay)

Description: Users reported getting a "Missing required fields" toast error even when all visible required fields were filled. The issue was traced to the `barangay` field being implicitly required in the backend API and database schema, despite being intended as optional.

Reported By: User Feedback

Date: 2026-02-24

Fix Implemented

Action:

  • Updated Prisma schema (`Address`, `StoreAddress`) to make `barangay` nullable (`String?`).
  • Updated API routes to remove `barangay` from required field checks.
  • Updated Frontend Zod validation schema to mark `barangay` as optional.
  • Replaced generic toast errors with field-specific Zod validation messages.

Status: Resolved

Issue: Prisma Client File Lock (EPERM)

Description: During development, `prisma generate` failed with `EPERM: operation not permitted` when trying to rename `query_engine-windows.dll.node` in `@prisma/client-custom`. This is caused by the file being locked by the running Next.js development server or other node processes.

Reported By: Developer (Internal)

Date: 2026-02-24

Fix Implemented

Action: Changed the Prisma client output directory to `@prisma/client-custom-v4` in `schema.prisma` and updated `lib/prisma.js` to import from the new location. This bypasses the locked file.

Status: Resolved (Workaround)
