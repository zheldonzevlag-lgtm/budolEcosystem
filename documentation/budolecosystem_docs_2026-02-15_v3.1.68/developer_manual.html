Version v3.1.68 - Focus on Reliability and Branding

Auth Service Stability

The quick registration flow was previously failing due to an undefined isLocal variable, which is critical for environment-specific logic. We implemented a robust detection mechanism that considers NODE_ENV, LOCAL_IP, and JEST_WORKER_ID.

Rationale

Ensuring environment parity between development, testing, and production is a core requirement of PCI DSS 6.4. This fix prevents runtime errors that could disrupt the user registration journey.

auth-service/index.js Open File
const isLocal = process.env.NODE_ENV === 'development' || process.env.NODE_ENV === 'test' || process.env.LOCAL_IP === '127.0.0.1' || !!process.env.JEST_WORKER_ID;

Centralized OTP Generation

Hardcoded OTP generation was replaced with a centralized generateOTP helper. This helper detects if it's running in a test environment and returns the standard test OTP (123456) to ensure reliable automated testing.

auth-service/index.js Open File
const generateOTP = () => {
    if (process.env.NODE_ENV === 'test' || !!process.env.JEST_WORKER_ID) return '123456';
    return Math.floor(100000 + Math.random() * 900000).toString();
};

Per user request, OTP values are now highlighted in yellow in all server console logs for easier debugging during local development.

auth-service/index.js Open File
if (isLocal) console.log(`[LOCAL] Quick-Reg OTP for ${maskPII(recipient)}: \x1b[33m${otpCode}\x1b[0m`);

NPC PII Masking Compliance

To comply with the NPC Data Privacy Act, we implemented a centralized maskPII helper across all services. This helper automatically masks Phone Numbers, Emails, and Names in all API responses and server logs.

auth-service/index.js Open File
const maskPII = (str, type = 'AUTO') => {
    if (!str) return 'N/A';
    if (type === 'AUTO') {
        if (str.includes('@')) type = 'EMAIL';
        else if (/\d/.test(str) && str.length >= 7) type = 'PHONE';
        else type = 'NAME';
    }
    // ... masking logic (e.g., 091*****123) ...
};

Compliance Rationale

NPC regulations require protecting Personal Identifiable Information (PII) from unauthorized disclosure. Masking PII in logs and non-essential API fields significantly reduces the risk of data leakage in development and monitoring environments.

Security & Compliance (PCI DSS / BSP)

Compliance with PCI DSS 10.2.2 (Audit logs for all actions) and BSP Circular 808/1108 (Multi-factor authentication) is maintained through:

  • Rate-limited OTP delivery via otpUpdatedAt field in Prisma schema.
  • Centralized audit logging for registration events.
  • Secure token handling for mobile sessions.
