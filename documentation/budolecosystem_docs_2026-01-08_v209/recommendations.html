Future Enhancement Recommendations

Strategic roadmap for the budolID Identity Ecosystem.

Multi-Factor Authentication (MFA)

Purpose

Enhance account security by requiring a second form of verification.

Explanation

Implement TOTP (Time-based One-Time Password) or SMS-based codes during the login flow.

Reason

Static passwords are vulnerable to phishing, credential stuffing, and social engineering.

Issues

Current system relies solely on passwords, which is a single point of failure.

Fixes

Add an MFA verification step after the initial password check in the SSO flow.

Risk

Increased login friction may lead to user frustration; SMS costs for the organization.

Mitigation

Allow "Remember this device" for 30 days; offer TOTP apps as a free alternative to SMS.

Dependencies

External SMS API (Twilio) or TOTP library (speakeasy); DB schema update for MFA secrets.

Rate Limiting & Bruteforce Protection

Purpose

Protect the login endpoints from automated brute-force and DDoS attacks.

Explanation

Limit the number of requests a single IP address can make to authentication routes within a timeframe.

Reason

The current /auth/sso/login route has no restrictions, allowing unlimited attempts.

Issues

High vulnerability to password spraying and service exhaustion.

Fixes

Integrate express-rate-limit middleware in the budolID service.

Risk

False positives blocking legitimate users on shared corporate or public networks.

Mitigation

Implement tiered limiting and use Redis for distributed tracking across service instances.

Dependencies

express-rate-limit package; Redis (optional but recommended for production).

Refresh Token Implementation

Purpose

Maintain long-term user sessions securely without persisting sensitive access tokens.

Explanation

Issue short-lived access tokens (e.g., 15m) and long-lived refresh tokens (e.g., 7d) stored in HttpOnly cookies.

Reason

Current 24h tokens are a balance but suboptimal for both security and user experience.

Issues

If a 24h token is stolen, it remains valid for the full duration; no easy way to revoke individual sessions.

Fixes

Implement a token rotation strategy and a refresh endpoint in budolID.

Risk

Refresh token theft could lead to permanent account access if not properly rotated.

Mitigation

Use Refresh Token Rotation; revoke all tokens if a reused refresh token is detected.

Dependencies

DB schema update for RefreshToken model; cookie-parser middleware.
