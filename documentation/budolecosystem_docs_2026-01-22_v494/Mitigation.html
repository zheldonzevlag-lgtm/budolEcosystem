🛡️ Mitigation Strategy v494

M-494-01: Realtime Event Deduplication

Risk: Duplicate notifications causing user confusion and UI flicker (R-494-03).

Mitigation: Implementation of a two-tier deduplication logic in RealtimeProvider.jsx. It uses a 5-second window for the same event/data pair and a lastSeenStatus map to prevent duplicate toasts for the same status transition.

M-494-02: Filter-Aware Cache Patching

Risk: Orders appearing in incorrect tabs after status change (Tab Desync).

Mitigation: The patchOrderCache function now dynamically evaluates the URL parameters of cached SWR keys. If an order's new status no longer matches the status filter in the URL, it is explicitly removed from that cache entry, ensuring UI consistency without full page reloads.

M-494-03: Universal Status Normalization

Risk: Inconsistent terminology between courier updates and user notifications.

Mitigation: Integration with statusMapper.js's getStatusLabel and UNIVERSAL_STATUS. All realtime notifications now pass through a normalization layer to ensure the user sees "To Ship" or "Delivered" regardless of the underlying technical status or courier-specific label.

M-494-04: Visual Consistency Standard

Risk: UI fragmentation where system notifications look different from the brand's intended aesthetic (R-494-04).

Mitigation: Established a "Notification Design Pattern" in RealtimeProvider.jsx that overrides default toast styles with a standard white-on-slate theme, ensuring all order status updates follow the professional "Image 2" specification.
