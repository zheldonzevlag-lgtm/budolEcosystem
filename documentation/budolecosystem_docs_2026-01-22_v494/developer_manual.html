Purpose & Rationale

The primary goal of this update is to solve the "Order Tab Desync" issue where orders remained in their old status tabs after a status update (e.g., a "DELIVERED" order staying in the "To Receive" tab). This provides a seamless, realtime user experience while adhering to cost-optimization standards by using targeted SWR cache patching instead of full-page reloads.

Technical Implementation

The core logic resides in RealtimeProvider.jsx. The patchOrderCache function was enhanced to be "Filter-Aware". It now inspects the SWR cache keys (which contain query parameters like status=PAID,PROCESSING) and determines if an updated order still belongs in that specific cache entry.

Additionally, notifications were enhanced to use the statusMapper.js utility, ensuring that technical status codes (e.g., PAID) are presented to users as friendly labels (e.g., Paid).

Dependencies & Software

  • SWR (swr): Used for data fetching and cache management. Its mutate function is the backbone of the realtime patching logic, allowing for optimistic UI updates and targeted cache invalidation.
  • Pusher (pusher-js): Chosen as the primary realtime provider for its managed infrastructure and reliability. It handles the pub/sub logic for order and return events.
  • Socket.io (socket.io-client): Supported as an alternative realtime provider for self-hosted or local development scenarios.
  • React Hot Toast (react-hot-toast): Used for displaying non-intrusive, realtime notifications to users.
  • statusMapper.js: A internal utility that provides a single source of truth for universal status mappings and user-friendly labels.
