v195 Implementation Plan

Subject: GCash-Style Universal Registration & SSO Propagation

Date: 2026-01-05 | Version: v195

Purpose

This plan outlines the technical steps to implement a seamless, multi-step registration flow in the budolPay mobile app that mirrors the GCash onboarding experience. Crucially, it ensures that any account created via budolPay is "Universal" across the budolEcosystem by leveraging the budolID SSO service.

Phase 1: budolID SSO Enhancements

1.1 Update User Registration API

Modify budolID/index.js's /auth/register endpoint to accept and validate phoneNumber. This ensures that the primary identifier for mobile users is captured at the ecosystem level.

1.2 Availability Check Endpoint

Implement GET /auth/check-availability to allow the mobile app to verify if an email or phone number is already registered before the user completes the form.

Phase 2: budolPay Universal Propagation

2.1 "Lazy" User Creation in budolPay

Update budolPay/services/auth-service/index.js's /verify endpoint. If a valid token from budolID is presented but the user doesn't exist in the budolPay database, auto-create the user record and a ₱0.00 wallet. This guarantees "Universal" access.

Phase 3: Flutter Mobile App Development

3.1 ApiService Integration

Add register() and checkAvailability() methods to lib/services/api_service.dart.

3.2 Multi-Step Registration Screen

Create lib/screens/registration_screen.dart with three steps:

  • Step 1: Mobile Number (GCash style).
  • Step 2: Personal Info (First Name, Last Name, Email).
  • Step 3: Password Selection.

Testing Protocol & Results

  • E2E Test 2: Centralized Registration - PASSED (Verified via test-universal-reg.js)
  • E2E Test 3: budolPay Propagation (Lazy Sync) - PASSED (Verified user/wallet auto-creation in budolPay database)
  • E2E Test 4: Mobile UI Flow Integration - PASSED (Registration screen successfully integrated with 3-step validation)

Risks & Mitigations

Risk Mitigation
Sync Delay Using "Lazy Creation" on first login/verify eliminates the need for complex webhooks.
Security All registration data is hashed via bcrypt in budolID before storage.
