Comparative Analysis: BudolPay vs GCash KYC

1. User Journey Comparison

Stage GCash Flow BudolPay (Current)
Entry Basic Account (OTP only) Basic Account (Email/Phone)
Trigger Gated by Features (Send, GSave, etc.) Manual Opt-in via Settings
Process ID Capture -> Selfie (Liveness) -> Personal Details Personal Details -> ID Capture -> Selfie (Face Match)
Verification Automated + Vendor (Real-time) ML Kit (On-device) + Admin Review

2. Technical Architecture Alignment

Key Difference: GCash uses a "KYC SDK/Module" that acts as a gatekeeper. BudolPay currently relies on UI-level suggestions which are not strictly enforced as feature blocks.

Current Architecture (BudolPay)

  • Mobile app captures data -> Sends to API Gateway.
  • Backend updates KYCStatus.
  • Admin Panel reviews VerificationDocument.

Target Architecture (GCash-Inspired)

  • KYC Gating Module: A middleware in the mobile app that checks kycLevel before navigation to sensitive features.
  • Tiered Leveling: Explicitly mapping kycStatus to kycLevel (BASIC, FULLY_VERIFIED, PLUS).
  • Liveness Check: Upgrading from simple selfie capture to active liveness detection (blink/smile) to match GCash standards.

3. Feature Gating Strategy

Following GCash's lead, BudolPay will implement the following gating logic:

Feature Required Level Trigger Message
Receive Money BASIC -
Send Money (P2P) FULLY VERIFIED "Get Fully Verified to start sending money to friends."
Cash Out / Bank Transfer FULLY VERIFIED "KYC is required for financial withdrawals."
Investment / Savings PLUS "Unlock BudolInvest by completing enhanced verification."
