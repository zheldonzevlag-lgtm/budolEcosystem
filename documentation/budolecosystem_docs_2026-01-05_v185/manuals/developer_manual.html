Developer Manual: Tiered KYC & Gating Logic

1. Schema Enhancements

To support tiered levels, we are introducing a kycLevel field to the User model. This allows for more granular control than just status (NONE/PENDING/VERIFIED).

enum KYCLevel {
  BASIC          // Registered, OTP verified
  FULLY_VERIFIED // ID + Selfie approved
  PLUS           // Enhanced verification / High limits
}

model User {
  // ...
  kycLevel  KYCLevel @default(BASIC)
  kycStatus KYCStatus @default(NONE)
}

2. Mobile Gating Logic (Flutter)

Implementing a central utility KYCGuard to manage feature access. This follows the pseudo-logic: IF user.feature IN gated_features THEN require KYC.

class KYCGuard {
  static bool canAccess(KYCLevel currentLevel, Feature feature) {
    switch (feature) {
      case Feature.sendMoney:
      case Feature.cashOut:
        return currentLevel >= KYCLevel.FULLY_VERIFIED;
      case Feature.invest:
        return currentLevel >= KYCLevel.PLUS;
      default:
        return true;
    }
  }

  static void checkAndPrompt(BuildContext context, KYCLevel level, Feature feature) {
    if (!canAccess(level, feature)) {
      Navigator.push(context, MaterialPageRoute(builder: (context) => const GetVerifiedPrompt()));
    }
  }
}
