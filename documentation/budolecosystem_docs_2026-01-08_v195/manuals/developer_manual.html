Developer Manual: Wallet Service Scoping Fix

Purpose

This manual documents the technical resolution of the ReferenceError: userId is not defined in the Wallet Service, which caused identification failures in the mobile application.

Problem Analysis

Issue: JavaScript Variable Scoping (Block Scoping)

In the getBalance route of wallet-service/index.js, the userId variable was declared using const inside the try block. When an error occurred within the try block, the execution jumped to the catch block. Since const is block-scoped, userId was not available in the catch block, leading to a secondary ReferenceError when attempting to log the error.

Technical Details

Affected File

services/wallet-service/index.js

Root Cause Code (v194 and earlier)

router.get('/balance/:userId', async (req, res) => {
    try {
        const { userId } = req.params; // Scoped only to the try block
        let wallet = await prisma.wallet.findUnique({
            where: { userId }
        });
        // ...
    } catch (error) {
        console.error(`[Wallet] Error in getBalance for user ${userId}:`, error); // ReferenceError: userId is not defined
        res.status(500).json({ error: error.message });
    }
});

Implementation of Fix

The Solution

The variable declaration was moved outside the try block to ensure it is accessible throughout the entire function scope, including the catch block.

router.get('/balance/:userId', async (req, res) => {
    const { userId } = req.params; // Now accessible in catch block
    try {
        let wallet = await prisma.wallet.findUnique({
            where: { userId }
        });
        // ...
    } catch (error) {
        console.error(`[Wallet] Error in getBalance for user ${userId}:`, error);
        res.status(500).json({ error: error.message });
    }
});

Audit & Verification

  • Wallet Service Audit: All routes were checked for similar scoping issues. No other instances were found.
  • Settlement Service Audit: Proactively audited settlement-service/index.js. No scoping issues identified.
  • Syntax Verification: Ran node -c on the modified file to ensure no syntax errors were introduced.
  • Functional Verification: Created a mock test script test_fix.js to simulate an error and verify that userId is correctly captured in the catch block.

Test Results

Dependencies

  • Express.js (Route handling)
  • Prisma ORM (Database operations)

Risks & Mitigation

Risk: Similar scoping issues might exist in other microservices not yet audited.

Mitigation: Establish a coding standard that mandates extracting request parameters at the top of route handlers before entering try-catch blocks.

Recommendations for Future Enhancement

Implement a centralized error handling middleware for Express that automatically captures request context (including params) to avoid redundant try-catch blocks in every route.
