Topic: Cart Variation Implementation

1. Database Schema Changes

We added a `variationId` column to the `CartItem` table to support SKU-level tracking.

Prisma Schema

File: budolshap-0.1.0/prisma/schema.prisma

model CartItem {
  id          String   @id @default(cuid())
  cartId      String
  productId   String
  variationId String?  // Added this field
  quantity    Int
  
  cart        Cart     @relation(fields: [cartId], references: [id], onDelete: Cascade)
  product     Product  @relation(fields: [productId], references: [id], onDelete: Cascade)

  // Composite unique constraint to prevent duplicates
  @@unique([cartId, productId, variationId]) 
}

2. Frontend State Management (Redux)

The cart state now uses a composite key format `productId_variationId` to track items.

Key Format Logic

  • Standard Product: Key = `productId`
  • Variation Product: Key = `productId_variationId`

Redux Slice

File: lib/features/cart/cartSlice.js

// Helper to generate key
export const generateCartKey = (productId, variationId) => {
    return variationId ? `${productId}_${variationId}` : productId;
};

// Helper to parse key
export const parseCartKey = (key) => {
    const parts = key.split('_');
    return {
        productId: parts[0],
        variationId: parts.length > 1 ? parts.slice(1).join('_') : null
    };
};

3. API Implementation

The API endpoint `/api/cart` handles the splitting and reconstruction of these keys to maintain compatibility with the frontend format.

File: app/api/cart/route.js

// PUT Request (Sync Cart)
const cartItems = Object.entries(cart).map(([key, quantity]) => {
    const parts = key.split('_');
    return {
        cartId: userCart.id,
        productId: parts[0],
        variationId: parts.length > 1 ? parts.slice(1).join('_') : null,
        quantity: Number(quantity)
    };
});

// Use transaction to ensure atomic updates
await prisma.$transaction([
    prisma.cartItem.deleteMany({ where: { cartId: userCart.id } }),
    prisma.cartItem.createMany({ data: cartItems })
]);

5. Cart Persistence & Synchronization

To ensure cart data persists across sessions (especially for guest-to-logged-in transitions) and is not lost during logout, we implemented a robust sync strategy.

Logout Sync Logic (Fix for Race Condition)

Previously, clearing the local cart state before the user session was fully terminated caused a race condition where the empty state was synced to the server. The corrected flow is:

  1. User clicks Logout: Navbar.jsx initiates the process.
  2. Force Sync: The current cart state is sent to the server via PUT /api/cart.
  3. Logout Action: await logout() is called, which invalidates the session and sets userId to null.
  4. Clear Local State: dispatch(clearCart()) is called only after userId is null.

This order ensures that the auto-sync effect in PublicLayout.jsx (which listens for cart changes) aborts because userId is missing, thus protecting the server-side cart data.

6. Troubleshooting & Deployment

Schema Updates (Prisma)

When modifying schema.prisma (e.g., adding `variationId`), you MUST follow these steps to apply changes:

  1. Stop the running server (`Ctrl+C` in the terminal).
  2. Run Generator: `npx prisma generate` (This updates the client in `node_modules`).
  3. Push Schema: `npx prisma db push` (This updates the actual database structure).
  4. Restart Server: `npm run dev` or `npm start`.

Common Error: EPERM: operation not permitted during generation means the server is still running and locking the client files.

Common Error: Unknown argument variationId means the server is running with an old client version.
