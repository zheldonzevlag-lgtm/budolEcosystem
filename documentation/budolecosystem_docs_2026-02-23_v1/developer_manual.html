Developer Manual: Geocoding Standardization

1. Overview

This update standardizes the geocoding API response format across all supported map providers (Geoapify, Google Maps, Radar, OSM/Nominatim). The goal is to ensure consistent address data (City, Province, Barangay, Street, Zip, District) regardless of the underlying provider.

2. Key Components

2.1 API Route: /api/system/geocode/route.js

The backend API has been updated to return a standardized address object in both forward (search) and reverse (coordinates) geocoding responses.

Standardized Address Object Structure:

address: {
    city: string,
    province: string,
    barangay: string,
    street: string,
    zip: string,
    district: string
}

Code Reference: app/api/system/geocode/route.js

Implementation Details:

  • Geoapify: Maps directly from properties (city, state, suburb, etc.).
  • Google Maps:
    • Reverse Geocoding: Extracts components from address_components arrays. Logic added to find 'barangay' from 'sublocality_level_1', 'neighborhood', or 'sublocality'.
    • Forward Search: Uses Places Text Search API to find specific buildings/POIs. Does not return full address components; relies on client-side reverse-geocoding (via MapPicker) to fill form details upon selection.
  • Radar: Maps from addresses array. Best effort mapping for barangay (neighborhood).
  • Nominatim: Maps from address object (suburb/neighbourhood -> barangay).

2.2 AddressFormManager.jsx

The frontend component responsible for handling address form data. It has been updated to consume the standardized address object.

Code Reference: components/address/AddressFormManager.jsx

Key Changes:

  • handleLocationChange: Now checks for result.address first before falling back to legacy raw data mapping.
  • handleSearchSelect: Updated to also use result.address for consistent behavior when selecting search results.
// Example of consumption in AddressFormManager.jsx
if (result.address) {
    const { city, province, barangay, zip, district, street } = result.address;
    if (city) newDetails.city = city;
    // ... maps other fields
}

2.3 Handling Search Selection vs. Reverse Geocode Overwrites

A specific challenge arose where selecting a precise building (e.g., "One Park Drive") from search results would trigger a map movement, which in turn triggered a reverse geocode that overwrote the building name with a generic street address (e.g., "9th Avenue").

Solution: Implemented a useRef flag isSearchSelection in AddressFormManager.jsx.

  • When a user selects a search result, isSearchSelection.current is set to true.
  • The form's "Detailed Address" is updated with the search result name.
  • When the map updates and triggers handleLocationChange, the system checks this flag.
  • If true, the existing "Detailed Address" is preserved instead of being overwritten by the reverse geocode result. The flag is then reset to false.
// Code Snippet from AddressFormManager.jsx
if (isSearchSelection.current) {
    if (formData.detailedAddress) {
        newDetails.detailedAddress = formData.detailedAddress;
    }
    isSearchSelection.current = false;
}

3. Compliance & Security

The implementation adheres to data privacy standards by ensuring only necessary address components are processed and stored. No sensitive personal data is exposed in the geocoding logs.
