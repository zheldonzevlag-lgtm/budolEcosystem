Issues & Fixes Log

Issue #1: Inconsistent Address Fields across Providers

Description: Geoapify, Google Maps, Radar, and OSM return address components with different property names (e.g., 'suburb' vs 'neighborhood' vs 'sublocality'). This caused the form fields to not populate correctly when switching providers.

Fix: Implemented a standardized address object in the backend API (/api/system/geocode). This object normalizes the data into city, province, barangay, street, zip, and district before sending it to the frontend.

Resolved

Issue #2: Google Maps Missing Barangay

Description: Google Maps API often nests the Barangay level under 'sublocality_level_1' or 'neighborhood', which wasn't being captured by the legacy mapping.

Fix: Updated the Google Maps extraction logic to check sublocality_level_1, neighborhood, and sublocality in that order to find the Barangay.

Resolved

Issue #3: Search Selection Not Syncing Fields

Description: Selecting a result from the search bar updated the map but sometimes failed to update the specific text fields (City, Zip, etc.) if the provider format wasn't recognized.

Fix: Updated AddressFormManager.jsx to use the same standardized address object for search results (forward geocoding) as it does for map clicks (reverse geocoding).

Resolved

Issue #4: Search Not Finding Specific Buildings (Google Maps)

Description: Searching for specific buildings like "One Park Drive" returned the street address ("11th Drive") instead of the building name when using the standard Geocoding API.

Fix: Switched the Google Maps search implementation to use the Places Text Search API, which is optimized for finding Points of Interest (POIs). The system now correctly identifies the building and uses client-side reverse-geocoding to fill the administrative address details.

Resolved

Issue #5: Barangay Field Not Updating (Stale Data)

Description: When moving the map to a location that lacks specific address components (e.g., missing Barangay), the form field would retain the previous value instead of clearing or updating. This caused misleading information (e.g., showing "Kasiyahan Homes" for a BGC location).

Fix: Updated AddressFormManager.jsx to explicitly reset all address fields (City, Province, Barangay, Zip, District) to the new values or empty strings whenever the location changes. This ensures no stale data persists.

Resolved

Issue #6: "Barangay is required" Error Preventing Save

Description: Users were unable to save addresses because the form validation required a Barangay field, even though some locations (or map providers) do not return this data.

Fix: Made the Barangay field optional in AddressFormManager.jsx by removing the strict validation requirement. The label was updated to "Barangay (Optional)".

Resolved

Issue #7: Address Modal Not Closing After Save

Description: After successfully saving an address, the modal remained open, confusing users about whether the action was completed.

Fix: Uncommented the setShowAddressModal(false) call in AddressModal.jsx to ensure the modal closes automatically upon successful submission.

Resolved

Issue #8: Search Result Details Overwritten by Reverse Geocode

Description: When selecting a specific building (e.g., "One Park Drive") from search results, the subsequent map update triggered a reverse geocode that overwrote the specific building name with a generic street address (e.g., "9th Avenue").

Fix: Implemented a flag isSearchSelection in AddressFormManager.jsx to detect when an update comes from a search selection. In this case, the system preserves the specific name from the search result as the "Detailed Address" instead of overwriting it with the reverse geocode result.

Resolved
