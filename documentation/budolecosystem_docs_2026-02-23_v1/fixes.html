Resolved Issues - v1 (2026-02-23)

  • Inconsistent Address Sync: Resolved an issue where address fields were not populating consistently depending on the selected map provider.
    Fix: Implemented a standardized `address` object in the Geocoding API response for all providers (Geoapify, Google Maps, Radar, OSM/Nominatim) and updated the frontend `AddressFormManager` to consume this standard format.
  • Google Maps Barangay Issue: Resolved an issue where Barangay data was missing for Google Maps locations.
    Fix: Updated the backend extraction logic to check sublocality_level_1, neighborhood, and sublocality fields.
  • Search Accuracy (Detailed Address): Resolved an issue where searching for a specific building (e.g., "One Park Drive") would result in the street address being saved instead of the building name.
    Fix: Implemented logic in `AddressFormManager.jsx` to preserve the search result name as the "Detailed Address" when a user selects a location from the search bar.
  • Stale Data Persistence: Resolved an issue where moving the map to a location with missing data (e.g., no Barangay) would display the previous location's data.
    Fix: Updated `AddressFormManager.jsx` to explicitly reset all address fields to empty strings before populating new data.
  • Validation Error Preventing Save: Resolved an issue where users could not save an address if the Barangay field was empty.
    Fix: Made the Barangay field optional in the form validation logic.
  • Modal Not Closing: Resolved an issue where the address modal remained open after a successful save.
    Fix: Ensured `setShowAddressModal(false)` is called upon successful submission.
