Architecture Review & Deployment Strategy (v14)

Subject: Deep-Dive Review of BudolEcosystem Regional Transition (Singapore ap-southeast-1)

1. Executive Summary

The v14 Singapore Rebuild transitions the BudolEcosystem from a monolithic infrastructure to a Partitioned Service Architecture. This strategy emphasizes service isolation, AWS Free Tier compliance, and regional low-latency for Southeast Asian users.

2. Service Topology

Identity budolID (Port 8000)

Central SSO provider. Deployed as a standalone Fargate service. All ecosystem subdomains query this service for session validation.

Payments budolPay (Ports 8001-8080)

A monorepo suite behind an Express API-Gateway (Port 8080). Handles wallets, transactions, and settlement services.

E-Commerce budolShap (Next.js)

Primary storefront. Integrates with budolPay for checkout and budolID for staff/customer login.

Financials budolAccounting

Post-transaction financial auditing and reporting service.

3. Multi-Layer Networking & Connectivity

We implement a Triple-Tier Network Perimeter:

  • Public Tier: Application Load Balancer (ALB) handles HTTPS (Port 443) traffic. Rules route requests by host/path.
  • Application Tier: ECS Fargate tasks running in private subnets. Communication is restricted to Cloud Map internal DNS (*.budol.local).
  • Data Tier: Single RDS instance (Consolidated) restricted to Port 5432 requests originating ONLY from the Application Security Group.

Implementation Phases

  1. Infrastructure (Done): VPC, RDS, and Bridge provisioned.
  2. Bridges (In-Progress): DuckDNS SSL acquisition and Cloud Map initialization.
  3. Service Launch: Deploying budolID first, followed by the budolPay gateway.
  4. Verification: End-to-end testing of cross-service API calls (e.g., Shap calling Pay).

6. SSL/TLS Strategy: Let's Encrypt vs. ACM

Primary Choice: AWS Certificate Manager (ACM)

Leverages AWS-managed certificate lifecycle. Standard for PCI DSS Compliance as it ensures certificates are never stored on persistent disk in plaintext. Requires CNAME-based DNS validation.

Secondary/Fallback: Let's Encrypt (Certbot)

Utilized for internal nodes or where DNS automation is restricted. In the v14 rebuild, we use Certbot on the Management Bridge for initial validation, but migrate to ACM for ALB binding to reduce "Side-Agent" overhead.

7. Compliance Standard (PCI DSS & BSP)

To meet PCI DSS Requirement 1.2.1, we restrict all inbound database traffic to specific service IPs. All sensitive variables (DB Passwords, JWT Secrets) are injected at runtime via ECS Environment Secrets (referencing AWS Secrets Manager) ensuring no credentials exist in images or code.
