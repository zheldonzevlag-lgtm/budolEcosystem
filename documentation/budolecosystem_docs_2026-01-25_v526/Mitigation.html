Mitigation Strategies: v526

This document outlines the specific technical and procedural measures taken to address the risks identified in the register.

Ref: RSK-526-01 (Infrastructure Failure)

Hybrid Polling-Event Strategy

Action: Implemented a fallback polling mechanism in QRCodeModal.jsx. If the real-time event is not received within 15 seconds, the frontend automatically falls back to manual status checks every 3 seconds.

Ref: RSK-526-02 (Race Conditions)

Idempotent Webhook Processing

Action: Modified payment-gateway-service to use a "status-lock" mechanism. The system ignores any status update if the transaction is already in a terminal state (COMPLETED/FAILED), ensuring that the first successful signal (webhook or internal update) wins without causing side effects.

Ref: RSK-526-03 (Data Integrity)

Unified Status Translation Layer

Action: Created a central mapping function in budolpay-adapter.js that translates all provider-specific statuses (e.g., "succeeded", "COMPLETED", "paid") into a unified internal enum before broadcasting to the frontend.

Ref: RSK-526-04 (Session Expiry)

Auto-Refresh Logic

Action: The mobile app now checks JWT validity before initiating the process-qr request. If the token is near expiry, it silently refreshes the session to ensure the payment flow is not interrupted.
