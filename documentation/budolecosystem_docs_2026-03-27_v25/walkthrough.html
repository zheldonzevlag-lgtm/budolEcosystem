Walkthrough – v25 Release

VERIFIED

Phase 0: Problem Identification

Issue: Phone numbers flagged as "ALREADY TAKEN" even when not in the database during Quick (Phone-Only) Registration. Standard Registration was unaffected.

Phase 1: Root Cause Analysis

Finding: Node.js 18+ resolves localhost to IPv6 [::1], but budolID binds to IPv4 0.0.0.0. The connection was silently refused. The /api/auth/check-phone route returned HTTP 503. The frontend defaulted to phoneExists = true to prevent security bypasses, blocking registration.

Phase 2: Backend Fixes (budolID & budolPay)

Fix 1: Added isQuickReg check in budolID/index.js to skip the password complexity regex for phone-only account creation flows.
Fix 2: Mirrored the same bypass in budolpay/services/auth-service/index.js for parity.

Phase 3: API Route Fix (budolShap)

Fix 3: Updated app/api/auth/check-phone/route.js and lib/api/budolIdClient.js to use:
BUDOL_ID_URL || NEXT_PUBLIC_SSO_URL || http://127.0.0.1:8000
This forces IPv4 locally and uses the production env var on Vercel.
Fix 4: Updated app/api/auth/register/route.js to pass registrationType through to the budolID client.

Phase 4: Frontend Polish (budolShap)

Change: Removed v25-DEBUG label from the "Create Account" button in AuthForm.jsx. The button now simply reads "Create Account".

Phase 5: Shield Challenge (MathCaptcha)

Feature: Added a math-based CAPTCHA gate before the registration form is revealed. This prevents bot registrations without requiring an external CAPTCHA service (BSP/Cybersecurity compliance).

Phase 6: Deployment

Result: npm run build ran successfully. Vercel deployment aliased to https://budolshap.vercel.app with Exit Code 0 ✅
