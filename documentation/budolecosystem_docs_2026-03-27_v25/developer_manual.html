Developer Manual – v25

1. Purpose & Rationale

This document explains the technical changes made in v25 to support phone-only Quick Registration without bypassing compliance security policies.

2. Root Cause: Node.js 18+ IPv6 Resolution Bug

Node.js 18 changed dns.lookup to prefer IPv6 (::1) when resolving localhost. The budolID SSO service was bound to 0.0.0.0 (IPv4 only), causing the connection to be silently refused. The check-phone API route then returned HTTP 503, and the frontend defensively assumed the phone was "ALREADY TAKEN" to prevent bypass.

4. Key Code Patterns

4.1 IPv4 Fallback Chain

// Why: Node 18+ resolves localhost → IPv6 [::1] but budolID listens on IPv4 only.
// What: Prefer explicit env vars; last resort is 127.0.0.1 (always IPv4).
const BUDOL_ID_URL = process.env.BUDOL_ID_URL
    || process.env.NEXT_PUBLIC_SSO_URL
    || 'http://127.0.0.1:8000';

4.2 Quick Registration Password Bypass (budolID)

// Why: Auto-generated passwords for phone-only accounts cannot satisfy
//      PH Cybersecurity password complexity requirements.
// What: Skip the regex check only when isQuickReg is explicitly true.
const isQuickReg = body.isQuickReg === true || body.registrationType === 'quick';
if (!isQuickReg && !passwordRegex.test(password)) {
    return res.status(400).json({ error: 'Password complexity not met.' });
}

4.3 registrationType Propagation

// budolshap register/route.js → budolIdClient.js → budolID /register
// Why: The identity service needs to know the registration context to
//      correctly apply (or bypass) password validation rules.
const { registrationType } = body; // 'standard' | 'quick'
await budolIdClient.register({ ...payload, registrationType });
