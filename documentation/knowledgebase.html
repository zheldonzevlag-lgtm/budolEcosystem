    Face Recognition & TFLite

    Our core biometric engine relies on TFLite for on-device inference. This section details optimizations and reliability patterns developed during the v360-v378 release cycle.

    v373 - v378

    TFLite Interpreter Reliability (The "Nuclear Option")

    Problem: Intermittent Interpreter is null errors on Android caused by native asset manager limitations and file locks.

    ✅
    Solution: Assets are now extracted to a unique temporary file per attempt (facenet_attempt_$attempt.tflite) to avoid lock contention.
    • Retry Loop: A 2-attempt retry mechanism with a 200ms delay between tries.
    • Health Checks: Validating the native handle by calling getInputTensors() immediately after initialization.

    Key File: face_embedding_service.dart

    v369 - v370

    Input/Output Normalization & Reshaping

    Insight: FaceNet models require strict input dimensions [1, 160, 160, 3] and RGB normalization (0.0 to 1.0).

    • Input Reshaping: Explicitly flattening and reshaping the Float32List to match model expectations.
    • RGB Normalization: Pixel values are divided by 255.0 for AI inference compatibility.

    Flutter Web Performance

    Optimizations for web environments to ensure smooth UI performance despite heavy initial asset loading.

    v346 - v347

    Main Thread De-blocking & Parallel Init

    Problem: Web apps hanging on a white screen during heavy initialization.

    • Parallel Initialization: Using Future.wait in the splash screen to run API init and branding delays concurrently.
    • Async Main: Moved heavy initialization out of main() into SplashScreen to allow the Flutter engine to warm up properly.

    Key File: splash_screen.dart

    Security & Biometrics

    Compliance and security patterns for handling sensitive biometric data.

    v558

    Session Locked UI & Idle Timeout Enforcement

    Security Standard: Enforcing a 3-minute idle timeout and grace period for mobile sessions. The UI is obscured with a blur effect and requires biometric re-authentication to resume.

    ✅
    Implementation:
    • SessionService: Modified to use _isLocked flag instead of hard logout.
    • SessionOverlay: Global root listener for activity detection and blur UI injection.
    • Compliance: Fully compliant with PCI DSS 8.1.8 and BSP Circular 808.

    Technical Document: Developer Manual v1.3.65

    v399

    GCash-Style "1 Device → 1 Account" Policy

    Security Standard: Enforcing a strict 1:1 relationship between a user account and a physical device to prevent unauthorized concurrent access.

    🛡️
    Policy Details:
    • Device Binding: Accounts are locked to a specific Device ID.
    • Mandatory Unlink: Logging into a new device requires unlinking the old one via OTP + Face Verification.
    • Compliance: Aligns with BSP Circular 958 and PCI DSS requirements for device fingerprinting.

    Strategic Document: Comparative Analysis: GCash vs budolPay

    v390

    Refined Face Login Registration Check

    Problem: Users who had already registered their face profile were unable to use face recognition login if the general "Enable Biometrics" toggle (tied to native OS settings) was disabled.

    ✅
    Solution: Decoupled the face recognition login availability from the general biometric toggle. The app now checks for a stored face template (_hasFaceTemplate) independently, allowing face login for registered users regardless of native toggle status.

    Key File: login_screen.dart

    v389

    Custom Face Login Prioritization

    Problem: User confusion when tapping the "Face" icon but being redirected to a generic "Biometric Login" (often fingerprint) prompt, with no way to specifically use the custom face recognition flow.

    ✅
    Solution: Prioritized the custom TFLite face recognition flow if a stored template exists. Redirect to native biometrics only occurs as a fallback if no custom template is found but native Face ID is available.

    Key File: login_screen.dart

    v387 - v388

    Biometric Redirection & Shape Alignment

    v387: Implemented hasEnrolledFace() to detect native system biometrics.

    v388: Fixed TFLite shape mismatch ([1, 512] vs [1, 128]) for FaceNet model compatibility.

    v366

    PIN Persistence & Registration Flow

    Problem: PIN codes were not persisting after registration.

    Fix: Ensured SharedPreferences commit before navigation. Integrated a synchronization check in ApiService to verify local credentials against the backend on startup.

    ⚠️
    Compliance Note: Raw biometric images must NEVER be persisted or transmitted. Only embeddings (hashes) are allowed to be processed.

    TFLite Model Validation & Security

    Enhanced security measures for biometric model integrity and validation.

    v386

    FaceNet Model Integrity Validation

    Challenge: TFLite model initialization failed due to 66-byte placeholder file instead of the expected 93.9MB FaceNet model.

    Solution: Implemented comprehensive model validation with size and hash verification:

    • Size Validation: Ensures model file is exactly 93,941,044 bytes
    • Hash Verification: SHA256 checksum validation with expected hash c3a5e56c
    • Format Validation: TFLite binary format verification
    • Health Check: Interpreter initialization validation

    Implementation: Added guard checks in face_embedding_service.dart to prevent corrupted models from being loaded.

    Security Impact: Prevents biometric authentication bypass and ensures model integrity compliance with PCI DSS requirements.

    v386

    Enhanced Debug Window

    Improvement: Upgraded Face Service Debug Window to full-screen mode for better troubleshooting experience.

    Features:

    • Full-screen camera preview with SafeArea protection
    • Real-time model loading status display
    • Error highlighting for quick issue identification
    • Model file information and validation results

    Implementation: Modified settings_screen.dart to use Positioned.fill and SafeArea for optimal display.

    Database Schema & Synchronization

    Managing schema drift and ensuring consistency across the Budol Ecosystem microservices.

    v382

    Manual SQL Synchronization vs. Migration Resets

    Challenge: Prisma migration drift occurred when manual changes were made to the production database without corresponding migration files, leading to initialization failures.

    ⚠️
    Decision: We prioritized manual SQL column additions over prisma migrate reset to preserve existing biometric and user data.
    • Face Recognition Support: Added faceTemplate (String?) to User and VerificationDocument models.
    • Command Used: npx prisma db execute --stdin --schema=... to apply non-destructive changes.

    Key File: schema.prisma

    v382

    Microservice Port Management

    Issue: Port conflicts (specifically 4000 and 3000) prevented concurrent execution of the API Gateway and Admin Dashboard.

    • Gateway: Standardized on port 4000 for API aggregation.
    • Admin Dashboard: Next.js frontend runs on port 3000.
    • Resolution: Updated api-gateway/index.js proxy targets and verified admin build scripts.

    Forensic Audit & Compliance

    Regulatory adherence to BSP Circular 808 and PCI DSS 10.2.2 requirements for financial integrity.

    v479

    Standardized Forensic Audit Logging

    Compliance Goal: Implement immutable audit trails with standardized Manila (UTC+8) timestamps as required by BSP and PCI DSS.

    📋
    Implementation:
    • Centralized Helper: Created createAuditLog utility to capture IP, User Agent, and Manila Time.
    • Real-time Alerts: Linked audit logs to notifyAdmin for immediate security visibility.
    • Cross-Service: Integrated into Auth, Wallet, and Transaction services.

    Strategic Document: v479 Developer Manual

    Risks & Mitigations

    Technical Risks

    • Risk: Native TFLite crashes on specific chipsets.
      Mitigation: Disabled NNAPI by default and added a "Health Check" loop.
    • Risk: TFLite Model Placeholder (v383).
      Mitigation: Resolved in v386 with official 93.9MB FaceNet model and comprehensive validation.
    • Risk: UI thread blocking on web.
      Mitigation: Parallelized initialization in SplashScreen.

    Operational Risks

    • Risk: Biometric data theft.
      Mitigation: Local-only processing of face embeddings.
