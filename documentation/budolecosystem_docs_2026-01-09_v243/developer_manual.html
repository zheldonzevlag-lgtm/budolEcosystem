Developer Manual - Technical Implementation Details

Architectural Decisions

Shift to Strongly Typed Service Methods

Previously, the ApiService used flexible but dangerous Map payloads for both inputs and outputs. While this allowed for rapid prototyping, it introduced significant runtime risk in financial transactions. In v243, we transitioned to strongly typed return values (double for balance) and named parameters for transaction methods.

Code Logic Explanations

Balance Retrieval Refactoring

The getBalance() method now handles the complexity of parsing various backend response formats (Direct double vs. Map with balance key) and returns a clean double.

// lib/services/api_service.dart
Future<double> getBalance() async {
  // ... authentication checks ...
  final response = await http.get(Uri.parse(url), ...);
  if (response.statusCode == 200) {
    final data = json.decode(response.body);
    if (data is Map && data.containsKey('balance')) {
      return (data['balance'] ?? 0.0).toDouble();
    }
    return (data ?? 0.0).toDouble();
  }
  // ... error handling ...
}

Named Parameters for Transactions

By using named parameters, we ensure that the compiler enforces the presence of required fields like amount and recipient, reducing bugs in the UI-to-Service bridge. This was applied to transfer, cashIn, and processPayment.

// lib/services/api_service.dart
Future<Map<String, dynamic>> transfer({
  required String recipient,
  required double amount,
  String? description,
}) async {
  // Logic here...
}

Impact Analysis

  • Reduced Boilerplate: UI screens no longer need double.tryParse() or Map casting for balance.
  • Improved Stability: Transactions are less likely to fail due to missing keys in a Map.
  • Breaking Change: Any screen calling cashIn or transfer must be updated to use the new named parameter syntax.

Future Enhancement Recommendations

  • Response Objects: Move from Map return types for transactions to dedicated TransactionResult classes.
  • Dio Migration: Consider replacing the http package with Dio for better interceptor support and request cancellation.
  • Unit Testing: Implement mock-based unit tests for ApiService to verify parsing logic across various edge cases.
