v293 Developer Manual

Overview

This manual covers the technical implementation of the Network Discovery and LAN Access features introduced in v293.

Network Utility Implementation

The core logic resides in scripts/network-util.js. It uses the os module to detect physical network interfaces and filters out virtual adapters (WSL, Docker, etc.).

function getLocalIP() { const interfaces = os.networkInterfaces(); // ... filtering logic ... return candidates[0].address; }

The utility now recursively updates all .env files in the ecosystem. This ensures that when a developer moves to a different network (e.g., from home to office), all microservices automatically point to the correct LAN IP.

SSO Flow Logic

In v293, we moved away from hardcoded 127.0.0.1 overrides in the frontend. Instead, we rely on environment variables injected at runtime. The budolID login page and the application callback routes now use the detected LOCAL_IP via NEXT_PUBLIC_SSO_URL.

Architecture Decision: We use 0.0.0.0 as the bind address for all Express and Next.js servers to ensure they are reachable from any interface, while using the specific LAN IP for cross-service redirects.

Impact Analysis

  • Authentication: SSO redirects are now dynamic. Users on the same network can log in via the server's IP.
  • Service Communication: Internal API calls between services (e.g., budolShap to budolID) now use the LAN IP, facilitating physical device testing.
  • Development Workflow: Running node scripts/network-util.js (or starting the ecosystem via start-ecosystem.js) is required whenever the host machine's IP changes.
