Developer Manual

Technical implementation details for v1.2.0

Dual-Channel OTP Delivery

The OTP delivery logic has been enhanced to support simultaneous delivery via SMS and Email. This ensures high availability and compliance with BSP/PCI DSS standards for multi-factor authentication.

Core Implementation

The main logic resides in the OTP API route: budolshap-0.1.0/app/api/auth/otp/route.js


// Dual-Channel Delivery (PCI DSS & BSP Compliant)
const isPhone = /^\+63/.test(identifier) || /^[0-9]{10,11}$/.test(identifier)

if (isPhone) {
    // Primary: SMS
    await sendOTPSMS(identifier, otp)
    
    // Secondary: Email (Always send to email as backup if we have it)
    if (user.email) {
        await sendOTPEmail(user.email, otp, user.name || 'User')
    }
} else {
    // Primary: Email
    await sendOTPEmail(user.email, otp, user.name || 'User')
    
    // Secondary: SMS (If phone number exists)
    if (user.phoneNumber) {
        await sendOTPSMS(user.phoneNumber, otp)
    }
}

Utilities

  • Email Utility: lib/email.js - Handles SMTP/Brevo delivery with fallback to console logging in development.
  • SMS Utility: lib/sms.js - Supports Zerix, iTextMo, and Viber providers with console fallback.
Rationale: By sending the OTP through two separate channels, we mitigate the risk of a single provider outage (e.g., SMS gateway delay) blocking the user login experience.

Database Schema

The VerificationCode table must exist in the public schema of the budolshap_1db database.


model VerificationCode {
  id         String   @id @default(cuid())
  identifier String   @unique
  code       String
  expiresAt  DateTime
  type       String   @default("LOGIN")
  createdAt  DateTime @default(now())
}
