Future Enhancements: Network & Authentication

1. HTTPS Support for Local Development

Purpose:

Enable secure communication (TLS/SSL) for local development to better mimic production environments.

Explanation:

Currently, all services run over HTTP. This prevents testing features like Secure/SameSite=None cookies and some modern browser APIs that require a secure context.

Reason:

Increasingly, browsers and external APIs (like OAuth providers) require HTTPS even for local testing.

Issues:

Local SSL certificates are difficult to manage across multiple microservices and devices.

Fixes:

Implement mkcert or a similar utility to generate locally-trusted certificates and update network-util.js to handle https:// protocols.

Risk:

Increased complexity in local setup for new developers.

Mitigation:

Provide a one-click setup script that installs certificates and configures the ecosystem.

2. Dynamic Service Discovery (mDNS/Bonjour)

Purpose:

Allow services to find each other without manual IP configuration using human-readable names like budolid.local.

Explanation:

Instead of hardcoding IPs in .env files, services would broadcast their presence and ports via mDNS.

Reason:

Eliminates the need to run network-util.js every time the IP changes.

Issues:

mDNS reliability can vary across different routers and OS configurations.

Fixes:

Integrate bonjour-service library into the startup runner to resolve .local names to IPs dynamically.

Risk:

Dependency on network hardware support for multicast.

Mitigation:

Keep the current IP-based discovery as a fallback mechanism.

3. Centralized Configuration Management

Purpose:

Move configuration away from 20+ individual .env files to a single source of truth.

Explanation:

A "Config Service" or a shared configuration file that all services read at startup.

Reason:

Reduces the risk of configuration drift and mismatches (like the JWT_SECRET issue resolved in v292).

Issues:

Adds a startup dependency (Config Service must be up first).

Fixes:

Implement a lightweight JSON-based configuration server or use a tool like etcd or Consul.

Risk:

Single point of failure for the entire ecosystem.

Mitigation:

Use local caching of the last known good configuration.

4. RBAC & Granular Permission Management UI

Purpose:

Provide a user-friendly interface for administrators to manage roles and granular permissions without direct database access.

Explanation:

Currently, RBAC assignments are handled via scripts (e.g., assign_rbac.mjs). A dedicated UI within the Admin Dashboard would allow for real-time adjustments.

Reason:

As the number of staff and managers grows, manual script execution becomes inefficient and error-prone.

Issues:

Need to ensure that the UI itself is heavily secured and only accessible by Super-Admins.

Fixes:

Develop a PermissionsMatrix component in the Admin Dashboard linked to the /api/admin/users/roles endpoint.

Risk:

Potential for accidental privilege escalation if the UI logic has bugs.

Mitigation:

Implement "Dual-Approval" for critical permission changes and maintain a strict audit log of all role modifications.

Dependencies:

Requires the AUDIT_LOGS_VIEW permission and backend logging infrastructure.
