Technical Walkthrough: Production Data Recovery

1. Initial Situation

The BudolShap storefront on Vercel was displaying "0 products" and user logins were failing with 500 errors. Investigation revealed that while the Vercel Postgres instance was running, it lacked the production data stored in the local PGDMP backups.

2. Data Conversion & Schema Isolation

PostgreSQL PGDMP (Custom Format) backups were converted to plain-text SQL to allow for manual editing and compatibility with Vercel's proxy connection limits.

pg_restore -f budolID.sql_txt G:\IT\database-backup\budolid.PGDMP

Since Vercel Postgres does not allow CREATE DATABASE, we implemented Schema Isolation, mapping each local database to a Postgres Schema: budolid, budolpay, and budolaccounting.

3. Manual Data Synchronization

Monolithic psql restoration timed out due to Vercel's 10s connection proxy limit. A custom Node.js sync script (manual-data-sync.js) was developed to parse COPY blocks and perform row-by-row INSERT ... ON CONFLICT operations.

  • Restored: 97 Categories
  • Restored: 14 Products (Smart Watch, Electronics, etc.)
  • Restored: 6 User accounts in public
  • Restored: 2 User accounts in budolid

4. Environment Variable Synchronization

Critical environment variables were added to Vercel Production to point services to their respective schemas:

  • BUDOLID_DATABASE_URL: ...?schema=budolid
  • BUDOLPAY_DATABASE_URL: ...?schema=budolpay
  • BUDOLACCOUNTING_DATABASE_URL: ...?schema=budolaccounting
  • JWT_SECRET: Restored for auth stability

5. Verification

The application was redeployed to apply the environment variables and clear API caches. Browser verification confirms that products are now live.

Live storefront at https://budolshap.vercel.app/shop showing 13+ restorted products.

Next Steps: Monitor transaction logs for any cross-schema foreign key constraints.
