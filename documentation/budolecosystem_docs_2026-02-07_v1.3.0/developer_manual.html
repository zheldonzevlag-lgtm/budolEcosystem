2. KYC Data Contracts (BSP Compliant)

Following BSP Circular 1022, the KYC process is split into levels. Developers must ensure all KYC data is encrypted at rest and in transit.

2.1. Verification Levels

  • Level 1 (Basic): { name, phone, email } - Stored in User model.
  • Level 2 (Fully Verified): { idType, idNumber, selfieUrl, dob, address, occupation, incomeSource } - Stored in KycVerification model.
Data Schema Proposal (Prisma) 
model KycVerification {
  id        String   @id @default(cuid())
  userId    String   @unique
  user      User     @relation(fields: [userId], references: [id])
  idType    String   // e.g., 'PhilID', 'Passport'
  idNumber  String
  selfieUrl String
  status    String   @default("pending") // pending, approved, rejected
  createdAt DateTime @default(now())
}

3. Compliance Requirements

  • PCI DSS: No storage of raw CVV or magnetic stripe data.
  • AMLA: Automated screening against Sanctions Lists during Level 2 verification.
  • NPC: Biometric data must have an explicit expiry and deletion policy.
