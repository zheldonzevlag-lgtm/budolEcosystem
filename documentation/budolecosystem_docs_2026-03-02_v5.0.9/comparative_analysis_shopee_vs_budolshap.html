Comparative Analysis: Shopee vs BudolShap

Date: 2026-02-20

Version: v1.0

1. Executive Summary

This document provides a comparative analysis between the industry-standard e-commerce processes (modeled after Shopee) and the current implementation of the BudolShap ecosystem. The analysis focuses on Multi-Store Checkout, Product Structure, and Financial Flows.

Key Finding: BudolShap's architecture is largely aligned with industry standards regarding multi-store order splitting and payment aggregation. However, a critical gap exists in Inventory Management, where stock is checked but not atomically deducted during order creation.

2. Process Comparison

2.1 Multi-Store Checkout & Payment

Feature Shopee (Industry Standard) BudolShap (Current) Status
Checkout Model Single Payment Event → Split into Multiple Orders (per store). Single Payment Event → Split into Multiple Orders (per store). Logic handled in ordersService.js. Aligned
Order IDs Generated per store (e.g., ORD-A, ORD-B). Generated per store (UUIDs). Linked via Payment Intent metadata. Aligned
Payment Aggregation Total Amount = Sum(Store A + Store B). Single Gateway Request. Total Amount aggregated on frontend/backend. Single Gateway Request (PayMongo/BudolPay). Aligned
Session Tracking Explicit Checkout entity (Table) linking all orders. Implicit linking via paymentId and Webhook Metadata. No dedicated Checkout table. Minor Gap

2.2 Product & Inventory

Feature Shopee (Industry Standard) BudolShap (Current) Status
Product Structure Product has 1 or more SKUs. Variants generate SKU matrix. Product has tier_variations and variation_matrix (JSON). Architectural Diff
Inventory Deduction Atomic deduction at the moment of order creation (payment pending). Checks inStock but DOES NOT DEDUCT stock in createOrder transaction. CRITICAL GAP
Variant Pricing Per SKU price. Per SKU price in variation_matrix JSON. Aligned

3. Critical Gap Analysis

🔴 Critical: Inventory Deduction Missing

Observation: The createOrder function in lib/services/ordersService.js verifies stock availability: 

if (selectedVariation.stock === 0) throw new Error(...)
However, it does not verify nor decrement the stock count in the database transaction.

Risk: Overselling. Multiple users can buy the last item simultaneously.

Fix Required: Implement atomic decrement of stock in the prisma.$transaction block.

🟡 Minor: JSON-based Variants

Observation: Variants are stored as JSON blobs (variation_matrix).

Impact: Harder to perform SQL-level aggregation (e.g., "Total stock across all red shirts"). Updates require reading the full JSON, modifying it, and writing it back, which is prone to concurrency issues.

Recommendation: Move to a normalized ProductVariant table in Phase 3.

5. Architecture Reference

Current Order Flow:

Cart -> Checkout Page -> OrderSummary.jsx 
    -> POST /api/payment/checkout (Aggregates Amount)
    -> Payment Gateway (PayMongo/BudolPay)
    -> Webhook (splits by metadata) -> Updates Order Status

Proposed Inventory Flow (Phase 1):

POST /api/orders
    -> Validate User & Address
    -> Transaction Start
        -> For each item:
            -> Lock Product/Variant
            -> Check Stock > Qty
            -> Decrement Stock
            -> Create Order
            -> Create OrderItems
            -> Clean Cart
    -> Transaction Commit
