Developer Manual: Forensic Audit Integration

Technical guide for implementing and maintaining real-time forensic audit logs across the Budol Ecosystem.

Mobile App Versioning

Mandatory Update: Version 1.2.94 is required for forensic logout audit compliance and real-time SWR stability.

The mobile app version has been incremented to ensure all security events are captured. Previous versions (v1.2.91 and below) only cleared sessions locally, which created a gap in the forensic audit trail.

Logout Implementation (Mobile)

The ApiService.logout() method now performs a synchronous backend notification before clearing local storage. This ensures the SECURITY_LOGOUT event is recorded even if the app is closed immediately after.

Future<void> logout() async { try { if (user != null && user!['id'] != null) { final url = '$authUrl/logout'; await http.post( Uri.parse(url), headers: {'Content-Type': 'application/json'}, body: json.encode({ 'userId': user!['id'], 'deviceId': deviceId, }), ).timeout(const Duration(seconds: 5)); } } catch (e) { print('ApiService: Backend logout notification failed: $e'); } // Clear session... }

1. Architecture & Data Contracts

v409 introduces a Provider-Agnostic Real-time Hub. Microservices no longer care whether the client is using WebSockets, Pusher, or SWR polling. They simply notify the Gateway, which orchestrates the distribution.

Data Contract: Internal Notification
POST /internal/notify { "userId": "uuid", "isAdmin": boolean, "event": "AUDIT_LOG_CREATED", "data": { ...auditLogObject } }

2. Real-time Pipeline Implementation

Clients (Mobile and Web) now fetch system settings to determine their real-time synchronization strategy. This ensures high availability even if a third-party provider like Pusher goes down.

Mobile SWR Implementation (Flutter)

The mobile app uses a "silent polling" mechanism as a fallback for SWR. It avoids UI flicker by suppressing loading states during background refreshes.

void _initSWR() { final method = apiService.systemSettings?['realtime']?['method'] ?? 'SWR'; if (method == 'SWR' || method == 'POLLING') { _swrTimer = Timer.periodic(Duration(milliseconds: interval), (_) { _fetchTransactions(silent: true); }); } }

4. Compliance Verification

The system is aligned with the following standards:

  • PCI DSS 10.2.2: Logging of all actions taken by any individual with root or administrative privileges.
  • BSP Circular 808: Real-time monitoring of financial transactions for AML/KYC compliance.
  • Cybersecurity Law: Immutable forensic audit trails for incident response.

5. Dependencies & Rationale

Dependency Purpose Rationale
Socket.io Bi-directional Real-time Chosen for low-latency, stateful connections between the Admin Dashboard and the Gateway. Supports "rooms" (e.g., admin) for targeted broadcasting.
Pusher (Channels) Managed Webhooks Used as a high-availability fallback. Since it's a managed service, it reduces server-side load and provides global edge distribution for mobile clients.
SWR (Stale-While-Revalidate) Client-side Polling Implements the "Silent Refresh" pattern on mobile. It ensures data is always "fresh enough" even if WebSocket connections are unstable in mobile environments.
Prisma ORM Database Interaction Provides type-safe access to the Audit Log schema, ensuring compliance data is correctly formatted before insertion.
JWT (Json Web Token) Session Management Stateless authentication enables microservices to verify users independently while logging security events (login/logout) to a centralized audit service.
