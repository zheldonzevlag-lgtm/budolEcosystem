P2P Transaction Implementation Details

Purpose

Enable secure and efficient budolPay-to-budolPay (P2P) fund transfers between users within the ecosystem.

Code Logic Explanation

The transaction flow involves several steps across the mobile app and backend services:

  1. Mobile Initiation: ApiService.transfer in the Flutter app sends a POST request to /transactions/transfer via the API Gateway.
  2. Gateway Proxy: The API Gateway routes the request to the transaction-service.
  3. Transaction Processing:
    • Resolves recipient (email or phone) to a User ID.
    • Checks KYC limits (BASIC users cannot send P2P).
    • Creates a PENDING transaction record in the database.
    • Calls wallet-service to subtract funds from the sender.
    • Calls wallet-service to add funds to the receiver.
    • Creates accounting ledger entries (Debit Liability 1010 for sender, Credit Liability 1010 for receiver).
    • Updates transaction status to COMPLETED.
    • Notifies Admin via Socket.io.

Architectural Decisions

  • Microservice Separation: Transactions are handled by a dedicated transaction-service, while balance updates are managed by the wallet-service.
  • RBAC Enforcement: Permissions like TRANSACTION_CREATE_P2P and WALLET_DEBIT are enforced at the service level using middleware.
  • Atomicity: Rollback logic is implemented if adding funds to the receiver fails after deducting from the sender.

Impact Analysis

  • Database: New records in Transaction and LedgerEntry tables.
  • Performance: Synchronous calls between services require low latency; timeouts are set to 10 seconds.
