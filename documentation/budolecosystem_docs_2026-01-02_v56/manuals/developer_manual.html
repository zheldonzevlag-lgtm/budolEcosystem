Developer Manual - Version 56

Session Stability Improvements

Version 56 introduces a multi-layered authentication strategy to resolve session persistence issues, particularly during the checkout flow and when testing on local IP addresses.

1. Multi-Layered Auth in OrderSummary

We've moved away from a cookie-only authentication strategy for critical checkout API calls. The OrderSummary.jsx component now explicitly sends the JWT token in the Authorization header.

// OrderSummary.jsx enhancement
const token = getToken();
const response = await fetch('/api/orders', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        ...(token ? { 'Authorization': `Bearer ${token}` } : {})
    },
    body: JSON.stringify(orderData)
});

2. Resilient Session Synchronization

The AuthContext.jsx and auth-client.js have been updated to distinguish between "Unauthorized" (401/403) and "Server/Network Errors" (5xx/Fetch failures).

  • Explicit Logout: Only occurs if the server returns 401 or 403.
  • Session Persistence: If a 500 error or network glitch occurs during background sync, the existing client-side session is preserved instead of being cleared.

3. Implementation Details

  • lib/auth-client.js: Updated getCurrentUser() to return null for auth failures and undefined for other errors.
  • context/AuthContext.jsx: Updated loadUser() to only trigger logout logic when result === null.

Impact Analysis

These changes significantly reduce "unexpected logouts" during the transition from order creation to payment gateway redirection. It also makes the platform more compatible with local network testing where cookie behavior might be inconsistent.
