KYC Rejection Pipeline

The KYC rejection process is handled by a shared API endpoint between the Admin and User services. When an admin triggers a rejection, the following database operation occurs:

Backend Logic (Prisma)

// apps/admin/app/api/users/route.ts
const updatedUser = await prisma.user.update({
  where: { id: userId },
  data: { kycStatus: 'REJECTED' },
});

Impact: The kycStatus field is strictly updated to "REJECTED". No historical data is currently moved to an audit table.

Mobile App Integration

The mobile app (Flutter) synchronizes this status during the next profile fetch or session refresh. The logic is located in the ApiService and reflected in the SettingsScreen.

  • Status Sync: SharedPreferences is updated with the latest user profile, including the new status.
  • Navigation: The "Verify Account" tile in settings_screen.dart remains active, allowing users to re-enter the KYCVerificationScreen.

OCR Status Logic

The "Extraction in progress or data not found" message in the Admin UI is a conditional render based on the presence of a source engine but absence of mapped fields (Name, ID, etc.).

// Logic check in renderOcrData
const hasActualData = Object.keys(rest).length > 0;
// If !hasActualData but source exists -> Display "Extraction in progress..."

Dependencies & Risks

  • Dependency: Mobile app relies on a successful API sync to show the latest KYC status.
  • Risk: Overwriting previous KYC data on re-submission makes it difficult to track repeated rejections.
  • Risk: Lack of "Rejection Reason" persistence prevents developers from providing specific UI feedback to users.
