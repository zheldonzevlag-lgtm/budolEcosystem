Developer Manual: v258 Implementation

1. Purpose & Reason

The primary purpose of this update was to resolve a regression in the API Gateway where the /system/settings route was missing. This route is critical for the mobile application to determine whether to use Pusher, Socket.io, or SWR for real-time updates. Without it, the mobile app defaulted to SWR (polling), which resulted in delayed or non-functional real-time transaction updates.

2. Technical Details & Code Changes

API Gateway Route Implementation

File: services/api-gateway/index.js

Added the following route to the Express router to fetch and format system settings:

router.get('/system/settings', async (req, res) => {
    try {
        const settings = await prisma.systemSetting.findMany();
        const settingsMap = settings.reduce((acc, curr) => {
            acc[curr.key] = curr.value;
            return acc;
        }, {});

        // Format for mobile app
        const response = {
            realtime: {
                method: settingsMap['REALTIME_METHOD'] || 'SWR',
                pusher: {
                    key: settingsMap['PUSHER_KEY'],
                    cluster: settingsMap['PUSHER_CLUSTER']
                },
                socketio: {
                    url: settingsMap['SOCKETIO_URL']
                },
                swr: {
                    refreshInterval: parseInt(settingsMap['SWR_REFRESH_INTERVAL'] || '5000')
                }
            },
            maps: {
                enabledProviders: JSON.parse(settingsMap['ENABLED_MAP_PROVIDERS'] || '[]'),
                googleMapsKey: settingsMap['GOOGLE_MAPS_API_KEY'],
                geoapifyKey: settingsMap['GEOAPIFY_API_KEY'],
                radarKey: settingsMap['RADAR_API_KEY']
            }
        };

        res.json(response);
    } catch (err) {
        console.error('[Gateway] Failed to fetch system settings:', err.message);
        res.status(500).json({ error: 'Failed to fetch system settings' });
    }
});

3. Impact Analysis

  • Mobile Integration: RealtimeService now correctly receives method: "PUSHER" and initializes the Pusher client.
  • Latency: Reduced UI update latency from 1-5 seconds (SWR) to near-instantaneous (Pusher).
  • Server Load: Decreased polling frequency from mobile clients, reducing unnecessary HTTP traffic.
