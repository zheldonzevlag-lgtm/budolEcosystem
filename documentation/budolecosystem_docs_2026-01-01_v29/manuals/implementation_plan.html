Implementation Plan: SSO & Ecosystem Integration

Version: 2026-01-01_v29 | Status: Active

Executive Summary

This plan outlines the migration from siloed application authentication to a centralized Single Sign-On (SSO) model using budolID. The goal is to provide a seamless user experience across BudolShap, BudolPay, and future ecosystem services.

Implementation Phases

Phase 1: Foundation & Core SSO

Deploy the standalone budolID service and its dedicated database.

  • M1.1: Initialize budolid_db with global User and App models.
  • M1.2: Implement JWT generation and public key verification endpoints.
  • M1.3: Deploy SSO Login Portal UI.

Dependency: PostgreSQL host availability.

Phase 2: BudolShap Integration

Modify BudolShap to support both local and SSO authentication.

  • M2.1: Register BudolShap as an authorized app in budolid_db.
  • M2.2: Add "Login with budolID" redirect logic to BudolShap frontend.
  • M2.3: Implement backend token verification and local session mapping.

Dependency: Phase 1 completion.

Phase 3: Data Synchronization & Cleanup

Ensure user data consistency across the ecosystem.

  • M3.1: Execute migration script to sync existing BudolShap users to BudolID.
  • M3.2: Implement webhook/trigger for real-time user profile sync.
  • M3.3: Finalize SSO redirect ports (8000 for SSO, 3000 for Shop).

Dependency: Phase 2 validation.

Testing & Validation Protocol

Test Case Scenario Success Criteria
SSO Redirect Click "Login with budolID" in BudolShap User reaches :8000/login with correct apiKey
JWT Verification Successful login in SSO portal User redirected back to Shop with valid JWT; Shop recognizes user
Cross-App Auth Login in Shop, then visit BudolPay User is automatically recognized (Session Persistence)

Rollback Strategy

In case of critical failure in SSO integration:

  1. Disable the "Login with budolID" button in the frontend.
  2. Revert NEXTAUTH_URL and provider configurations to local-only mode.
  3. Restore budolshap_1db from pre-sync backup if data corruption occurs.
