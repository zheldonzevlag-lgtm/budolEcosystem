Developer Manual: Address System & Validation Updates

Version 3.3.24 introduces significant improvements to the address handling system, focusing on optional fields, validation consistency, and database schema flexibility.

1. Optional Address Fields (Barangay & Zip)

The `barangay` and `zip` fields are now optional across the entire stack (Frontend, Backend, Database). This was done to accommodate addresses where this information is not available or necessary.

Database Schema Changes

The `Address` and `StoreAddress` models in `schema.prisma` have been updated to make `barangay` and `zip` nullable (`String?`).

prisma/schema.prisma 
model Address {
  // ...
  barangay     String?
  zip          String?
  // ...
}

2. Zod Validation Integration

We have replaced toast-based validation with Zod schemas for robust, type-safe validation in `AddressFormManager.jsx`. This ensures consistent error handling and user feedback.

Validation Schema

components/address/AddressFormManager.jsx 
        const getSchema = () => {
        const isStoreMode = mode === 'store';

        return z.object({
            name: isStoreMode ? z.string().min(1, "Store name is required") : z.string().optional(),
            firstName: !isStoreMode ? z.string().min(1, "First name is required") : z.string().optional(),
            lastName: !isStoreMode ? z.string().min(1, "Last name is required") : z.string().optional(),
            email: !isStoreMode ? z.string().email("Invalid email format").min(1, "Email is required") : z.string().optional(),
            phone: z.string().refine((val) => {
                const code = COUNTRY_CODES.find(c => val.startsWith(c.code))?.code || '+63';
                const numberPart = val.slice(code.length).replace(/\D/g, '');
                return numberPart.length === 10;
            }, "Phone number must be exactly 10 digits"),
            city: z.string().min(1, "City is required"),
            province: z.string().min(1, "Province is required"),
            zip: z.string().optional(),
            barangay: z.string().optional(),
            detailedAddress: z.string().min(1, "Street name, building, house no. is required"),
            password: z.string().min(8, "Password must be at least 8 characters").optional().or(z.literal('')),
        });
    };

3. Prisma Client Custom Path (File Lock Mitigation)

To resolve `EPERM` file lock issues during development when regenerating the Prisma client, we have changed the output directory of the Prisma client to `@prisma/client-custom-v4`.

Configuration

prisma/schema.prisma 
generator client {
  provider = "prisma-client-js"
  output   = "../node_modules/@prisma/client-custom-v3"
}

Client Instantiation

lib/prisma.js 
import { PrismaClient } from '@prisma/client-custom-v3'
// ...

6. Critical Fixes (Post-Release v3.3.24)

Missing Required Fields Fix

Resolved an issue where users could not save addresses due to "Missing required fields" error.

  • Issue 1: `AddressModal` was ignoring the email entered in the form and sending the (potentially empty) email from the initial user metadata.
  • Fix: Updated `handleSaveAddress` to prioritize `formData.email`.
  • Issue 2: The API required `state` (Province) and `zip` code, but `AddressFormManager` lacked input fields for them (relying solely on map data which could be incomplete).
  • Fix: Added manual input fields for Province/State and Zip Code to `AddressFormManager` and added Zod validation to ensure they are provided.

AddressModal.jsx Fix

components/AddressModal.jsx 
// Before
email: userMeta.email,

// After
email: formData.email || userMeta.email,

Missing Required Fields Fix (Store Address)

Resolved an issue where store addresses could not be saved due to "Missing required fields" error, specifically `district` and `zip` which were required by the API but not provided by the form.

  • Issue: The API endpoint `app/api/stores/[storeId]/addresses/route.js` required `district` and `zip`, but the frontend `AddressFormManager` does not enforce them for store addresses.
  • Fix: Made `district` and `zip` optional in the `StoreAddress` schema and updated the API validation to allow them to be null.

Store Address API Update

app/api/stores/[storeId]/addresses/route.js 
// Before
if (!phone || !district || !city || !detailedAddress || !zip) { ... }

// After
if (!phone || !city || !detailedAddress) { ... }

7. Address Label and Default Status

Added `label` (e.g., "Home", "Work") and `isDefault` fields to the Address model to better manage user addresses.

Database Updates

prisma/schema.prisma 
model Address {
  // ...
  label        String?
  isDefault    Boolean  @default(false)
}

model StoreAddress {
  // ...
  label        String?
}

Functionality

  • Label Selection: Users can now categorize addresses (Home, Work, Other).
  • Default Address Logic: Setting an address as default automatically unsets any previous default address for the user.
  • Form Validation: The "Save" button is now disabled until all required fields are filled, preventing incomplete submissions.
