Technical Implementation Details

1. Robust Input Validation

To prevent financial errors and potential exploits, we implemented multi-layer validation for payment amounts. This ensures that only valid, positive numeric values (in centavos) are processed.

Backend Implementation

In the Payment Gateway Service, we added strict parsing and checks for the amount field.

// File: budolpay-0.1.0/services/payment-gateway-service/index.js const parsedAmount = parseFloat(amount); if (isNaN(parsedAmount) || parsedAmount <= 0) { return res.status(400).json({ error: 'Invalid amount: must be a positive number' }); }

View payment-gateway-service/index.js

Frontend/Library Validation

In BudolShap, we added validation at the initiation level to catch errors early.

// File: budolshap-0.1.0/lib/services/paymentService.js const parsedAmount = parseInt(amount, 10); if (isNaN(parsedAmount) || parsedAmount <= 0) { throw new Error('Valid amount (in centavos) is required'); }

View paymentService.js

2. Duplicate Transaction Prevention

Duplicate payments are a significant risk in e-commerce. We implemented a robust check using orderId stored in transaction metadata.

// Robust metadata search for duplicate PENDING transactions const orderIdSearch = `"orderId":"${metadata.orderId}"`; const altOrderIdSearch = `"orderId": "${metadata.orderId}"`; const existingPending = await prisma.transaction.findFirst({ where: { status: 'PENDING', type: 'MERCHANT_PAYMENT', OR: [ { metadata: { contains: orderIdSearch } }, { metadata: { contains: altOrderIdSearch } } ] } });

View payment-gateway-service/index.js

3. QR Payment Integrity

During QR processing in the Wallet Service, we now verify that the requested payment amount matches the original transaction intent created by the gateway.

// Verify amount consistency in wallet-service if (parseFloat(transaction.amount) !== amountToPay) { return res.status(400).json({ error: 'Amount mismatch: QR amount does not match original transaction' }); }

View wallet-service/index.js

Compliance Verification

These changes satisfy BSP Circular 808 Section 5 (Transaction Integrity) and PCI DSS Requirement 6.4.1 (Input Validation).
