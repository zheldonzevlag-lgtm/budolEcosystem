Security Mitigation Plan

1. Multi-Tier Input Validation (NaN/Negative Fix)

Risk: R-PAY-001

Strategy: Implement validation at the UI layer (BudolShap), Library layer (PaymentService), and API Gateway layer (Payment Gateway Service). Use strict parsing (parseInt/parseFloat) and range checks (amount > 0).

2. Idempotency & Duplicate Check

Risk: R-PAY-002

Strategy: Use orderId as a unique identifier for pending transactions. Before creating a new intent, query existing PENDING transactions with the same orderId in metadata. Return the existing record instead of creating a new one.

3. Transaction-Intent Reconciliation (QR Security)

Risk: R-PAY-003

Strategy: When processing a QR payment, the Wallet Service must fetch the original transaction intent from the database using the paymentIntentId and compare its amount with the amount provided in the QR data. If they differ, the transaction is rejected.
