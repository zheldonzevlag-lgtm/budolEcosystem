v224: RBAC Defense in Depth & Internal Enforcement

Date: 2026-01-08 | Status: Implemented

Purpose

To implement "Defense in Depth" (PCI DSS Requirement 7.2.1) by enforcing Role-Based Access Control (RBAC) not just at the Gateway, but within internal microservices (Wallet & Transaction Services). This ensures that even if internal security is bypassed, unauthorized actions are blocked at the resource level.

Architectural Rationale: Multi-Admin RBAC

The decision to maintain separate RBAC management in budolShap and budolPay is a deliberate security and compliance measure.

  • Segregation of Duties (SoD): Per BSP Circular 808, financial oversight must be isolated from retail operations. Marketplace staff (approving products) should not have inherent access to financial treasury or KYC data.
  • Blast Radius Mitigation: In the event of a credential compromise in the Marketplace portal, the Financial Ecosystem remains protected by a separate set of role-based permissions and enforcement logic.
  • Compliance Mapping:
    • budolShap RBAC is optimized for DPA 2012 (Customer Data Privacy).
    • budolPay RBAC is hardened for PCI DSS v4.0 and BSP Circular 808 (Financial Integrity).

Shared Identity: Both portals utilize budolID for Single Sign-On (SSO), ensuring a unified identity while maintaining granular, application-specific authorization.

Technical Implementation

1. Unified Auth Middleware

Created packages/database/auth-middleware.js and exported via @budolpay/database/auth. This middleware handles both JWT verification and granular permission checks.

// Usage in Microservices
const { verifyToken, authorize } = require('@budolpay/database/auth');
app.use(verifyToken);
router.post('/transfer', authorize(PERMISSIONS.TRANSACTION_CREATE_P2P), ...);

2. Microservice Hardening

  • Wallet Service: Restricted /process-qr to roles with WALLET_DEBIT and /config to SYSTEM_SETTINGS_READ.
  • Transaction Service: Restricted /transfer to TRANSACTION_CREATE_P2P.

3. RBAC Testing Suite

Developed scripts/test-rbac.mjs to automate verification of access control policies across all roles (ADMIN, STAFF, USER). This satisfies BSP Circular 808 requirements for periodic security testing.

Compliance Validation

Zero-Trust Architecture: By enforcing permissions at the service level, we mitigate the risk of "Lateral Movement" within the cluster. Even if an attacker compromises one service, they cannot execute commands in another without a valid role-scoped token.

Issues & Fixes

  • Issue: Internal services were previously relying solely on Gateway validation.
  • Fix: Integrated verifyToken and authorize in all sensitive microservice routes.
  • Risk: Performance overhead of redundant JWT verification.
  • Mitigation: Minimal impact as JWT verification is a stateless, local CPU operation (no DB call required for simple role checks).
