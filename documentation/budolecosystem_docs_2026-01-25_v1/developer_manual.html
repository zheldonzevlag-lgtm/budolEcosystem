Developer Manual - Unified Payment UI

Overview

This implementation unifies the post-payment flow for budolPay with existing providers like PayMongo (GCash, Maya). Key changes include provider-specific status polling and redirection to the shared /payment/return page.

1. Status API Enhancement

File: app/api/paymongo/status/route.js

The status API now supports an explicit provider parameter. When provider=budolpay is passed, it routes the status check through the budolPayAdapter and translates the internal COMPLETED status to the standard succeeded status used by the frontend.

if (intent_id.startsWith('INTENT-') || intent_id.startsWith('JON-') || provider === 'budolpay') {
    const budolPayAdapter = paymentService.adapters['budolpay'];
    const statusData = await budolPayAdapter.getPaymentIntent(intent_id);
    const status = statusData.status === 'COMPLETED' ? 'succeeded' : statusData.status;
    return NextResponse.json({ status });
}

2. Unified Return Page Polling

File: app/payment/return/page.jsx

The return page now captures the provider from URL search parameters and passes it to the pollStatus function, ensuring accurate verification for budolPay orders.

const providerParam = provider ? `&provider=${provider}` : '';
const resp = await fetch(`/api/paymongo/status?intent_id=${id}${providerParam}`);

3. QR Modal Redirection

File: components/OrderSummary.jsx

Instead of a simple toast and redirect to /orders, the success handler now redirects to the unified return page to show the "Payment Successful" UI.

const handleQRSuccess = () => {
    const provider = (qrData?.paymentMethod === 'BUDOL_PAY') ? 'budolpay' : 'paymongo';
    router.push(`/payment/return?payment_intent_id=${intentId}&orderId=${orderId}&provider=${provider}`);
};

Compliance Notes

  • PCI DSS: No sensitive card data is stored. All transactions are handled via secure payment adapters.
  • Cybersecurity Law: Polling logic is rate-limited and includes max-retry logic to prevent resource exhaustion.
