Developer Manual - v496

1. Session Data Optimization

The /api/auth/me endpoint was updated to include the user's store information in the initial session load. This is critical for the RealtimeProvider to correctly identify and subscribe to the store's private notification channel.

File: app/api/auth/me/route.js

// Enhanced Prisma query to include store data
const user = await prisma.user.findUnique({
    where: { id: userId },
    select: {
        id: true,
        // ... existing fields ...
        store: {
            select: {
                id: true,
                name: true,
                username: true,
                logo: true
            }
        }
    }
});

2. Realtime Event Normalization

To ensure consistent handling in the frontend RealtimeProvider.jsx, all order completion events have been normalized to order-updated with a status of COMPLETED.

File: lib/services/ordersService.js

// v496 Change: Unified event name to 'order-updated'
await triggerRealtimeEvent(`store-${order.storeId}`, 'order-updated', {
    orderId: order.id,
    status: 'COMPLETED'
});

await triggerRealtimeEvent(`user-${order.userId}`, 'order-updated', {
    orderId: order.id,
    status: 'COMPLETED'
});

3. Frontend Listeners

The RealtimeProvider.jsx automatically re-subscribes to channels when the user object changes, thanks to the dependency array in its primary useEffect hook.

File: components/RealtimeProvider.jsx

useEffect(() => {
    if (!user) return;

    const pusherInstance = initPusher();
    if (!pusherInstance) return;

    // Store Channel Subscription (v496 fix: user.store.id is now available)
    if (user?.store?.id) {
        const channelName = `store-${user.store.id}`;
        const storeChannel = pusherInstance.subscribe(channelName);
        storeChannel.bind('order-updated', (data) => handleGlobalEvent('order-updated', data));
    }
    
    // ... existing user channel subscription ...
}, [user]);

4. Data Contracts

The order-updated event payload now consistently includes:

  • Event Normalization: Unified order-updated events are now triggered across all services (orders, returns, disputes) to ensure consistent cache patching in the frontend.
  • Realtime Sync Logic: The RealtimeProvider.jsx now suppresses generic toasts for return-related status changes, prioritizing more specific notifications while ensuring the patchOrderCache logic always moves orders to the correct status tabs.

Realtime Sync Architecture

The system uses a combination of SWR for data fetching and Pusher/Socket.io for realtime updates. To avoid UI flicker and ensure instant tab routing, we use a Cache Patching strategy.

// components/RealtimeProvider.jsx - Simplified Sync Logic
const patchOrderCache = (data) => {
    mutate(
        (key) => key.startsWith('/api/orders'),
        (current) => {
            // Update status or remove from filtered list
            return updatedData;
        }
    );
};

// handleGlobalEvent handles deduplication and toast prioritization
if (event === 'order-updated') {
    if (!isReturnStatus(data.status)) showToast(data);
    patchOrderCache(data);
}
if (event === 'return-requested') {
    showReturnToast(data);
    patchOrderCache({ ...data, status: 'RETURN_REQUESTED' });
}

Returns & Refunds Integration

Return requests trigger a complex state change. The following flow ensures the Seller UI stays in sync:

  1. Buyer creates return -> returnsService.js updates order status to RETURN_REQUESTED.
  2. Backend triggers return-requested and order-updated events.
  3. Frontend RealtimeProvider receives both.
  4. Deduplication logic ensures only one informative toast is shown.
  5. patchOrderCache updates the local SWR store.
  6. StoreOrders.jsx re-filters and moves the order to the Return / Refund tab instantly.
