Developer Manual - v53

Purpose

To eliminate floating-point precision issues in financial calculations by standardizing amount handling across all microservices and frontend components.

Implementation Details

1. Shift from parseFloat to Number

parseFloat() can introduce subtle rounding errors. We have migrated to using Number() for initial parsing and performing calculations using integer arithmetic (centavos).

2. Integer Arithmetic (Centavos Pattern)

In wallet-service and transaction-service, balances are now updated using the following pattern:

// Convert to centavos (integer)
const currentBalanceCents = Math.round(Number(wallet.balance) * 100);
const amountCents = Math.round(Number(amount) * 100);

// Perform integer math
const newBalanceCents = type === 'add' ? currentBalanceCents + amountCents : currentBalanceCents - amountCents;

// Convert back to decimal for storage
const newBalance = newBalanceCents / 100;

3. Frontend Standardization

Updated OrderSummary.jsx and WalletPage.jsx to ensure amounts are passed as validated numbers, preventing JSON parsing errors in the backend.

// OrderSummary.jsx
const amountInCentavos = Math.round(Number(finalAmount).toFixed(2) * 100);

Impact Analysis

  • Affected Services: payment-gateway-service, wallet-service, transaction-service, budolshap (frontend).
  • Database: No schema changes; Prisma's Decimal type remains compatible.
  • API: APIs still accept decimal strings/numbers but handle them as integers internally.

Recommendations for Future Enhancement

  • Implement a global Currency utility class to centralize these conversions.
  • Add unit tests specifically for boundary cases (e.g., 0.1 + 0.2).
  • Consider storing amounts as BigInt (centavos) directly in the database for absolute precision.
