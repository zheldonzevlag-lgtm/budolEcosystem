Centralized Transaction Logging Logic

Architecture Overview

The budolPay Admin application serves as the central source of truth for all financial transactions in the ecosystem. It achieves this by reading directly from the Transaction table in the shared budolpay_db.

Data Flow

1. Intent Logging

Every time a checkout is initiated (via PayMongo, BudolPay, etc.), the payment-gateway-service creates a PENDING record in the Transaction table.

// payment-gateway-service/index.js
const transaction = await prisma.transaction.create({
  data: {
    amount: parseFloat(amount),
    type: 'MERCHANT_PAYMENT',
    status: 'PENDING',
    referenceId: `INTENT-${Date.now()}`,
    metadata: JSON.stringify(metadata)
  }
});

2. Webhook Updates

Regardless of the provider, all completion signals (webhooks) are routed through the gateway, which updates the record to COMPLETED.

// Webhook handler in Gateway
const transaction = await prisma.transaction.update({
  where: { referenceId: referenceId },
  data: { 
    status: 'COMPLETED',
    completedAt: new Date()
  }
});

3. Admin Display

The budolPay Admin fetches these records using a standardized API route that filters by status, type, and date.

// apps/admin/app/api/transactions/route.ts
const transactions = await prisma.transaction.findMany({
  where,
  orderBy: { createdAt: "desc" },
  include: { sender: true, receiver: true }
});

Verification

✅ All PayMongo intents are logged before redirection.

✅ All internal BudolPay wallet payments are logged.

✅ Transaction history in Admin app reflects real-time status updates from webhooks.
