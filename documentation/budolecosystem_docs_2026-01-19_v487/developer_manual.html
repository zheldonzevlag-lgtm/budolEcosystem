Developer Manual - In-Page QR Modal Trigger

1. Purpose

To standardize the payment initiation flow by replacing external page redirections with an in-page modal trigger. This ensures a seamless user experience across checkout and order management views.

2. Rationale

Redirection to a separate payment page (Image 1 style) often leads to higher bounce rates and friction. Standardizing on a modal (Image 2 style) keeps the user in context, improves perceived performance, and aligns with modern fintech UX patterns used by GCash and Maya.

3. Technical Explanation

The implementation utilizes a centralized QRCodeModal component that handles QR rendering, status polling, and countdown timers. State management in OrderSummary.jsx and orders/page.jsx was updated to intercept successful checkout responses and display the modal instead of triggering a router push.

Key Logic:

  • Timer: 600-second (10-minute) countdown using useEffect and setInterval.
  • Polling: Background status checks every 5 seconds to detect payment completion.
  • Data Normalization: Ensures QR data from different providers (PayMongo, budolPay) is mapped to a consistent schema.

4. Dependencies and Software Rationale

The following software and libraries were selected to ensure scalability, security, and developer productivity:

  • Next.js (React Framework): Chosen for its hybrid rendering capabilities (SSR/CSR) and built-in routing, which allows for fast page loads and seamless modal transitions.
  • Prisma ORM: Provides type-safe database access, ensuring that payment logs and order statuses are updated with zero schema mismatches.
  • Axios: Used for standardized HTTP requests with interceptors, crucial for handling authentication and error logging across payment gateways.
  • PayMongo API: Selected for its robust Philippine market support (GCash, Maya, QRPH) and clean JSON API structure.
  • Lucide React: Provides a consistent, lightweight icon set that aligns with the "Atlassian-inspired" professional UI/UX.

Rationale for Modal Approach:

The decision to move from a page-based to a modal-based payment trigger was driven by Conversion Rate Optimization (CRO). By keeping the user on the same page, we eliminate the risk of session loss during redirection and provide a faster feedback loop for payment status updates.

5. Changes

5.1 Backend Services

  • Modified payment-gateway-service/index.js: Updated the create-intent endpoint to return QR data for internal budolPay payments instead of a checkout URL. This enables the in-page modal flow for budolPay.

5.2 Frontend Components

  • Modified QRCodeModal.jsx: Standardized UI, added instructions and timer.
  • Modified OrderSummary.jsx: Integrated modal trigger for "Place Order".
  • Modified orders/page.jsx: Integrated modal trigger for "Pay Now".

7. Compliance

  • BSP Circular 808: Maintains forensic audit trails for all payment initiation events.
  • NPC: No PII is stored in LocalStorage; only transaction-specific identifiers are used for session recovery.
  • Cybersecurity Law: Implements secure status polling with timeout protection.
