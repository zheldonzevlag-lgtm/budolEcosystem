Technical Explanation: Status Standardization

The ecosystem previously suffered from "Status Drift" where different payment gateways and services updated the Order status to either PAID or PROCESSING inconsistently.

Architecture Decision: In v489, we have standardized on PROCESSING as the primary state for paid orders that are ready for fulfillment. This aligns with the UNIVERSAL_STATUS.TO_SHIP mapping.

Implementation: Realtime Synchronization

To ensure the buyer sees their order move from "To Pay" to "To Ship" without a manual refresh, webhooks now broadcast to the user channel:


// Notify User (Buyer) for UI sync
await triggerRealtimeEvent(`user-${order.userId}`, 'order-updated', {
    orderId: order.id,
    status: order.status
});
