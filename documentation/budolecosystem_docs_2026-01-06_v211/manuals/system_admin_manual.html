System Admin Manual: Auth Service Infrastructure

Purpose

To provide operational guidelines for managing the upgraded GoTyme-aligned auth-service, including database schema changes and environment configuration.

Explanation

The auth-service now manages critical security metadata (PIN hashes, device trust lists). Admins must ensure high availability of the database to prevent login lockouts.

Reason

Correct infrastructure configuration is vital for maintaining "Bank-Grade" security. Misconfigured environment variables or exposed database ports could compromise user credentials.

Infrastructure Changes
  • Database: Prisma schema updated with pinHash, trustedDevices, and biometricKeyId fields.
  • Environment Variables: Ensure JWT_SECRET and DATABASE_URL are securely managed in production (e.g., via HashiCorp Vault or Vercel Secrets).
Operational Runbook

Deployment Procedure:

  1. Run npx prisma generate to update the client.
  2. Run npx prisma db push to apply schema changes to the target DB.
  3. Restart auth-service using pm2 restart auth-service or similar process manager.
Issues & Fixes

Issue: Database connection timeouts during high registration volume.
Fix: Increase connection pool size in DATABASE_URL (e.g., &connection_limit=20).

Risk & Mitigation

Risk: Exposure of trustedDevices JSON blob in logs.
Mitigation: Ensure log sanitization filters are active for the auth-service output.

Dependencies
  • PostgreSQL 14+ (Required for JSONB support).
  • Node.js 18+ LTS.
Technical Details

The trustedDevices field is stored as a JSON string for flexibility. Admins can manually reset a user's trusted devices by clearing this field in the User table if a device is stolen.

Test Results

Verified that prisma db push successfully adds new columns without data loss for existing users.

Recommendation for Future Enhancement

Purpose: Proactive security monitoring.
Explanation: Implement real-time alerting for failed PIN attempts across multiple devices for a single account.
Reason: Helps detect distributed brute-force attacks.
Issues: High volume of false positives.
Fixes: Use a threshold of 5 failed attempts within 10 minutes.
Risk: Alert fatigue for admins.
Mitigation: Route alerts to a dedicated security Slack channel.
Dependencies: Redis for rate limiting/event counting.
