Developer Manual: GoTyme-Aligned Auth Implementation

Purpose

To implement a bank-grade, mobile-centric authentication system for budolPay, aligning with GoTyme's security standards. This includes transitioning to 6-digit PINs, device-bound sessions, and biometric integration hooks.

Explanation

The system shifts from email/password to using the mobile number as the primary identity. It uses a multi-phase login flow: Identification -> Device Trust Check -> Challenge (OTP/PIN/Biometric). This ensures that only trusted devices can bypass OTP, mimicking high-security banking apps.

Reason

Traditional email-password flows are vulnerable to credential stuffing and phishing. Mobile-centric auth with device binding provides a higher security ceiling and a more seamless user experience for mobile users.

Issues & Fixes
Issue Fix
Initial 4-digit MPIN had low entropy. Upgraded to 6-digit PIN enforced by regex and logic in /register.
No way to identify if a device was previously used. Implemented trustedDevices JSON field in User model and device identification logic.
Risk & Mitigation

Risk: Users losing access to their registered phone number.
Mitigation: Implement a secondary recovery flow via verified email or manual support (planned for v212).

Dependencies
  • bcryptjs: For PIN hashing.
  • jsonwebtoken: For session management.
  • prisma: For database persistence.
  • Express.js: Core service framework.
Technical Details

The auth-service was modified to include new routes for mobile identification and biometric challenges. The database schema was updated to support pinHash, trustedDevices, and biometricKeyId.

Test Results
  • Registration: Successfully validates 6-digit numeric PINs.

  • Biometrics: Challenge/Verify simulation returns valid JWTs.
Recommendation for Future Enhancement

Purpose: Strengthen PIN security.
Explanation: Replace BCrypt with Argon2id for PIN hashing.
Reason: Argon2id provides better resistance against GPU-based brute-force attacks.
Issues: Higher CPU/Memory usage during login.
Fixes: Fine-tune cost parameters (time=1, memory=64MB).
Risk: Potential latency on low-end servers.
Mitigation: Horizontal scaling of auth-service.
Dependencies: argon2 library.
