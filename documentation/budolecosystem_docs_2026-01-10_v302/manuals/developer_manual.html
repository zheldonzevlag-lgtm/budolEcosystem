Developer Manual: LAN Network Configuration

1. Technical Implementation Details

The core change involved moving away from static localhost bindings to dynamic network interfaces. This was achieved by updating the server initialization logic in each microservice.

1.1. Dynamic IP Detection

All services now utilize a centralized .env file (resolved relative to the service directory) to fetch the LOCAL_IP variable. This variable is populated by the start-ecosystem.js runner or env-manager.js script based on the host's active network interface.

// Example from services/api-gateway/index.js
require('dotenv').config({ path: path.resolve(__dirname, '../../.env'), override: true });
const LOCAL_IP = process.env.LOCAL_IP || '127.0.0.1';

1.2. 0.0.0.0 Server Binding

To accept connections from any IP on the local network, the app.listen or server.listen calls were updated to explicitly bind to 0.0.0.0 instead of the default (which often defaults to 127.0.0.1 in some environments).

// Standardized listener across all services
app.listen(PORT, '0.0.0.0', () => {
    console.log(`Service running on http://0.0.0.0:${PORT} (LAN-accessible)`);
});

2. Code Logic Explanations

2.1. API Gateway Proxying

The API Gateway uses http-proxy-middleware to route requests. Previously, these targets were hardcoded to localhost. They now use the LOCAL_IP variable to ensure that internal requests are routed correctly even when accessed via the LAN IP.

2.2. mDNS Service Discovery

The Bonjour library is used to advertise the budolpay-gateway service. This allows the mobile application to automatically discover the API Gateway's IP and port without manual configuration.

3. Impact Analysis

  • Network Security: By listening on 0.0.0.0, services are exposed to the local network. Developers should ensure that firewall rules (like Windows Defender) allow traffic on the specified ports (8080, 8000-8009).
  • Backward Compatibility: The LOCAL_IP || '127.0.0.1' fallback ensures that the services still work in isolated local development environments without a network connection.
  • Service Discovery: The mobile app can now connect to the backend by simply being on the same Wi-Fi, greatly improving the development and testing experience for physical devices.

4. Fixes & Issues Resolved

  • Issue: API Gateway inaccessible from physical mobile devices (Connection Refused).
  • Fix: Changed listener to 0.0.0.0 and updated LOCAL_IP configuration.
  • Issue: Hardcoded localhost in SSO_URL preventing remote login.
  • Fix: Replaced with environment-driven LOCAL_IP.
