Developer Manual

Technical Explanation: Phone Standardization

The system now uses a unified normalizePhone utility across the ecosystem to ensure all mobile numbers are stored in E.164 format (+639XXXXXXXXX).

Frontend Implementation

File: AuthForm.jsx

const normalizePhone = (phone) => { if (!phone) return '' const raw = phone.trim().replace(/[^0-9]/g, '') if (raw.startsWith('0')) return '+63' + raw.substring(1) if (raw.startsWith('63') && raw.length === 12) return '+' + raw if (raw.startsWith('9') && raw.length === 10) return '+63' + raw return '+63' + raw.replace(/^63/, '') }

Backend Logic (budolID)

File: index.js

The backend now performs a multi-schema search (budolid and public) to find users even if they were registered in different services.

Database Migration

File: standardize-phones.js

A script was developed to iterate through all users and update their phone numbers to the standardized format using raw SQL queries to bypass model constraints.

Phase 8: Dependencies & Software Choices

The following software and libraries were chosen for this implementation:

  • Tailwind CSS: Used for AuthForm animations. Chosen for its utility-first approach which allows rapid UI development with minimal custom CSS.
  • Prisma Client: Used for database operations. Chosen for its type-safety and multi-schema support which was critical for standardizing data across budolid and public schemas.
  • Jest & React Testing Library: Used for verifying AuthForm logic. Chosen for their robustness and ability to mock complex user interactions like switching login methods.
  • E.164 Standard: Chosen as the phone number format to ensure global compatibility and uniqueness in the database.

Compliance

  • PCI DSS: No sensitive payment data is stored in the phone normalization flow.
  • NPC (Data Privacy Act): Personal identifiable information (phone numbers) is handled securely and only used for authentication.
  • Cybersecurity: Rate limiting is applied to phone check and OTP endpoints to prevent enumeration attacks.
