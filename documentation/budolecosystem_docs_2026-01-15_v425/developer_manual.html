Developer Manual - Release v425

Purpose & Rationale

The primary purpose of this update is to resolve module resolution issues in the Next.js App Router. Next.js uses its own internal bundling system that often conflicts with standard Node.js require() calls when using path aliases (e.g., @/lib/...).

Technical Explanation

In the App Router environment, all server-side routes and components should prefer ES module import statements. Using require() with @/ aliases can lead to MODULE_NOT_FOUND errors because the alias resolution is handled during the compilation step, which dynamic require() may bypass or incorrectly resolve at runtime.

Code Changes

1. Store Verification Route

File: route.js

Replaced dynamic require with top-level import for the realtime library.

import { triggerRealtimeEvent } from '@/lib/realtime'

// ... later in the code ...
await triggerRealtimeEvent(`user-${store.userId}`, 'store-status-updated', {
    storeId: store.id,
    status: verificationStatus,
    isActive: store.isActive
})

2. Shipping Order Updater

File: shippingOrderUpdater.js

Converted internal require for returnsService to a top-level import.

import { receiveReturn } from '@/lib/services/returnsService';

// ... used in syncOrderStatus ...
await receiveReturn({
    returnId: activeReturn.id,
    storeId: order.storeId
});

3. Order Summary Component

File: OrderSummary.jsx

Standardized auth-client imports to avoid dynamic require on the client-side.

import { getUser, getToken, setAuth } from '@/lib/auth-client';
