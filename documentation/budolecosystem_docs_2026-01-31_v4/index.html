2. Implementation Plan: Dual-Identifier Login & Uniqueness

Step 1: Database Normalization & Constraint Reinforcement

Ensure all phone numbers are stored in E.164 format (+639...). Add unique composite indexes if necessary.

Timeframe: 1 Day

Step 2: Budol ID SSO API Enhancement

Modify /auth/login to accept a generic identifier field instead of just email. The backend will query User where email == identifier OR phoneNumber == identifier.

Timeframe: 2 Days

Step 3: Real-time Ecosystem Check (Registry)

Enhance check-email and check-phone to be cross-app aware. Implement atomic check-and-reserve logic.

Timeframe: 1 Day

Step 4: Frontend UI/UX Update (Shap & Pay)

Update AuthForm.jsx and Mobile Login screens to support dual-identifier input. Add visual cues for "Email/Phone".

Timeframe: 2 Days

3. Risk Register

  • Risk: Confusion between international vs local phone formats. Mitigation: Strict server-side normalization.
  • Risk: Data collision during registration. Mitigation: Transactional database locks on registration.

4. Future Recommendations

Integrate biometric hashing (FaceID/Fingerprint) as a 3rd identifier for even higher security, similar to GoTyme's onboarding.
