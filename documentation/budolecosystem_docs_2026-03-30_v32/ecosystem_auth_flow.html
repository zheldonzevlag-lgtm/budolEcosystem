Ecosystem Authentication Flow

Comprehensive findings and analysis of the authentication mechanism bridging the localized schemas (budolpay, budolshap) with the unified Single Sign-On (budolID).

1. Architecture Overview

The Budol Ecosystem operates utilizing a federated approach to identity management. The central identity provider is the budolID SSO service, which acts as the ultimate source of truth for user registration data across all interconnected applications.

┌─────────────────────────────────────────────────────────────────────────────┐
│                       Budol Ecosystem Auth Architecture                     │
├──────────────────┬────────────────────────────────┬─────────────────────────┤
│ Application      │ Target Auth Provider Route     │ Underlying DB Schema    │
├──────────────────┼────────────────────────────────┼─────────────────────────┤
│ budolPayMobile   │ budolpay.vercel.app/api/auth   │ Neon → budolpay schema  │
│ budolShap        │ budolID SSO (Render/Local)     │ Neon → budolshap schema │
│ budolPay Admin   │ budolID SSO (Render/Local)     │ Neon → budolpay schema  │
│ budolID SSO      │ Self (Central Identity)        │ Neon → budolid schema   │
└──────────────────┴────────────────────────────────┴─────────────────────────┘

While the SSO holds the master records, each application (like BudolPay and BudolShap) maintains local, isolated schemas in the Neon PostgreSQL database. This isolation is crucial for multi-tenant scalability and microservice architecture but introduces challenges when an identity from one schema needs to transact in another for the first time.

2. The Mobile Authentication Bridge Discovery

The Missing Link

During the Vercel migration for the Next.js `budolpay-0.1.0` application, Next.js API Routes were established to supplant the legacy Express `auth-service` for mobile login. Initially, these routes only searched the localized `budolpay` schema for users.

However, users entering the ecosystem via `budolShap` or directly registering via `budolID` do not inherently possess a `budolpay` schema user record until they synchronize or explicitly register on the mobile app. This resulted in the "Account not found" error previously encountered.

Auto-Sync & Provisioning

We restored the full Express logic within the Next.js backend, which features a critical cross-check fallback:

  1. Local Check: Route first hunts for an array of potential phone number variations (e.g. `0948...`, `+63948...`, `948...`) within the local `budolpay` schema.
  2. SSO Fallback: If the local look-up yields no results, the Next.js `auth` API routes seamlessly query the centralized `budolID` SSO.
  3. JIT Provisioning: If `budolID` confirms the user exists in the greater ecosystem, the Next.js backend executes a Just-In-Time (JIT) provisioning procedure. It pulls the user metadata and auto-creates the minimal necessary user structures (including a `PHP` Wallet instance) inside the local `budolpay` schema prior to evaluating trust devices or generating OTPs.

Code Location: apps/admin/app/api/auth/login/mobile/identify/route.ts

3. Cross-Application Authentication Mechanisms

BudolShap (E-Commerce Web)

  • Entry: Connects directly to `budolID` via the `/auth/sso/callback` route.
  • Mechanism: Token verification via `budolID/auth/verify`. Uses the verified Email as a unique identifier.
  • Sync: Performs a mirrored JIT provisioning logic inside the `budolshap` schema, granting the `BUYER` or `ADMIN` role locally. Stores a localized `budolshap_token` alongside ecosystem cookies.

BudolPay Admin Dashboard

  • Entry: Like BudolShap, relies firmly on `budolID` for SSO login.
  • Mechanism: Handled securely over the standard OAuth flow. User sessions are persisted contextually to manage merchants, POS metrics, and system ledgers.

BudolPay Mobile App (Android/iOS)

  • Entry: Mobile apps interact with the specialized `/api/auth/setup-pin`, `/verify-pin`, and `/identify` endpoints on the `budolpay` backend directly.
  • Mechanism: Uses PCI-DSS aligned multi-factor strategies. Combines device trust verification (`deviceId` lookup) with real OTP challenges before demanding a 6-digit bcrypt-hashed PIN.

4. Conclusion & Compliance

This implementation of cross-schema verification and JIT auto-provisioning assures unified user mobility within the Budol Ecosystem. A user can create a profile buying items on BudolShap, download the BudolPay Mobile App, input their connected phone number, securely verify their device via OTP, and immediately start transacting without redundantly registering. This aligns with modern federated identity patterns and guarantees robust scale capabilities adhering strictly to BSP standard data partitioning recommendations.
