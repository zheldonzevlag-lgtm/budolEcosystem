⚠️ Risk Register - v524

Risk assessment for Payment Exclusion and UI Changes

ID Risk Description Impact Probability Severity Mitigation Strategy
R-524-001 User Confusion on Hidden Orders: Users might panic if they don't see an order they thought they placed but actually failed payment. High (Customer Support Load) Medium Medium Display a toast notification or a "Recent Failed Orders" link (Future Enhancement). Currently managed via User Manual education.
R-524-002 Grace Period Timing Issues: If server time drifts, the 30-minute grace period might be inaccurate, hiding orders too early or too late. Medium (Data Consistency) Low Low Ensure server NTP synchronization. Logic uses database createdAt which is consistent.
R-524-003 Valid Orders Hidden by Mistake: If a payment gateway callback is delayed > 30 mins, a paid order might be hidden if status is still awaiting_payment. High (Lost Revenue/Trust) Low High Critical: Ensure payment webhooks update status to paid immediately. The logic excludes paid orders from being hidden, so once paid, they reappear.
R-524-004 Name Mismatch in Logistics: Prioritizing User Name over Shipping Name in UI might confuse users about where the item is going if they didn't read the "Shipping To" label. Low (User Error) Low Low Added explicit "Shipping To:" label in the UI to differentiate.
R-524-005 Mobile/Backend API Desync: If the Mobile App is updated but the Wallet Service is not, the PaymentStatusScreen breakdown card will display placeholder data. Medium (UX Inconsistency) Low Medium Phased rollout: Deploy Wallet Service update before releasing Mobile App v1.3.33.
