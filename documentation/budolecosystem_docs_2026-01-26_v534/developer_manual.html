Technical Overview

The primary technical change in this version is the transition from a "push-and-replace" navigation model for payment processing to a "state-callback" model. This ensures the PaymentStatusScreen maintains a stable lifecycle during the asynchronous verification process.

Code References

1. PaymentStatusScreen Architecture Update

Modified payment_status_screen.dart to support an onVerify callback. This allows the screen to initiate the payment process itself while displaying the "Verifying" UI.


// Added onVerify callback to the constructor
final Future<Map<String, dynamic>> Function()? onVerify;

// In initState
if (widget.onVerify != null) {
  _performVerification();
}

// Logic to handle verification with a minimum UX delay
Future<void> _performVerification() async {
  try {
    await Future.delayed(const Duration(milliseconds: 1500)); // UX feedback delay
    final result = await widget.onVerify!();
    if (mounted) {
      updateStatus(PaymentStatus.success, updatedData: result['transaction']);
    }
  } catch (e) {
    if (mounted) {
      updateStatus(PaymentStatus.failed, errorMessage: e.toString());
    }
  }
}

2. QR Scanner Navigation Simplification

Modified qr_scanner_screen.dart to eliminate redundant navigation calls. It now only performs a single Navigator.push to the status screen.


void _handlePayment(BuildContext context) async {
  Navigator.push(
    context,
    MaterialPageRoute(
      builder: (context) => PaymentStatusScreen(
        initialStatus: PaymentStatus.verifying,
        transactionData: paymentData,
        onVerify: () => apiService.processPayment(qrData: paymentData),
      ),
    ),
  );
}

Rationale for Delay

A 1500ms delay was introduced in _performVerification. This is not for technical reasons (the API is faster), but for User Experience (UX). It ensures that the user has enough time to recognize the "Verifying" state, which builds trust that the system is actually performing work before showing the final success/failure result.
