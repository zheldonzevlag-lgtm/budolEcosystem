Developer Manual

Technical implementation details for the Provider-Agnostic Notification Engine.

1. Architecture Overview

The system uses a centralized notification package @budolpay/notifications that fetches dynamic settings from the Prisma database. This allows administrators to switch providers (e.g., from Zerix to iTextMo) without code changes.

Dynamic Settings Sync

Implemented in notifications/index.js, the getSystemSettings function implements a 60-second in-memory cache to optimize performance.

const getSystemSettings = async (forceRefresh = false) => {
    const now = Date.now();
    if (!forceRefresh && settingsCache && (now - lastFetch < CACHE_TTL)) {
        return settingsCache;
    }
    // ... prisma fetch logic ...
};

2. OTP Delivery Flow

OTP delivery is dual-channel (SMS + Email) by default. The logic is handled in the sendOTP function.

Provider Agnostic SMS

The system supports Zerix, iTextMo, Brevo, and Viber. The implementation handles E.164 normalization for Philippine numbers (prefixing with 63).

File Reference: SMS Provider Logic

if (provider === 'ZERIX') {

4. Security & Compliance

  • Rate Limiting: Database-backed rate limiting with dynamic windows. rate-limit.js
  • Data Masking: Sensitive payloads are never logged to the console in production mode.
  • BSP Circular 808: Multi-factor authentication is enforced for all financial transactions.
