Developer Manual: Phone-Primary Identity (v1.3.4)

1. Architecture Overview

This update moves away from hardcoded "User" strings in favor of a dynamic fallback system. The phone number is now treated as a valid display name during the "Quick Registration" phase.

2. Backend Implementation

BudolShap Registration API

File: route.js

Logic change: Removed the template literal that prepended "User " to the phone number.

// Before
if (!name) name = `User ${phoneNumber}`;

// After
if (!name) name = phoneNumber;

BudolID SSO Service

File: index.js

Modified the /auth/register/quick endpoint to use the normalized phone number as the firstName.

// Updated Prisma creation logic
firstName: firstName || normalizedPhone,
lastName: firstName ? (req.body.lastName || '') : '',

3. Frontend UI Fallbacks

Navbar Component

File: Navbar.jsx

The UI now checks for user.name and falls back to user.phoneNumber before defaulting to "Guest".

<span>{user.name || user.phoneNumber || 'Guest'}</span>

4. Data Contracts

The User object now implicitly supports null or empty names, requiring downstream consumers to handle fallbacks via the phone number field.
