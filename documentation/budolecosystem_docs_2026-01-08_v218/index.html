1. Purpose

The purpose of this update is to resolve a data discrepancy where the user galvezjon59@gmail.com was present in the central BudolID service but missing from the local BudolShap application database, leading to registration errors and visibility issues in the Admin panel.

2. Explanation & Reason

In a previous synchronization phase (v216), the user was added to BudolID to enable SSO. however, the local BudolShap database was not updated simultaneously. This caused the "Create Account" form to report "User with this email already exists" (from the BudolID check) while the Admin "Manage Users" list showed the user as missing (reading from the local database).

3. Issues Identified

  • Partial Synchronization: User existed in identity provider (BudolID) but not in application service (BudolShap).
  • Admin Visibility: The Admin panel user list was inconsistent with the central identity service.
  • Registration Blocking: Legitimate attempts to register with existing SSO emails were correctly blocked by BudolID but incorrectly handled in local state.

4. Fixes Implemented

  • Local User Creation: Executed a script budolshap-0.1.0/scripts/create-galvez-locally.mjs to manually provision the user in the local budolshap_1db database.
  • Data Integrity: Ensured consistent IDs, roles (ADMIN), and passwords (hashed) across both services.
  • Verification: Confirmed the user's presence in the local database using verify-user-password.mjs.

5. Technical Details

Local Provisioning Script

budolshap-0.1.0/scripts/create-galvez-locally.mjs:

const user = await prisma.user.upsert({
    where: { email: 'galvezjon59@gmail.com' },
    update: { isAdmin: true, accountType: 'ADMIN' },
    create: {
        id: 'galvez-jon-59-id',
        name: 'Jon Galvez',
        email: 'galvezjon59@gmail.com',
        password: hashedPassword,
        isAdmin: true,
        accountType: 'ADMIN'
    }
});

7. Risks & Mitigation

  • Risk: Manual provisioning can lead to out-of-sync state if not automated.
  • Mitigation: Future registrations should automatically sync to local DB upon successful BudolID creation (already implemented in registration route, this was a legacy data fix).

8. Dependencies

  • BudolID Service
  • BudolShap Service
  • PostgreSQL (budolid and budolshap_1db)

9. Recommendations for Future Enhancement

  • Implement a background "Account Reconciliation" job to periodically verify consistency between BudolID and application services.
  • Centralize user profile data entirely in BudolID to minimize local duplication.
