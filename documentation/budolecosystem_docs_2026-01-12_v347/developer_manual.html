Technical Architecture & Fixes

This version focuses on resolving a critical performance issue on Flutter Web where the UI would hang during initialization.

1. Parallel Initialization in Splash Screen

To prevent the UI thread from blocking, we moved from sequential await calls to Future.wait. This allows the branding delay and the service initialization to occur simultaneously.

File: splash_screen.dart


Future<void> _navigateToNext() async {
  final apiService = context.read<ApiService>();
  
  // Parallelize init and delay
  await Future.wait([
    apiService.init().timeout(const Duration(seconds: 10)),
    Future.delayed(const Duration(seconds: 2)),
  ]);

  if (!mounted) return;
  // ... navigation logic
}

2. Web Compatibility Optimization

The dart:io library is not supported on web and its inclusion can lead to runtime errors or stalls. We removed unnecessary imports and ensured the ApiService is web-ready.

File: api_service.dart

3. Main Thread De-blocking & Route Fix

Previously, main.dart attempted to initialize the ApiService before calling runApp(). On web, this could lead to a blank screen if the initialization stalled. We've moved this responsibility to the SplashScreen. Additionally, we fixed an assertion error regarding redundant route definitions.
