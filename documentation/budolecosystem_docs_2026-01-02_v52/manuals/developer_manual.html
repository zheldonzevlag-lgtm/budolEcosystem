Developer Manual - v52

Release Date: 2026-01-02

Overview

This update focuses on making the payment flow more resilient by handling PayMongo API failures with a local mock fallback and ensuring consistent data synchronization between budolPay and the budolID SSO provider.

Technical Changes

1. Payment Gateway Resiliency

Fix Resolved the PayMongo "Checkout not found" 404 error that occurred when the system fell back to production URLs in a local environment.

  • File: budolpay-0.1.0/services/payment-gateway-service/index.js
  • Implementation: Replaced production fallback URLs with localShapUrl (http://localhost:3001). If the PayMongo API call fails in development, the system now redirects to a local mock QR payment page.
  • Success/Cancel URLs: Standardized success and cancel redirect URLs to use the local budolShap instance.

2. Payment Method Normalization

Fix Resolved "invalid payment_method" errors caused by inconsistent string casing between the frontend and the gateway.

  • File: budolshap-0.1.0/components/OrderSummary.jsx
  • Logic: Added .toLowerCase().replace('_', '') to the payment method selection logic, ensuring that "BUDOL_PAY" and "budolpay" are both treated as budolpay.

3. budolPay Admin SSO Bi-directional Sync

Feature Expanded the SSO synchronization logic to include the budolPay Admin service, preventing name reversion after profile edits.

  • New Client: Created budolpay-0.1.0/apps/admin/lib/budolIdClient.ts to communicate with the budolID PATCH /users/:userId/profile endpoint.
  • API Update: Modified budolpay-0.1.0/apps/admin/app/api/users/route.ts to include an UPDATE_PROFILE action that pushes local name changes to budolID.
  • SSO User Linkage: Verified that the ssoUserId is correctly stored in the kycData field of the budolPay User model during the OAuth callback.

Code Example: Mock Fallback Logic

// payment-gateway-service/index.js
if (isDevelopment) {
  console.log('[BudolPay Gateway] Using local mock fallback for PayMongo');
  providerResponse = {
    checkout_url: `${localShapUrl}/payment/qr?reference=${transaction.referenceId}&amount=${amount}&provider=paymongo&mock=true`,
    id: `pm_intent_mock_${transaction.id}`
  };
}

Impact Analysis

Developers can now test the full payment flow locally without relying on external API availability. Admin profile management is now consistent across all major ecosystem services (budolShap and budolPay), ensuring a unified user identity.

Recommendations for Future Enhancement

  • Centralize the budolIdClient into a shared package to reduce code duplication across services.
  • Implement a unified payment configuration object to manage URLs and provider settings across the ecosystem.
