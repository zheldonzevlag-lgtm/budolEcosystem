Technical details for Logout Session Guarding

Index Developer Manual System Admin Manual User Manual

Problem Description

In version v351 and earlier, a race condition existed during the logout process. When a user initiated a logout from the SettingsScreen, the HomeScreen (which remained in the navigation stack until pushNamedAndRemoveUntil completed) would occasionally trigger a data refresh. Since the ApiService had already cleared the token, the backend request would fail, triggering an error snackbar on the LoginScreen.

Technical Analysis

The issue was traced to the BottomNavigationBar handler in HomeScreen:


// home_screen.dart (Old Code)
} else if (index == 4) {
    await Navigator.pushNamed(context, Routes.settings);
    _fetchData(); // This would execute AFTER returning from settings, even if logged out!
}

Solution: Authentication Guarding

We implemented a defensive check within the data-fetching methods to ensure that no API calls are made if the user is not authenticated.

Modified Code: HomeScreen

File: home_screen.dart


Future<void> _fetchBalance({bool silent = false}) async {
  // ...
  final apiService = Provider.of<ApiService>(context, listen: false);
  
  // Do not fetch if not authenticated
  if (!apiService.isAuthenticated) {
    if (kDebugMode) print('HomeScreen: Skipping _fetchBalance - not authenticated');
    return;
  }
  // ...
}

Modified Code: WalletScreen

File: wallet_screen.dart


Future<void> _fetchBalance({bool silent = false}) async {
  final apiService = Provider.of<ApiService>(context, listen: false);
  
  // Do not fetch if not authenticated
  if (!apiService.isAuthenticated) {
    return;
  }
  // ...
}

Core Service Update

The app version has been incremented to 1.2.58 in pubspec.yaml and api_service.dart to reflect these stability improvements.

Dependencies

  • Provider: Used for authentication state management.
  • ApiService: Provides the isAuthenticated property.
