Consolidated Recommendations: KYC & Profile Security

Future enhancements, risks, and technical roadmap for the BudolPay identity ecosystem.

1. Identity & Compliance Enhancements

High Priority Automated ID Cross-Verification

Purpose: To automate the approval of high-risk requests like phone number changes.

Explanation: Implement a facial recognition check between the ID uploaded for the phone change and the original KYC ID image.

Reason: Manual review is a bottleneck. Automation reduces turnaround time from 48 hours to seconds while maintaining security.

Risks: False negatives due to aging or poor lighting. Mitigation: Manual fallback for failed automated matches.

Dependencies: Advanced ML Kit face embedding model, existing KYC ID database access.

Medium Priority Real-Time Address Validation

Purpose: To replace the mock "Pin Address" feature with real geographic data.

Explanation: Integrate Google Maps API or OpenStreetMap to fetch precise coordinates and formatted address strings.

Reason: Current "pinning" is simulated. Real data prevents address fraud and enables proximity-based services (e.g., nearest Cash-In point).

Risks: API costs for high volume. Mitigation: Cache common address patterns and use free-tier limits wisely.

Dependencies: Maps API Key, Location Permissions in Flutter.

2. Infrastructure & Security Roadmap

Technical S3/Object Storage Migration

Purpose: To move identity documents out of the main database.

Explanation: Store images in AWS S3 or Minio and save only the URL/Reference in the Prisma VerificationDocument model.

Reason: Storing binary data in BYTEA columns bloats database backups and slows down queries.

Issues: Current migration scripts need to handle data transfer from DB to S3.

Fixes: Use multer-s3 for direct uploads and a background worker for legacy data migration.

Medium Priority Dual-Factor Phone Change

Purpose: To add an extra layer of security before the manual review phase.

Explanation: Send an OTP to the NEW phone number to verify its existence before the user even uploads their ID.

Reason: Prevents users from requesting changes to numbers they don't own or have typos in.

Dependencies: SMS Gateway integration (e.g., Twilio or Infobip).

3. Summary of Technical Details (v170)

  • Current State: Request-based phone change with ID proof, calendar-based DOB entry, simulated address pinning, and admin document viewer.
  • Immediate Issues: Storage bloat due to image blobs, manual review latency for phone changes.
  • Mitigation Strategy: Image compression (implemented), future S3 migration (planned), automated face matching (recommended).
