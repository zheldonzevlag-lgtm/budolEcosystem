Risk Mitigation Strategies - v491

Proactive Measures for System Stability

M-491-01: Realtime Provider Redundancy

Risk: Infrastructure failure or quota limits.

Strategy: The system implements a fallback to POLLING mode if Pusher or Socket.io is unavailable. Frontend hooks (useRealtimeBuyerOrders) detect the configuration and adjust automatically.

M-491-02: Webhook Integrity Verification

Risk: Malicious payload injection.

Strategy: All payment webhooks (PayMongo, BudolPay) and shipping webhooks (Lalamove) implement provider-specific signature verification using HMAC-SHA256 with secret keys stored in secure environment variables.

M-491-03: Race Condition Prevention

Risk: UI desync during simultaneous updates.

Strategy: SWR mutate() is used globally. Realtime events trigger an immediate cache invalidation and re-fetch, ensuring that the latest database state is always shown regardless of the event source.

M-491-04: Channel Authorization

Risk: Unauthorized data eavesdropping.

Strategy: Channels are prefixed with user- or store- and include unique IDs. Future enhancements will implement Pusher Private Channels with JWT-based authentication for tighter security.

M-491-05: Redundant Notification Suppression

Risk: Notification fatigue and UI noise.

Strategy: Implemented a two-tier suppression system: 1) Backend status verification in service layers to skip redundant database writes and event triggers, and 2) Frontend event fingerprinting with a 3-second TTL to deduplicate concurrent broadcasts from multiple realtime providers.
