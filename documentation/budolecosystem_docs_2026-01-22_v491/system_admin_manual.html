Overview

The v491 update enhances the system's ability to handle high-frequency status updates through automated realtime broadcasting. Admins must ensure that the Realtime Provider is correctly configured to maintain these features.

1. Realtime Provider Configuration

Admins can choose between POLLING, PUSHER, and SOCKET_IO in the Admin Settings panel.

Important: For "To Ship" tab auto-switching and instant status updates, PUSHER or SOCKET_IO is highly recommended. POLLING will have a 5-10 second latency.

2. Webhook Monitoring

The following endpoints are critical for the automated status transition flow:

  • /api/webhooks/paymongo - Handles credit card and e-wallet payments.
  • /api/webhooks/budolpay - Handles native BudolPay transactions.
  • /api/webhooks/lalamove - Handles courier status updates (Picked Up, Delivered).

3. Database Audit

Admins should monitor the SystemSettings table for the realtimeProvider value. Ensure that pusherAppId, pusherKey, pusherSecret, and pusherCluster are valid if Pusher is selected.

4. Troubleshooting Realtime Latency

  • Verify the Pusher dashboard for event throughput.
  • Check the server logs for [Realtime] Subscribing to user-xxx entries.
  • Ensure that the SOCKET_URL is accessible if using Socket.io.

5. Notification Efficiency Policy

Starting with v491, the system enforces a "Silent Update" policy for redundant operations. Notifications are suppressed if the incoming status matches the current database state. This reduces unnecessary UI noise and optimizes browser resource usage.
