Technical Architecture

The v491 release implements a robust realtime notification system using a channel-based subscription model. It ensures that payment events and shipping status changes are broadcasted immediately to relevant parties (Buyer and Store).

1. Realtime Provider Expansion

The RealtimeProvider.jsx was updated to subscribe to user-specific channels (user-{userId}) and store-specific channels (store-{storeId}).

// components/RealtimeProvider.jsx
if (user?.id) {
    const userChannelName = `user-${user.id}`
    const userChannel = pusherInstance.subscribe(userChannelName)
    userChannel.bind('order-updated', (data) => handleGlobalEvent('order-updated', data))
    userChannel.bind('order-created', (data) => handleGlobalEvent('order-created', data))
}

2. Webhook Integration

Payment and shipping webhooks now trigger realtime events upon successful status transitions.

  • PayMongo Webhook
  • BudolPay Webhook
  • Lalamove Webhook
// Example Webhook Trigger
await triggerRealtimeEvent(`store-${order.storeId}`, 'order-updated', {
    orderId: order.id,
    status: order.status
});

3. Frontend Automated Navigation

In the Buyer Orders Page, the handleQRSuccess handler now explicitly switches the active tab to TO_SHIP to provide immediate visual confirmation of the status change.

4. Service Layer Updates

The shippingService.js now includes realtime triggers during the bookShipping operation, ensuring the store dashboard reflects "To Ship (Booked)" status immediately after booking.

5. Realtime Deduplication (Frontend)

To prevent multiple notifications for the same event (e.g., when Pusher and Socket.IO both broadcast the same update), a two-tier deduplication mechanism was implemented in RealtimeProvider.jsx.

Tier 1: Rapid-Fire Suppression

Uses a timestamp-bucketed fingerprint to ignore identical events received within a 5-10 second window. This handles race conditions from multiple channels.

const timestampBucket = Math.floor(Date.now() / 5000)
const fingerprint = `${event}:${orderId}:${status}:${timestampBucket}`
if (seenEvents.current.has(fingerprint)) return;

Tier 2: Persistent Status Suppression

Tracks the last displayed status for each order. If an event is received for a status that has already been notified (e.g., repeated location updates from a courier while the status is still "IN_TRANSIT"), the notification toast is skipped, but the data is still refreshed in the background.

const isRepeatStatus = lastSeenStatus.current.get(statusKey) === status;
if (isRepeatStatus) {
    // Refresh data silently using Cache Patching
    patchOrderCache(data);
    return; // No toast
}

6. Backend Status Change Verification

Service layers (ordersService.js, shippingService.js, returnsService.js) now verify if a status update actually changes the current state before performing database writes or triggering realtime events.

// Example from returnsService.js
const currentReturnStatus = returnRequest.status;
const nextReturnStatus = 'REFUNDED';
const hasReturnStatusChanged = currentReturnStatus !== nextReturnStatus;

if (hasReturnStatusChanged) {
    await triggerRealtimeEvent(`store-${storeId}`, 'return-updated', { ... });
}

7. Targeted Cache Patching (Flicker Prevention)

To eliminate UI flickering (layout shifts and loading skeletons) during realtime updates, a targeted cache-patching strategy was implemented in RealtimeProvider.jsx. Instead of a global router.refresh() or mutate() with no arguments, we specifically update the affected order in the SWR cache.

const patchOrderCache = (data) => {
    const orderId = data.orderId || data.id
    const newStatus = data.status
    if (!orderId || !newStatus) return

    mutate(
        (key) => typeof key === 'string' && (key.startsWith('/api/orders') || key.startsWith('/api/dashboard/seller/orders')),
        (currentData) => {
            if (!currentData || !currentData.orders) return currentData
            const updatedOrders = currentData.orders.map(order => 
                order.id === orderId ? { ...order, status: newStatus } : order
            )
            return { ...currentData, orders: updatedOrders }
        },
        { revalidate: true } // Background revalidation only
    )
}

8. Centralized Realtime Management

Responsibility for Pusher/Socket.io listeners has been moved from individual hooks (useRealtimeBuyerOrders.js and useRealtimeOrders.js) to the global RealtimeProvider.jsx. This prevents duplicate connections and ensures a unified update strategy across the entire ecosystem.

9. Cloud Cost & Bandwidth Optimization

To ensure the system remains cost-effective when hosted on cloud providers like AWS or Vercel, several optimization strategies were implemented in RealtimeProvider.jsx:

A. Smart Partial Revalidation

The patchOrderCache function now defaults to revalidate: false for status updates. Since the realtime payload already contains the updated status, we update the local SWR cache immediately without triggering a follow-up API call to the server. This significantly reduces API requests and bandwidth usage.

// Optimization: No API call if payload is sufficient
mutate(key, updateFn, { revalidate: false });

B. Tab Visibility Guard

The system monitors the browser's visibility state. When a tab is hidden (in the background), the polling interval is automatically increased from 5 seconds to 30 seconds. This prevents unnecessary resource consumption for inactive users.

const handleVisibilityChange = () => {
    const isHidden = document.visibilityState === 'hidden'
    setSwrOptions(prev => ({ 
        ...prev, 
        refreshInterval: isHidden ? 30000 : 5000 
    }))
}

C. Minimal Payload Design

All realtime events broadcasted from the backend (ordersService.js) use a minimal payload structure (e.g., only orderId, status, and isPaid). This minimizes the bandwidth consumed by Pusher/Socket.IO messages.

Dependencies & Software Rationale

The following technical choices were made to ensure a robust, scalable, and compliant realtime ecosystem:

  • Pusher (Managed WebSockets): Chosen for its zero-infrastructure overhead and high reliability in broadcasting events across disparate geographic locations. Essential for BSP-compliant payment transparency.
  • Socket.IO (Self-Hosted Fallback): Provides a secondary realtime layer for environments where managed services might be restricted or for local development consistency.
  • SWR (Client-Side State Management): Enables "optimistic" UI updates and efficient cache invalidation. When a realtime event is received, SWR triggers a revalidation of the specific order data, ensuring the UI is always accurate without expensive full-page refreshes.
  • Prisma ORM: Ensures data integrity during status transitions. Using Prisma's transaction capabilities (where applicable) prevents race conditions between concurrent status updates.
  • Webhook Orchestration Layer: Centralizing status updates in ordersService.js ensures that whether an update comes from a manual admin action, a PayMongo webhook, or a Lalamove event, the logic remains consistent and auditable.
