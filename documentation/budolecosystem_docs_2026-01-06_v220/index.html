budolEcosystem v220

Admin API Gateway & Settings Resolution

← Previous (v219) Main Index Next (v220) → Next →

Purpose

This release resolves the 404 error observed in the Gateway API logs for the /system/settings endpoint. It ensures that the Admin console can correctly fetch system configurations even when the database is in an initial or empty state.

Technical Specification & Fixes

1. Gateway API Route Fix

Issue: The /api/system/settings endpoint in the Admin app was returning a 404 or failing to provide a consistent response structure when certain settings were missing from the database.

Resolution: Updated budolpay-0.1.0/apps/admin/app/api/system/settings/route.ts to include fallback values and a status indicator.

// Updated Response Structure
return NextResponse.json({
  mapProvider: settingsMap['MAP_PROVIDER'] || 'OSM',
  enabledMapProviders: JSON.parse(settingsMap['ENABLED_MAP_PROVIDERS'] || '["OSM"]'),
  // ... other keys
  status: 'operational',
  timestamp: new Date().toISOString()
});

2. Path Verification

Discovery: Verified that the Admin app uses /settings/environment for its main configuration UI, while the API is located at /api/system/settings. The 404 in the logs was likely due to a mismatch between the expected internal service path and the actual Next.js route structure.

Validation & Test Results

  • Endpoint Test: Successfully ran curl http://localhost:3000/api/system/settings and received a 200 OK with the expected JSON payload.
  • Database Resilience: Confirmed the API handles empty systemSetting tables by providing safe defaults (e.g., "OSM" for map provider).
  • Service Status: Gateway API now reports status: "operational".

Recommendation

Monitor the Gateway API logs for any remaining 404s. If the mobile app requires these settings, ensure it points to http://[LOCAL_IP]:3000/api/system/settings.
