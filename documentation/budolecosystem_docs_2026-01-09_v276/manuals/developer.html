Developer Manual Updates - v276

Purpose

To resolve the "Failed to fetch" TypeError in the product upload flow and provide better diagnostic tools for developers.

Technical Details

1. Client-Side Enhancement (page.jsx)

The fetch call was wrapped in a dedicated try-catch block to distinguish between network errors and server errors. We also added logic to extract detailed error messages from the server response even if it's not valid JSON.

// Enhanced error handling in page.jsx
try {
    uploadResponse = await fetch('/api/upload', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ image: base64ToUpload })
    })
} catch (fetchError) {
    throw new Error(`Network error: Could not reach the upload server. ${fetchError.message}`)
}

2. Server-Side Route Handler (route.js)

Added maxDuration export to increase the execution limit on Vercel and similar platforms. Integrated console.log for tracing payload size and Cloudinary response status.

export const maxDuration = 60; // 60s timeout

export async function POST(request) {
    console.log('Received upload request');
    // ... logging image length and status ...
}

Risk & Mitigation

  • Risk: Large payloads (10MB+) might still exceed browser memory or client-side network limits.
  • Mitigation: Implemented client-side compression using canvas.toDataURL('image/webp', 0.8) before upload.

Recommendations for Future Enhancement

  • Implement chunked uploads for extremely large files to avoid body size limits entirely.
  • Add a progress bar to the UI using XHR or fetch with readable streams (if supported).
  • Integrate Sentry logging for client-side fetch failures to track occurrences in the wild.
