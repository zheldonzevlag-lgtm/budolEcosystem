Timezone Standardization Architecture

The Budol Ecosystem has transitioned from "Fake UTC" (hardcoded +8h offsets in backend storage) to "True UTC" with "Manila Display Utility". All backend timestamps are now stored as standard UTC, and the Asia/Manila timezone is applied only at the presentation layer.

1. Central Utility: dateUtils.js

The core logic resides in dateUtils.js. It handles the Intl.DateTimeFormat configuration and resolves conflicts between dateStyle/timeStyle and individual components. 

export function formatManilaTime(date, options = {}) {
    if (!date) return 'N/A';
    const d = new Date(date);
    const defaultOptions = {
        timeZone: 'Asia/Manila',
        year: 'numeric',
        month: 'short',
        day: '2-digit',
        hour: '2-digit',
        minute: '2-digit',
        hour12: true
    };
    const mergedOptions = { ...defaultOptions, ...options };
    if (options.dateStyle || options.timeStyle) {
        delete mergedOptions.year;
        delete mergedOptions.month;
        delete mergedOptions.day;
        delete mergedOptions.hour;
        delete mergedOptions.minute;
    }
    return d.toLocaleString('en-US', mergedOptions);
}

2. Backend Implementation (Lalamove Webhook)

Backend services now use getNowUTC() (which returns new Date()) instead of manual offset calculations. See lalamove/webhook/route.js. 

// OLD (Hardcoded +8h)
// updatedAt: new Date(new Date().getTime() + (8 * 60 * 60 * 1000)).toISOString()

// NEW (True UTC)
updatedAt: getNowUTC().toISOString()

3. Admin UI (Transactions Page)

The Transactions Page has been updated to use formatManilaTime for both the data table and the CSV export feature.

Regulatory Compliance

  • BSP Circular 808: All transaction logs are timestamped with accurate UTC time for audit trail integrity.
  • PCI DSS 10.2.2: System clock synchronization across services ensures non-repudiation of security logs.
  • NPC DPA 2012: Access logs for sensitive user data now include verifiable standardized timestamps.
