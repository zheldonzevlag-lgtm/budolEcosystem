Developer Manual: Secure Reference ID Generation

Purpose

The purpose of this update is to replace the legacy, insecure Math.random() based reference ID generation with a cryptographically secure, unique, and standardized format (JON-YYYYMMDDHHMMSS-RANDOM) to meet security and compliance requirements.

Technical Details

Reference ID Structure

The new referenceId follows this pattern: JON-{TIMESTAMP}-{RANDOM}

  • Prefix: JON- (Standardized identifier for the ecosystem).
  • Timestamp: 14-digit UTC timestamp (YYYYMMDDHHMMSS) for temporal uniqueness and auditability.
  • Random Segment: 8-character uppercase hexadecimal string generated from 4 cryptographically secure random bytes.

Implementation

The generation logic is implemented in payment-gateway-service/index.js using the Node.js crypto module.

const crypto = require('crypto');

function generateSecureReferenceId() {
  const now = new Date();
  const timestamp = now.toISOString()
    .replace(/[-T:.Z]/g, '')
    .slice(0, 14);
  
  const randomBytes = crypto.randomBytes(4).toString('hex').toUpperCase();
  return `JON-${timestamp}-${randomBytes}`;
}

Impact Analysis

  • Database: The Transaction model in Prisma already has a @unique constraint on referenceId, which is now better supported by the new high-entropy IDs.
  • Downstream Systems: budolPay-adapter and paymongo/status routes are compatible as they rely on the string value of the referenceId.

Recommendations for Future Enhancement

  • Centralized ID Service: Consider moving ID generation to a dedicated microservice if more entities require secure unique identifiers.
  • Check Digits: Implement a Luhn-like algorithm or checksum at the end of the ID to allow for offline validation of manual entry.
  • Audit Logs: Ensure every generated ID is logged with its associated request context for full traceability.
