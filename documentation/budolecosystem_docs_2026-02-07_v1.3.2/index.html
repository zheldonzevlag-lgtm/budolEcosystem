Release v1.3.2: Browser Validation Fix & Smart UX Refinement

Date: 2026-02-07

Released

This version addresses a critical UI conflict where browser-level email validation was interfering with mobile number logins. We've also refined the "Smart Identifier Detection" to ensure a seamless Lazada-style transition between login methods.

The Issue: Browser Validation Conflict

During mobile number login, browsers were triggering an error: "Please include an '@' in the email address. '9484099400' is missing an '@'."

Rationale:

  • Hidden Field Validation: Even when visually hidden, type="email" fields with required attributes can trigger browser validation if they are not explicitly disabled.
  • Cross-Pollination: If a user typed a phone number into the email field before switching tabs, the invalid value remained in the hidden email input, causing the validation failure on form submission.

Technical Solution

1. Input Disabling (The "Nuclear" Option)

We implemented dynamic disabled attributes on all inactive authentication fields. Browsers ignore disabled fields during form validation.

<input name="email" type="email" required={isLogin && loginMethod === 'email'} disabled={isLogin && loginMethod !== 'email'} ... />

2. Automated Field Transition

Refined the handleChange logic in AuthForm.jsx to automatically move values and clear invalid fields when switching types.

if (name === 'email' && isLogin) { if (value.includes('@')) { setLoginMethod('email'); } else if (/^\d{3,}/.test(value)) { setLoginMethod('phone'); setFormData(prev => ({ ...prev, phoneNumber: value.replace(/[^0-9]/g, '').substring(0, 10), email: '' // Prevent stray data })); } }

Compliance Verification

  • PCI DSS: Data integrity maintained; no sensitive info stored in stray fields.
  • UX Standard: Follows Lazada's "One-Field" high-conversion philosophy.
  • Accessibility: Maintained tabIndex management for keyboard navigation.
