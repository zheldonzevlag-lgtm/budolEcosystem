Developer Manual - Version v255

Purpose

The primary goal of this update was to verify and fix the real-time provider switching logic between Pusher and Socket.io. Previously, the app required a restart or manual trigger to switch providers when system settings were updated via the backend.

Technical Implementation

1. Reactive Service Architecture

Modified RealtimeService to extend ChangeNotifier and listen to ApiService updates. This ensures that whenever fetchSystemSettings() is called and settings change, the real-time engine reacts immediately.

// lib/services/realtime_service.dart
RealtimeService(this._apiService, this._socketService, this._pusherService) {
  _apiService.addListener(_onApiServiceChanged);
}

void _onApiServiceChanged() {
  final settings = _apiService.systemSettings;
  final method = settings?['realtime']?['method'] ?? 'SWR';
  final userId = _apiService.user?['id']?.toString();

  if (method != _currentMethod || userId != _currentUserId) {
    init(); // Re-initializes with new settings
  }
}

2. Clean Disconnection Logic

To prevent memory leaks and duplicate event listeners, both SocketService and PusherService now implement a explicit disconnect() method that is called by RealtimeService before a new provider is initialized.

  • Socket.io: Disconnects the socket and disposes of the instance.
  • Pusher: Unsubscribes from user channels and calls the native disconnect method.

Test Results

  • Verification: Simulated switching from SOCKETIO to PUSHER via system settings update.
  • Outcome: RealtimeService correctly detected the change, called _socketService.disconnect(), and successfully initialized _pusherService.
  • Resource Usage: No orphaned WebSocket connections were detected in the background.

Impact Analysis

This change improves system resilience. If one real-time provider goes down, administrators can switch to the fallback provider via the dashboard, and all active mobile clients will transition automatically without user intervention.
