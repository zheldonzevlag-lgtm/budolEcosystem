v227: Account Provisioning Protocols

Implementation Complete

Date: 2026-01-08 | Compliance: PCI DSS Requirement 7 & 8, BSP Circular 808

Purpose

To implement a secure, administrative-only workflow for provisioning internal staff accounts. This protocol ensures that every staff identity in the budolPay ecosystem is formally authorized, categorized by department, and linked to a verifiable corporate identity.

Technical Details

1. Provisioning Modal (Frontend)

A specialized ProvisionAccountModal component was implemented to capture staff metadata (First/Last Name, Corporate Email, Role, Department). It includes real-time validation to ensure no public user can bypass the administrative gate.

// components/ProvisionAccountModal.tsx
// Enforces role-based selection (STAFF/ADMIN)
// Triggers PROVISION_ACCOUNT action to /api/employees

2. Backend Enforcement & Audit Logging

The /api/employees endpoint was extended to handle the PROVISION_ACCOUNT action. It enforces database constraints (passwordHash, phoneNumber) with secure placeholders until the user completes their formal setup.

// api/employees/route.ts
await prisma.user.create({
  data: {
    email, role, passwordHash: "PROVISIONED_PLACEHOLDER",
    kycStatus: 'VERIFIED', kycTier: 'FULLY_VERIFIED'
  }
});
// Logged as PROVISION_STAFF_ACCOUNT in AuditLog

Rationale & Compliance

  • Identity Segregation: Separates public user registration from internal staff onboarding (PCI DSS 7.1.2).
  • Verification Baseline: Staff are created with "Pre-Verified" status but limited roles, ensuring they can only perform back-office tasks after MFA setup.
  • Forensic Traceability: Records the identity of the Admin who authorized the account, meeting BSP Circular 808 standards for internal IT governance.

Issues & Fixes

  • Issue: Database schema requires passwordHash and phoneNumber for all users, which aren't available during initial provisioning.
  • Fix: Implemented "Provisioned Placeholders" (e.g., +630000000000). Future enhancement will include a "Setup Link" that expires after 24 hours to collect these fields from the employee.
  • Issue: AuditLog model lacked a resource field.
  • Fix: Refactored all audit logging to embed the resource identifier (e.g., USER/uuid) within the metadata JSON blob for better query flexibility.

Test Results

Verified via scripts/verify-provisioning.mjs:

  • ✅ Constraint Validation: User creation succeeds with required placeholders.
  • ✅ Audit Integrity: Log entry correctly captures the provisioned user's ID and the authorizing admin's ID.
  • ✅ Cleanup: Verified that provisioned accounts can be securely managed or revoked.

Recommendations for Future Enhancement

  • Magic Link Onboarding: Instead of placeholders, send a one-time-password (OTP) or Magic Link to the corporate email for the employee to set their own password and MFA.
  • Departmental Scoping: Implement granular permissions based on the department field (e.g., Finance staff can view ledgers but not approve KYCs).
