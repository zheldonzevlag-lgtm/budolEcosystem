Developer Manual: Robust Reference Uniqueness

The Problem: Collision Risk

In the previous implementation (v99), the referenceId was generated using JON-${Date.now()}. While this provides a unique ID per millisecond, it poses a risk in high-concurrency environments:

  • Multiple Requests: Two or more payment requests arriving within the same millisecond would generate the same referenceId.
  • Distributed Systems: Clock drift between servers could increase the likelihood of collisions in a load-balanced setup.

The Solution: Randomized Suffix

We have enhanced the generator to include a 4-character random alphanumeric suffix. This increases the entropy significantly.

// Old Implementation (v99) referenceId: `JON-${Date.now()}` // New Implementation (v100) referenceId: `JON-${Date.now()}-${Math.random().toString(36).substring(2, 6).toUpperCase()}`

The Math.random().toString(36) generates a random base-36 string, from which we take 4 characters. This adds 1,679,616 (36^4) possible combinations per millisecond.

Impact Analysis

Safety Guaranteed: The probability of a collision within the same millisecond is now roughly 1 in 1.6 million, making it statistically impossible for our current scale.

System Impacts:

  • Database: The referenceId column is indexed. Longer IDs slightly increase index size but improve search reliability.
  • Webhooks: The webhook logic in payment-gateway-service uses exact matches, so it remains fully compatible with the new format.
  • Simulations: Scripts like simulate-mobile-scan.js will continue to work as they fetch the referenceId from the database rather than predicting it.

Future Recommendations

For even higher scale, we recommend migrating to UUID v4 or NanoID, which are industry standards for unique identifier generation across distributed systems.
