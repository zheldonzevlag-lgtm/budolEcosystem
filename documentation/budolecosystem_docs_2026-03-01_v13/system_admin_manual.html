System Admin Manual: AWS Hosting

Infrastructure Management

1. Billing & Free Tier Monitoring

Administrators must check the AWS Billing Dashboard weekly. Ensure the "Estimated monthly charge" remains at $0.00. Set up billing alarms for any amount over $0.01.

2. RDS Instance Management

The db.t3.micro instance allows for 750 free hours per month. Only one instance should be running to stay within this limit. Automatic backups should be enabled with a 7-day retention period.

4. Creating Multi-AZ Subnets

Based on your budol-vpc configuration (VPC ID: vpc-0f65f8cafca200de6), follow these steps:

  1. In the VPC Dashboard, click Subnets in the left sidebar.
  2. Click the Create subnet button.
  3. VPC ID: Select vpc-0f65f8cafca200de6 (budol-vpc).
  4. Subnet Settings: Click "Add new subnet" 4 times and enter exactly:
    Subnet Name Availability Zone IPv4 CIDR
    Public-A ap-southeast-1a 10.0.1.0/24
    Public-B ap-southeast-1b 10.0.2.0/24
    Private-A ap-southeast-1a 10.0.10.0/24
    Private-B ap-southeast-1b 10.0.11.0/24
  5. Click Create subnet at the bottom.
Crucial Step: After creation, select Public-A and Public-B, go to Actions -> Edit subnet settings, and check "Enable auto-assign public IPv4 address".

External DNS Management (Hybrid Strategy)

1. Manual DuckDNS Portal Setup (Primary)

Register the first 5 subdomains at duckdns.org:

  • budolshap, budolid, budolpay, budolws, budoladmin

2. Manual FreeDNS (Afraid.org) Setup (Secondary)

Since DuckDNS has a 5-domain limit, use FreeDNS for the Accounting service:

  1. Sign up for a free account at FreeDNS.
  2. Click Subdomains -> Add.
  3. Type budolaccounting and choose a domain (e.g., mooo.com).
  4. Point the Destination to your AWS Public IP.

4. IP Auto-Update Script (Automated)

The scripts/setup_websocket_ec2.sh script automatically creates an update utility and schedules it. To update manually or check status:

# Run manually
~/budol-ws/update_duckdns.sh

# Check crontab schedule
crontab -l

Ensure the TOKEN in ~/budol-ws/update_duckdns.sh matches your DuckDNS portal token.

2. Connecting to the Instance (No SSH Key Required)

Access is managed exclusively via AWS Systems Manager Session Manager. No SSH key pairs are configured.

# Connect interactively
aws ssm start-session --target i-00e50e8bb90c9397a

# Or run a remote command
aws ssm send-command --instance-ids i-00e50e8bb90c9397a \ --document-name AWS-RunShellScript --parameters 'commands=["YOUR_CMD"]'

3. Service Status & Logs

The WebSocket server runs as Docker container budol-ws-websocket-server-1.

cd /home/ubuntu/budol-ws

# View running containers
docker compose ps

# View real-time logs
docker compose logs -f websocket-server

# Restart after code changes
docker compose restart websocket-server

# Full rebuild
docker compose up -d --build

4. Critical Trigger Endpoint

Internal services trigger events via http://ws.budol.internal:4000/trigger. If notifications fail, verify reachability from APP EC2 instances:

curl -X POST http://ws.budol.internal:4000/trigger \ -H 'Content-Type: application/json' \ -d '{"channel":"test","event":"ping","data":"pong"}'

# Expected: {"success":true}

6. SSL Certificate Management (Let's Encrypt)

SSL is managed via Certbot. DuckDNS domains use the dns-duckdns plugin (DNS-01 challenge), while FreeDNS domains use the HTTP-01 challenge via Nginx.

A. DuckDNS SSL (DNS-01)

Use for budolws, budolid, budolpay.

# Obtain certificate for a new DuckDNS domain
sudo certbot certonly --non-interactive --agree-tos -m admin@budolecosystem.com \
--preferred-challenges dns --authenticator dns-duckdns \
--dns-duckdns-token-file /etc/letsencrypt/duckdns.token \
-d budolpay.duckdns.org

B. FreeDNS SSL (HTTP-01)

Use for budolaccounting.mooo.com. Requires Nginx to be running.

# Obtain certificate via HTTP-01
sudo certbot certonly --nginx -d budolaccounting.mooo.com

C. General Maintenance

# List all certificates
sudo certbot certificates

# Test renewal process (Dry Run)
sudo certbot renew --dry-run

Config Path: /etc/letsencrypt/live/

Microservice Containerization (ECS Fargate)

1. Deployment Strategy

Backend microservices are containerized using a unified multi-stage Docker build process. This ensures consistent dependencies and smaller image sizes.

# Build all services from the root
docker build -f Dockerfile.backend --target budol-id -t budolid:latest .
docker build -f Dockerfile.backend --target budol-accounting -t budolaccounting:latest .
docker build -f Dockerfile.backend --target budol-pay -t budolpay:latest .

2. ECS Cluster (budol-cluster)

Services run on AWS Fargate with the following resource limits to stay within the Free Tier or minimize costs:

  • CPU: 0.25 vCPU
  • Memory: 0.5 GB
  • Network: AWSVPC mode

Frontend Operations (AWS Amplify)

1. Mobile Web Deployment (budolPayMobile)

The Flutter Web version is deployed via AWS Amplify using the following custom build specification:

Buildspec: amplify-budolpaymobile-buildspec.yml

To trigger a manual redeploy:

  1. Commit changes to the main branch.
  2. Amplify will detect the change and run the preBuild and build phases.
  3. Artifacts in budolPayMobile/build/web will be uploaded to the Amplify CDN.
