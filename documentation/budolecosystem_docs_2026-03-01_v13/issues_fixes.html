Issues & Fixes Log

Issue #001: Lack of Domain Name

Description: Production environment lacked a professional domain name for external access, causing cross-origin issues between local dev and cloud APIs.

Fix: Implemented a 6-subdomain DuckDNS strategy. Mapped all microservices to specific subdomains (budolshap, budolid, budolpay, etc.) ensuring stable external access at zero cost.

Issue #002: Inter-service Communication Overhead

Description: Services were attempting to communicate via public URLs, increasing latency and cost.

Fix: Configured Route 53 Private Hosted Zones with .internal aliases. Services now communicate securely within the VPC over private IPs.

Issue #003: SSM Agent "Offline" on Private EC2

Description: The budol-redis instance was provisioned in a private subnet and appeared as "Offline" in AWS Systems Manager, preventing administrative access.

Fix: Corrected VPC Routing. The private subnets were not associated with the **Private Route Table**. 1. Associated Private Subnets with a dedicated Route Table. 2. Added a route for 0.0.0.0/0 targeting the **NAT Gateway**. This allowed the SSM agent to communicate with the AWS Systems Manager service endpoints securely.

Issue #004: Invalid AMI ID for Ubuntu 24.04 Launch

Phase: Phase 3 — EC2 Provisioning

Description: Initial EC2 launch for budol-ws failed with InvalidAMIID.NotFound error because a hardcoded AMI ID was used that was either expired or not available in ap-southeast-1.

Fix: Queried AWS SSM Parameter Store to get the latest valid Ubuntu 24.04 LTS AMI ID dynamically for the target region:
aws ssm get-parameters-by-path --path /aws/service/canonical/ubuntu/server/24.04/stable/current/amd64/hvm/ebs-gp3/ami-id
Result: AMI ID ami-0ed0867532b47cc2c was used for successful launch.

Issue #005: Windows PowerShell Shell-Escaping in SSM send-command

Phase: Phase 3 — Application Deployment

Description: Attempting to pass multi-line bash scripts (containing single quotes, double quotes, and heredocs) via --parameters in PowerShell caused persistent CommandNotFoundException and malformed shell script errors on the EC2 instance.

Fix: Adopted a two-step strategy:
  1. Write the SSM command payload to a local JSON file (ssm-input.json) using write_to_file.
  2. Reference it via --cli-input-json file://ssm-input.json to completely bypass PowerShell's argument parsing.
For injecting file content, source files were Base64-encoded locally using [Convert]::ToBase64String([System.IO.File]::ReadAllBytes(...)) and decoded on the remote via echo BASE64 | base64 -d > target.file.

Issue #006: SSM Agent "Offline" for budol-ws Instance on Initial Launch

Phase: Phase 3 — Software Setup

Description: After the EC2 instance was launched, attempting to send SSM commands resulted in InvalidInstanceId as the SSM agent had not yet fully initialized and registered with the Systems Manager service.

Fix: Added a polling wait loop using:
aws ssm describe-instance-information --filters Key=InstanceIds,Values=INSTANCE_ID
Waited until the instance appeared with PingStatus: Online before sending commands. Typically requires 2-4 minutes after launch for the SSM agent to register.

Issue #007: DuckDNS Token Placeholder Not Replaced After Deployment

Phase: Phase 3 — DuckDNS Automation

Description: The setup_websocket_ec2.sh script creates update_duckdns.sh with a hardcoded YOUR_DUCKDNS_TOKEN_HERE placeholder. After the initial deployment, the placeholder was not replaced, causing DuckDNS updates to silently fail with an auth error.

Fix: Retrieved the actual DuckDNS token from duckdns.org (account: galvezjon59@gmail.com, token appears on the dashboard after login). Applied the token remotely via SSM:
cat <<'EOF' > update_duckdns.sh ... TOKEN="db0964a1-387c-419a-9272-f87e38b8a542" ... EOF
Immediately ran the script — DuckDNS responded with OK, confirming successful IP update. Cron job is now active and runs every 5 minutes.
