AWS Production Infrastructure Setup Guide

A comprehensive, step-by-step guide to provisioning and configuring the budolEcosystem on AWS Free Tier.

Project Context & Compliance

Purpose

Migration of the BudolEcosystem to AWS Free Tier to achieve enterprise-grade reliability, security, and scalability while maintaining zero-cost hosting for the Phase 0-3 implementation.

Rationale

Current Vercel/Local hosting models lack the granular networking control (VPC) and database consolidation capabilities required for PCI DSS and BSP compliance. AWS provisioned infrastructure allows for a Gold Standard configuration with clear segregation of duties and data sovereignty.

Technical Explanation

The architecture leverages a Multi-Subnet VPC for isolation. Core services run on ECS Fargate and AWS Lambda, while databases are consolidated into a single RDS PostgreSQL instance with logical isolation. Real-time traffic is handled by a dedicated Socket.io server on EC2, bypassing expensive third-party providers like Pusher.

Architectural Deep Dive Summary

Based on the architectural review completed in v13:

  • API Gateway (Port 8080): Central hub for RBAC and routing. Proxies to 7 core microservices.
  • SSO (budolID - Port 8000): Central identity provider with multi-app redirect logic.
  • Financial Core (budolPay): Split into 7 specialized services (Auth, Wallet, TX, Payment, KYC, Settlement, Gateway).
  • Accounting (budolAccounting): Centralized double-entry ledger.
  • Real-time (Socket.io - Port 4000): Transitioned from Pusher to a dedicated EC2 microservice.

Risk & Mitigation Register

Identified Risks

  • Free Tier Limits: Over-provisioning could lead to unexpected costs.
  • Data Latency: Consolidated database might face I/O bottlenecks.
  • Connectivity: NAT Gateway costs if data transfer exceeds 1GB/month.

Mitigation Strategies

  • Billing Alerts: Integrated CloudWatch alarms at $0.01 threshold.
  • Sticky Sessions: ALB configured for websocket session persistence.
  • Data Squeezing: S3 Intelligent-Tiering and RDS Storage Auto-scaling.

Compliance & Standards

PCI DSS & BSP Alignment

All database traffic is restricted to the private subnet with no public ingress. Secrets are managed via AWS Secrets Manager with automatic rotation. MFA is enforced for all IAM roles associated with the production environment.

Phase 1: Networking & VPC Setup

1.1 Create VPC & Subnets

  1. Navigate to **VPC Dashboard** -> **Create VPC**.
  2. Select **VPC only**, Name: budol-vpc, CIDR: 10.0.0.0/16.
  3. Create **4 Subnets**:
    • Public-A (10.0.1.0/24), Public-B (10.0.2.0/24)
    • Private-A (10.0.10.0/24), Private-B (10.0.11.0/24)

1.2 Internet & NAT Gateway

  1. Create an **Internet Gateway (IGW)** and attach it to budol-vpc.
  2. Create a **NAT Gateway** in Public-A. Allocate an Elastic IP.
  3. Route Tables:
    • Public-RT: Add route 0.0.0.0/0 -> IGW. Associate with Public subnets.
    • Private-RT: Add route 0.0.0.0/0 -> NAT Gateway. Associate with Private subnets.
Operational Note: For SSM Session Manager to function on EC2 instances in private subnets, ensure the subnet is explicitly associated with Private-RT and that the route to the NAT Gateway is active. Lack of association defaults to the Main Route Table which typically lacks an egress path, causing the instance to appear "Offline".

1.3 Internal DNS (Route 53)

  1. Go to **Route 53** -> **Hosted Zones** -> **Create Hosted Zone**.
  2. Domain: budol.internal, Type: **Private Hosted Zone**.
  3. Associate with budol-vpc.
  4. Create A-Records (Alias) for internal load balancers or EC2 IPs (e.g., db.budol.internal).
