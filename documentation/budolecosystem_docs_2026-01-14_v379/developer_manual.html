Developer Manual – v379

Technical Explanation: Propagation of Initialization Errors

In previous versions, the _initModel() method used a Completer to manage asynchronous initialization. However, if all retry attempts failed, the method would complete the future with an error but would then exit the loop and return normally to the caller. This caused the caller (the first invocation of generateEmbedding) to proceed as if initialization had succeeded, eventually hitting a null pointer check on the interpreter.

Code Change: Explicit Exception Throwing

The fix involves explicitly throwing an exception after the last retry attempt in _initModel(). This ensures that the first caller also receives the error immediately.

File: face_embedding_service.dart

// In _initModel() inside the catch block for retry attempts if (attempt == maxRetries) { debugPrint('FaceEmbeddingService: All attempts exhausted.'); _isModelLoaded = false; _interpreter = null; if (_loadingCompleter != null && !_loadingCompleter!.isCompleted) { _loadingCompleter!.completeError(e); } // NEW: Re-throw the error so the first caller knows initialization failed throw Exception('TFLite initialization failed after $maxRetries attempts: $e'); }

Code Change: Improved Error Context in generateEmbedding

We also wrapped the call to _initModel() in a try-catch to provide better context to the UI layer.

try { await _initModel(); } catch (e) { debugPrint('FaceEmbeddingService: _initModel() threw: $e'); throw Exception('AI Inference failed to initialize: $e'); }

Rationale

By ensuring that initialization failures are never "swallowed," we prevent the application from entering an inconsistent state where it believes a model is ready when it is not. This improves the reliability of the Face Registration flow and provides developers with clearer diagnostic information when hardware or file-system issues occur.
