Technical Architecture: Provider-Agnostic Realtime

The core of this update is the realtime.ts client library. It abstracts the underlying communication protocol, allowing the application to switch providers without changing component logic.

1. Realtime Client Abstraction

The RealtimeClient class fetches system settings on initialization and sets up the appropriate provider (Socket.io or Pusher). If no provider is available, it defaults to SWR-like behavior (handled via event emission patterns).

// apps/admin/lib/realtime.ts
class RealtimeClient {
  private method: RealtimeMethod = 'SWR';
  // ...
  async init() {
    const res = await fetch('/api/system/settings');
    const settings = await res.json();
    this.method = settings.realtime.method;
    // Initialize Socket.io or Pusher based on this.method
  }
  // ...
}

2. Backend Notification Logic

Microservices like auth-service now use a notifyAdmin helper (provided via shared utils or direct call to API Gateway) to broadcast events.

// services/auth-service/index.js
const notifyAdmin = async (event, data) => {
  await axios.post(`${GATEWAY_URL}/internal/notify`, {
    event, data, isAdmin: true
  });
};

// On new user registration:
await notifyAdmin('new_user', { count: updatedCount });

3. Component Integration

Components now subscribe to events using the realtime.on() pattern, which is agnostic of the underlying provider.

// apps/admin/components/LiveStats.tsx
useEffect(() => {
  realtime.init().then(() => {
    realtime.on("new_user", (data) => {
      setStats(prev => ({ ...prev, userCount: data.count }));
    });
  });
}, []);
