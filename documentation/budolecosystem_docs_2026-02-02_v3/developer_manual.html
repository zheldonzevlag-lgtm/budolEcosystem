Developer Manual - Version 2.2.0

1. Purpose & Rationale

This release implements Security Hardening Phase 3 of the Budol Ecosystem Security Standard. The primary focus is on protecting the authentication infrastructure from brute-force attacks and providing administrative controls for security policies in compliance with PCI DSS Requirement 8.1.8 and BSP Circular 808.

2. Technical Architecture: Rate Limiting

We implemented a database-backed rate limiting system using the Token Bucket algorithm. This approach was chosen over memory-based solutions (like lru-cache) to ensure consistency across serverless environments (Vercel) and multi-instance deployments.

2.1 Data Contract (Prisma Schema)

A new RateLimit model was added to track hits per key (e.g., IP address or User ID).

model RateLimit {
  id        String   @id @default(uuid())
  key       String   @unique
  hits      Int      @default(1)
  expiresAt DateTime
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([key])
  @@index([expiresAt])
}

4. Compliance Verification

  • PCI DSS 8.1.8: Implementation of session idle timeout settings.
  • BSP Circular 808: Enforcement of multi-factor authentication (MFA) policy controls.
  • Cybersecurity Law: Protection of critical infrastructure via rate limiting.
