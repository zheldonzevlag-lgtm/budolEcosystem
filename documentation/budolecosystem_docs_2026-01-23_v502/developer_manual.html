Technical Architecture

The real-time synchronization system uses a provider-agnostic approach, supporting Pusher, Socket.io, or Polling. The frontend uses SWR for data fetching and cache management.

Payload Standardization

All order-updated events must now include isPaid, paymentStatus, and shipping data to allow the frontend to correctly filter orders into tabs (e.g., "To Pay", "To Ship", "Cancelled").

Dependencies & Rationale

  • Next.js: Chosen for its server-side rendering (SSR) capabilities, which improve SEO and initial load performance for the BudolShap marketplace. API routes provide a unified backend-frontend development experience.
  • Prisma: Selected as the ORM because of its type-safety and auto-generated client, which reduces runtime errors and accelerates development of complex order/shipping schemas.
  • Pusher: Utilized for real-time event broadcasting due to its managed infrastructure, which handles scale and reliability (e.g., connection fallbacks) better than a custom-built WebSocket server.
  • SWR: Integrated for React state management and data fetching. Its built-in cache mutation (mutate) is essential for the "flicker-free" UI updates implemented in this release.
