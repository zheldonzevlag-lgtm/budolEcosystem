Developer Manual - v395

Technical Rationale

In v394, we prioritized OS-level biometrics to avoid redundant photo capture. However, feedback indicated that the Face icon should remain the primary entry point for the custom Native Face Recognition (TFLite) feature, as it includes proprietary liveness checks.

v395 implements a "Smart Fallback" logic: The Face icon will attempt the custom TFLite flow first if a template is present and enabled. If not, it falls back to the system's biometric prompt (LocalAuth).

Code Changes

1. Logic Update in Login Screen

The _handleFaceRecognitionLogin method in login_screen.dart has been restructured. 

// Prioritize Custom Face Recognition (Native to App)
if (_hasFaceTemplate && _isBiometricEnabled) {
  // Use Custom TFLite Flow (Manual Photo Capture)
  // ...
} else {
  // Fallback to Biometric Login (OS Prompt)
  final hasNativeFace = await biometricService.hasEnrolledFace();
  if (hasNativeFace && _isBiometricEnabled) {
    return _handleBiometricLogin();
  }
}

Compliance

  • NPC: Biometric data remains encrypted and stored locally.
  • Cybersecurity Law: Liveness detection (blink/smile) remains the primary defense against spoofing in the custom flow.
