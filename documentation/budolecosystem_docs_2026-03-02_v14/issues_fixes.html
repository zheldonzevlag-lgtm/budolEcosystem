Issues & Fixes Log

This document chronicles all issues encountered during the ECS Fargate migration and their resolutions.

Issue #001: Lack of Domain Name (v6)

Phase 1 Date: Pre-v7.0.0

Description: Production environment lacked a professional domain name for external access, causing cross-origin issues between local dev and cloud APIs.

Fix: Implemented a 6-subdomain DuckDNS strategy. Mapped all microservices to specific subdomains (budolshap, budolid, budolpay, etc.) ensuring stable external access at zero cost.

Issue #002: Inter-service Communication Overhead (v6)

Phase 1 Date: Pre-v7.0.0

Description: Services were attempting to communicate via public URLs, increasing latency and cost.

Fix: Configured Route 53 Private Hosted Zones with .internal aliases. Services now communicate securely within the VPC over private IPs.

Issue #003: SSM Agent "Offline" on Private EC2 (v6)

Phase 2 Date: Pre-v7.0.0

Description: The budol-redis instance was provisioned in a private subnet and appeared as "Offline" in AWS Systems Manager, preventing administrative access.

Fix: Corrected VPC Routing. The private subnets were not associated with the Private Route Table.
  1. Associated Private Subnets with a dedicated Route Table.
  2. Added a route for 0.0.0.0/0 targeting the NAT Gateway.
This allowed the SSM agent to communicate with the AWS Systems Manager service endpoints securely.

Issue #004: Invalid AMI ID for Ubuntu 24.04 Launch (v6)

Phase 3 Date: Pre-v7.0.0

Description: Initial EC2 launch for budol-ws failed with InvalidAMIID.NotFound error because a hardcoded AMI ID was used that was either expired or not available in ap-southeast-1.

Fix: Queried AWS SSM Parameter Store to get the latest valid Ubuntu 24.04 LTS AMI ID dynamically for the target region:
aws ssm get-parameters-by-path --path /aws/service/canonical/ubuntu/server/24.04/stable/current/amd64/hvm/ebs-gp3/ami-id
Result: AMI ID ami-0ed0867532b47cc2c was used for successful launch.

ECS Fargate Migration Issues

Issue #005: ECR Repository Creation Permissions

Phase 2 Date: v7.0.0

Description: Attempting to create ECR repositories failed with AccessDeniedException when the IAM user didn't have ecr:CreateRepository permissions.

Fix: Added the following IAM permissions to the deployment user: 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ecr:CreateRepository",
        "ecr:DeleteRepository",
        "ecr:DescribeRepositories",
        "ecr:PutLifecyclePolicy"
      ],
      "Resource": "arn:aws:ecr:ap-southeast-1:*:repository/budolEcosystem/*"
    }
  ]
}

Issue #006: Docker Build Failures - Missing Dependencies

Phase 5 Date: v7.0.0

Description: Docker build failed for some services with errors like cannot find module 'express' because node_modules weren't being included in the image.

Fix: Updated Dockerfile to use multi-stage builds: 
# Build stage
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .

# Production stage
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app .
CMD ["node", "server.js"]

Issue #007: ECS Task Definition - Missing IAM Role

Phase 4 Date: v7.0.0

Description: ECS tasks failed to start with error AccessDeniedException: The role 'ecsTaskExecutionRole' cannot be assumed.

Fix: Created the ECS Task Execution IAM role with the proper trust relationship: 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ecs-tasks.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
And attached the AmazonECSTaskExecutionRolePolicy managed policy.

Issue #008: ALB Target Group Health Check Failures

Phase 6 Date: v7.0.0

Description: ECS tasks were showing as unhealthy in ALB target groups. The health check was failing because the default path / didn't return 200 for some services.

Fix:
  1. Configured service-specific health check paths in the target group:
    • budolID: /api/health
    • budolPay Gateway: /health
    • budolAccounting: /health
  2. Added health check endpoints to each service that returns HTTP 200 with {"status": "ok"}
  3. Increased health check interval to 30 seconds and timeout to 10 seconds

Issue #009: Container Networking - Services Can't Reach Each Other

Phase 6 Date: v7.0.0

Description: Inter-service communication was failing with ECONNREFUSED errors. Containers couldn't resolve service names.

Fix:
  1. Ensured all ECS services are in the same security group with inbound rules allowing all traffic on all ports
  2. Updated service URLs to use the ECS service discovery names (e.g., http://budolid:8000)
  3. Added awsvpc network mode to task definitions for proper networking

Issue #010: Secrets Not Available in Container

Phase 4 Date: v7.0.0

Description: Environment variables for database passwords were showing as undefined in running containers.

Fix:
  1. Used AWS Secrets Manager instead of SSM Parameter Store for sensitive data
  2. Updated task definitions to use secrets instead of environment for sensitive values:
"secrets": [
  {
    "name": "DATABASE_PASSWORD",
    "valueFrom": "arn:aws:secretsmanager:ap-southeast-1:123456789012:secret:/budolEcosystem/production/DATABASE_PASSWORD"
  }
]

Issue #011: Task Definition Revision Conflicts

Phase 4 Date: v7.0.0

Description: Registering new task definitions failed with ClientException: The container name 'budolid' is already in use.

Fix:
  1. Ensured each task definition uses a unique container name
  2. Used JSON file approach for task definitions instead of CLI to avoid naming conflicts
  3. Registered with --cli-input-json instead of individual parameters

Issue #012: Security Group Rules Too Restrictive

Phase 3 Date: v7.0.0

Description: ALB couldn't connect to ECS tasks because the security group only allowed inbound from the ALB's security group, but the tasks were in a different security group.

Fix:
  1. Created a dedicated security group for ECS tasks
  2. Added inbound rule allowing all traffic from ALB security group
  3. Added inbound rule allowing all traffic from VPC CIDR for internal communication

Issue #013: CloudFormation Stack Creation Timeout

Phase 3 Date: v7.0.0

Description: CloudFormation stack creation timed out while waiting for ALB to become available.

Fix:
  1. Increased stack creation timeout to 60 minutes
  2. Broke down CloudFormation template into smaller nested stacks
  3. Added --timeout-in-minutes 60 to the deployment command

Issue #014: Cost Optimization - Fargate vs EC2

Phase 3 Date: v7.0.0

Description: Fargate tasks are running but costs may exceed Free Tier limits with constant 9 running tasks.

Mitigation:
  1. Set desired task count to 1 for each service (minimum)
  2. Configure auto-scaling to scale down to 0 during non-peak hours
  3. Consider using Spot instances with ECS EC2 launch type for cost savings
  4. Monitor AWS Cost Explorer for actual usage

Issue #015: Route 53 Private Zone DNS Resolution

Phase 3 Date: v7.0.0

Description: ECS containers couldn't resolve .budol.internal domain names.

Fix:
  1. Associated the Private Hosted Zone with the VPC
  2. Added enableDnsHostnames and enableDnsSupport VPC attributes
  3. Used awsvpc network mode which provides enhanced DNS resolution

Troubleshooting Guide

Task Not Starting

Symptoms: ECS task stays in PENDING state or fails to start.

Troubleshooting Steps:
  1. Check task definition for syntax errors: aws ecs describe-task-definition --task-definition
  2. Verify image exists in ECR: aws ecr describe-images --repository-name
  3. Check CloudWatch logs for container errors
  4. Verify security group allows outbound to ECR
  5. Ensure task execution role has ECR pull permissions

Service Not Reaching Healthy State

Symptoms: Target group shows 0/0 healthy or unhealthy instances.

Troubleshooting Steps:
  1. Check if task is running: aws ecs list-tasks --cluster --service-name
  2. Review CloudWatch logs for health check failures
  3. Verify health check path returns HTTP 200
  4. Check security group allows traffic from ALB
  5. Increase health check grace period period

Inter-Service Communication Failing

Symptoms: Services return 502 Bad Gateway or ECONNREFUSED.

Troubleshooting Steps:
  1. Verify target service is running: aws ecs describe-services --cluster --services
  2. Check security groups allow internal traffic
  3. Test DNS resolution from within container
  4. Verify correct service URLs in environment variables
  5. Check ALB listener rules are correctly configured

Image Pull Failures

Symptoms: Tasks fail with CannotPullContainerError.

Troubleshooting Steps:
  1. Verify ECR repository exists and image tag exists
  2. Check VPC has gateway endpoint for ECR
  3. Ensure IAM role has ecr:GetAuthorizationToken and ecr:BatchGetImage
  4. Verify security group allows outbound HTTPS (443) to ECR endpoints
