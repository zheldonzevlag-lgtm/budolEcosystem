System Administrator Manual: ECS Fargate

Comprehensive guide for system administrators managing the budolEcosystem ECS Fargate infrastructure.

Daily Operations

Monitoring ECS Services

Regular monitoring tasks to ensure system health:

  1. Check ECS Dashboard - Verify all 9 services are running
  2. Review CloudWatch Logs - Check for errors or warnings
  3. Monitor ALB Metrics - Response times, error rates, request counts
  4. Check RDS Connections - Monitor database connection pool
  5. Review Redis Memory - Ensure memory usage is within limits

Checking Service Status

# List all services in the cluster
aws ecs list-services --cluster budolEcosystem-production

# Describe service status
aws ecs describe-services \
  --cluster budolEcosystem-production \
  --services budolid budolaccounting budolpay-gateway

# Get running tasks
aws ecs list-tasks \
  --cluster budolEcosystem-production \
  --desired-status RUNNING

Backup & Recovery

Database Backups

RDS PostgreSQL automated backups are configured with:

  • Daily automated backups
  • Backup retention: 7 days
  • Point-in-time recovery enabled
# Create manual snapshot
aws rds create-db-snapshot \
  --db-instance-identifier budol-postgres \
  --db-snapshot-identifier budol-manual-backup-$(date +%Y%m%d)

# List snapshots
aws rds describe-db-snapshots \
  --db-instance-identifier budol-postgres

Redis Backups

Redis ElastiCache supports automatic snapshots:

  • Daily automated snapshots
  • Retention: 1 day
# Create manual snapshot
aws elasticache create-snapshot \
  --cache-cluster-id budol-redis \
  --snapshot-name budol-redis-manual-$(date +%Y%m%d)

# List snapshots
aws elasticache describe-snapshots \
  --cache-cluster-id budol-redis

Disaster Recovery

In case of complete infrastructure failure:

  1. Restore RDS from snapshot - Use latest automated snapshot
  2. Redeploy ECS cluster - Run CloudFormation template
  3. Restore ECR images - Images are persisted in ECR
  4. Update DNS records - Route 53 will auto-recover

Scaling Operations

Manual Scaling

# Scale up a service
aws ecs update-service \
  --cluster budolEcosystem-production \
  --service-name budolid \
  --desired-count 3

# Scale down a service
aws ecs update-service \
  --cluster budolEcosystem-production \
  --service-name budolid \
  --desired-count 1

Auto Scaling Configuration

Auto scaling is recommended for production workloads:

# Register scalable target
aws application-autoscaling register-scalable-target \
  --service-namespace ecs \
  --resource-id service/budolEcosystem-production/budolid \
  --scalable-dimension ecs:service:DesiredCount \
  --min-capacity 1 \
  --max-capacity 10

# CPU-based scaling policy
aws application-autoscaling put-scaling-policy \
  --policy-name budolid-cpu-scaling \
  --service-namespace ecs \
  --resource-id service/budolEcosystem-production/budolid \
  --scalable-dimension ecs:service:DesiredCount \
  --policy-type TargetTrackingScaling \
  --target-tracking-scaling-policy-configuration '{
    "TargetValue": 70,
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ECSServiceAverageCPUUtilization"
    }
  }'

# Memory-based scaling policy
aws application-autoscaling put-scaling-policy \
  --policy-name budolid-memory-scaling \
  --service-namespace ecs \
  --resource-id service/budolEcosystem-production/budolid \
  --scalable-dimension ecs:service:DesiredCount \
  --policy-type TargetTrackingScaling \
  --target-tracking-scaling-policy-configuration '{
    "TargetValue": 80,
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ECSServiceAverageMemoryUtilization"
    }
  }'

Security Operations

Updating Secrets

# Update database password in Secrets Manager
aws secretsmanager update-secret \
  --secret-id /budolEcosystem/production/DATABASE_PASSWORD \
  --secret-string '{"password":"new-password-here"}'

# Restart service to pick up new secret
aws ecs update-service \
  --cluster budolEcosystem-production \
  --service-name budolid \
  --force-new-deployment

Managing IAM Roles

# List task execution roles
aws iam list-roles --path-prefix /aws-service-role/ecs.amazonaws.com/

# Check role permissions
aws iam get-role --role-name ecsTaskExecutionRole

Security Group Management

# List security groups
aws ec2 describe-security-groups \
  --filters "Name=vpc-id,Values=vpc-xxxxxxxx"

# Add inbound rule
aws ec2 authorize-security-group-ingress \
  --group-id sg-xxxxxxxx \
  --protocol tcp \
  --port 8000 \
  --cidr 10.0.0.0/16

Maintenance Operations

Rolling Updates

ECS performs rolling updates automatically when you update a service:

# Update service with new task definition
aws ecs update-service \
  --cluster budolEcosystem-production \
  --service-name budolid \
  --task-definition budolid:latest

# Monitor deployment progress
aws ecs describe-services \
  --cluster budolEcosystem-production \
  --services budolid \
  --query 'services[0].deployments'

Log Management

# View container logs
aws logs get-log-events \
  --log-group-name /ecs/budolEcosystem-production/budolid \
  --log-stream-name ecs/budolid/xxxxxxxxx \
  --limit 50

# Create log retention policy
aws logs put-retention-policy \
  --log-group-name /ecs/budolEcosystem-production/budolid \
  --retention-in-days 30

Image Updates

# Pull new image from ECR
aws ecr batch-get-image \
  --repository-name budolEcosystem/budolid \
  --image-tags latest \
  --query 'images[0].imageManifest' \
  --output text | docker load

# Update task definition with new image
# Edit ecs/task-definitions/budolid.json with new image URI
aws ecs register-task-definition \
  --cli-input-json file://ecs/task-definitions/budolid.json

# Deploy new version
aws ecs update-service \
  --cluster budolEcosystem-production \
  --service-name budolid \
  --force-new-deployment

Troubleshooting

Service Won't Start

  1. Check task definition is valid
  2. Verify image exists in ECR
  3. Check security group allows ECR access
  4. Review CloudWatch logs for errors
# Check task failure reason
aws ecs describe-tasks \
  --cluster budolEcosystem-production \
  --tasks  \
  --query 'tasks[0].stopCode'

High CPU/Memory Usage

  1. Identify affected service
  2. Check for memory leaks in application
  3. Scale up the service temporarily
  4. Review CloudWatch metrics
# Get service metrics
aws cloudwatch get-metric-statistics \
  --namespace AWS/ECS \
  --metric-name CPUUtilization \
  --start-time 2024-01-01T00:00:00Z \
  --end-time 2024-01-02T00:00:00Z \
  --period 3600 \
  --statistics Average,Maximum

ALB Health Check Failures

  1. Verify target is registered
  2. Check health check path
  3. Ensure security group allows ALB
  4. Review target health description
# Check target health
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:ap-southeast-1:123456789012:targetgroup/budolid-tg/xxxxxxxx

Cost Optimization

Cost Monitoring

# Check current month costs
aws ce get-cost-and-usage \
  --time-period Start=2024-01-01,End=2024-02-01 \
  --granularity DAILY \
  --metrics UnblendedCost \
  --group-by Type=DIMENSION,Key=SERVICE

Cost Saving Tips

  • Use Fargate Spot - Save up to 70% with spot instances
  • Right-size services - Match CPU/memory to actual usage
  • Scale to zero - Scale down during off-hours
  • Use S3 for storage - Cheaper than EBS for persistent data
  • Monitor Free Tier - Stay within Free Tier limits

Emergency Procedures

Critical Emergency: If the entire ECS cluster is down:
  1. Check AWS Service Health Dashboard
  2. Verify VPC and networking configuration
  3. Check for CloudFormation stack failures
  4. Contact AWS Support if needed
Service Outage: If a single service is down:
  1. Check service status in ECS console
  2. Review CloudWatch logs
  3. Force restart the service
  4. Rollback to previous task definition if needed
