Non-Serverless Infrastructure Strategy v158

This document outlines the alternative "Always-On" hosting strategy for the budolEcosystem. While serverless (Vercel/Lambda) is great for scaling, a persistent server approach provides superior stability for real-time financial applications.

Why Choose Non-Serverless?

1. Native Socket.io Support

Unlike Vercel, a persistent server (EC2/VPS) keeps the Node.js process alive indefinitely. This allows for stable WebSocket connections without the need for complex workarounds or external providers.

2. Zero "Cold Start" Latency

In serverless, the first request after a period of inactivity can take 2-5 seconds (cold start). A persistent server is always "warm," ensuring sub-100ms response times for every transaction.

3. State Management

Persistent servers can maintain in-memory caches (like local Redis or internal JS maps) which are lost in serverless environments between function calls.

4. Predictable Billing

Instead of paying per-request or per-millisecond, you pay a fixed monthly fee for the server (e.g., $5/mo for a VPS or EC2 instance), making budgeting much easier.

Proposed Non-Serverless Stack

Component Technology Role
Compute AWS EC2 / DigitalOcean Droplet Persistent Virtual Private Server (VPS).
Orchestration Docker + Docker Compose Packaging all 11 microservices into containers.
Process Manager PM2 Keeping Node.js processes alive and auto-restarting on failure.
Reverse Proxy Nginx Handling SSL (HTTPS), Load Balancing, and routing traffic to containers.
Database Managed PostgreSQL Still recommended to use RDS for data durability.

Implementation Path

1. Dockerization

Each service in the ecosystem (Auth, Transaction, Wallet, etc.) will have its own Dockerfile. A master docker-compose.yml will launch all services simultaneously on the server.

2. Unified API Gateway

The api-gateway will act as the single entry point, routing requests to internal containers using Docker's internal networking (faster than public HTTP calls).

3. Nginx Configuration

# Example Nginx Routing to the Gateway server { listen 443 ssl; server_name api.budolpay.com; location / { proxy_pass http://localhost:8000; # Points to the Docker Gateway proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; } }

Recommendation

If the user prefers simplicity and maximum reliability for Sockets, we recommend the Docker on EC2 approach. It mirrors the local development environment (start-ecosystem.js) most closely, making debugging and deployment significantly easier than serverless refactoring.
