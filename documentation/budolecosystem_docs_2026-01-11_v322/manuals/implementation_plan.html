Biometric Login Implementation Plan & Strategy

Phase-by-phase plan to implement biometric login using OS biometrics (open-source first) while maintaining PCI DSS, BSP, BIR, NPC, DTI, and cybersecurity best practices.

Purpose

To define a phased, executable strategy for enabling biometric login in budolPay Mobile in a way that is secure, compliant, testable, and maintainable.

Key Clarification: “Capture fingerprint” vs OS biometric verification

Important: Standard Android/iOS apps cannot access or “capture” raw fingerprint images/templates. The OS biometric subsystem (Android BiometricPrompt / iOS LocalAuthentication) only returns a yes/no result and never exposes biometric data to apps.

Given financial compliance and modern mobile OS security models, the correct approach is to use OS biometrics for user verification, then unlock a device-bound secret (token/PIN surrogate) stored in the secure enclave/keystore.

If you truly need raw fingerprint capture, it requires external fingerprint hardware and a vendor SDK, and it significantly increases privacy and regulatory risk under NPC. This plan focuses on OS biometrics (recommended).

Current Baseline (Already Present in Codebase)

  • Biometric service wrapper: biometric_service.dart
  • PIN-step biometric trigger and enable dialog: login_screen.dart
  • Settings toggle (enable flag + clear stored PIN on disable): settings_screen.dart
  • E2E checklist script: test_biometric_login.dart
Status: Biometric login currently uses OS biometrics plus secure storage to retrieve a stored PIN. The following phases strengthen security, reduce risk, and improve operational behavior.

Strategy Overview

Track Open-source approach Primary goal
Track A: OS biometrics + secure storage local_auth + flutter_secure_storage Fast login by unlocking a device-bound secret after biometric verification
Track B: Passwordless (FIDO2/Passkeys) Passkeys / WebAuthn (client + server challenge flow) Eliminate stored PIN reuse risk and improve phishing resistance

This plan implements Track A first (fastest, least intrusive), then evolves toward Track B for higher assurance.

Phased Implementation Plan

Phase 1 — Security Model Hardening (No raw PIN-at-rest)

Goal: Replace “store raw PIN for biometric login” with a biometric-unlocked session secret while keeping PIN as fallback.

Approach: After successful manual OTP+PIN login, request a backend-issued device-bound refresh token (or a biometric login token) and store it in secure storage. On biometric login, exchange the token for a session without ever storing the PIN.

Dependencies: Gateway/auth-service endpoint for token issuance & exchange, device binding identifier, secure storage, OS biometrics.

Entry criteria: Existing biometric flow is working (v319) and stable.

Exit criteria: Biometric login works without storing a raw PIN in secure storage.

Phase 2 — Backend Device Binding & Policy Controls

Goal: Make biometric login enforceable and auditable.

  • Introduce per-device binding: userId + deviceId + public metadata (model/OS) + token.
  • Introduce global policy flags (enable/disable biometrics, max attempts, forced fallback).
  • Add server-side monitoring and anomaly detection hooks (failed biometric exchanges, token reuse).

Compliance alignment: Supports PCI DSS access controls, supports audit evidence for BSP/NPC incident analysis.

Phase 3 — UX & Reliability Improvements

Goal: Reduce user support tickets and improve deterministic behavior across devices.

  • Expose a “Biometric status” screen: supported, enrolled, enabled, token present.
  • Clear, localized user messaging when biometrics are unavailable (not enrolled / no screen lock).
  • Implement timeout rules (require biometric re-auth after background idle).

Exit criteria: Support can diagnose 90% of biometric issues using in-app status + logs (without collecting secrets).

Phase 4 — High Assurance Path (Open Standards: Passkeys / FIDO2)

Goal: Move toward passwordless authentication using device biometrics as the local user verification for key operations.

  • Implement WebAuthn/FIDO2 challenge-response with server stored public keys.
  • Allow passkey registration after successful OTP+PIN login.
  • Biometric unlock triggers signing, not secret retrieval.

Risk reduction: Removes reliance on PIN reuse and reduces phishing susceptibility.

Issues, Fixes, Risks, and Mitigations

Item Risk/Issue Mitigation/Fix
Raw fingerprint capture Not supported by OS APIs; privacy/regulatory exposure under NPC Use OS biometric verification only; do not store/process biometric templates
Storing raw PIN Increases blast radius if device is compromised Phase 1 shifts to biometric-unlocked refresh token instead of PIN
Biometric enrollment variability Users expect it to work without enrollment Phase 3 adds status screen and clear user messages
Replay of token Stolen token could be abused Phase 2 device binding + rotation + server-side anomaly detection

Testing Protocol (Quality Gates Per Phase)

Unit tests

  • Biometric availability and enabled flag logic
  • Secure storage read/write/delete behaviors (mocked)

Integration tests

  • Token exchange endpoints (Phase 1/2)
  • Device binding enforcement and failure modes

End-to-end tests

Use the existing checklist script (manual E2E):

cd \budolEcosystem
dart run scripts\test_scripts\test_biometric_login.dart

Acceptance criteria: biometric login works without exposing secrets; fallback PIN login always works; disabling biometrics revokes stored materials.

Rollback Strategy

  • Phase 1: keep PIN login intact; if token exchange fails, disable biometric token usage and fall back to PIN.
  • Phase 2: server-side policy flag to disable biometric login instantly (feature kill switch).
  • Phase 3: UX-only; safe to roll back via feature flags.
  • Phase 4: introduce as opt-in; retain OTP+PIN as fallback for account recovery.

Recommendations for Future Enhancement (Consolidated)

  • Adopt token-based biometric login (no PIN storage) as the default security model.
  • Add in-app biometric diagnostics for support (no secrets, no biometric data).
  • Implement passkeys (WebAuthn/FIDO2) for high assurance, phishing-resistant authentication.
  • Centralize policy controls (disable biometrics globally, max attempts, forced re-auth intervals).
