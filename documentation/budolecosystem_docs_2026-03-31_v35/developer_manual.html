Developer Manual: Build Stabilization & Vercel Nexus

Rationale

The transition to Next.js 14 (App Router) on Vercel requires strict separation between Build-time (Static) and Runtime (Dynamic) environments. Prerendering failues occur when static analysis triggers database queries in environments without DATABASE_URL (e.g. Preview build machines).

Phase 0: Build Resilience Architecture

We implemented a Dummy Connection Fallback in the core database client. If DATABASE_URL is missing during the next build phase, the client returns a local mock string instead of crashing the process.

📂 packages/database/index.ts 
const dbUrl = process.env.DATABASE_URL && process.env.DATABASE_URL.length > 0 
  ? process.env.DATABASE_URL 
  : "postgresql://postgres:postgres@localhost:5432/budolpay?schema=public";

Phase 1: Route Dynamic Forcing

To prevent DynamicServerUsage errors during static optimization, all pages fetching live data from the database or accessing request.url are marked with force-dynamic.

📂 apps/admin/app/page.tsx 
export const dynamic = 'force-dynamic';
// Root dashboard fetches count() from prisma during render
📂 apps/admin/app/api/audit-logs/route.ts 
export const dynamic = 'force-dynamic';
// API routes using request.url must be dynamic

Phase 2: MathCaptcha Persistence

Re-instated the Shield Challenge on the admin login page to fulfill PCI DSS 10.2.1 requirements for bot mitigation and session protection.
