Post-Mortem: v35 Build & Postgres Migration

Issue 1: PrismaClientInitializationError during Static Analysis

Root Cause: Next.js 14 attempted to evaluate database-fetching pages (/accounting, /) during the build phase on Vercel preview machines, which do not have DATABASE_URL accessibility.

Fix applied: Core database client now uses a resilient fallback to localhost:5432 if DATABASE_URL is empty during the PHASE_PRODUCTION_BUILD.

Issue 2: DynamicServerUsage Error in API Routes

Root Cause: API routes in /api/audit-logs and /api/transactions accessed request.url. In Next.js 14, this forces a route to be dynamic, but since it wasn't explicitly marked, the static optimizer failed.

Fix applied: Added export const dynamic = 'force-dynamic'; to both API routes.

Issue 3: Prisma Accelerate Latency & Billing Concerns

Root Cause: The production environment was using db.prisma.io intermediate proxy, causing unnecessary latency and potential cost overhead.

Fix applied: Integrated native Vercel Postgres (Postgres Nexus) storage directly into the project environment.

Issue 4: Admin Search Params Type Errors

Fix applied: Refactored SettingsPage props in settings-orig to correctly handle searchParams as a promise-compatible or synchronous object depending on Next.js exact versioning needs.
