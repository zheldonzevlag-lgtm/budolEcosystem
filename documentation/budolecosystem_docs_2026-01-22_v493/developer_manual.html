Developer Manual - Version 493

1. Admin Settings Enforcement

Version 493 ensures that the realtime infrastructure strictly adheres to the configuration set in the Admin UI. This is achieved through a dual-enforcement strategy on both the frontend and backend.

Frontend Enforcement (RealtimeProvider)

The RealtimeProvider.jsx fetches settings via SWR and dynamically switches between PUSHER, SOCKET_IO, and POLLING modes.

const { data: configData } = useSWR('/api/system/settings', fetcher);
useEffect(() => {
    const provider = configData?.realtimeProvider || 'POLLING';
    if (provider === 'POLLING') {
        setSwrOptions({ refreshInterval: 5000 });
    } else {
        setSwrOptions({ refreshInterval: 0 });
        // Initialize Pusher or Socket.io...
    }
}, [configData]);

Backend Enforcement (lib/realtime.js)

The lib/realtime.js module caches settings for 1 minute and routes all events through the active provider.

export async function triggerRealtimeEvent(channel, event, data) {
    const settings = await getSettings();
    if (settings.realtimeProvider === 'PUSHER') {
        // Trigger Pusher...
    } else if (settings.realtimeProvider === 'SOCKET_IO') {
        // Trigger Socket.io...
    }
}

2. Centralized Realtime Architecture

All realtime logic is centralized in the global RealtimeProvider. Individual hooks like useRealtimeOrders no longer manage their own connections, reducing redundant listeners and memory usage.

Reference: useRealtimeOrders.js

3. Cost & Bandwidth Optimization

To minimize AWS/Vercel hosting costs, the following features are implemented:

  • Visibility-Aware Polling: Throttles polling from 5s to 30s when the browser tab is hidden.
  • SWR Cache Patching: Updates local UI state without full API re-fetches.
  • Minimal Payloads: Realtime events only carry essential fields (e.g., ID, Status).
