System Administrator Manual

This document outlines the monitoring and troubleshooting procedures for the new authentication and normalization features.

1. Monitoring Login Activity

The system logs all login attempts, including successful fallbacks to BudolID. These logs are crucial for identifying potential sync issues or malicious activity.

Identifying Fallback Logins

When a user logs in via email and their local password fails but BudolID succeeds, the following log entry is generated:

[Login] Local password failed for user@example.com. Attempting fallback to BudolID. [Login] BudolID fallback successful for user@example.com

Action: Frequent occurrences of this log for the same user indicate that the local password sync is working correctly. However, if you see repeated failures after success, investigate potential database write issues.

Identifying Failed Fallbacks

If both local and BudolID authentication fail, or if BudolID is unreachable:

[Login] BudolID fallback failed: Connection timed out [Login] BudolID fallback failed: Invalid credentials

Action: Check BudolID service status if multiple users report login failures simultaneously.

2. Phone Number Normalization Audit

The mobile app now enforces E.164 format. Monitor API logs for rejected phone numbers which might indicate older app versions are still in use.

API Log Analysis

Look for 400 Bad Request errors on /api/auth/register or /api/auth/login with messages related to phone format.

Error: Invalid phone number format. Expected E.164 (e.g., +639...)

Action: Encourage users to update their app if they encounter persistent issues.

3. Database Maintenance

Periodically, run a script to identify users with non-normalized phone numbers in the database.

SQL Query to Check for Non-Normalized Numbers:

SELECT id, phoneNumber FROM "User" WHERE phoneNumber NOT LIKE '+639%' AND phoneNumber IS NOT NULL;

Mitigation: Use the provided migration script (see Developer Manual) to batch update these records during a maintenance window.
