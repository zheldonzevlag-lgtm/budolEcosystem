Deployment & Scalability Roadmap

This document outlines the transition from local development to a production-grade AWS infrastructure.

Phase 1 Local Development (Current)

Developing all "Lego Blocks" on the local machine using Docker for databases and Localhost for app communication.

- Database: PostgreSQL (Local Docker)
- Communication: http://localhost:3000-3005
- Env Vars: .env.local
Phase 2 Vercel Hobby Plan (MVP)

Deploying frontend and API routes to Vercel. Database moves to a managed provider (e.g., Supabase, Neon, or Railway) to stay within free/hobby tiers.

- Hosting: Vercel Serverless Functions
- Database: Managed PostgreSQL (External)
- SSO: budolID hosted as a separate Vercel project
Hobby Note: Watch out for Serverless Function execution limits (10s). Atomic transactions must be optimized.
Phase 3 Vercel Pro / AWS (Production)

Migration to AWS for full control over networking (VPC), specialized services (S3 for receipts, SQS for transaction queues), and global scalability.

- Hosting: AWS ECS (Docker) or Amplify Pro
- Database: AWS RDS (PostgreSQL) with Read Replicas
- Cache: AWS ElastiCache (Redis) for SSO sessions

Environment Configuration Strategy

To make the move from Vercel to AWS seamless, we will use Dynamic Environment Variables:

  • NEXT_PUBLIC_SSO_URL: Changes from localhost:3000 to auth.budolecosystem.com.
  • DATABASE_URL: Injected by Vercel/AWS at runtime.
  • INTERNAL_SECRET_KEY: Used for secure app-to-app communication.

Scalability Considerations

The "Lego" architecture is perfect for Vercel/AWS because each app is a separate deployment. If budolShap gets a traffic spike, only its Vercel project or AWS cluster needs to scale up.
