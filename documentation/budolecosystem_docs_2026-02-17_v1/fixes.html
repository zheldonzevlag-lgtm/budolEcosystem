Critical Fixes

1. Cart Persistence Bug (Critical)

Issue: Ordered items remained in the cart after successful payment/order placement.

Root Cause: The cart cleanup logic in createOrder failed to match items for products without variations. The frontend or request payload might send variationId as undefined or garbage, while the database expects null. The mismatch caused deleteMany to skip these items.

Fix: Implemented robust variation ID normalization in lib/services/ordersService.js. The logic now checks if the product has a variation matrix. If empty, it forces variationId to null regardless of the input payload.

// If product has no variations, we must target the base item (variationId: null)
const targetVariationId = hasVariations ? (item.variationId || null) : null;

const deleteResult = await tx.cartItem.deleteMany({
    where: {
        cartId: userCart.id,
        OR: cartCleanupConditions
    }
});

2. Return Service Audit Log User ID Mismatch

Issue: The receiveReturn function in lib/services/returnsService.js was passing storeId instead of userId to the audit log service, causing potential foreign key violations or incorrect attribution.

Fix: Updated the function to traverse the relation returnRequest.order.store.userId to get the correct User ID of the store owner.

// Before
createAuditLog(returnRequest.order.storeId, 'RETURN_RECEIVED', ...

// After
createAuditLog(returnRequest.order.store.userId, 'RETURN_RECEIVED', ...

3. Return Service DB Update Optimization

Issue: buyerRespondToProposal was performing two separate DB updates to the Return model (one for escrow unlock, one for status), causing inefficiency and race conditions in tests.

Fix: Merged the updates into a single prisma.return.update call.
