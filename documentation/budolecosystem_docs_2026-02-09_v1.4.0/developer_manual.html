Developer Manual: Role-Based OCR Visibility

← Previous Version (v1.3.9) | Next Version (v1.5.0) →

This manual details the technical implementation of restricting sensitive AI-extracted data to admin users only.

Technical Overview

The implementation spans across the frontend UI (React components) and the backend API (Next.js Routes) to ensure multi-layered security.

1. Frontend Restriction

File: KYCWizard.jsx

We use the useAuth hook to determine the current user's role and conditionally render the AI Extraction Results section.

const { user } = useAuth()
const isAdmin = user?.isAdmin || user?.accountType === 'ADMIN' || user?.role === 'ADMIN'

// Conditional Rendering logic
{isAdmin ? (
  // Show full OCR results
) : (
  // Show "Confidential Processing" message
)}

2. Backend Data Masking

File: route.js (User KYC)

The GET handler now fetches the user's role from the database and masks sensitive fields if the requester is not an admin.

// Data Masking Logic
if (!isAdmin && kycDetails) {
  const maskedDetails = { ...kycDetails }
  if (maskedDetails.fullName) maskedDetails.fullName = '*** PROTECTED ***'
  if (maskedDetails.idNumber) maskedDetails.idNumber = '*** PROTECTED ***'
  if (maskedDetails.ocrLines) maskedDetails.ocrLines = []
  kycDetails = maskedDetails
}

3. Admin API Extension

File: utils.js (Admin API)

Updated the adminUserInclude to include kycDetails and kycStatus so that the admin dashboard can retrieve full data for verification.

export const adminUserInclude = {
    // ... other includes
    kycDetails: true,
    kycStatus: true,
}

Rationale

Limiting the visibility of raw OCR results to regular users reduces the risk of accidental exposure of PII (Personally Identifiable Information) on the client side, while still providing feedback that the automated process is working. This aligns with the principle of Least Privilege.
