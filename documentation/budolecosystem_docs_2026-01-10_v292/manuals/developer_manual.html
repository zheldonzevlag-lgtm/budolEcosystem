Developer Manual Updates - v292

Technical Implementation Details

The core fix involved standardizing the SSO callback logic across microservices to handle IPv6/IPv4 resolution issues common in local development environments.

1. SSO Callback Logic Standardization

Modified the GET handler in the callback routes of budolPay and budolShap. The logic now ensures that the ssoUrl used for server-to-server token verification always points to 127.0.0.1 if localhost is provided.

// Standardized URL resolution in budolPay & budolShap let ssoUrl = process.env.NEXT_PUBLIC_SSO_URL || 'http://127.0.0.1:8000'; if (ssoUrl.includes('localhost')) { ssoUrl = ssoUrl.replace('localhost', '127.0.0.1'); }

2. Token Verification Endpoint (budolID)

The /auth/verify endpoint in budolID-0.1.0/api/index.js was audited for security and performance. It performs a two-step validation:

  • JWT Verification: Validates the signature using JWT_SECRET.
  • Database Session Check: Verifies the token exists in the Session table and has not expired.

3. Database Synchronization

To ensure consistency during testing, a seed script seed-galvez.js was implemented using the shared @budolpay/database package to sync the test user reynaldomgalvez@gmail.com with the correct passwordHash and role.

Architectural Impact

This change moves the ecosystem towards a more robust URL resolution strategy, reducing dependence on system-level host file configurations and improving developer experience across different operating systems.

Issues & Fixes

  • Issue: "Invalid Token" error during SSO redirect.
    Fix: Standardized JWT_SECRET across all 21 .env files.
  • Issue: Fetch failures to localhost:8000 from Next.js server components.
    Fix: Forced use of 127.0.0.1 to bypass IPv6 resolution bugs.
