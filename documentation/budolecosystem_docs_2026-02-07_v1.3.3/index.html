Release v1.3.3: OTP Prefill Fix & State Synchronization

Date: 2026-02-07

Released

This version fixes a critical UX bug where the Verification Code (OTP) field was being prefilled with old values (passwords or previous OTPs) after clicking the "Send Code" button.

The Issue: Persistent State in Shared Input

The password field in the authentication state was shared between the standard password input and the OTP input. When switching methods or requesting a new code, the old value remained in memory, causing the field to appear pre-filled.

Rationale:

  • State Reuse: formData.password was used for both password and OTP for simplicity, but lacked proper cleanup logic.
  • Browser Autofill: Browsers sometimes attempt to fill fields that look like passwords or verification codes, exacerbating the issue if the state isn't explicitly cleared.

Technical Solution

1. Post-Send Cleanup

Modified handleSendOtp in AuthForm.jsx to explicitly clear the password state upon successful OTP request.

setIsOtpSent(true); // Clear the password field so it's ready for the new OTP setFormData(prev => ({ ...prev, password: '' }));

2. Method/Mode Switch Reset

Added a useEffect hook to monitor loginMethod and mode. This ensures that any time the user switches between Email/Mobile or Login/Register, all sensitive or temporary fields are wiped clean.

useEffect(() => { setFormData(prev => ({ ...prev, password: '' })); if (isLogin) { setIsOtpSent(false); // Reset OTP flow state } }, [loginMethod, mode, isLogin]);

Compliance Verification

  • Cybersecurity Law: Prevents sensitive data (passwords) from being visible in plain text OTP fields during method switching.
  • BSP Circular 1170: Ensures OTP flow integrity by forcing fresh input for every authentication attempt.
  • UX Standard: Eliminates user confusion caused by pre-filled stale data.
