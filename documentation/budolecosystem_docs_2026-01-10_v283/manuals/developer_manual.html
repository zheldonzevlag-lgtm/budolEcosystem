Developer Manual - v283

Purpose

This version focuses on enabling full LAN-wide SSO accessibility and resolving Prisma schema inconsistencies that caused 401 errors during token verification.

Prisma Schema Enhancements

A critical missing relation was identified in the Session model. The budolID service was attempting to include the app relation when verifying tokens, but this relation was not defined in the shared Prisma schema.

Issue

Unknown field app for include statement on model Session error in budolID logs.

Fix

Modified budolpay-0.1.0/packages/database/prisma/schema.prisma to include the app relation in the Session model and the reciprocal sessions array in EcosystemApp.

model Session {
  id        String        @id @default(uuid())
  userId    String
  appId     String?
  token     String        @unique
  expiresAt DateTime
  createdAt DateTime      @default(now())
  user      User          @relation(fields: [userId], references: [id])
  app       EcosystemApp? @relation(fields: [appId], references: [id])
}

model EcosystemApp {
  // ... other fields
  sessions  Session[]
}

LAN SSO Configuration

To support testing on a secondary computer in the LAN, all apps were updated to use the host computer's LAN IP (192.168.1.24) for authentication redirects and verification calls.

Code Changes

  • budolShap: Updated .env.local to set NEXT_PUBLIC_SSO_URL=http://192.168.1.24:8000.
  • budolPay Admin: Updated app/api/auth/callback/route.ts to prioritize NEXT_PUBLIC_SSO_URL and fall back to the dynamic request host if necessary.
  • budolPay Admin Login: Integrated the "Login with budolID" button into app/login/page.tsx to provide a consistent SSO entry point.

Verification & Testing

Verification was performed by accessing the services from a LAN IP and confirming the redirect flow:

  1. User visits http://192.168.1.24:3001/login.
  2. User clicks "Login with budolID" → Redirected to http://192.168.1.24:8000/login?apiKey=bs_key_2025.
  3. User logs in → Redirected back to http://192.168.1.24:3001/auth/callback?token=....
  4. Callback service verifies token via LAN IP and establishes session.

Risks & Mitigations

  • Risk: Camera access for KYC/Face verification is blocked by browsers on non-localhost/non-HTTPS origins (like a LAN IP).
  • Mitigation: Documented the chrome://flags workaround for developers testing on LAN.
