v503 - Dependencies & Software Rationale

This document outlines the software choices and dependencies implemented in version 503 to support Dynamic Page-Specific Search and Real-time Sync Integrity.

React Context API
Why: Chosen for global search state management.
Rationale: Since the search query needs to be accessible by the Navbar (input) and various page components (consumers), a global state was necessary. Context API provides a lightweight, built-in solution without the complexity or bundle size increase of Redux or Zustand.
Prisma ORM (insensitive filtering)
Why: Backend database querying.
Rationale: Prisma's mode: 'insensitive' for contains filters allowed us to implement case-agnostic search directly at the database level. This is significantly more performant than fetching all records and filtering in JavaScript.
Next.js usePathname Hook
Why: Dynamic placeholder and search reset.
Rationale: By utilizing usePathname within the SearchProvider, we can automatically detect navigation events, reset the search query, and update the navbar placeholder to match the current page context (e.g., "Search orders..." vs "Search users...").
SWR (Stale-While-Revalidate)
Why: Data fetching and cache mutation.
Rationale: SWR's mutate function is used to trigger instant UI updates when real-time events are received. By aligning the search query with the SWR cache key, we ensure that real-time updates respect the current search filters.
Pusher Channels
Why: Real-time event broadcasting.
Rationale: Pusher provides a managed WebSocket infrastructure that handles connection scaling and fallback mechanisms. In v503, we ensured that Pusher events are prioritized over SWR polling to minimize latency.
