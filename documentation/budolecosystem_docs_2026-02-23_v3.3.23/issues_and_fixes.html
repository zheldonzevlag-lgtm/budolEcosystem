Issues & Fixes: Version 3.3.22

Issue: MapPicker "Failed to fetch"

Description: When dragging the pin on the map, the reverse geocoding request would sometimes fail with a generic "Failed to fetch" error.

Root Cause: Client-side requests to Nominatim (OSM) were failing due to:

  • Missing User-Agent header (required by Nominatim usage policy).
  • Strict CORS policies blocking direct browser-to-API requests.
  • Browser security restrictions on mixed content (if using HTTP).

Fix: Implemented a server-side proxy (/api/system/geocode) that handles the request on behalf of the client. This allows adding required headers and avoids CORS issues.

Verification: Verified using scripts/test_scripts/test_geocode_api.js and manual testing on the frontend.

Issue: AddressAutocomplete "Failed to fetch"

Description: Similar to MapPicker, the address autocomplete component was failing when searching for addresses due to direct client-side requests to Nominatim/Geoapify.

Fix: Refactored components/address/AddressAutocomplete.jsx to use the centralized /api/system/geocode endpoint.

Issue: Geoapify Key Exposure

Description: Previously, the Geoapify API key was exposed to the client to make direct requests.

Fix: Moved API key handling to the backend. The key is retrieved from system settings and used server-side only.

Issue: AddressModal Save Button Flicker/Reload

Description: Clicking "Save" on the profile or address modal caused the entire page to reload, leading to a visible flicker and closing the modal abruptly.

Root Cause: The handleSaveProfile function in components/AddressModal.jsx explicitly called window.location.reload() to refresh the user data.

Fix: Removed the page reload and modal close logic. Instead, the local component state (userMeta) is updated directly with the new form data, providing an instant, flicker-free update without closing the modal.

Issue: Map Provider Hardcoding (OSM/Geoapify)

Description: The MapPicker component was tightly coupled to OpenStreetMap and Geoapify, making it difficult to switch to other providers like Google Maps or Radar.

Fix: Refactored MapPicker.jsx to dynamically determine the tile layer based on system settings. Added fallback logic for Google Maps (using OSM tiles until JS API integration is complete).

Issue: Commercial Provider Rate Limiting

Description: The system applied a strict 1.5-second rate limit to all geocoding requests, including paid commercial providers which support much higher throughput.

Fix: Updated MapPicker.jsx to bypass the rate limiter when a valid commercial API key (Google Maps, Radar, Geoapify) is present.
