Detailed mitigation protocols for identified security risks within the Budol Ecosystem.

Resolved (Phase 7)

M-SEC-007: Refresh Token Rotation

Risk: Session hijacking via long-lived static JWTs.

Mitigation: Replaced 30-day static tokens with short-lived (15 min) access tokens and 30-day refresh tokens. Implemented One-Time Use refresh token rotation. Every time a token is refreshed, the old refresh token is invalidated and a new one is issued. This detects and neutralizes stolen refresh tokens immediately upon the next refresh attempt.

Resolved (v337)

M-SEC-337-01: SSL/TLS Certificate Pinning

Risk: Man-in-the-Middle (MITM) attacks via compromised Certificate Authorities.

Mitigation: Implemented SHA-256 hash pinning in the Flutter mobile application. The app now explicitly verifies the server's certificate against a hardcoded list of trusted hashes, bypassing the device's default trust store for BudolPay API domains.

Resolved (v337)

M-SEC-337-02: Multi-Hash Strategy for Zero-Downtime

Risk: App lockout during planned certificate rotation or emergency replacement.

Mitigation: ApiService supports a list of multiple valid hashes. This allows the security team to perform "Pre-rotation" by shipping an app update containing the next certificate's hash before the actual server-side switch occurs.

Resolved (Historical)

M-SEC-005: Secure Local Storage

Risk: Extraction of sensitive data (PINs, tokens) from physical device storage.

Mitigation: Use of flutter_secure_storage which leverages Keychain (iOS) and AES encryption in Keystore (Android). Sensitive data is encrypted at rest using hardware-backed security modules where available.

Active Monitoring

M-SEC-004: SIM Swap Protection

Risk: Account takeover via unauthorized SIM cloning.

Mitigation: Implemented Device Fingerprinting (Level 1). Every login is tied to a unique device ID. SIM swaps trigger a "High Risk" flag requiring mandatory biometric re-verification and a 24-hour cooldown for high-value transactions.

Resolved (Historical)

M-SEC-001: API Rate Limiting

Risk: Distributed Brute-force attacks on authentication endpoints.

Mitigation: Backend-enforced rate limiting. IP-based and User-ID based limits (10 attempts per 15 minutes). Excessive failures trigger automatic account locking and email notification to the user.

Resolved (v338)

M-SEC-338-01: Liveness Detection (Anti-Spoofing)

Risk: Unauthorized access using high-resolution photos or videos of the legitimate user.

Mitigation: Implemented multi-metric liveness detection using Google ML Kit. The system requires active participation (blink/smile/head-turn) to verify the user is a living person. Static images are rejected based on probability thresholds for eye opening and smiling metrics.

Resolved (v338)

M-SEC-338-02: Service Port Isolation

Risk: System instability and startup failures due to port 8080 contention.

Mitigation: Assigned dedicated ports to all microservices (8001-8003) and the API Gateway (8080). This eliminates port conflicts and allows for independent service scaling and monitoring.
