Developer Manual: v339 Technical Implementation

1. Admin Build Optimization

To resolve build-time errors related to Prisma and static generation, the following changes were implemented:

1.1 Next.js Configuration

Updated next.config.mjs to explicitly handle the Prisma client and define the DATABASE_URL for build-time schema loading.

const nextConfig = {
  experimental: {
    serverComponentsExternalPackages: ['@prisma/client'],
  },
  output: 'standalone',
};
View next.config.mjs

1.2 Dynamic Route Enforcement

Forced dynamic rendering on the Realtime Settings page to prevent Prisma connection errors during static site generation.

export const dynamic = 'force-dynamic';
View page.tsx

2. Role-Based Access Control (RBAC)

Phase 1 of RBAC integration ensures that only authorized roles can access sensitive administrative functions.

2.1 Permission Mapping

Defined core permissions for ADMIN, SUPERVISOR, and USER roles.

const permissions = {
  ADMIN: ['manage_users', 'view_reports', 'manage_settings'],
  USER: ['view_profile', 'make_payment']
};
View rbac-config.js

3. LAN Connectivity Standardization

The system now defaults to 192.168.1.17 to facilitate cross-device testing on the local network. This required updates to both configuration and runtime logic.

3.1 Environment Configuration

LOCAL_IP=192.168.1.17
NEXT_PUBLIC_SSO_URL=http://192.168.1.17:8000
View .env

3.2 Dynamic Auth Redirects

Updated the auth callback to handle 0.0.0.0 fallbacks by redirecting to the standard LAN IP.

const baseUrl = new URL(request.url);
if (baseUrl.hostname === '0.0.0.0') {
    baseUrl.hostname = '192.168.1.17';
}
View route.ts

3.3 Login Page SSO Integration

Ensured the login page uses the NEXT_PUBLIC_SSO_URL for redirects, with a hardcoded fallback to the LAN IP for stability.

const ssoUrl = process.env.NEXT_PUBLIC_SSO_URL || 'http://192.168.1.17:8000';
View page.tsx
