Comparative Analysis: "1 Device → 1 Account" Policy

Overview

This document compares the strict 1 Device → 1 Account security policy (as enforced by GCash) against the current budolPay implementation. The goal is to align budolPay with global security standards while maintaining user experience.

Comparison Matrix

Technical Deep Dive

GCash Implementation Analysis

GCash uses a centralized device registry. When a user logs in, the backend checks if the deviceId matches the one stored in the account metadata. If it doesn't, the login is blocked until the previous device is unlinked. This prevents session hijacking even if the MPIN is compromised.

budolPay Current Implementation

Currently, budolPay stores a list of trustedDevices in the database. While secure, it allows a user to be logged in on multiple devices (e.g., a phone and a tablet). This increases the attack surface.

// Current logic in auth-service/index.js
const devices = JSON.parse(user.trustedDevices);
const device = devices.find(d => d.deviceId === deviceId && d.isVerified);
if (device) isDeviceTrusted = true;

Proposed v399 Logic

We will transition to a 1:1 binding. The trustedDevices field will be deprecated in favor of a single linkedDeviceId. Registering a new device will automatically prompt for the unlinking of the existing one using the 512-dim Face Recognition verification.

Recommendations & Next Steps

  • Phase 1: Database migration to add linked_device_id and device_binding_timestamp.
  • Phase 2: Implement /api/v1/auth/unlink-device endpoint in Auth Service.
  • Phase 3: Update budolPay Mobile to enforce the "1 Device" check during the splash screen/login flow.
  • Phase 4: Integrate Face Recognition (FaceNet 512-dim) as the primary factor for unlinking.
