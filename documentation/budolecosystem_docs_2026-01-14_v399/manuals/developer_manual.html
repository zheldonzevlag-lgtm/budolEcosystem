Developer Manual: "1 Device → 1 Account" Policy

This manual details the technical implementation of the strict device binding policy introduced in v399.

1. Architecture Overview

The core objective is to ensure that a budolPay account is active on exactly one device at any given time. This is achieved by introducing a primaryDevice flag in the trustedDevices JSON array within the User model.

Key Components:

  • Identification Service: Checks if the incoming deviceId matches the verified primary device.
  • Unlink Coordinator: Manages the transition from an old device to a new one.
  • Biometric Verifier: Performs 512-dimensional face embedding matching for unlinking.

4. Mobile Web Compatibility Fix

During the development of v399, a compatibility issue was identified with the tflite_flutter package when targeting the Web platform. The library attempted to load native binaries which are not supported in a browser environment.

Fix Applied:

Modified face_embedding_service.dart to detect the Web platform using kIsWeb and bypass the TFLite interpreter initialization. Face recognition features are gracefully disabled on Web to maintain application stability.

// [face_embedding_service.dart](file:///d:/IT Projects/budolEcosystem/budolPayMobile/lib/services/face_embedding_service.dart)
Future<void> _initModel() async {
  if (kIsWeb) {
    _log('Face embedding is not supported on web.');
    _isModelLoaded = false;
    return;
  }
  // ... rest of the initialization
}

5. Mandatory Environment Selection

To prevent accidental deployment or startup in the wrong environment, the npm start command now enforces a manual selection process. The script start-ecosystem.js has been updated to ignore pre-set ENV_CHOICE variables and always prompt the developer.

Startup Flow:

  1. Developer runs npm start.
  2. System presents environment options (LOCAL, PRODUCTION/VERCEL, SKIP).
  3. Developer must enter a valid choice (1, 2, or 3). The script will loop until a valid input is received.
// [start-ecosystem.js](file:///d:/IT Projects/budolEcosystem/start-ecosystem.js)
let choice = '';
while (!['1', '2', '3'].includes(choice)) {
    choice = (await askQuestion('\nEnter choice (1, 2, or 3): ')).trim();
}

6. Rationale & Technical Details

Purpose: To enforce a 1:1 account-to-device ratio for maximum security against concurrent access and credential abuse.

Technical Rationale: By using a unique hardware-backed identifier (UUID) as a secondary authentication factor, we effectively turn the physical phone into a "Something You Have" factor. This prevents an attacker with only the "Something You Know" (PIN) and "Something You Get" (OTP) from accessing the account from a rogue device.

Code References & Implementation:

  • login_screen.dart: Handles the DEVICE_UNLINK_REQUIRED state by redirecting to the UnlinkDeviceScreen.
  • api_service.dart: Sends the persistent deviceId in the identifyUser payload.
  • face_embedding_service.dart: Generates the 512-dimensional vector used for high-assurance identity verification during the unlinking process.

6. Compliance & Security Standards

This implementation adheres to the following regulatory standards:

  • BSP Circular 958: Implements Strong Customer Authentication (SCA) through device binding and biometrics.
  • PCI DSS: Ensures that authentication tokens and biometric data are handled with industry-standard encryption.
  • Cybersecurity Law: Protects user accounts from unauthorized access through proactive device management.
