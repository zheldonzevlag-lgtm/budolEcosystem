System Admin Manual: Device Management Policy

1. Policy Enforcement

Admins must ensure that the "1 Device → 1 Account" policy is active across all environments. This is controlled via the AUTH_STRICT_DEVICE_BINDING environment variable in the auth-service.

Setting: AUTH_STRICT_DEVICE_BINDING=true

2. Handling Manual Unlink Requests

In cases where a user loses their device and cannot perform AI Face Verification (e.g., facial injury or hardware failure), Admins can perform a manual unlink via the Admin Dashboard.

Standard Operating Procedure (SOP):

  1. Verify User Identity via secondary means (e.g., government ID upload, security questions).
  2. Navigate to User Management → [User ID] → Security Settings.
  3. Click "Force Unlink Primary Device".
  4. The user will be required to verify OTP and set a new primary device on their next login.
Warning: Manual unlinking should be logged and audited for security compliance.

3. Monitoring & Logs

Monitor logs for the following events to detect potential fraud:

  • DEVICE_UNLINK_ATTEMPT: Logged when a user starts the unlink flow.
  • DEVICE_UNLINK_SUCCESS: Logged upon successful primary device migration.
  • DEVICE_UNLINK_FAILURE: Logged if Face Verification or OTP fails multiple times.

4. Compliance Audit

This policy supports BSP Circular 958 compliance. Audit reports can be generated monthly to show the distribution of active devices and unlinking history.

5. Environment Safeguards

The ecosystem startup process includes a strictly mandatory environment selection gate. This ensures that administrators explicitly confirm the target environment before services initialize. There is no default selection; the system will repeat the prompt until a valid choice is made.

Startup Requirement: Running npm start will always prompt for environment selection. Administrators must enter 1, 2, or 3 to proceed.
