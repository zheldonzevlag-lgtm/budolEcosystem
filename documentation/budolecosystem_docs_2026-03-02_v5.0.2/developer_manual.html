Developer Manual - BudolEcosystem v5.0.2

Release Date: 2026-03-02

Status: Regression Fix & Schema Sync

Bug Fixes Task List

✔ Fix ReferenceError: prisma is not defined in checkoutService.js
✔ Fix ReferenceError: activeCheckoutId is not defined in checkout/route.js
✔ Fix database schema mismatch (Transaction.status) in budolpay Gateway
✔ Sync budolpay services to use "budolpay" schema
✔ Verify database migrations for budolpay schema

Purpose

This hotfix addresses critical regressions discovered after the v5.0.1 release, including undefined variable references and a database schema mismatch between budolShap and budolPay.

Technical Explanation

1. Prisma Reference Fix

In checkoutService.js, the prisma object was used without being imported or initialized. This was corrected by importing the shared prisma instance from @/lib/prisma.

File: lib/services/checkoutService.js

2. Scope Correction for activeCheckoutId

In the checkout API route, activeCheckoutId was declared inside a try block but accessed in the catch block, causing a ReferenceError when the try block failed early. The variable declaration was moved to the function scope.

File: app/api/payment/checkout/route.js

3. Database Schema Sync (budolPay)

A conflict occurred because both budolShap and budolPay were using the same database (budolshap_1db) and the same schema (public). Since they have different Transaction model definitions, budolPay failed when it couldn't find the status column.

Fix: All budolPay services were migrated to use a dedicated budolpay schema within the same database. All .env files were updated to include ?schema=budolpay.

Action: Ran npx prisma migrate deploy in budolpay-0.1.0/packages/database.

Changes Summary

  • Updated lib/services/checkoutService.js to import shared prisma instance.
  • Corrected activeCheckoutId variable scope in app/api/payment/checkout/route.js.
  • Migrated budolPay services to schema=budolpay in budolshap_1db database.
  • Updated all .env files across budolPay monorepo services.

Compliance & Risk

Compliance: Data isolation is improved by moving budolPay to its own schema, aligning with PCI DSS requirements for logical data separation.

Risk: Low. Services must be restarted to pick up the new database configuration.
