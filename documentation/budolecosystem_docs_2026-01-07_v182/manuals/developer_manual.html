Developer Manual v182

Purpose

To ensure that production-focused changes (e.g., Vercel deployment configurations) do not interfere with the local development environment.

Technical Details: Environment Isolation

The ecosystem uses an env-manager.js script to switch between environments. To prevent accidental overwriting of local settings, we have implemented .env.connection.local as a permanent source of truth for local configurations.

1. Implementation of .env.connection.local

Every major application (budolpay, budolID, budolshap, budolAccounting) now has a .env.connection.local file containing local database URLs and development-specific flags.

# Example .env.connection.local
DATABASE_URL="postgresql://postgres:r00t@localhost:5432/budolpay?schema=public"
NODE_ENV=development

2. Safe Environment Switching

When running node scripts/env-manager.js local, the script now pulls from .env.connection.local instead of relying on existing .env files, ensuring a clean restore of the development state.

Risk & Mitigation

  • Risk: Developers accidentally pushing .env to production.
  • Mitigation: .env is added to .gitignore, and env-manager.js enforces the use of .env.vercel for production deployments.
