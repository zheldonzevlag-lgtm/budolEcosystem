Developer Manual - v70

Overview

This version addresses the issue where the budolPay mobile app failed to parse QR codes generated by the system. The root cause was identified as a combination of broken external checkout URLs (budolpay.com) and missing local infrastructure to serve valid QR data for internal wallet payments.

Technical Implementation

1. New Checkout Route

A new GET /checkout/:referenceId endpoint was added to the payment-gateway-service. This route:

  • Retrieves transaction details from the database using the referenceId.
  • Constructs a JSON payload compatible with the budolPay mobile app.
  • Serves an HTML page with an embedded QR code generated via qrcode.js.

2. QR Payload Structure

The mobile app expects a specific JSON format to process payments. The generated QR code contains:

{
  "type": "budolpay_payment",
  "orderId": "ORD-123",
  "amount": 150.50,
  "merchant": "budolShap",
  "paymentIntentId": "uuid-of-transaction"
}

3. Dynamic Redirects

The POST /payments/create-intent route was updated to use a BASE_URL environment variable (defaulting to localhost:8004) to generate the checkoutUrl. This ensures that users are redirected to the local checkout page instead of the production domain during development.

Code Changes

File: services/payment-gateway-service/index.js

  • Added app.get('/checkout/:referenceId', ...) route.
  • Modified app.post('/payments/create-intent', ...) to return local URL.

Verification

Verified using fetch and manual inspection of the generated QR data. The payload was confirmed to contain all required fields for the mobile app's process-qr flow.
