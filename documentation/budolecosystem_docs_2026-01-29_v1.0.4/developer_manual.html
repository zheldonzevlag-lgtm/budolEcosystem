Developer Manual v1.0.4

Technical Architecture

The SSO flow relies on JWT token exchange between budolID (Identity Provider) and budolShap (Service Provider).

Node.js Express Next.js Prisma PostgreSQL

Critical Code Changes

1. JWT Secret Synchronization

The JWT_SECRET was updated in budolID to match budolShap's configuration to ensure tokens generated by one can be verified by the other.

Open File 
// budolID-0.1.0/index.js
const JWT_SECRET = process.env.JWT_SECRET || 'GJ7Lxn0/kdV/KuZJ5xJ7Ip0RvMerrGW5n0gf44mfHgc=';

// Generating SSO Token
const token = jwt.sign(
    {
        sub: user.id,
        email: user.email,
        iss: 'budolID',
        jti: crypto.randomUUID()
    },
    JWT_SECRET,
    { expiresIn: '7d' }
);

2. Database Schema Isolation

To prevent table collisions in the shared PostgreSQL database, budolID now uses a dedicated budolid schema via search_path.

Open File 
# budolID-0.1.0/.env
DATABASE_URL="postgresql://postgres:r00t@localhost:5432/budolshap_1db?schema=budolid&search_path=budolid"

3. SSO Callback Logic

The callback in budolShap verifies the budolID token and creates/updates a local user record.

Open File 
// budolshap-0.1.0/app/api/auth/sso/callback/route.js
const verifyResponse = await axios.get(`${ssoUrl}/auth/verify`, {
    headers: { 'Authorization': `Bearer ${token}` }
});

if (verifyResponse.data.valid) {
    const { user: ssoUser } = verifyResponse.data;
    // Sync to local DB and generate local token
    const localToken = generateToken({ userId: localUser.id, ... });
}

Data Contracts

SSO Token Payload:

  • sub: User UUID from budolID
  • email: User email
  • iss: 'budolID'
  • jti: Unique token identifier
