Developer Manual - v58.0.0

Purpose: To resolve session/cookie loss and email sending errors during the payment flow, specifically when the application is accessed via an IP address instead of localhost.

1. IP-Based Host Rewriting in BudolPayAdapter

In local development environments where developers access the site via a LAN IP (e.g., 192.168.1.24), internal services like the BudolPay Gateway might return localhost-based URLs because they are called server-to-server.

We implemented a rewriting logic in the BudolPayAdapter to intercept these localhost URLs and replace them with the actual host the user is browsing on.

// In lib/payment/adapters/budolpay-adapter.js if (checkoutUrl && options.successUrl) { const successUrlObj = new URL(options.successUrl); const checkoutUrlObj = new URL(checkoutUrl); if (successUrlObj.hostname !== checkoutUrlObj.hostname && checkoutUrlObj.hostname === 'localhost') { checkoutUrlObj.host = successUrlObj.host; // Replaces host:port checkoutUrl = checkoutUrlObj.toString(); } }

2. Async Transporter Fix in Email Service

The sendOrderPlacedEmail function was failing with TypeError: transporter.sendMail is not a function because the async createTransporter() helper was not being awaited.

3. Impact Analysis

  • Auth State: Cookies are now preserved because the user stays on the same origin (IP) throughout the redirect to /payment/qr.
  • LocalStorage: QR payment data is now accessible on the QR page because the origin remains consistent.
  • Email Reliability: Order confirmation emails will now be correctly sent (or logged) without crashing the payment flow.
