    Technical Architecture

    Phase 2 shifts the icon resolution from a purely name-based heuristic to a Hybrid Model where explicit Database ID overrides take precedence over the fallback mapping.

    1. Database Schema Update

    The `Category` model in prisma/schema.prisma now stores the Lucide component name as a dynamic override.

    model Category {
      id        String    @id @default(cuid())
      name      String
      slug      String    @unique
      icon      String?   // Stores Lucide icon name (e.g., 'Smartphone')
      // ...
    }

    2. Hybrid Resolution Logic

    The resolver in components/CategoryIcons.jsx was updated to prioritize `dbIcon`.

    export const getCategoryLucideIcon = (slug, name, dbIcon = null) => {
      // 1. Prioritize explicit DB override
      if (dbIcon) {
        const component = resolveIconByName(dbIcon);
        if (component) return component;
      }
      // 2. Fallback to name/slug mapping...
    }

    3. API Data Fetching

    The Product API in app/api/products/route.js now includes the category relationship to ensure icons are available on product listings.

    include: {
      categoryData: {
        select: { icon: true }
      }
    }

    4. Searchable Icon Picker

    The Admin interface in app/admin/categories/page.jsx implements a searchable grid of icons using `PROFESSIONAL_ICON_MAP`. It uses local state filter to match user input against icon names.

    5. Framer Motion Integration

    In components/CategoriesSection.jsx, we use `motion.div` for staggered entry of category tiles.

    Dependencies

    • lucide-react: Core icon set.
    • framer-motion: Orchestrating animations.
    • zod: Validating the new `icon` field in API payloads.
