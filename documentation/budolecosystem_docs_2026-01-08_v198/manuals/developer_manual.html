Developer Manual Updates - v198

Purpose

To refine the mobile authentication flow, ensuring users are never prompted for a PIN if they haven't set one, even on trusted devices.

Logic Change: /login/mobile/identify

Previously, the code only checked isDeviceTrusted. It now checks if a pinHash exists before suggesting PIN authentication.

// BEFORE
if (!isDeviceTrusted) { ... trigger OTP ... }

// AFTER
if (!isDeviceTrusted || !user.pinHash) { ... trigger OTP ... }

Logic Change: /login/mobile/verify-pin

The error handling was split to differentiate between a missing user and a missing PIN, providing better diagnostic info for the mobile client.

if (!user) return res.status(401).json({ error: 'User not found' });
if (!user.pinHash) return res.status(401).json({ error: 'PIN not set for this account. Please verify via OTP.' });

Impact Analysis

This change prevents the "Invalid authentication request" (401) error observed in the mobile app. It forces a fallback to OTP verification, which is a secure way to re-authenticate and eventually prompt the user to set a PIN if required by the business logic.

Verification

Verified using a test user (09484099388) who has a trusted device but pinHash: null. The identification endpoint now correctly returns status: OTP_REQUIRED.
