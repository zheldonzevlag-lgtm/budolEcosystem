Developer Manual - v282

Purpose

This manual documents the fix for the budolID authentication failure and the procedure for maintaining schema consistency across multiple databases in the ecosystem.

Database Synchronization

The ecosystem uses multiple PostgreSQL databases (budolid, budolpay, budolaccounting, budolshap_1db). While most share the same Prisma schema, they must be individually synced to ensure all tables and columns exist.

Issue Encountered

The User table in the budolid database was missing the isFaceVerified column, leading to a P2022 Prisma error during login attempts.

Fix Implemented

Performed a forced reset and push of the shared schema to both budolid and budolpay databases:

$env:DATABASE_URL="postgresql://.../budolid"; npx prisma db push --force-reset
$env:DATABASE_URL="postgresql://.../budolpay"; npx prisma db push --force-reset

Authentication Logic

The budolID service uses the prisma client from @budolpay/database. Ensuring that prisma generate is run after any schema change is critical. In this version, we ensured the client was generated while no services were locking the binary files.

Issue 2: Invalid API Key

After a force-reset, the EcosystemApp table was empty, causing budolID to reject authentication requests from other apps with an "Invalid API Key" error.

Fix Implemented (Seeding)

Created a new LAN-specific seeding script seed_lan.js that automatically detects the current LOCAL_IP and registers the following apps with matching redirect URIs:

  • budolPay: bp_key_2025
  • budolShap: bs_key_2025
  • budolExpress: be_key_2025
// seed_lan.js logic
const LOCAL_IP = process.env.LOCAL_IP || 'localhost';
const redirectUri = `http://${LOCAL_IP}:3001/auth/callback`; // Example for budolShap

Verification: Login Test

A programmatic test was conducted to verify the full SSO login flow. The test confirmed that budolID correctly validates credentials, identifies the calling app, and redirects with a valid JWT token.

Test Case: Programmatic SSO Login
Endpoint: POST /auth/sso/login
Status: 302 Found (Success)
Redirect URL: http://192.168.1.24:3001/auth/callback?token=...

budolShap SSO Callback Fix

Two critical issues were identified and resolved in budolShap to support SSO:

  • 404 Not Found: Created the missing app/auth/callback/route.js to handle the redirect from budolID.
  • 401 Unauthorized (api/auth/me): Fixed a logic bug where the system was strictly checking for userId in the token, but budolID provides it as sub. Updated the check to support userId, id, or sub.

The test script test-login.js was used to perform this validation, ensuring that the database seeding was effective.

LAN/mDNS Technical Details

Services are now verified to be reachable at 192.168.1.24. The verify-lan.js script was updated to handle potential timeouts from Next.js services during their initial compilation phase.

Future Recommendations

  • Implement a centralized database migration script that handles all ecosystem databases in a single command.
  • Automate the seeding of EcosystemApps after every database reset to prevent "Invalid API Key" errors.
