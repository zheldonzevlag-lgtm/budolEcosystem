Version 226

Settings Crash Fix & Navigator Reset Implementation

Issue Description

Users reported a type 'Null' is not a subtype of type 'String' crash when navigating to the Settings screen after logging in as "Tony Stark". Additionally, clicking the "Reset App & Logout" button on the error screen brought users to an empty Settings screen instead of the Login screen.

Root Cause Analysis

  • Null Subtype Error: In settings_screen.dart, the user['email'] field was being assigned directly to a non-nullable String variable. If the backend returned a null email (as with the "Tony Stark" test account), the widget crashed during build.
  • Navigation Inconsistency: The global error handler's reset button called logout() but didn't force a navigation reset. Since the SettingsScreen was already on top of the navigation stack, it rebuilt after logout. With no user data, it showed the default "User" state (Image 2) instead of redirecting to Login.

Technical Fixes

1. Robust Data Parsing

Updated settings_screen.dart to use safe parsing for user profile data:

String userName = 'User';
String email = '';
if (user != null) {
  final firstName = user['firstName']?.toString() ?? '';
  final lastName = user['lastName']?.toString() ?? '';
  userName = '$firstName $lastName'.trim();
  email = user['email']?.toString() ?? '';
}

2. Navigator State Management

Implemented a GlobalKey<NavigatorState> in main.dart. This allows the global error handler to force a navigation reset even when a build context is unavailable or invalid.

navigatorKey.currentState?.pushNamedAndRemoveUntil('/login', (route) => false);

Test Results

  • Scenario: Login as Tony Stark (null email) and navigate to Settings.
  • Result: Screen loads successfully with "Tony Stark" as name and empty email field. No crash.
  • Scenario: Click "Reset App & Logout" on any error screen.
  • Result: App immediately clears session and redirects to the Login screen.
