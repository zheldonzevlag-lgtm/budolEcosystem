Version 321

Developer Manual

Clarifications for biometric availability vs OS enrollment, plus E2E checklist alignment.

Purpose

To clarify biometric login behavior that depends on OS-level enrollment (fingerprint/face setup), align documentation with the actual in-app toggle behavior, and standardize a reproducible E2E checklist script for release verification.

Explanation & Rationale

On Android/iOS, “biometric available” is a combination of hardware support and OS enrollment state. This update documents the expected behaviors so support teams and developers can correctly diagnose cases where biometric login is not shown or fails.

For security and PCI DSS-aligned handling of authentication secrets, the PIN is only stored in OS secure storage after a successful manual PIN verification and only used after a successful biometric prompt.

Issues & Fixes

  • Issue: Confusion between device support vs biometric enrollment causing inconsistent expectations.
    Fix: Documented the OS enrollment dependency and negative test path (no enrolled biometrics).
  • Issue: Prior docs showed a toggle enable dialog flow not matching the current implementation.
    Fix: Documented the actual enable semantics: settings toggle enables the feature flag; PIN is stored on next successful manual PIN login.
  • Issue: E2E validation steps were not centralized in a deterministic script.
    Fix: Updated scripts/test_scripts/test_biometric_login.dart to locate repo root reliably and print a complete checklist with prerequisites.

Dependencies

  • local_auth (Flutter) for OS biometric prompts.
  • flutter_secure_storage for Keystore/Keychain-backed PIN storage.

Technical Details

Availability vs Enrollment

  • Hardware supported: device can technically use biometrics.
  • Biometrics enrolled: user has added at least one fingerprint/face in OS settings.
  • App toggle enabled: stored flag (biometric_enabled) is true.
  • Stored PIN present: stored value (biometric_stored_pin) exists in secure storage.

Biometric login requires: supported + enrolled + enabled + stored PIN present.

Settings toggle behavior

// lib/screens/settings_screen.dart
// Enable: sets enabled flag, but PIN is stored on next successful manual PIN login
// Disable: clears enabled flag and deletes stored PIN

Login flow behavior

// lib/screens/login_screen.dart
// If biometrics enabled and supported:
// - biometric icon triggers OS prompt
// - on success, reads stored PIN and verifies PIN via API

Risk & Mitigation

  • Risk: Users assume biometrics will work without OS enrollment.
    Mitigation: Document prerequisites; add negative test and support steps.
  • Risk: PIN reuse if biometrics are enabled but user changes MPIN.
    Mitigation: Current implementation overwrites stored PIN after successful PIN verification; recommend storing on MPIN change flow as a future enhancement.
  • Risk: Device compromise could expose authentication flows.
    Mitigation: Rely on OS secure storage and OS-level biometric prompt; no biometric data is stored/handled by the app.

Recommendation for Future Enhancement

Consolidate all biometric-related user messaging into a single UX flow: show an inline explanation when biometrics are unavailable (not enrolled), and re-store the PIN automatically after MPIN changes to prevent stale secure-storage values.
