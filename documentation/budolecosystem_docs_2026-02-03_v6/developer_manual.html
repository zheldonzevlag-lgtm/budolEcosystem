Architecture Overview

The Budol Ecosystem now uses a unified, provider-agnostic realtime event system. Developers can trigger events from the backend without worrying about the active provider (Pusher, Socket.io, or SWR Polling).

Primary Goal: Seamless switching between hosted (Pusher), self-hosted (Socket.io), and zero-config (SWR Polling) methods with automated fallback.

Backend Implementation

All realtime events should be triggered via the triggerRealtimeEvent helper in lib/realtime.js.

File Reference: lib/realtime.js

import { triggerRealtimeEvent } from "@/lib/realtime";

// Triggering an audit log event
await triggerRealtimeEvent('admin', 'AUDIT_LOG_CREATED', {
    action: 'LOGIN',
    userId: 'user_123',
    // ... metadata
});

The backend automatically checks SystemSettings to determine the active provider and validates credentials before attempting to trigger.

Frontend Implementation

Components should use the useRealtime hook to listen for events. This hook manages the subscription lifecycle and fallback polling automatically.

File Reference: hooks/useRealtime.js

import { useRealtime } from "@/hooks/useRealtime";

const { provider, isConnected, isPolling } = useRealtime({
    channel: 'admin',
    event: 'AUDIT_LOG_CREATED',
    onData: (data) => {
        console.log("New realtime data received:", data);
        // Update local state
    }
});

Configuration & Database

The SystemSettings model stores the configuration for all providers.

File Reference: prisma/schema.prisma

  • realtimeProvider: ENUM ('POLLING', 'PUSHER', 'SOCKET_IO')
  • swrPollingInterval: Default 10000ms
  • socketUrl: Dedicated socket server URL
  • pusher*: Pusher-specific credentials
