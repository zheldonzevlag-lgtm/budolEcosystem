Developer Manual (v3.4.1)

This manual details the technical changes implemented to resolve the cart persistence issue for asynchronous payments (GCash, Maya, GrabPay) via PayMongo.

1. Issue Overview

Previously, items remained in the user's cart even after a successful payment. This occurred because the createOrder service relied on exact matching of variationId during the initial order creation phase, which could fail if data was inconsistent. Furthermore, the webhook handler explicitly disabled cart clearing due to safety concerns about deleting all product variations.

2. Implementation Details

2.1 Webhook Logic Update

The PayMongo webhook handler has been updated to include a safe and robust cart clearing mechanism. Instead of deleting items based solely on productId (which risks deleting unpurchased variations), the new logic deletes items matching both productId AND variationId of the purchased items.

Key Change: The webhook now fetches the OrderItems associated with the paid order and removes the corresponding CartItems from the user's cart. This acts as a reliable fallback for any failures in the initial createOrder cleanup process.

// 2. Clear Cart (Safe Implementation)
try {
    // Need to fetch order items first if not present
    const orderItems = await prisma.orderItem.findMany({
        where: { orderId: order.id }
    })

    const userCart = await prisma.cart.findUnique({ where: { userId: order.userId } })
    
    if (userCart && orderItems.length > 0) {
        console.log(`[Webhook] Clearing cart for user ${order.userId} based on order ${order.id}`);
        
        // Execute deletes in parallel
        // We use deleteMany with specific matching to ensure we only delete exactly what was bought
        // (matching both productId and variationId)
        await Promise.all(orderItems.map(item => {
            return prisma.cartItem.deleteMany({
                where: {
                    cartId: userCart.id,
                    productId: item.productId,
                    variationId: item.variationId // Exact match (null matches null)
                }
            });
        }));
        
        console.log(`[Webhook] Cleared ${orderItems.length} items from cart for user:`, order.userId)
    }
} catch (e) {
    console.error('Failed to clear cart in webhook:', e)
}

2.2 Why this is Safe

  • Granular Deletion: By including variationId in the where clause, we ensure that if a user buys a "Red" shirt, the "Blue" shirt in their cart remains untouched.
  • Idempotency: deleteMany operations are safe to run multiple times. If the item is already gone, it simply returns a count of 0 without error.
  • Parallel Execution: Using Promise.all ensures the cleanup is fast and doesn't significantly delay the webhook response.

3. Related Services

The ordersService.js still attempts to clear the cart during order creation. The webhook logic serves as a critical redundancy for async payments where the user session might be disconnected or the initial transaction logic might face race conditions.

4. Webhook Monitoring & Logging

To support debugging and monitoring of payment events, a comprehensive logging system has been implemented.

4.1 Database Schema: WebhookEvent

All incoming webhook requests are logged to the WebhookEvent table. This allows for:

  • Audit Trails: Every request is stored with its payload, provider, and timestamp.
  • Status Tracking: Events are marked as PENDING, PROCESSED, or FAILED.
  • Error Capturing: Any errors during processing (including signature verification failures) are stored in the error column.

4.2 Admin Interface

An admin interface is available at /admin/webhooks to view these logs. It features:

  • Real-time status updates (Success/Failed)
  • Payload inspection
  • Retry mechanism for failed events
  • Simulation tool for testing

5. Testing Requirements

Developers should verify this fix using the PayMongo Test Mode credentials provided in the project environment variables. Simulate a successful payment and check the database logs for the "Cleared cart for user" message.

6. Dependencies & Software Decisions

The following key technologies are used in this implementation:

  • Prisma ORM: Chosen for its type-safe database access and ease of schema migration. It handles the complex relationships between Cart, CartItem, Order, and OrderItem.
  • PayMongo API: The selected payment gateway for the Philippines market (GCash, Maya, GrabPay). We use the webhook mechanism to handle asynchronous payment notifications.
  • Next.js API Routes: Used to host the webhook endpoint (/api/webhooks/paymongo), providing a serverless-ready architecture.

7. Compliance & Security

  • PCI DSS: No sensitive card data is stored on our servers. All processing is handled by PayMongo (a PCI DSS compliant provider).
  • Data Privacy: Customer data in logs is minimal and necessary for transaction verification.
  • Webhook Security: All incoming webhooks are verified using the paymongo-signature header to prevent replay attacks and spoofing.
